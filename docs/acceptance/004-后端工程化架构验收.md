# 背景

后端工程化落地不只是“代码能运行”，还必须验证统一错误处理、统一响应、统一上下文、OpenAPI 主线、数据库边界和 Worker 运行面是否真正收敛。

# 目标

1. 为 `009/010/011/018` 建立统一验收出口。
2. 定义横切规范、契约、数据库、Worker 和模块迁移的验收清单。
3. 明确每个阶段需要保留的验证证据。

# 非目标

1. 本文档不替代阶段计划。
2. 本文档不记录一次性临时排障过程。

# 验收范围

1. 横切工程骨架
2. 契约与 HTTP 规范
3. 数据访问与迁移边界
4. Worker 运行面
5. 业务模块迁移结果

# 验收清单

## 1. 横切工程骨架

- [x] 存在统一错误模型与错误码注册表：`internal/platform/http/errors.go`
- [x] 存在 recover middleware：`internal/platform/http/middleware.go`
- [x] 成功/失败/分页响应格式统一：`internal/platform/http/response.go`
- [x] request id 与 trace id 注入生效：`RequestIDMiddleware` 与 `internal/platform/runtime/context.go`
- [x] 统一日志字段生效：`internal/platform/logging/logger.go`

## 2. 契约与 HTTP 规范

- [x] `docs/openapi.json` 已生成
- [x] 契约中显式表达 Bearer/JWT：`components.securitySchemes.bearer`
- [x] 错误响应结构在契约中统一：`ErrorBody` 与各 operation `default` response
- [x] DTO 未直接复用 ORM 模型：HTTP 层使用 request/response DTO 与 service/query interface

## 3. 数据访问与迁移边界

- [x] `db/migrations` 为正式 schema 主线：`db/migrations/000_schema.sql` 覆盖 `db/schema.sql` 表集合
- [x] 业务层未直接读取 `*gorm.DB`：`scripts/validate-architecture-boundaries.sh` 白名单限制在 `internal/database` 与 `internal/app` 装配层
- [x] 复杂查询收敛在 repository 内部：`Raw/Exec/Table/Model` 仅允许出现在 `internal/database`

## 4. Worker 运行面

- [x] 存在统一 scheduler / job runner 入口：`internal/jobs/runner.go` 与 `internal/app/worker_jobs.go`
- [x] 任务统一通过 job/use-case service、domain service 或 repository adapter 边界执行，runner 不承载 SQL、外部 API 细节或业务判断
- [x] 关键任务具备幂等保护：显式 `run_key`、进程内成功去重和 skipped audit
- [x] 失败记录与重试策略可验证：`RunRecorder`、`RunStatusFailed`、`RetryPolicy` 与 runner 单测
- [x] 单实例运行边界在文档和运行方式中一致：当前只实现单进程 runner，不引入 leader election 或多副本调度

## 5. 业务模块迁移结果

- [x] 第一批模块已接入统一错误、响应、上下文和契约规范：`auth/monitor/content/topic/trend/notify`
- [x] 模块级单测通过：`go test ./...`
- [x] 模块级 API 测试通过：`tests/integration/api` 与 `scripts/smoke-api.sh`

# 验收证据

至少保留以下证据：

1. `go test ./...`：由 `bash scripts/validate-repository.sh` 执行通过。
2. `go test -race ./...`：在 PR #237 最新提交 `1ed5a40` 后执行通过。
3. `go build ./...`：由 `bash scripts/validate-repository.sh` 执行通过。
4. `docker compose config`：由 `bash scripts/validate-repository.sh` 执行通过。
5. `docs/openapi.json`：存在且通过 `make openapi-validate` / repository validation。
6. `bash scripts/validate-architecture-boundaries.sh`：通过，且已验证无 `rg` 的 CI 环境 fallback。
7. `bash scripts/validate-repository.sh`：通过完整仓库验证、OpenAPI 验证、架构边界验证和 runtime API smoke。
8. GitHub PR #237：`Repository structure check` 通过，PR 状态为 open / mergeable。

# 风险与边界

1. 如果只验“能跑”，很容易漏掉工程主线是否真正统一。
2. 如果不要求证据，后续回归时无法确认规范是否被持续遵守。
3. 当前 Worker 边界保持单实例，不声明分布式调度能力。
4. `AuthMiddleware` 暂时保留 legacy `UserIDKey` 与 `platformruntime.WithUserID` 双写，因为 `auth_helpers.go` 仍读取 legacy key；后续迁移读取链路后可删除双写。
5. OpenAPI 当前使用统一 `ResponseEnvelope`，后续可继续细化每个 operation 的 `data` schema。
