# 背景

后端工程化落地不只是“代码能运行”，还必须验证统一错误处理、统一响应、统一上下文、OpenAPI 主线、数据库边界和 Worker 运行面是否真正收敛。

# 目标

1. 为 `009/010/011/018` 建立统一验收出口。
2. 定义横切规范、契约、数据库、Worker 和模块迁移的验收清单。
3. 明确每个阶段需要保留的验证证据。

# 非目标

1. 本文档不替代阶段计划。
2. 本文档不记录一次性临时排障过程。

# 验收范围

1. 横切工程骨架
2. 契约与 HTTP 规范
3. 数据访问与迁移边界
4. Worker 运行面
5. 业务模块迁移结果

# 验收清单

## 1. 横切工程骨架

- [ ] 存在统一错误模型与错误码注册表
- [ ] 存在 recover middleware
- [ ] 成功/失败/分页响应格式统一
- [ ] request id 与 trace id 注入生效
- [ ] 统一日志字段生效

## 2. 契约与 HTTP 规范

- [ ] `docs/openapi.json` 已生成
- [ ] 契约中显式表达 Bearer/JWT
- [ ] 错误响应结构在契约中统一
- [ ] DTO 未直接复用 ORM 模型

## 3. 数据访问与迁移边界

- [ ] `db/migrations` 为正式 schema 主线
- [ ] 业务层未直接读取 `*gorm.DB`
- [ ] 复杂查询收敛在 repository 内部

## 4. Worker 运行面

- [ ] 存在统一 scheduler / job runner 入口
- [ ] 任务统一调用 service
- [ ] 关键任务具备幂等保护
- [ ] 失败记录与重试策略可验证
- [ ] 单实例运行边界在文档和运行方式中一致

## 5. 业务模块迁移结果

- [ ] 第一批模块已接入统一错误、响应、上下文和契约规范
- [ ] 模块级单测通过
- [ ] 模块级 API 测试通过

# 验收证据

至少保留以下证据：

1. `go test ./...`
2. `go build ./...`
3. `docker compose config`
4. `docs/openapi.json`
5. 阶段性测试记录或脚本输出摘要

# 风险与边界

1. 如果只验“能跑”，很容易漏掉工程主线是否真正统一。
2. 如果不要求证据，后续回归时无法确认规范是否被持续遵守。
