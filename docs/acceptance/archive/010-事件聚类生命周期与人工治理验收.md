# 事件聚类、生命周期与人工治理验收

## 当前结论

PLAN-010 的范围验收已完成。`804cac2` 补齐重复关系合并时保留最高分和最早关系时间的最后一项 Important 发现；独立 Reviewer 已对完整实现范围给出 `APPROVED`，无未解决 Critical/Important。

已落地的可复核范围包括 Schema/领域契约、四通道有界候选召回、确定性决策记录、生命周期与人工治理 API、合并/拆分事务，以及以下补齐项：

- `1b73d00`：候选决策与 Event/成员/Monitor 关系在同一 PostgreSQL 事务中应用；重试和并发重放不重复创建 Event。
- `4b646c0`：active 证据变化事务化重算生命周期、代表成员和 retained 审计；人工锁不被自动覆盖。
- `961f2a2`：固定聚类评测集覆盖跨语言同事件、同实体不同事件、迟到内容、最优候选和灰区复核，并强制 F1 ≥ 75% 与候选上限。
- `261b3df`、`3922caf`：拆分成员按确定顺序加锁并拒绝陈旧成员；合并先解析 canonical target，再校验提交时的目标版本，避免历史 merged target 误报冲突。
- `3dd9acf`：事件 API 将输入错误、缺失资源和版本/状态冲突稳定映射为 400/404/409 numeric code。
- `973b3d8`：以 PostgreSQL trigger 注入成员迁移失败，证明 merge 和 split 都回滚成员、事件版本和治理审计，不留下半迁移关系。
- `4c046e0`、`f1be1e5`：split 和 merge 后都在原事务内重算受影响 Event 的生命周期、代表成员与 retained 审计；源 Event 不再指向已迁出的代表内容。
- `eca4cad`：曾为 split Event 复制 `content_dedupe_v1`；该初始实现已被 `b751b10` 的版本化事实指纹取代，不再代表当前契约。
- `23afa13`：候选召回用 PostgreSQL fixture 验证词法/时间/指纹通道各自的 16/12/10 上限；没有匹配的 Content/Event 向量空间时显式标记降级，确定性候选仍可继续处理。
- `de93a00`：向量通道先锁定精确的 profile/version 向量空间，再以 `event_embeddings_active_hnsw_idx` 取最多 12 个候选，并以已接受的 Monitor 关系过滤；集成 `EXPLAIN` 拒绝 event_embeddings 全表扫描。
- `ede6f1a`：PLAN-010 Schema 升级手册与固定 catalog 对齐，且从 PLAN-009 非空历史库完成 backup、升级、约束验证、未准备回退失败、准备后 restore 与历史 catalog 验证演练。
- `8eee5b2`：以 13 条公开发布元数据替换 6 条合成评测输入。v2 语料覆盖 2023-09-22 至 2024-08-05、5 个现实事件组、英文/中文、4 个以上独立发布机构、跨语言、同名不同时间、同实体不同事件、迟到更新和人工复核；每条样本保存 HTTPS 来源、发布时间、事件组、分区与标注理由。测试拒绝缺少来源、标签、冲突决议、必要覆盖项或将同一现实事件泄漏到 calibration/acceptance 两侧的 fixture；8 条 acceptance 样本含 5 条链接和 3 条非链接/复核，确定性决策 F1 为 1.00。
- `f00fcce`：候选去重保留所有命中的召回通道及其分数、Event 代表 Content 证据；最终候选仍只占一个精排名额。确定性决策为每个候选补齐 `recalled_*`/阈值/硬冲突原因、`recall_channels`/`recall_scores`/冲突标记快照和去重后的证据 Content ID；PostgreSQL 集成验证这些字段会写入 append-only `event_clustering_decisions`，而不改变候选上限或引入 Worker。
- `a1f718a`：使用真实 PostgreSQL Repository 串联 SQL 候选召回、三通道去重、Event 代表证据、确定性评分和 `ApplyClustering` 事务写入；验证持久化决策同时含三条召回通道、四条原因码和两条 Content 证据，且无向量空间时仍显式降级。
- `2694034`：HNSW `EXPLAIN` fixture 改为逐列解释包含 `representative_content_id` 的真实向量候选 SQL；继续证明 `event_embeddings_active_hnsw_idx` 被使用且没有 `event_embeddings` 全表扫描。
- `5e8d844`：成员查询在 Event 不存在时返回稳定 404；所有治理 `reason` 在 DTO/应用层复用审计长度与非空规则，超长输入稳定返回 400；OpenAPI 同步 lifecycle/merge 的 404。
- `d92dcbc`：合并先按稳定 `event_key` 锁 source 与完整 canonical 链，再比较 source 与 root 版本；历史 target 不再绕过 root optimistic version。普通成员迁移、关系去重和 Monitor 关系均记录带 Content/Monitor ID 的审计，合并 source 同时清空已迁出代表内容。
- `6fad39d`：补齐 split 运行时 `ErrNotFound` 的 404 Swagger 响应，并由架构 OpenAPI 测试固定该错误契约。
- `a5db148`：无可用 Event 向量空间时，`vector_unavailable` 会经 Recall/Execution 写入每条 `event_clustering_decisions.feature_snapshot`；确定性规则链仍可执行。
- `b751b10`：词法通道必须命中 accepted Monitor 的 approved entity/keyword/phrase 规则；时间通道以同 `source_connection_id` 或共享 Monitor region 限定 30 日候选。事实指纹由版本化 entity/action rule terms、region 与 UTC 日桶生成；旧版本或跨日 hash 均不召回，缺少实体或动作词时显式不生成指纹并继续其他通道。新 Event 与 split Event 使用相同事实函数，不再将 `dedupe_key` 冒充事实指纹。
- `d2c39de`：Schema trigger 拒绝 `event_key` 更新及 `event_clustering_decisions`、`event_governance_audits` 的 UPDATE/DELETE；PostgreSQL 集成直接验证三类写入均被拒绝。
- `804cac2`：重复 `event_contents` 与 `monitor_events` 合并均保留最高分和最早 `created_at`；PostgreSQL `-race` 回归覆盖两类关系，且验收记录明确旧 `content_dedupe_v1` 已被版本化事实指纹取代。

本地已在可丢弃 PostgreSQL/Redis fixture 上通过相关单元、`-race` 集成和 `make ci`；关键成员迁移故障注入已覆盖。v2 语料是真实公开报告的最小可复核元数据集，覆盖的是 PLAN-010 的确定性聚类契约。实际采集、Cron/River 编排与长期线上质量属于 PLAN-013/017 的验收责任，不能反向阻塞已完成的 PLAN-010 范围验收。

## 验收范围与禁止越界

- 只验收已通过 Monitor 相关性判断的 active Content 到 Event 的聚类、成员、生命周期、合并、拆分、证据重算和人工锁。
- 不验收热度趋势、实体/Claim、AI 摘要、队列/调度、知识库、报告投递或新增 AI task type。
- 合并/拆分只验证 `event_contents`、`monitor_events` 和 `merged_into_id`；实体、Claim、Topic、Report 只验证 `downstream_reconcile_required` 审计事件，不验证其迁移实现。
- 记录准确实现提交 SHA、Schema/record/OpenAPI 变更；不记录真实 DSN、密钥、Provider 原文或本机绝对路径。
