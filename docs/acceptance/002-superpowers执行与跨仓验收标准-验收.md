# 背景

HotKey 需要一套能长期复用的正式验收标准，确保任务不是“改完就算完”，而是按照 `server -> web -> miniapp -> 回归` 的顺序真正完成交付。

本标准用于回答三件事：

1. server 任务什么时候算完成。
2. web 与 miniapp 什么时候算同步完成。
3. 工程化检查、浏览器回归和代码审核怎样留下可追溯证据。

# 目标

1. 给出 `hotkey-server` 的完成判定清单。
2. 给出 `hotkey-web` 与 `hotkey-miniapp` 的同步完成判定清单。
3. 给出 `vercel:agent-browser` 的统一记录格式。
4. 给出代码审核与残余风险记录格式。

# 非目标

1. 本文档不替代具体业务功能的用例设计。
2. 本文档不限制各仓使用更细粒度的测试方法。
3. 本文档不将聊天记录视为验收证据。

# 验收范围

本标准适用于：

1. server 契约、错误结构、业务行为或鉴权规则变更。
2. 需要同步更新 web 或 miniapp 的跨仓任务。
3. 需要使用 `@superpowers` 完成设计、计划、执行和回归的 HotKey 任务。

# 验收顺序

固定顺序如下：

1. `hotkey-server`
2. `hotkey-web`
3. `hotkey-miniapp`
4. 跨仓回归

不允许：

1. 未完成 server 验收就先宣布端侧完成。
2. 未完成自动化检查就只用浏览器点击代替。
3. 未完成代码审核就只凭测试结果交付。

# `hotkey-server` 验收清单

## A. 代码与脚本验证

- [ ] 相关 Go 测试通过
- [ ] 相关构建检查通过
- [ ] 仓内验证脚本通过
- [ ] 关键成功路径与失败路径都有验证证据

建议记录：

```text
命令:
- go test ./...
- bash scripts/validate-repository.sh

结果:
- PASS / FAIL
- 若失败，记录阻塞原因与影响范围
```

## B. 工程化规范检查

- [ ] 存在统一异常处理或错误出口
- [ ] 对外错误结构保持一致
- [ ] handler / service / repository / worker 边界清晰
- [ ] 关键失败路径未散落在调用点临时处理
- [ ] 可观测性或日志路径未被本次改动破坏

建议记录：

```text
检查项:
- 全局异常处理
- 统一错误结构
- 边界分层
- 失败路径
- 可观测性

结论:
- 通过 / 不通过
- 对应证据文件与说明
```

## C. 契约事实源检查

- [ ] `docs/openapi.json` 已根据本次实现更新
- [ ] OpenAPI 变更与真实代码一致
- [ ] 已明确列出新增、修改、删除的接口或字段
- [ ] 已明确哪些下游页面或能力需要同步

建议记录：

```text
契约变更摘要:
- 接口:
- DTO:
- 错误结构:
- 鉴权:

下游影响:
- hotkey-web:
- hotkey-miniapp:
```

## D. 代码审核与高可用性检查

- [ ] 已显式审核本次修改代码
- [ ] 已检查未兜底错误出口
- [ ] 已检查状态不一致或跨层泄漏风险
- [ ] 已检查回归破坏点
- [ ] 已记录已修复项与残余风险

建议记录：

```text
审核结论:
- 发现问题:
- 修复情况:
- 残余风险:
```

只有 A、B、C、D 四组全部通过，`hotkey-server` 才允许进入下游同步阶段。

# `hotkey-web` 同步验收清单

## A. 契约同步

- [ ] 使用最新 server OpenAPI 重新生成客户端
- [ ] 未手写漂移类型绕过生成客户端
- [ ] 已记录生成命令与生成结果

## B. 端侧工程化检查

- [ ] 全局请求封装与错误处理已同步
- [ ] 全局状态接入点清晰
- [ ] 受影响页面覆盖加载态、空态、错误态
- [ ] AI 失败降级态可验证

## C. 自动化验证

- [ ] 相关测试通过
- [ ] 类型检查通过
- [ ] 构建通过

## D. `vercel:agent-browser` 回归

- [ ] 受影响页面逐项验证
- [ ] 主链路操作结果与预期一致
- [ ] 页面异常、空数据或失败降级路径至少验证一次
- [ ] 验证结果写入正式记录

# `hotkey-miniapp` 同步验收清单

## A. 契约同步

- [ ] 使用最新 server OpenAPI 重新生成客户端
- [ ] 未手写漂移类型绕过生成客户端
- [ ] 已记录生成命令与生成结果

## B. 端侧工程化检查

- [ ] 全局请求封装与登录适配已同步
- [ ] 全局状态接入点清晰
- [ ] 受影响页面覆盖加载态、空态、错误态
- [ ] 登录失效态或端侧受限态可验证

## C. 自动化验证

- [ ] 相关测试通过
- [ ] 构建通过

## D. 回归验证

- [ ] 若存在浏览器可承载入口，使用 `vercel:agent-browser` 完成主链路验证
- [ ] 若不存在浏览器可承载入口，已记录专属小程序验证路径、步骤与结果
- [ ] 验证结果写入正式记录

# `vercel:agent-browser` 记录模板

每次浏览器回归至少使用以下格式记录：

```text
页面/链路:
- 例如: 登录 -> 热点榜单 -> 热点详情 -> AI 快速理解

前置条件:
- 使用的账号
- 使用的环境
- 依赖的 server 版本或契约版本

步骤:
1. 打开页面
2. 执行操作
3. 观察结果

预期结果:
- 页面成功加载
- 数据正确展示
- 错误或降级路径符合预期

实际结果:
- 通过 / 失败
- 如失败，记录现象、截图位置和阻塞范围
```

如果页面发生跳转或 DOM 变化，记录中应体现重新 snapshot 并继续验证的步骤。

# 交付说明最低要求

每次跨仓任务交付说明至少包含：

1. 修改了什么
2. 如何验证
3. 工程化检查结果
4. `vercel:agent-browser` 回归结果
5. 代码审核结论
6. 未验证内容或残余风险
7. 涉及关键文件
8. Git 提交状态和 PR 状态

# 验收门禁

本文档本身的完成门禁如下：

1. 已覆盖 server 完成判定。
2. 已覆盖 web 与 miniapp 的同步判定。
3. 已给出 `vercel:agent-browser` 的记录模板。
4. 已给出代码审核与残余风险的记录格式。
5. 已明确交付说明的最低要求。

# 风险与边界

1. 若下游仓未按最新 OpenAPI 重新生成客户端，本标准无法替代真实同步工作。
2. 若没有可用的浏览器承载入口，miniapp 仍需要额外的小程序专属验证路径。
3. 若代码审核缺失，即使测试和浏览器回归通过，仍不能视为达到高可用性交付标准。
