# docs

本目录用于存放项目文档。建议按文档类型放到对应子目录，避免 PRD、计划、设计、验收和运维材料混放。

## 当前目录说明

1. `product/`：产品需求、目标与范围定义、阶段化需求。
2. `engineering/`：技术方案、架构边界、实现约束和验收标准。
3. `plans/`：执行计划、阶段拆解、任务清单、排期安排。
4. `research/`：调研材料。
5. `acceptance/`：验收标准、测试记录、验证报告、回归证据（目前按阶段沉淀）。
6. `archive/`：归档材料与历史文档。

## 编写规范

1. 新文档先判断类型，再放入对应子目录。
2. 文档标题和文件名应能表达主题，避免使用 `new.md`、`temp.md` 等模糊命名。
3. 文档内容应保持 MVP 风格，只写当前阶段需要的信息。
4. OpenSpec proposal、tasks、spec delta 仍放在 `openspec/`，不要混入 `docs/`。

## 收录边界

1. `docs/` 只收录会对项目产生长期真实影响的文档。
2. 不收录临时任务清单、过程性进展记录、会议记录、一次性排查记录。
3. 过程性任务应写入 OpenSpec change 的 `tasks`，完成后随 change 归档。
4. 如果文档不能影响后续开发、验收、发布或维护决策，默认不应放入 `docs/`。

## 说明

每份正式文档建议使用统一头部信息（如 `yaml frontmatter`）描述来源、用途、版本、责任人、输入输出与触发条件，便于长期追踪与验收回溯。
