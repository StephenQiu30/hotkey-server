# AI 热点监控工具技术方案

## 1. 技术方案概述

本项目从零重建，不保留旧实现、旧目录结构、旧数据库结构、旧 OpenAPI 契约和旧示例数据路径。

目标是实现一个可自部署的 AI 热点监控 MVP：关键词管理、多源抓取、AI 查询扩展、真假识别、相关性分析、热点列表、筛选排序、即时搜索、SMTP 邮件通知、日报/周报、手动触发和简单定时触发。

功能闭环围绕本项目 AI 热点监控 MVP 自主设计。

## 2. 技术选型

| 层级 | 选型 |
| --- | --- |
| 前端 | `Next.js + TypeScript` |
| 后端 | `Python + FastAPI` |
| 数据库 | `PostgreSQL` |
| ORM | `SQLAlchemy 2.0` |
| Schema 管理 | `sql/001_init_schema.sql` + `SQLAlchemy 2.0` models（无迁移） |
| AI | OpenAI 兼容模型 API |
| 邮件 | `SMTP` |
| 定时任务 | P0 轻量调度，后续再评估 Celery；Redis 如有启用以本机实例为先 |
| 部署 | 本机 PostgreSQL + 本地进程；Redis 使用本机 Homebrew 实例（如有启用）；Docker Compose 仅作为可选 API/Web 容器启动方式 |

- 运行边界：Python/Node 默认不要求仓库级虚拟环境；PostgreSQL/Redis 默认连接本机 Homebrew 实例（`localhost`）；Docker 仅为可选启动方式。

该阶段不引入 schema 迁移工具。`sql/001_init_schema.sql` 是数据库表结构唯一事实源，空库启动时优先执行该 SQL 文件创建表；SQLAlchemy models 必须与 SQL 文件保持一致。`Base.metadata.create_all()` 仅作为开发兜底，不作为主要 schema 事实源。

运行边界约束：

- 不依赖仓库级 Python 虚拟环境；Node 亦按本机执行环境启动命令。
- PostgreSQL 与 Redis 优先连接本机 Homebrew 实例；若未配置 Redis 不阻塞主流程。
- Docker 仅作为可选部署入口，不默认创建数据库容器。

## 3. 新目录职责

- `server`
  - FastAPI 后端入口、路由、依赖注入、数据库访问和任务触发入口
- `apps/web`
  - Next.js 控制台
- `packages/core`
  - 轻量共享常量、类型或规则说明，不承载旧复杂分层
- `sql`
  - PostgreSQL 建表 SQL，保存 P0 全部表结构
- `infra`
  - 环境变量模板、本机 PostgreSQL/Redis 连接说明、Docker 可选配置
- `docs/plans`
  - 可执行任务计划，后续实现必须按计划文件推进

## 4. 数据模型

P0 新数据库模型从空 PostgreSQL 创建，不兼容旧结构。

- `keywords`
  - 关键词、查询模板、启停、优先级、创建时间、更新时间
- `sources`
  - 来源名称、来源类型、启停、配置 JSON、创建时间、更新时间
- `hotspots`
  - 标题、URL、来源、关键词、发布时间、抓取时间、状态、唯一约束
- `ai_analyses`
  - 热点 ID、真实性、相关性分数、相关性理由、关键词是否命中、重要性、摘要、模型名
- `notifications`
  - 热点 ID、报告 ID、邮件状态、收件人、错误信息、发送时间
- `reports`
  - 报告类型、周期开始、周期结束、状态、标题、摘要、正文、热点数量、发送时间
- `check_runs`
  - 触发类型、开始时间、结束时间、状态、成功数量、失败数量、错误摘要
- `settings`
  - 非敏感设置；敏感信息仍通过环境变量注入

## 5. 核心流程

1. 用户创建并启用关键词。
2. 用户手动触发或简单定时触发热点检查。
3. 后端读取启用关键词和启用来源。
4. AI 将关键词扩展为 2-5 个相关查询词。
5. RSS、Hacker News、X/Twitter、Bing、Bilibili、Sogou-style adapter 抓取候选内容，单个来源失败只记录，不中断整体任务。
6. 内容标准化后按 URL + 来源去重。
7. 调用 OpenAI 兼容模型输出真实性、相关性、重要性和中文摘要。
8. 低于 `RELEVANCE_THRESHOLD` 或被 AI 判定为不真实的热点标记为 `filtered`；达到阈值且 `is_real is not False` 的热点标记为 `active`。
9. SMTP 配置存在时只为 `active` 热点发送事件邮件并记录通知状态。
10. 日报/周报只聚合 `active` 热点；报告内容以本地 Markdown 模板作为 P0 必需能力。
11. 当前阶段优先跑通后端检测、搜索、报告和通知闭环；控制台保留最小可视化闭环（关键词、来源、热点、搜索、任务、报告、通知、设置）用于验收，不作为企业级扩展目标。

## 6. API 范围

P0 API 以新实现为准，由 FastAPI 自动生成 OpenAPI 文档。

- 关键词 CRUD 和启停
- 来源配置和启停
- 热点列表、详情、筛选、排序
- 全网搜索
- 手动触发热点检查
- 任务执行记录查询
- 通知列表和邮件状态查询
- 日报/周报生成、发送、列表和详情，统一入口为 `/api/reports`
- 设置项读取和更新

旧 `contracts/openapi` 不作为新实现约束。

## 7. 降级要求

- SMTP 未配置时，热点检查仍可运行，只跳过邮件发送。
- X/Twitter 未配置时，系统仍可运行，只跳过该来源。
- Bing 未配置时，系统仍可运行，只跳过该来源。
- 单个来源失败不能中断整体热点检查。
- AI 调用失败必须记录到 `check_runs` 或等价任务状态中。
- 敏感配置必须通过环境变量注入，不写入数据库。

## 8. 当前实现状态

- 已实现并可用现有自动化检查验证：后端检测链路、即时搜索、统一报告 API、模板报告、SMTP 缺失降级、查询扩展降级、AI 分析降级、前端工作台页面。
- 需要真实配置后验收：X/Twitter、Bing、SMTP 发送和 OpenAI 兼容模型调用。
- P0 不要求 AI 生成整篇报告；报告 AI 增强属于后续可选能力。

## 9. 非目标

- 不保留旧实现。
- 不保留旧数据库结构。
- 不做旧数据迁移。
- P0 不做多租户、复杂权限、计费、复杂工作流、向量库、复杂队列治理。
- P0 不引入 Celery/Redis 作为必需组件。

## 10. 运维可观测

- 默认提供运行时可观测端点：`GET /api/ops/metrics`。
- 运行时快照包含：
  - 请求总数
  - 状态码分桶
  - 限流触发次数（429）
  - 活跃限流客户端数量
- 指标为内存快照，服务重启后清零；长期行为由日志与数据库审计补充。
