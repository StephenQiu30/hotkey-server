# 邮箱认证与会话安全设计

## 1. 背景

当前 `hotkey-server` 已提供 `/api/v1/auth/register` 和 `/api/v1/auth/login`：注册直接创建用户，密码使用 bcrypt 保存，登录签发单个有效期 24 小时的 JWT。当前实现没有邮箱验证、验证码限流、忘记密码、Refresh Session、可靠会话撤销和真实邮件发送。

`hotkey-web` 已有邮箱密码登录页，但将 Token 写入 `localStorage`，没有注册、忘记密码和安全的无感刷新流程。`hotkey-miniapp` 仍保留旧版平台登录契约，本期不实现小程序认证。

本文定义目标认证方案，并以 `hotkey-server` 为契约事实源，按 `server -> web -> 回归` 顺序交付。

## 2. 目标

1. 用户先完成邮箱验证码验证，再设置密码并创建账号。
2. 用户使用邮箱和密码登录，并获得无感续期的 Web 会话。
3. 用户可以通过邮箱验证码安全重置密码。
4. Access Token 有效期 15 分钟；Refresh Session 滑动有效期 7 天、绝对最长 30 天。
5. 验证码、登录和刷新流程具备限流、并发控制、重放防护和账号探测防护。
6. 使用 163 邮箱 SMTP 发送注册验证码、重置验证码和密码变更通知。
7. ErrorCode、全局异常、统一响应、DTO、VO 和转换遵循项目统一规范。
8. 密码、验证码、Token 和 SMTP 授权码不以明文持久化或写入日志。

### 2.1 核心验收场景

```gherkin
Given 一个未注册邮箱
When 用户完成验证码验证并设置合法密码
Then 系统创建唯一用户、创建会话并进入登录状态
```

```gherkin
Given 用户持续使用 Web
When Access Token 到期且会话未空闲超过 7 天并且未达到 30 天上限
Then 系统自动轮换 Refresh Token 并无感继续访问
```

```gherkin
Given 会话已经达到绝对 30 天
When Web 尝试刷新
Then 系统返回 AUTH_SESSION_EXPIRED 并要求重新输入密码
```

```gherkin
Given 用户完成密码重置
When 任意旧设备尝试刷新
Then 该用户的全部旧会话均已撤销
```

```gherkin
Given 请求使用不存在邮箱或已注册邮箱
When 用户请求重置密码验证码
Then HTTP状态、ErrorCode、响应结构和公开文案保持一致
```

## 3. 非目标

本期不实现：

1. 微信小程序登录或微信身份绑定。
2. 手机号、GitHub、Apple 等其他登录方式。
3. 登录二次验证码或 MFA。
4. 邮箱修改。
5. 用户自行查看和管理全部设备。
6. 企业级风控引擎。
7. 验证码异步消息队列。
8. 邮箱字段级可逆加密与复杂密钥轮换系统。

模型边界允许后续扩展，但本期不提前建设未使用的身份表和账号合并流程。

## 4. 总体架构

采用“Redis 验证码 + PostgreSQL Session + SMTP”方案：

1. `AuthService`：注册、登录和重置密码的业务编排。
2. `VerificationService`：验证码、Ticket、Redis 锁与限流。
3. `SessionService`：Refresh Session 创建、轮换和撤销。
4. `TokenService`：Access Token 签发与验证。
5. `Mailer`：通过 163 SMTP 发送认证邮件。
6. `UserRepository`、`AuthSessionRepository`：用户和会话持久化。

数据按生命周期存储：

| 存储 | 内容 |
|---|---|
| PostgreSQL | 用户、密码哈希、邮箱验证状态、Refresh Session |
| Redis | 验证码摘要、Ticket、失败次数、发送锁和限流计数 |
| HttpOnly Cookie | 原始 Refresh Token |
| Web 内存 | Access Token 和当前用户 VO |

Redis 不可用时，验证码申请、确认、注册和密码重置失败关闭；已注册用户的密码登录、Access Token 鉴权、Session 刷新和退出仍可工作。

## 5. 模型与目录规范

### 5.1 模型分层

统一使用：

```text
internal/model/
├── entity/
├── dto/
├── enum/
└── vo/
```

| 目录 | 职责 | 约束 |
|---|---|---|
| `entity/` | PostgreSQL/GORM 持久化模型 | 不带 HTTP tag，不直接返回 API |
| `dto/` | Controller、Service、Repository 间输入输出 | 可包含请求绑定和基础校验 tag，不带 GORM tag |
| `enum/` | ErrorCode、验证码用途、账号状态、撤销原因 | 禁止业务层散落裸字符串 |
| `vo/` | 对客户端公开的视图模型 | 不包含任何密码、Token 哈希或内部安全状态 |

依赖方向为：

```text
Controller -> DTO -> Service -> Repository -> Entity
Entity -> Convert -> DTO -> Convert -> VO -> ResponseBody
```

### 5.2 DTO 命名

资源 CRUD 请求统一使用：

```text
{Resource}AddRequest
{Resource}EditRequest
{Resource}QueryRequest
{Resource}DetailRequest
{Resource}DeleteRequest
```

公共 DTO 只抽取真正一致的形状，例如 `PageQueryRequest`、`IDRequest` 和 `DeleteRequest`。存在版本、所有权或级联参数时，必须定义资源专属 DeleteRequest。

认证不是 CRUD，使用语义化动作名称：

```text
VerificationSendRequest
VerificationConfirmRequest
EmailRegisterRequest
EmailLoginRequest
PasswordResetRequest
TokenRefreshRequest
LogoutRequest
```

所有 HTTP 入参以 `Request` 结尾，HTTP 出参使用 VO。Entity 不得作为 Request 或 Response。

### 5.3 认证目录

```text
internal/
├── controller/
│   ├── auth_controller.go
│   └── swagger_response.go
├── service/
│   ├── auth_service.go
│   ├── verification_service.go
│   └── session_service.go
├── repository/
│   ├── user_repository.go
│   └── auth_session_repository.go
├── platform/
│   ├── email/smtp_mailer.go
│   └── security/
│       ├── password.go
│       ├── token.go
│       └── digest.go
├── convert/auth_convert.go
└── model/
    ├── entity/
    ├── dto/
    ├── enum/
    └── vo/
```

Service 不依赖 Gin；Repository 不返回 VO；Controller 不直接构造 Entity。Entity、DTO 和 VO 的转换统一放在 `internal/convert/`。

## 6. PostgreSQL 数据模型

项目继续以 `db/schema.sql` 作为数据库结构唯一事实源，不建立长期 migration 目录。

### 6.1 users 扩展

新增：

| 字段 | 类型 | 用途 |
|---|---|---|
| `email_verified_at` | `timestamptz null` | 邮箱验证完成时间 |
| `password_changed_at` | `timestamptz not null default now()` | 密码最近变更时间 |
| `last_login_at` | `timestamptz null` | 最近一次密码登录时间 |

邮箱写入前必须去除首尾空白、转为小写、校验标准格式并限制为 254 个字符。不对 `.` 或 `+suffix` 做服务商特定归一化。

### 6.2 auth_sessions

新增 `auth_sessions`：

| 字段 | 用途 |
|---|---|
| `id` | Session ID |
| `user_id` | 关联用户 |
| `token_hash` | 当前 Refresh Token 的 SHA-256 哈希 |
| `family_id` | Token 轮换族 |
| `created_at` | 首次登录时间 |
| `last_used_at` | 最近成功刷新时间 |
| `idle_expires_at` | 滑动 7 天到期时间 |
| `absolute_expires_at` | 固定 30 天上限 |
| `revoked_at` | 撤销时间 |
| `revoke_reason` | `logout`、`password_reset`、`token_reuse` 等枚举值 |
| `user_agent_hash` | 客户端摘要，不保存完整 User-Agent |
| `ip_prefix` | 可选脱敏 IP 网段，仅用于安全审计 |

`token_hash` 建唯一索引；`user_id`、`family_id` 和到期字段建立查询索引。过期和撤销记录由定时任务清理，请求正确性不得依赖清理任务。

## 7. Redis 验证码与限流

Key 不包含完整邮箱，使用规范化邮箱的 HMAC 摘要：

```text
auth:verification:{purpose}:{email_key}
auth:ticket:{purpose}:{ticket_hash}
auth:send_lock:{purpose}:{email_key}
auth:send_limit:{purpose}:{email_key}:{hour}
auth:ip_limit:{purpose}:{ip_key}:{hour}
```

`purpose` 只允许 `register` 和 `reset_password`。

规则：

1. 验证码为 6 位数字，有效期 10 分钟。
2. 发送锁有效期 60 秒，使用 `SET NX EX` 原子获取。
3. 同一邮箱、同一用途每小时最多发送 5 次。
4. 同一 IP、同一用途每小时默认最多发送 20 次。
5. 连续校验错误 5 次后立即删除验证码。
6. 新验证码覆盖同一邮箱和用途的旧验证码。
7. 注册和重置验证码不能混用。
8. 校验成功后原子删除验证码并签发一次性 Ticket。
9. Ticket 有效期 10 分钟，绑定邮箱和用途，只能使用一次。

验证码使用 `crypto/rand` 生成。锁、限流、失败计数、验证码消费和 Ticket 消费使用 Lua 脚本或等价原子操作，禁止 `GET` 后再 `SET` 的竞态实现。

### 7.1 发送一致性

1. 原子获取发送锁并检查限流。
2. 生成验证码和 HMAC 摘要。
3. 将验证码状态以短 TTL 写入 Redis。
4. 调用 163 SMTP。
5. 明确发送失败时删除验证码并释放锁。
6. 发送成功后增加邮箱和 IP 计数。
7. SMTP 超时导致结果不确定时保留验证码并计算一次额度，因为邮件可能已经送达。

## 8. 敏感信息保护

| 信息 | 保护方式 |
|---|---|
| 密码 | bcrypt 不可逆哈希，永不保存或记录明文 |
| 验证码 | Redis 只保存 `HMAC-SHA256(用途 + 邮箱 + 验证码)`；HMAC 密钥作为 pepper |
| Refresh Token | 256 位安全随机数；PostgreSQL 只保存 SHA-256 哈希 |
| Access Token | JWT 签名，短期有效，不持久化 |
| SMTP 授权码 | 环境变量或 Secret Manager 注入，不进入数据库、Git、响应或日志 |
| 邮箱 | 数据库保存规范化明文用于登录、唯一约束和发信；磁盘与备份加密；日志脱敏 |

邮箱随机可逆加密会破坏稳定查询和唯一约束。本期采用数据库与备份加密、最小权限和日志脱敏，不引入字段级可逆加密系统。

禁止记录密码、验证码、原始 Token、完整 Cookie、SMTP 授权码和完整邮箱。安全日志使用 `request_id`、ErrorCode、操作、用户 ID、脱敏邮箱、Session ID、依赖类型和内部 cause。

## 9. API 契约

### 9.1 发送验证码

`POST /api/v1/auth/verifications`

请求包含 `email` 和 `purpose`。无论邮箱是否存在均返回 `202 Accepted`、`SUCCESS` 和统一文案“如果该邮箱可用，验证码将发送”。

注册用途仅在邮箱不存在时发送；重置用途仅在邮箱存在且账号状态正常时发送。公开状态、ErrorCode 和文案不得暴露邮箱是否已注册。

### 9.2 确认验证码

`POST /api/v1/auth/verifications/confirm`

请求包含邮箱、用途和验证码。成功后返回一次性 `verification_ticket` 和 `expires_in=600`。Ticket 是安全随机字符串，Redis 只保存哈希。前端后续不再提交验证码。

### 9.3 注册

`POST /api/v1/auth/register`

请求包含 `verification_ticket`、`password` 和 `display_name`。密码为 8 至 64 个 Unicode 字符且 UTF-8 编码不得超过 bcrypt 的 72 字节上限，必须至少包含一个 ASCII 字母和一个数字，允许其他 Unicode 字符、特殊字符与空格，不强制大小写或特殊字符。密码按用户原始输入参与哈希，不做 trim 或 Unicode 归一化。

注册成功后直接创建 Session，返回用户 VO 和 Access Token，并设置 Refresh Cookie。

### 9.4 登录

`POST /api/v1/auth/login`

请求包含邮箱和密码。邮箱不存在、密码错误或账号不可用统一返回 `AUTH_INVALID_CREDENTIALS` 和“邮箱或密码错误”。邮箱不存在时仍执行一次固定 bcrypt 比较，降低时序探测风险。

### 9.5 刷新

`POST /api/v1/auth/token/refresh`

从 HttpOnly Cookie 读取 Refresh Token，原子轮换并返回新 Access Token和新 Cookie。旧 Token 重放时撤销整个 Session family。

### 9.6 当前用户

`GET /api/v1/auth/me`

需要 Access Token，返回当前用户公开 VO。

### 9.7 退出

`POST /api/v1/auth/logout`

撤销当前 Session 并清除 Cookie；接口幂等。本期不提供用户可见的全部设备管理页面。

### 9.8 重置密码

验证码申请和确认复用通用接口，purpose 为 `reset_password`。`POST /api/v1/auth/password/reset` 接收 Ticket 和新密码。

成功后在同一数据库事务中更新密码哈希和 `password_changed_at`，撤销全部 Session；随后消费 Ticket、清除 Cookie，并尽力发送密码变更通知。邮件失败不得回滚已经成功修改的密码。

## 10. ErrorCode、全局异常与统一响应

### 10.1 统一 Envelope

成功：

```json
{
  "code": 200,
  "error_code": "SUCCESS",
  "data": {}
}
```

失败：

```json
{
  "code": 401,
  "error_code": "AUTH_INVALID_CREDENTIALS",
  "data": null
}
```

分页响应在相同 Envelope 中增加 `page`、`page_size` 和 `total`。`code` 与实际 HTTP status 保持一致；`error_code` 是前端业务分支的唯一稳定依据。响应体不包含 `message` 和 `request_id`；`request_id` 仅保留在 `X-Request-Id` 响应头和安全日志中。

Envelope 字段固定语义：

| 字段 | 类型 | 规则 |
|---|---|---|
| `code` | integer | 必须与实际 HTTP status 一致 |
| `error_code` | string | 必须来自 Server 中央 ErrorCode 注册表；成功固定为 `SUCCESS` |
| `data` | object/array/null | 成功时为业务 VO，失败时固定为 `null` |

### 10.2 ErrorCode 枚举

通用枚举：

```text
SUCCESS
BAD_REQUEST
UNAUTHORIZED
FORBIDDEN
NOT_FOUND
CONFLICT
RATE_LIMITED
INTERNAL_ERROR
SERVICE_UNAVAILABLE
```

认证枚举：

```text
AUTH_INVALID_INPUT
AUTH_INVALID_CREDENTIALS
AUTH_EMAIL_ALREADY_REGISTERED
AUTH_VERIFICATION_INVALID
AUTH_VERIFICATION_EXPIRED
AUTH_VERIFICATION_TOO_MANY_ATTEMPTS
AUTH_VERIFICATION_SEND_TOO_FREQUENT
AUTH_SESSION_EXPIRED
AUTH_SESSION_REVOKED
AUTH_TOKEN_INVALID
AUTH_TOKEN_REUSED
AUTH_ACCOUNT_DISABLED
AUTH_PASSWORD_POLICY_VIOLATION
```

中央注册表为每个 ErrorCode 固定映射 HTTP 状态、是否可重试和是否记录安全日志。Controller 不得重复维护映射，不得通过自定义文案改变错误语义。

`AUTH_EMAIL_ALREADY_REGISTERED` 只允许在已经持有有效注册 Ticket 的最终注册提交中使用；验证码发送和密码重置申请不得返回该错误，避免账号探测。

### 10.3 全局异常

全局处理器覆盖 DTO 校验、AppError、PostgreSQL、Redis、SMTP、JWT、未知错误、panic、404 和 405。内部 cause 只进入脱敏日志，外部只返回注册过的 `error_code`。

Controller 只绑定 Request DTO、调用 Service、转换成功 VO，并将错误交给全局处理器。禁止在 Controller 中使用大段错误 switch、直接暴露 `err.Error()` 或自行拼装 JSON。

响应已经提交时只记录错误，禁止追加第二段 JSON。

### 10.4 DTO、VO 与 Swagger

认证 Request DTO 包括：

```text
VerificationSendRequest
VerificationConfirmRequest
EmailRegisterRequest
EmailLoginRequest
PasswordResetRequest
TokenRefreshRequest
LogoutRequest
```

VO 包括：

```text
VerificationSendData
VerificationTicketData
AuthenticatedUserData
AuthTokenData
LoginData
SessionData
OperationResultData
```

VO 不包含密码哈希、Token 哈希、family ID、SMTP 信息、Redis Key 或内部撤销原因。Swagger 在 `controller/swagger_response.go` 声明显式包装类型，但必须与统一 Envelope 同构，不建立第二套运行时响应模型。

## 11. Token、Session 与 Cookie

### 11.1 Access Token

Access Token 为有效期 15 分钟的 JWT，至少包含 `iss`、`aud`、`sub`、`sid`、`jti`、`iat`、`nbf` 和 `exp`。验证时固定允许算法，并校验签名、算法、签发方、受众和时间声明。

JWT 不携带邮箱、昵称和权限详情。Access Token 只保存在 Web 内存。

### 11.2 Refresh Token

Refresh Token 使用 `session_id.random_secret` 形式，secret 至少 256 位。数据库通过 Session ID 定位记录，并恒定时间比较 Token 哈希。

Session 滑动有效期 7 天，每次成功刷新后延长，但不得超过登录时确定的绝对 30 天。轮换必须使用数据库事务和行锁或条件更新，确保同一 Token 只能成功一次。

Web 必须使用单航班刷新，同一时刻多个请求共享一个 Refresh Promise。每个业务请求最多自动重放一次。

Session 撤销会立即阻止 Refresh，但已经签发的无状态 Access Token 最多仍可使用 15 分钟。首版接受这一明确窗口；不为每次 API 请求增加数据库或 Redis Session 查询。后续若出现高风险操作，应在该操作入口额外校验 Session，而不是把所有接口改为有状态鉴权。

### 11.3 Cookie

生产配置：

```text
Name: hk_refresh
HttpOnly: true
Secure: true
SameSite: Lax
Path: /api/v1/auth
Max-Age: 7 days
```

Cookie 默认不设置宽泛父域。本地 HTTP 可通过明确配置关闭 Secure。退出和重置密码必须使用完全相同的 Domain、Path 和 SameSite 参数清除 Cookie。

首选部署要求 Web 与 API 位于同一站点下的 HTTPS 域名，例如 `app.example.com` 与 `api.example.com`。如果实际部署为跨站域名，必须显式改为 `SameSite=None; Secure`，并增加双提交 CSRF Token；不得只修改 Cookie 属性后直接上线。

## 12. CSRF、CORS 与接口限流

Refresh 和 Logout 依赖 Cookie，必须校验 `Origin`、要求 `X-HotKey-Client: web`，使用 `SameSite=Lax`，并拒绝不受信来源。刷新、退出和密码修改不得使用 GET。

当前 `Access-Control-Allow-Origin: *` 与 `Access-Control-Allow-Credentials: true` 的组合无效，必须改为 `WEB_ALLOWED_ORIGINS` 白名单。命中时回显具体 Origin，增加 `Vary: Origin`；生产环境禁止 `*`。

默认限流：

| 操作 | 限制 |
|---|---|
| 验证码发送 | 邮箱/用途每小时 5 次，IP/用途每小时 20 次，重发间隔 60 秒 |
| 验证码确认 | 单验证码最多错误 5 次，并增加 IP 总量限制 |
| 登录 | IP 每分钟 20 次；邮箱连续失败采用递增短时锁定 |
| Refresh | Session 每分钟限流；客户端单航班 |
| 注册与重置提交 | Ticket 单次使用，并受 IP 限制 |

账号锁定只用于短时暴力破解，不永久锁死账号。限流响应使用 `RATE_LIMITED` 或认证专用 ErrorCode，并设置 `Retry-After`。

## 13. 事务与并发边界

### 13.1 注册

1. Redis 原子把 Ticket 从 `ready` 改为 `processing`。
2. PostgreSQL 事务再次检查邮箱唯一性，创建用户和首个 Session。
3. 提交后将 Ticket 标记为 `completed` 并设置短 TTL。
4. 数据库失败时把 Ticket 恢复为 `ready`。
5. 数据库已提交但 Redis 最终确认失败时仍返回成功；唯一约束阻止重复账号。

### 13.2 登录

查询用户、校验密码和状态后，在同一事务中创建 Session 并更新 `last_login_at`。JWT 签发失败时撤销刚创建的 Session。

### 13.3 Refresh

条件更新至少包含：Token 哈希匹配、未撤销、滑动期限未到、绝对期限未到。成功更新行数必须为 1，否则映射为过期、撤销或重放 ErrorCode。

### 13.4 重置密码

密码哈希更新、`password_changed_at` 更新和全部 Session 撤销在一个 PostgreSQL 事务中完成。安全通知邮件是事务后的尽力操作。

## 14. 163 SMTP

配置：

```text
SMTP_HOST=smtp.163.com
SMTP_PORT=465
SMTP_USERNAME=
SMTP_AUTH_CODE=
SMTP_FROM_EMAIL=
SMTP_FROM_NAME=HotKey
SMTP_TLS_MODE=implicit
SMTP_CONNECT_TIMEOUT=5s
SMTP_SEND_TIMEOUT=10s
```

使用 163 客户端授权码，不使用邮箱登录密码。必须验证证书和主机名，禁止关闭 TLS 校验。邮件模板变量必须 HTML 转义，并同时提供 HTML 与纯文本内容。

首版模板：注册验证码、重置密码验证码、密码已修改通知。邮件包含用途、6 位验证码、10 分钟有效期、安全提醒和非本人操作说明；不得包含密码、Token、Ticket 或可直接完成敏感操作的 GET 链接。

## 15. Web 交互与状态管理

### 15.1 页面

```text
/login
/register
/forgot-password
/reset-password
```

注册和忘记密码均采用“发送验证码 -> 确认验证码 -> 使用内存 Ticket 完成操作”的三步流程。Ticket 不进入 URL、日志、LocalStorage 或 SessionStorage；页面刷新后安全返回第一步。

### 15.2 Auth 状态

状态只使用：

```text
initializing
authenticated
unauthenticated
```

内存保存 `accessToken`、`accessTokenExpiresAt` 和用户 VO，不单独持久化 `isAuthenticated`。应用启动先 Refresh，再调用 `/auth/me`；初始化期间显示加载态，禁止受保护页面闪烁。

### 15.3 请求层

统一请求层负责注入 Access Token、解析统一 Envelope、把 `error_code` 转为前端异常、主动或被动刷新、单航班等待和一次请求重放。登录、注册、验证码和刷新请求禁止递归刷新。

前端维护一个集中的 `error_code -> 中文文案` 映射，页面只传递 ErrorCode 和展示位置，不重复编写错误文案。OpenAPI 生成的 ErrorCode 联合类型是前端枚举事实源；未识别 ErrorCode 统一展示“操作失败，请稍后重试”。

Refresh 失败时清空内存状态并跳转登录。站内 redirect 只允许相对路径，防止开放重定向。

### 15.4 安全交互

密码和验证码不得进入 URL、Analytics、埋点和错误上报。表单支持密码管理器语义、键盘操作、提交防重和字段级 ErrorCode 提示。

## 16. 测试与验收

### 16.1 Server

单元测试覆盖邮箱规范化、密码策略、bcrypt、验证码 HMAC、JWT claims、Refresh Token、ErrorCode 映射、转换泄漏和 Cookie 参数。

Service 测试覆盖注册、登录、刷新、退出、重置主流程，以及 Ticket 过期/误用/重放、验证码 5 次失败、并发注册、并发刷新、Session 期限、密码重置撤销、依赖失败和账号探测防护。

集成测试使用真实 PostgreSQL 和 Redis，Fake SMTP Server 验证邮件与 TLS 行为；验证 CORS、Cookie、统一 Envelope、404、405、panic 和敏感信息不进入日志。

### 16.2 Web

测试注册三步流程、倒计时、登录回跳、密码重置、启动恢复、Token 自动刷新、并发单航班、Refresh 失败、ErrorCode 映射、未知 ErrorCode 降级、AuthGuard 和浏览器存储中不存在 Token。

必须有一条可重复的端到端验收链路：

```text
发送验证码 -> 确认验证码 -> Ticket 注册并建立会话 -> /auth/me
-> 退出 -> 密码登录 -> Access Token 失效 -> Cookie 自动刷新 -> /auth/me
```

此链路必须同时验证 HTTP status、数字 `code`、稳定 `error_code`、Cookie 轮换与前端文案展示。旧的“直接提交邮箱注册”路径必须移除，不得作为兼容分支保留。

### 16.3 门禁

Server：

```text
make openapi
make openapi-validate
bash scripts/validate-repository.sh
go test -race ./...
git diff --check
```

Web：类型检查、Lint、单元/组件测试、生产构建、浏览器主链路验收和 `git diff --check`。

运行时使用真实 PostgreSQL、Redis 和测试 SMTP 走通；上线前使用真实 163 邮箱完成收件验证。

## 17. 迁移与发布

现有用户回填：

1. `email_verified_at = created_at`，避免旧用户升级后无法登录。
2. `password_changed_at = created_at`。
3. 保留已有 bcrypt 密码哈希。
4. 新 Session 表不导入旧 Session；旧 24 小时 JWT 在切换窗口结束后失效。

已有数据库不会仅靠 `CREATE TABLE` 自动增加字段。部署升级需提供一次性、可重复执行的 schema 升级脚本或启动兼容步骤；执行前备份并验证字段、索引和回填结果，完成后不保留长期 migration 链。

交付顺序：

1. Server 统一 ErrorCode、Envelope、DTO/VO。
2. Server 数据结构、Redis、Session 与安全组件。
3. Server SMTP 和认证 API。
4. 生成并校验 OpenAPI。
5. Web 请求层、Auth Store 和状态恢复。
6. Web 注册、登录和忘记密码页面。
7. Server + Web 集成回归。
8. miniapp 本期不实现；其旧 `/api/auth/*` 契约留待后续专门同步。

全局 Envelope 是破坏性契约变化，Server 和 Web 必须同步发布。

## 18. 监控、风险与边界

聚合指标包括验证码发送、限流、SMTP 错误与延迟、登录成功/失败、Refresh 成功/过期/撤销/重放、密码重置和依赖错误。指标不包含邮箱、验证码或 Token。

主要风险：

1. 163 SMTP 的频率、异地登录和授权码风控需上线前验证。
2. Refresh 轮换需要 Web 单航班和 Server 原子更新同时正确。
3. Redis 是注册和重置密码关键依赖，需要健康检查和告警。
4. 跨域 Cookie 依赖真实 Web/API 域名和 HTTPS 配置，必须在预发布环境验证。
5. 全局 Envelope 变化会影响所有现有 API 消费方，必须完成全量回归。
