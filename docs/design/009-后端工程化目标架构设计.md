# 背景

`hotkey-server` 作为 HotKey 体系中的后端服务，需要支撑认证、监控任务、内容采集、热点聚合、趋势分析、通知投递和日报导出等能力。对于这类后端系统，真正决定长期可维护性的，不是单点框架是否流行，而是是否建立了稳定的工程主线、清晰的职责边界和统一的基础设施规范。

本设计不以当前实现细节为前提，也不以历史遗留结构为约束，而是从软件工程角度重新定义：对于 `hotkey-server` 这类 Go 后端服务，什么样的架构设计与工程规范才是长期正确的目标状态。

# 目标

1. 定义 `hotkey-server` 的目标后端架构风格与工程主线。
2. 定义适合 Go 生态、同时具备 Spring Boot 式工程纪律的目录组织与职责边界。
3. 冻结 HTTP、ORM、配置、错误处理、OpenAPI、迁移、异步任务和可观测性的统一规范。
4. 为后续实施计划、重构拆解、验收标准和跨仓协作提供权威设计事实源。

# 非目标

1. 本文档不描述当前代码已经如何实现。
2. 本文档不直接拆分具体实施任务和排期。
3. 本文档不将系统拆分为微服务。
4. 本文档不采用重型的全展开 Clean Architecture 目录套娃。
5. 本文档不引入后端之外的 Web 页面职责。

# 架构设计原则

## 1. 务实工程化分层

系统采用务实工程化分层，而不是重型理论分层。分层的目标是控制依赖方向、职责边界和协作复杂度，而不是追求目录层级本身。

## 2. 模块化单体

系统以单体方式部署与演进，但在代码组织上按照业务能力模块化拆分，为未来拆分运行面或扩展能力预留边界。

## 3. 统一正式主线

每一项核心工程能力只允许有一条正式主线，包括 HTTP、ORM、配置、契约、迁移、日志和任务执行。禁止多套并行长期共存。

## 4. 工程约束优先于目录形式

长期可维护性主要来自统一错误模型、统一契约、统一配置、统一迁移、统一上下文和统一测试门禁，而不是来自过深的目录层级。

## 5. 边界清晰优于抽象过度

模块之间通过稳定接口协作；只有在复杂度真实增长时才引入更细分的子目录或抽象层。

# 总体架构总览

目标架构采用 `Gin + GORM + 模块化单体 + API/Worker 一体化后端系统`。

```text
+------------------------------------------------------------------+
| hotkey-server                                                    |
|                                                                  |
|  +----------------------+     +--------------------------------+ |
|  | Gin HTTP API         |     | Worker / Scheduler Runtime     | |
|  | auth monitor ...     |     | poll aggregate notify digest   | |
|  +----------------------+     +--------------------------------+ |
|                  shared services / repositories / platform       |
+------------------------------------------------------------------+
            |                     |                    |
            v                     v                    v
       PostgreSQL              Redis              External APIs
```

系统由以下运行面组成：

1. HTTP API：对 Web、Miniapp、管理端和外部集成暴露接口。
2. Worker Runtime：承载采集、聚合、通知、日报等异步任务。
3. Shared Business Core：供 API 和 Worker 共同调用的业务服务和数据访问能力。
4. Infrastructure Platform：配置、日志、数据库、缓存、队列、契约和上下文等工程基础设施。

# 技术选型冻结

1. HTTP 主框架：`Gin`
2. ORM 主框架：`GORM`
3. 配置主线：统一配置模块
4. 契约主线：`OpenAPI`
5. Schema 主线：`db/migrations`
6. 缓存与任务辅助：`Redis`
7. 日志主线：结构化日志
8. 任务执行主线：统一 Worker / Queue / Scheduler 机制

选型原则如下：

1. 选择 Go 社区成熟、团队易理解、文档和生态完善的方案。
2. 不为假设中的未来复杂度提前引入过重基础设施。
3. 框架只服务业务，不反向主导业务结构。

# 目录结构设计

推荐目录骨架如下：

```text
cmd/
  hotkey/
    main.go

internal/
  app/                  # 应用装配、生命周期、优雅退出
  config/               # 配置定义、加载、校验
  platform/
    http/               # Gin、middleware、response、recover、errors
    db/                 # GORM、事务、连接池、分页、基础数据库支持
    cache/              # Redis client 与缓存支撑
    queue/              # 异步任务与队列支撑
    logging/            # 结构化日志
    runtime/            # request context、trace、request metadata
    openapi/            # OpenAPI 契约维护与导出
    x/                  # 外部平台客户端适配
  auth/
  monitor/
  content/
  topic/
  trend/
  notify/
  digest/
  ops/

db/
  migrations/

docs/
  openapi.json
```

目录原则如下：

1. `cmd` 只承载程序入口。
2. `app` 只负责装配与生命周期。
3. `platform` 只负责技术基础设施。
4. 业务能力直接按模块组织在 `internal/*` 下。
5. 模块默认轻量，复杂后再局部细分。
6. 不采用全局 `controller/service/repository/model` 大平铺。
7. 不一开始将每个业务模块都拆成四层深目录。

# 业务模块设计

建议先冻结以下核心业务模块：

1. `auth`
   - 用户身份、登录、授权、用户偏好
2. `monitor`
   - 关键词监控任务、启停、配置、运行边界
3. `content`
   - 内容采集结果、标准化、命中、原始素材归档
4. `topic`
   - 热点主题聚合、主题实体与事实沉淀
5. `trend`
   - 趋势窗口、快照、查询、升温与回落判断
6. `notify`
   - 通知事件、投递渠道、发送记录
7. `digest`
   - 日报、摘要、导出、知识沉淀
8. `ops`
   - 后台运行状态、人工重跑、健康信息、审计接口

模块内部推荐先采用轻量组织：

```text
internal/monitor/
  model.go
  enums.go
  service.go
  repository.go
  handler.go
  dto.go
  errors.go
```

当模块复杂度明显上升时，再局部升级为子目录结构，而不是在项目启动阶段统一做深拆分。

# 分层职责边界

## Handler

职责：

1. 路由绑定
2. 参数解析与校验
3. 读取鉴权和请求上下文
4. 调用 service
5. 输出统一响应

禁止：

1. 直接数据库读写
2. 编写核心业务规则
3. 自行拼接非统一错误响应

## Service

职责：

1. 业务用例编排
2. 业务校验与规则执行
3. 协调多个 repository、外部客户端和任务能力
4. 定义事务边界
5. 产生领域结果和应用结果

禁止：

1. 直接依赖 `gin.Context`
2. 暴露 `*gorm.DB` 到外层
3. 直接读取环境变量

## Repository

职责：

1. 提供面向业务语义的数据访问接口
2. 封装 GORM 查询细节
3. 隔离表结构细节和复杂查询实现
4. 承载事务内持久化操作

禁止：

1. 编写业务流程判断
2. 处理 HTTP 请求与响应
3. 直接承载外部平台访问

## Platform

职责：

1. 提供 HTTP、DB、Cache、Queue、Logging、Runtime、OpenAPI 等基础设施
2. 统一框架接入方式
3. 为上层业务提供一致的工程能力

# 统一错误处理设计

本节只冻结总体目标与边界，错误模型、统一响应、请求上下文、全局状态、枚举与配置等横切规范的详细设计以下游文档 [`010-工程化横切规范设计.md`](010-工程化横切规范设计.md) 为准。

Go 中不采用 Java 式异常体系，而采用统一错误模型。

建议建立以下能力：

1. `AppError`
2. `ErrorCode`
3. `RecoverMiddleware`
4. 统一错误响应器
5. 错误码注册表

错误分层建议如下：

1. 业务错误：如任务不存在、状态不允许、配额不足
2. 应用错误：如创建监控任务失败、发布日报失败
3. 基础设施错误：如数据库失败、缓存失败、第三方 API 失败

统一错误响应格式建议为：

```json
{
  "code": "MONITOR_NOT_FOUND",
  "message": "monitor not found",
  "request_id": "req_xxx"
}
```

要求如下：

1. 对外错误码必须稳定。
2. 原始内部错误链保留在日志中。
3. 不将 GORM、SQL 或外部平台底层错误直接返回客户端。
4. panic 必须被全局 recover 并转换为统一错误响应。

# 统一响应与请求上下文设计

本节只定义必须具备的能力范围；响应包络、上下文注入字段、错误码注册表和日志字段等执行细节以下游文档 [`010-工程化横切规范设计.md`](010-工程化横切规范设计.md) 为准。

系统应建立统一响应规范与请求级上下文能力。

统一成功响应：

1. 成功结果格式一致。
2. 分页响应格式一致。
3. 所有响应可带 `request_id`。

请求上下文必须统一注入以下信息：

1. `request_id`
2. `trace_id`
3. `user_id`（如已登录）
4. `module`
5. `operator`（如适用）

这些上下文信息应在 handler、service、日志和审计记录中贯穿，但不通过业务全局变量传递。

# 全局状态设计

本节定义允许和禁止的状态边界；应用级状态容器、请求级上下文和跨模块共享常量的归位规则以下游文档 [`010-工程化横切规范设计.md`](010-工程化横切规范设计.md) 为准。

系统允许的“全局级别状态”仅限于应用基础设施层：

1. 配置对象
2. logger
3. db / redis / queue client
4. 应用装配容器
5. 请求级上下文载体

明确禁止：

1. 业务可变全局变量
2. 跨模块共享写入的全局 map
3. 隐式业务单例
4. 到处可读写的运行时状态对象

原则是：允许应用级受控状态，不允许业务级无边界全局状态。

# 枚举与常量规范

本节定义总体原则；枚举落位、命名规则、`IsValid()` 约束和 OpenAPI 枚举映射细节以下游文档 [`010-工程化横切规范设计.md`](010-工程化横切规范设计.md) 为准。

为避免魔法字符串散落系统，应采用强类型常量定义关键状态与协议值。

适用对象包括：

1. 监控任务状态
2. 通知渠道
3. 内容来源
4. 趋势周期
5. 任务执行状态
6. 用户状态

推荐形式：

```go
type MonitorStatus string

const (
    MonitorStatusActive MonitorStatus = "active"
    MonitorStatusPaused MonitorStatus = "paused"
)
```

要求：

1. 枚举归属于对应业务模块。
2. 跨模块共享协议常量放在统一公共位置。
3. 必要时提供 `String()`、`IsValid()`、序列化映射能力。
4. OpenAPI 中应同步反映枚举约束。

# 配置体系设计

本节定义统一配置主线；环境变量分层、配置对象拆分、启动期校验和敏感字段规范以下游文档 [`010-工程化横切规范设计.md`](010-工程化横切规范设计.md) 为准。

配置由统一模块集中定义、加载和校验。

规则如下：

1. 环境变量为正式主线。
2. `.env` 仅用于本地开发辅助。
3. 配置加载在启动时一次性完成。
4. 缺失关键配置时应用必须直接启动失败。
5. 业务代码不得直接调用 `os.Getenv`。

配置域至少包括：

1. app
2. db
3. redis
4. jwt/auth
5. x-platform
6. queue
7. observability

# 数据库与迁移设计

数据库设计遵循以下原则：

1. `GORM` 是唯一 ORM 主线。
2. `db/migrations` 是唯一 schema 事实源。
3. `AutoMigrate` 不作为生产正式主线。
4. repository 隔离数据表结构细节。

查询策略如下：

1. 常规 CRUD 使用 GORM 标准能力。
2. 聚合、统计、趋势等复杂查询允许在 repository 内部使用 Raw / builder。
3. 原生 SQL 只允许收敛在 repository 内部。
4. handler、service、worker 不允许散落 SQL。

# OpenAPI 契约设计

本节定义契约必须存在且为唯一事实源；契约维护方式、DTO 归位、鉴权方案表达和生成落盘机制以下游文档 [`011-接口契约与任务运行设计.md`](011-接口契约与任务运行设计.md) 为准。

OpenAPI 是跨仓契约事实源，供 `hotkey-web` 与 `hotkey-miniapp` 生成客户端。

要求如下：

1. 服务端维护唯一正式 OpenAPI 产物。
2. 接口变更必须同步更新契约。
3. Web 和 Miniapp 不手写后端 API 类型。
4. 契约中的 DTO、错误码、鉴权方案必须稳定一致。
5. 契约维护方式必须与 Gin 路由实现保持同源或强绑定。

# 异步任务与 Worker 设计

本节定义 Worker 是正式运行面；调度主线、单实例约束、幂等保护、任务编排边界和重试策略以下游文档 [`011-接口契约与任务运行设计.md`](011-接口契约与任务运行设计.md) 为准。

异步任务是正式运行面，不是附属脚本能力。

Worker 负责：

1. 采集任务
2. 聚合任务
3. 趋势计算
4. 通知投递
5. 日报生成
6. 人工重跑任务

执行原则：

1. Job 只负责任务编排和调度入口。
2. 任务核心业务逻辑调用 service。
3. job 不直接堆积 SQL、HTTP client 和业务规则。
4. 任务必须具备失败处理、重试和可审计能力。

# 日志与可观测性设计

系统默认采用结构化日志。

最低要求包括：

1. request logging
2. error logging
3. panic logging
4. health check
5. 慢请求/慢查询监控基础
6. worker 执行日志

统一日志字段建议包括：

1. `request_id`
2. `trace_id`
3. `user_id`
4. `module`
5. `action`
6. `error`
7. `duration_ms`

# 测试策略与验收门禁

测试按层次组织：

1. unit tests
   - service、规则、工具函数
2. integration tests
   - repository、database、外部适配器
3. API tests
   - handler、router、错误响应、契约一致性
4. smoke tests
   - 启动、健康检查、关键链路

验收门禁建议为：

1. 配置缺失能正确失败
2. 统一错误响应生效
3. 统一上下文与 request id 生效
4. OpenAPI 契约可用
5. 关键业务模块具备基础单测与集成验证
6. worker 关键任务链路可验证

# 明确禁止事项

1. handler 直接操作数据库
2. handler 编写核心业务规则
3. service 直接暴露或传递 `*gorm.DB`
4. 业务层直接读取环境变量
5. 魔法字符串散落各模块
6. 业务可变全局变量到处共享
7. GORM/SQL/第三方底层错误直接返回客户端
8. 将 OpenAPI、迁移、配置、日志做成多套并行主线

# 验收门禁

1. 本文档完整覆盖架构风格、目录结构、职责边界、基础设施规范与禁止事项。
2. 本文档可以直接作为后续计划拆解和验收标准的上游依据。
3. 文档中的工程主线没有相互冲突或多头事实源。
4. 文档不依赖当前实现细节也能成立。

# 风险与边界

1. 若后续实施阶段不配套清晰计划，设计容易停留在原则层。
2. 若团队在实现中回到“哪里方便写哪里”的习惯，架构边界会很快失效。
3. 若后续业务规模显著增长，部分轻量模块需要局部升级拆分。
