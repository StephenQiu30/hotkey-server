# AI 任务、证据与模型运行设计

## 1. 目标

AI 为确定性流程提供受控的语义判断和内容生成，但不成为采集、去重、热度或最终事实裁决者。所有生产 AI 运行必须结构化、可重现、可引用、可降级且可计算成本。

## 2. 任务目录

| task_type | 输入 | 输出 | 失败降级 |
|---|---|---|---|
| `term_expansion` | Monitor意图和已确认词项 | 翻译、别名和候选相关词 | 仅使用用户词项 |
| `relevance_review` | Content片段、Monitor意图和规则分 | 决策、分数和原因码 | 保留规则结果或人工复核 |
| `cluster_review` | Content与有限候选Event特征 | 候选ID、决策和理由 | 使用确定性阈值 |
| `event_summary` | Event结构和证据集 | 双语标题、摘要和时间线 | 基于代表Content生成简化摘要 |
| `entity_claim_extraction` | 有效证据片段 | 实体、角色、主张、立场和引用 | 不产生新主张 |
| `report_draft` | 冻结事件条目 | 报告摘要和分节草稿 | 生成结构化事件列表 |
| `knowledge_proposal` | 当前修订、新Event和证据 | 变更提案 | 延迟处理，不改写Vault |
| `embedding` | 版本化文本 | 1024维向量 | 词法与实体链路继续 |

新任务必须先在本目录、JSON Schema 和数据约束中注册，不允许运行时接受任意任务名称或用户 Prompt。

## 3. 统一运行包

```text
task_type, target_type, target_id
model_profile_id, model_profile_version, provider, model_name, model_version
prompt_version, input_schema_version, schema_version, parameters_version
input_hash, evidence_set_hash
reuse_key, attempt, max_attempts, retry_after, repair_attempted, lease_expires_at
budget_day, reserved_cost, cost, timeout, error_code
```

Provider 只接收经过清洗的最小必要输入。MinIO对象键、本地绝对路径、密钥、用户密码、完整原始响应和无关个人信息不进入 Prompt。

PLAN-008 的完整 Schema 与 Application 仅允许 `embedding` 与 `term_expansion`。它使用静态 `*-input-v1`/`*-output-v1` JSON Schema 和同版本 task instruction，`additionalProperties:false`，并限制字段长度/数组大小；输入先校验，输出失败最多按字段级校验错误修复一次。repair input 只包含第一次结构化 output 与有限 field-path violations。该计划不保存 AI request/response MinIO payload，`ai_runs` 对象键始终为 NULL，也不保存 Provider 原始错误或完整 Prompt。

## 4. 版本化 JSON Schema

- 每个任务有独立的输入和输出 Schema，作为代码内静态资源。
- `schema_version` 只增不改；破坏性字段变化创建新版本。
- 未知字段、非法枚举、超长文本和无效引用导致校验失败。
- 允许一次基于校验错误的结构化修复；修复仍属于同一 `ai_run`。

## 5. 证据引用

每个证据片段分配运行内ID，例如 `E01`。摘要句、时间线节点、实体角色和主张必须引用一个或多个已提供ID。服务端必须验证：

1. 引用ID存在于当次输入。
2. Content 仍为 `active` 且未超过来源保留边界。
3. `evidence_locator` 可定位到传入片段。
4. 短摘录不超过长度和版权边界。
5. 主张状态由多来源规则确定，不直接接受模型的“已证实”标签。

引用校验失败时，该条结论不进入业务事实。

## 6. 幂等复用与失效

```text
reuse_key = sha256(
  task_type + target_type + target_id + model_profile_id + model_profile_version + model_version
  + prompt_version + input_schema_version + schema_version + parameters_version
  + input_hash + evidence_set_hash
)
```

只复用 `succeeded` 且证据仍有效的运行。所有 claim、retry、settle、cancel 与 reclaim 事务先取得 `ai-budget:<profile>:<UTC-day>`，再取得 `ai-run:<reuse_key>` advisory lock：成功记录直接复用，queued/running/validating/retry_wait 记录返回 in-flight，其他情况才创建新的 queued 记录；Provider 调用在事务提交后进行。证据删除或撤回、事件合并拆分、人工锁变更、Prompt/Schema/模型配置切换都会使结果失效。失效只产生新运行，不改写历史。

## 7. 模型配置与选择

`ai_model_profiles` 按任务配置 Provider、模型、凭据引用、超时、最大尝试、回退优先级、单次预算和日预算。首批 profile 只允许 `openai` 和可选 `onnx`：OpenAI 的凭据引用固定为 write-only `env:OPENAI_API_KEY`，ONNX 为 NULL；它只在 Create 请求接受，PATCH 携带它为 `70000`。不接受 endpoint、任意环境变量或自由 parameters。OpenAI 的 `model_name` 是 SDK request/response 精确相等的 provider model ID，`model_version` 仅为本地语义/reuse 元数据，ID 不符为 `70000`。每个 profile 必有正数 max_cost，daily_budget 为空即无每日 cap 但保留 ledger；非空时必须不小于 max_cost。任务、Provider、模型名/版本、凭据与 embedding 维度不可变，语义改变必须 archive 后新建 profile。选择顺序为：任务匹配 -> 未删除/已启用 -> 凭据或 build 可用 -> 预算可用 -> 回退优先级。第三方 SDK 类型不穿透 Infrastructure。

## 8. 运行状态与重试

```text
queued -> running -> validating -> succeeded
   |         |           |
   +-------> retry_wait -> running
             +---------> failed
queued/running -> cancelled
```

- 预算不足、配置错误、非法输入和证据缺失不重试。
- 超时、429 和临时 5xx 在任务上限内按 `min(2^(attempt-1),4)` 秒退避；稳定错误码分别是 70001 模型不可用、70002 预算耗尽/超额封账、70003 限流、70004 临时 Provider、70005 timeout、70006 输出无效、70007 in-flight、70008 向量无效、70009 lease expired。
- Schema 失败修复一次，再失败进入规则降级或人工复核。
- 每次调用保存耗时、Token、估算成本和脱敏错误类型。

## 9. 预算与负载保护

预算在调用前按 profile+UTC-date ledger 预留、调用后结算。reserve/settle/release 使用同一个 `ai-budget:<profile-id>:<day>` 后 `ai-run:<reuse_key>` advisory lock：reserve 要求 `overage_blocked=false` 且（daily cap 为空或 `settled + reserved + max <= daily`）；失败/取消/lease recovery 释放 run 的 reserved，成功将真实 cost 加 settled。实际费用超过 reserve 时记录真实 cost、标记 `failed/70002`、加入 settled 并将同 profile+UTC-day `overage_blocked=true`，使 NULL daily 或仍有余额也不能继续 reserve；只有新 UTC-day ledger 自动清除封账，绝不伪造较小金额。每个 in-flight run 有 lease：create/queued→running/running→validating/retry_wait→running 刷新为 `now+timeout+30s`，进入 retry_wait 刷新为 `retry_after+timeout+30s`。worker-only reclaimer 以 budget 后 run 的锁序每 30 秒只回收已过期 run、标记 `failed/70009` 并释放 reservation，不重新外呼；有效 retry_wait 不得被回收。超额时依次保留：活跃事件摘要、聚类和相关性边界复核、报告草稿、扩词和知识提案。降级不影响采集、证据查询、规则匹配、基础聚类和热度计算。

## 10. 人工修正和写入边界

- AI 输出先写入 `ai_runs.structured_result`，再由 Application 校验并应用。
- 人工锁定字段、成员和主张状态不得被AI覆盖。
- 长期 Topic、关系、合并和归档只生成变更提案。
- 报告发布前校验结构、证据和冻结条目，不发布原始模型文本。

## 11. 可观测与保留

指标包含各任务成功率、降级率、Schema和引用失败率、P50/P95耗时、Token、成本和重试数；标签只允许 task、provider、outcome 和 stable error code。日志不打印 Prompt、完整证据、密钥、凭据引用或 Provider 原始响应。需要保留请求/响应的后续任务必须另行定义受控 MinIO 对象；PLAN-008 仅保留哈希、版本、受限结构化结果、用量和安全错误码。

## 12. 测试与验收门禁

- Provider契约测试覆盖成功、超时、429、5xx、非法JSON和部分结果。
- 每个任务有固定评测集，验证结构、引用、幻觉率和降级行为。
- 所有任务有静态、版本化的输入输出 Schema。
- 摘要、时间线、实体、主张和报告核心结论能定位到有效Content。
- 相同复用键不重复调用，证据失效后不复用旧结果。
- LLM 全部不可用时，热点主链路仍可采集、匹配、聚类、计算热度和查询证据。
- 人工锁和长期知识审核边界通过集成测试。
