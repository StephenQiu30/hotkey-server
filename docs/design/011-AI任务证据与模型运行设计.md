# AI 任务、证据与模型运行设计

## 1. 目标

AI 为确定性流程提供受控的语义判断和内容生成，但不成为采集、去重、热度或最终事实裁决者。所有生产 AI 运行必须结构化、可重现、可引用、可降级且可计算成本。

## 2. 任务目录

| task_type | 输入 | 输出 | 失败降级 |
|---|---|---|---|
| `term_expansion` | Monitor意图和已确认词项 | 翻译、别名和候选相关词 | 仅使用用户词项 |
| `relevance_review` | Content片段、Monitor意图和规则分 | 决策、分数和原因码 | 保留规则结果或人工复核 |
| `cluster_review` | Content与有限候选Event特征 | 候选ID、决策和理由 | 使用确定性阈值 |
| `event_summary` | Event结构和证据集 | 双语标题、摘要和时间线 | 基于代表Content生成简化摘要 |
| `entity_claim_extraction` | 有效证据片段 | 实体、角色、主张、立场和引用 | 不产生新主张 |
| `report_draft` | 冻结事件条目 | 报告摘要和分节草稿 | 生成结构化事件列表 |
| `knowledge_proposal` | 当前修订、新Event和证据 | 变更提案 | 延迟处理，不改写Vault |
| `embedding` | 版本化文本 | 1024维向量 | 词法与实体链路继续 |

新任务必须先在本目录、JSON Schema 和数据约束中注册，不允许运行时接受任意任务名称或用户 Prompt。

## 3. 统一运行包

```text
task_type, target_type, target_id
model_profile_id, provider, model_name
prompt_version, schema_version, parameters_version
input_hash, evidence_set_hash
request_id, trace_id, requested_at
budget_limit, timeout
```

Provider 只接收经过清洗的最小必要输入。MinIO对象键、本地绝对路径、密钥、用户密码、完整原始响应和无关个人信息不进入 Prompt。

## 4. 版本化 JSON Schema

- 每个任务有独立的输入和输出 Schema，作为代码内静态资源。
- `schema_version` 只增不改；破坏性字段变化创建新版本。
- 未知字段、非法枚举、超长文本和无效引用导致校验失败。
- 允许一次基于校验错误的结构化修复；修复仍属于同一 `ai_run`。

## 5. 证据引用

每个证据片段分配运行内ID，例如 `E01`。摘要句、时间线节点、实体角色和主张必须引用一个或多个已提供ID。服务端必须验证：

1. 引用ID存在于当次输入。
2. Content 仍为 `active` 且未超过来源保留边界。
3. `evidence_locator` 可定位到传入片段。
4. 短摘录不超过长度和版权边界。
5. 主张状态由多来源规则确定，不直接接受模型的“已证实”标签。

引用校验失败时，该条结论不进入业务事实。

## 6. 幂等复用与失效

```text
reuse_key = sha256(
  task_type + target_id + model_profile_version
  + prompt_version + schema_version + parameters_version
  + input_hash + evidence_set_hash
)
```

只复用 `succeeded` 且证据仍有效的运行。证据删除或撤回、事件合并拆分、人工锁变更、Prompt/Schema/模型配置切换都会使结果失效。失效只产生新运行，不改写历史。

## 7. 模型配置与选择

`ai_model_profiles` 按任务配置 Provider、模型、凭据引用、超时、最大尝试、参数、回退优先级、单次预算和日预算。选择顺序为：任务匹配 -> 已启用 -> 预算可用 -> 回退优先级。第三方SDK类型不穿透 Infrastructure。

## 8. 运行状态与重试

```text
queued -> running -> validating -> succeeded
   |         |           |
   +-------> retry_wait -> running
             +---------> failed
queued/running -> cancelled
```

- 预算不足、配置错误、非法输入和证据缺失不重试。
- 超时、429 和临时 5xx 在任务上限内指数退避。
- Schema 失败修复一次，再失败进入规则降级或人工复核。
- 每次调用保存耗时、Token、估算成本和脱敏错误类型。

## 9. 预算与负载保护

预算在调用前预留、调用后结算。超额时依次保留：活跃事件摘要、聚类和相关性边界复核、报告草稿、扩词和知识提案。降级不影响采集、证据查询、规则匹配、基础聚类和热度计算。

## 10. 人工修正和写入边界

- AI 输出先写入 `ai_runs.structured_result`，再由 Application 校验并应用。
- 人工锁定字段、成员和主张状态不得被AI覆盖。
- 长期 Topic、关系、合并和归档只生成变更提案。
- 报告发布前校验结构、证据和冻结条目，不发布原始模型文本。

## 11. 可观测与保留

指标包含各任务成功率、降级率、Schema和引用失败率、P50/P95耗时、Token、成本和重试数。日志不打印 Prompt、完整证据或Provider原始响应。需保留的请求和响应写入受控 MinIO 对象，数据库仅保留哈希、版本、结构化结果和对象键。

## 12. 测试与验收门禁

- Provider契约测试覆盖成功、超时、429、5xx、非法JSON和部分结果。
- 每个任务有固定评测集，验证结构、引用、幻觉率和降级行为。
- 所有任务有静态、版本化的输入输出 Schema。
- 摘要、时间线、实体、主张和报告核心结论能定位到有效Content。
- 相同复用键不重复调用，证据失效后不复用旧结果。
- LLM 全部不可用时，热点主链路仍可采集、匹配、聚类、计算热度和查询证据。
- 人工锁和长期知识审核边界通过集成测试。
