# 如何执行并归档 HotKey Server 任务计划

本文定义 HotKey Server 的单计划完整闭环。执行者一次只激活一个依赖已满足的 Plan，并依次完成审核、测试先行实施、接口验证、独立复核、Acceptance 和原路径归档。

## 治理目标与范围

本设计解决 001–017 任务链的执行一致性。它让每个状态都能由仓库文件、测试命令和 Git 提交复核。

本设计覆盖：

- `hotkey-server` 的 Design、PRD、Plan、Acceptance 和 Operations
- PLAN-006 至 PLAN-017 的依赖顺序、任务拆解和验收归档
- Schema、记录模型、OpenAPI、运行时接口和质量门禁的同步要求
- 文档元数据、依赖和验收引用的自动校验

本设计不覆盖：

- `hotkey-web` 或 `hotkey-miniapp` 的实现和验收
- 多个 Plan 并行修改同一 Schema 或模块
- 用聊天记录、临时日志或未提交补丁替代长期证据
- 因归档而移动 PRD、Plan 或 Acceptance 文件

## 权威事实与文档职责

每类工件只回答一个问题。下游文档不能修改上游事实。

| 工件 | 权威问题 | 完成证据 |
|---|---|---|
| Design | 系统和治理必须遵守什么设计 | `status: accepted` |
| PRD | 单项任务必须交付什么 | 范围、非目标、依赖和可验证验收 |
| Plan | 修改哪些文件以及如何执行 | 文件级任务、红绿命令和提交边界 |
| Code facts | 当前系统实际实现了什么 | 代码、`db/schema.sql`、记录模型和 OpenAPI |
| Acceptance | 哪个提交通过了哪些验收 | 环境、命令、结果、提交和残余风险 |
| Operations | 如何重复运行、发布、恢复和回滚 | 可执行命令和适用边界 |

`db/schema.sql` 是数据库唯一事实源。生成并校验后的 `docs/openapi/swagger.json` 是公共接口唯一事实源。

## 单计划状态机

任务使用文档成熟度、执行进度和 Plan 审核三个独立维度。任何自动化不得把三个维度合并成一个字段。

```text
候选
  -> Design accepted
  -> PRD accepted + backlog
  -> Plan review + in_review
  -> Plan accepted + approved + ready
  -> PRD/Plan in_progress
  -> 实施与验证完成
  -> Acceptance accepted
  -> PRD/Plan archived + done
  -> 解锁下一依赖 Plan
```

状态转换必须满足以下规则：

1. Design 未 accepted 时，相关 PRD 和 Plan 只能处于 draft 或 review。
2. PRD 未 accepted 时，对应 Plan 不能进入 approved 或 ready。
3. 前置 Plan 未 done 时，下游 Plan 不能进入 ready。
4. Plan 的范围、文件、接口、依赖或验收发生实质变化时，`review_status` 重置为 `pending`。
5. 只有一个 Plan 可以处于 `in_progress`，避免并行修改完整 Schema 和公共契约。
6. Acceptance 的 `result` 不是 `accepted` 时，PRD 和 Plan 不能进入 done 或 archived。
7. 归档必须同时更新 PRD、Plan 和两个索引，不能只修改正文 frontmatter。
8. `superseded` 必须关联替代任务。被替代任务不能继续实施，也不能伪装为完成归档。

## 开工定义

Plan 只有同时满足以下条件才能从 backlog 转为 ready：

- 相关 Design 全部为 accepted
- PRD 为 accepted，范围和非目标没有待定项
- Plan 为 accepted，且独立 Reviewer 将 `review_status` 批准为 approved
- 所有 `depends_on` Plan 均为 done
- 执行文件清单与当前仓库一致，不把已有文件错误标记为创建
- 每个任务都定义输入、输出、准确路径和独立测试边界
- 正常、失败、权限、并发、幂等、删除、恢复和降级边界均已覆盖或标记为不适用
- 每项验收都有红灯、绿灯或明确的替代证据
- 验收命令在当前 Makefile 或脚本中真实存在
- Git 基线、测试服务和可丢弃数据环境已经确认

Reviewer 必须独立于主要 Plan 编写者。审核记录保存在 Pull Request、ticket、Workpad 或对应 Acceptance 的复核章节中。

## 任务拆解标准

一个 Task 交付一个可独立拒绝或批准的结果。设置、配置、实现和文档步骤应归入需要它们的 Task，不能拆成没有业务结果的独立 Task。

每个 Task 必须包含：

1. **目标**：一句话描述可观察结果
2. **文件**：逐项列出 Create、Modify、Delete 和 Test 的准确路径
3. **接口**：列出输入、输出、函数、数据结构或 HTTP 契约
4. **红灯**：先运行能证明能力缺失的测试或结构校验
5. **绿灯**：实现最少代码并让目标测试通过
6. **重构**：在行为不变的前提下整理命名、边界和重复
7. **回归**：运行模块测试、受影响集成测试和仓库门禁
8. **提交**：只提交当前 Task 文件，并使用中文提交主题

Task 内的步骤使用 Markdown checkbox。每一步只执行一个动作，并写明命令、预期失败或通过信号。

## 单计划执行闭环

每个 Plan 按以下顺序执行，不跨越验收或归档步骤：

1. **预检**：确认分支、工作树、基线提交、依赖状态和可丢弃测试环境
2. **激活**：将 PRD 和 Plan 更新为 `in_progress`，同步索引
3. **测试先行**：按 Task 提交可复现红灯，再提交最小实现
4. **事实源同步**：同步完整 Schema、记录模型、OpenAPI、配置和架构校验
5. **范围验证**：运行模块、集成、故障、竞态和安全测试
6. **仓库验证**：运行 `make ci`、适用的 smoke、`make clean` 和 Git 检查
7. **接口验证**：对新增或变化接口执行真实 HTTP 请求，核对状态码、业务码、Result、权限和错误边界
8. **独立复核**：审查该 Plan 的全部提交、测试、Schema、OpenAPI 和残余风险
9. **长期验收**：创建同编号 Acceptance，并记录准确提交和可重复命令
10. **原路径归档**：将 PRD 和 Plan 更新为 `archived + done`，同步索引并提交

任一步失败时，任务保持 `in_progress` 或进入 `blocked`。执行者先修复根因并重跑受影响门禁，不能跳过失败项完成归档。

## 验收证据标准

Acceptance 必须让未参与实施的 Reviewer 在新工作区复核结论。摘要不能替代可执行命令。

每份 Acceptance 必须记录：

- 被验收的 Design、PRD 和 Plan
- 操作系统、Go、PostgreSQL、Redis 和外部依赖版本
- Fixture、可丢弃数据库和秘密隔离方式
- 至少一个完整 40 位实施提交 SHA
- 红灯命令、实际失败信号、恢复方式和绿灯结果
- 模块测试、集成测试、竞态测试、故障测试和 `make ci` 结果
- Schema fingerprint、OpenAPI 漂移结果和运行时 HTTP 证据
- 未执行项、原因、责任边界和残余风险
- 独立复核范围、发现、修复提交和最终结论
- `result: accepted`、`rejected` 或 `accepted_with_risk`

只有 `result: accepted` 允许自动归档。`accepted_with_risk` 需要先补充修复 Plan 或由维护者明确改变本规则，不能自动视为全部完成。

## 接口验收标准

公共接口的验收同时检查静态契约和运行时行为。仅通过 Handler 单元测试或 OpenAPI 生成不能证明接口可用。

每个新增或变化接口至少验证：

- 正常请求的 HTTP 状态码、业务 `code: 0` 和 `data`
- 参数、状态冲突、无权限和不存在资源的 HTTP 状态与稳定业务码
- `X-Request-ID` 只出现在响应头和日志，不进入 JSON
- 未认证、过期会话、撤销会话和角色不足的适用行为
- 重复请求、并发更新和幂等键的适用行为
- 日志、Result 和 OpenAPI 不泄露 SQL、密钥、堆栈或第三方原始错误
- OpenAPI operation、请求模型、响应模型、安全方案和实际路由一致

PLAN-017 在全部前置任务归档后执行完整 Server 端到端接口矩阵、故障恢复和容量基线。前置 Plan 的局部验收不能替代最终全链验收。

## 自动治理校验

仓库需要新增单一文档治理校验，并接入本地与 GitHub Actions 的 `make ci`。校验必须输出准确文件和错误原因。

校验至少覆盖：

- 正式文档 frontmatter 必填字段和值域
- 同目录 `doc_no` 和 `canonical_path` 唯一性
- Markdown 链接、inputs、downstream 和 depends_on 指向有效文件或任务
- PRD 与 Plan 的一一映射和依赖无环
- Design、PRD、Plan、Acceptance 组合状态符合本设计
- done Plan 存在同编号且 `result: accepted` 的 Acceptance
- Acceptance 的 40 位提交 SHA 能在完整 Git 历史中解析
- README 索引状态与正文 frontmatter 一致
- 计划中的 Make 目标和脚本在仓库中存在
- archived 文档仍位于 canonical path，没有被移动到第二套归档目录

GitHub Actions 必须检出验证提交所需的历史。校验器必须解析 URL 编码路径，不能把 `%20` 当作真实文件名。

## 已知治理缺口

当前审计确认以下问题必须在 PLAN-006 开工前修复：

1. 仓库没有文档 frontmatter、状态、依赖和 Acceptance 一致性校验。
2. PLAN-013 和 PLAN-017 引用了当前不存在的 `make smoke`。
3. PLAN-017 计划创建的 Operations `doc_no: 001` 与现有 CI 手册冲突。
4. PLAN-017 把部分现有 Operations 文件误标为创建。
5. 后续 Plan 没有明确区分复用目标 Schema、补充约束和新增结构。
6. 006–017 的执行步骤仍是大粒度编号，缺少可勾选 Task、接口和逐步红绿证据。

治理实施完成后，逐项审核并修订 006–017。只把 PLAN-006 转为 ready，后续 Plan 保持 backlog，直到其依赖和即时审核门禁全部满足。

## 归档与恢复规则

归档通过状态表达，不移动文件。原路径保留所有引用、Git 历史和后续依赖。

完成归档时必须：

1. 确认 Acceptance 为 accepted，且准确实施提交可解析
2. 将 PRD 和 Plan 的 `status` 改为 `archived`
3. 将 PRD 和 Plan 的 `execution_status` 改为 `done`
4. 保持 Plan 的 `review_status: approved`
5. 同步 PRD 和 Plan README 索引
6. 运行文档治理校验、`git diff --check` 和适用的仓库回归
7. 使用独立归档提交保存状态转换

若归档后发现回归，保留原 Acceptance 和归档提交。新建修复任务或前向 Plan，不能改写历史证据，也不能恢复旧双轨实现。
