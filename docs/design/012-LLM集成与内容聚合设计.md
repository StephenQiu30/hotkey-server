# LLM 集成与内容聚合设计规范

> **日期：** 2026-07-09
> **状态：** 已实现
> **关联：** [014-日志系统设计规范](014-日志系统设计规范.md)

---

## 1. 目标

在 hotkey-server 中集成 LLM 能力，通过设计模式实现 Provider 可切换、业务场景可组合的内容聚合管道。首个落地场景为**内容聚合**（摘要 + 主题标签 + 日报生成）。

## 2. 非目标

- 不支持实时流式输出（Streaming），当前仅做 request/response
- 不引入 Python sidecar 或外部 LLM 编排服务
- 不接入 langchaingo 的 Agent / Memory / Tool 等高级特性
- 不修改现有的内容采集、存储、计算数据流

## 3. 技术选型

| 组件 | 选择 | 理由 |
|------|------|------|
| LLM 框架 | langchaingo | 社区主力 Go 方案，多 provider 支持 |
| 模型 | DeepSeek v4-flash（OpenAI-compatible） | 高性价比，langchaingo 原生支持 |
| 设计模式 | Strategy + Factory + Adapter + Pipeline | 各自解决一个明确的关注点 |
| DI | go.uber.org/fx | 项目已有，保持一致 |

## 4. 架构

```
业务调用方（topic.Service / hotevent.Service / report.Service）
    │
    ▼
internal/llm/
├── Service（接口）—— Summarize() / LabelTopics() / GenerateDigest()
├── Chain（Pipeline）
│     FetchPost → Summarize → Label → Compile
├── Provider（Strategy 接口）—— Chat(ctx, prompt) → (string, error)
│     ├── "openai" → langchaingo openai.New()
│     └── "anthropic" → langchaingo anthropic.New()
├── Factory —— NewProvider(cfg) → Provider
└── Adapter —— langchainAdapter 封装 langchaingo 差异
```

## 5. 设计模式

### 5.1 Strategy — Provider 接口

```go
type Provider interface {
    Chat(ctx context.Context, prompt string, opts ...Option) (string, error)
}
```

不同模型通过同一个接口调用，切换 Provider 即切换后端模型。

### 5.2 Factory — NewProvider

```go
func NewProvider(cfg config.LLMConfig) (Provider, error)
```

- `cfg.Provider = "openai"` → langchaingo openai.New (可指向 DeepSeek API)
- `cfg.Provider = "anthropic"` → langchaingo anthropic.New
- 新增 provider 只加一个 case branch

### 5.3 Adapter — langchainAdapter

将 langchaingo `llms.Model` 的差异 API 统一到 `Provider` 接口，解耦业务代码与 langchaingo 的具体类型。

### 5.4 Pipeline — Chain

```go
type DigestInput struct {
    Posts  []PostRaw
    Topics []TopicRaw
}

type DigestOutput struct {
    Summary      string
    Labels       []string
    PostDigests  []PostDigest
    GeneratedAt  time.Time
}
```

Pipeline 步骤：
1. SummarizePosts(ctx, posts) → 每篇文章的摘要
2. LabelPosts(ctx, posts, summaries) → 每篇文章的主题标签
3. CompileDigest(ctx, posts, summaries, labels, topics) → 合并为结构化日报

## 6. 配置

```go
type LLMConfig struct {
    Provider    string   // "openai" | "anthropic" | "ollama"
    APIKey      string
    BaseURL     string   // DeepSeek: https://api.deepseek.com
    Model       string   // "deepseek-v4-flash"
    MaxTokens   int
    Temperature float64
}
```

环境变量：`LLM_PROVIDER`、`LLM_API_KEY`、`LLM_BASE_URL`、`LLM_MODEL`、`LLM_MAX_TOKENS`、`LLM_TEMPERATURE`。

DeepSeek 走 OpenAI-compatible API，因此 `LLM_PROVIDER=openai` + 改 `BASE_URL` 和 `MODEL` 即可。

## 7. 错误处理

- `ProviderError`：底层 API 调用失败（超时、认证、限流）
- `ContentTooLongError`：输入超过模型上下文窗口
- `EmptyResponseError`：LLM 返回空内容

调用方统一捕获 `ProviderError`，日志记录后降级：摘要为空则跳过，标签为空返回空列表，日报失败不阻塞整体流程。

## 9. 设计决策

| 决策 | 理由 |
|------|------|
| langchaingo 而非封装 OpenAI SDK 直接调用 | Provider 切换零改动，减少 provider-specific 胶水代码 |
| Adapter 隔一层而非直接暴露 langchaingo 类型 | 业务代码不直接依赖 langchaingo，方便未来替换 |
| Pipeline 为显式的步骤编排而非隐式链 | 每一步可测试、可跳过、可观测 |
