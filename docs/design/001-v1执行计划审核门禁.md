# 001 V1 执行计划审核门禁

## 目的

本计划文档用于审核后续 Linear ticket 的执行计划，确保 HotKey V1 按 SDD、TDD 和 E2E 方式交付。任何功能 ticket 如果不能通过本门禁，不应进入实现或交付。

## 适用范围

1. server 功能、OpenAPI、采集器、异步任务、AI 摘要、Git 同步、邮件、数据删除。
2. web 工作台功能。
3. miniapp 只读功能。
4. 跨端回归和上线验收。

## Ticket 开工门禁

每个 ticket 开工前必须具备：

1. 明确引用一个主 PRD，必要时引用依赖 PRD。
2. 明确用户故事、范围、非范围和验收标准。
3. 明确会修改的契约：OpenAPI、数据模型、状态机、任务队列、对象存储或 UI 路径。
4. 明确测试分层：unit、integration、contract、E2E、manual smoke。
5. 明确外部依赖：平台授权、MinIO、邮件 provider、Git provider、AI provider。
6. 明确不可用降级状态和用户可见提示。

缺少以上任一项时，只能做事实收集和 design，不允许实现。

## Ticket 切片原则

1. 一个 ticket 只能绑定一个主 PRD。
2. 如果一个 PRD 涉及多个独立状态机，应拆成多个 ticket。
3. 如果一个 ticket 同时需要 server、web 和 miniapp 改动，必须先完成 server 契约 ticket，再开端侧 ticket。
4. 平台适配器 ticket 必须先依赖 `004-平台适配器通用契约.md`，再依赖具体平台 PRD。
5. 外部平台 live smoke 不应阻塞 simulator E2E 的实现，但上线前必须作为发布门禁处理。
6. 任何 ticket 如果无法在一个 PR 中完整证明 TDD 和 E2E，应拆小。

## SDD 审核清单

1. 是否先写或更新 design 文档。
2. 是否先定义状态模型和失败状态。
3. 是否先定义 OpenAPI 字段、错误结构和状态码。
4. 是否先定义幂等键、重试策略和任务取消策略。
5. 是否先定义数据留存、权限和删除影响。
6. 是否先定义 provider simulator 或测试替身契约。
7. 是否说明 web/miniapp 客户端如何从 OpenAPI 生成类型。

审核结论只能是 `pass`、`revise_required` 或 `blocked_by_external_dependency`。

## TDD 审核清单

每个功能 ticket 的计划必须列出红绿顺序：

1. `test:` 先提交失败测试或可执行验收。
2. `impl:` 或 `feat:` 只实现使测试通过的最小功能。
3. `refactor:` 只在测试绿色后进行。
4. `docs:` 或 `chore:` 用于文档、生成客户端或配置更新。

测试必须覆盖：

1. happy path。
2. 授权失效。
3. 限流或 provider 失败。
4. 重试和幂等。
5. 数据删除或撤权影响。
6. 权限隔离。

只写 happy path 测试不得通过审核。

## E2E 审核清单

每个核心链路必须至少有一个自动化 E2E 或等价可执行验收：

1. server E2E：API、数据库、队列、MinIO、provider simulator。
2. web E2E：Playwright 覆盖配置、状态展示和用户操作路径。
3. miniapp E2E：只读查看、事件详情和状态展示路径。
4. cross-system E2E：采集到事件、事件到 Obsidian、Obsidian 到邮件、Web/miniapp 查看。

外部平台测试采用两层门禁：

1. PR 合并门禁：provider simulator E2E 必须稳定通过。
2. 上线门禁：允许范围内的测试账号或沙箱 smoke 必须通过，或明确标记平台能力为受限并在产品中展示。

## E2E 环境门禁

后续实现计划必须准备以下可重复环境：

1. 本地或 CI 数据库。
2. 本地或 CI MinIO。
3. 队列/异步任务测试运行器。
4. provider simulator，覆盖正常、限流、授权失效、schema 变化和空结果。
5. Git 测试仓库或本地 Git remote fixture。
6. 邮件测试 provider 或本地 SMTP sink。
7. AI provider mock 和受控 fixture。
8. Web Playwright 测试用户和测试数据种子。
9. miniapp 构建和只读页面自动化验收入口。

## 验收证据格式

每个 ticket 的 workpad 必须保留：

1. `PRD`：引用文件路径。
2. `Design`：引用设计文件路径或说明不需要 design 的理由。
3. `Acceptance Criteria`：可勾选验收项。
4. `Test-first Evidence`：失败测试名称、失败命令和失败原因。
5. `Green Evidence`：通过命令和关键输出摘要。
6. `E2E Evidence`：E2E 命令、截图/录屏/日志路径或 CI 链接。
7. `Residual Risk`：未验证内容和原因。

## 审核输出模板

执行计划审核必须输出以下结论：

```markdown
## Plan Review

Result: pass | revise_required | blocked_by_external_dependency

PRD:
- 主 PRD:
- 依赖 PRD:

SDD:
- 契约:
- 状态模型:
- 数据与权限:
- 失败状态:

TDD:
- Red 测试:
- Green 实现:
- 回归测试:

E2E:
- server:
- web:
- miniapp:
- cross-system:

Risks:
- 残余风险:
- 上线前 smoke:
```

## 不可交付条件

存在以下任一情况，不允许标记 Done：

1. 没有先写测试就实现功能，且没有在 workpad 说明合理例外。
2. OpenAPI 变更后没有重新生成 web/miniapp 客户端。
3. 外部平台失败时用户不可见。
4. 删除、撤权或最小化留存没有可执行验收。
5. E2E 只覆盖 mock UI，不覆盖 server 实际链路。
6. 测试依赖真实外部平台且没有 simulator 替代。
7. 工作区包含本 ticket 无关文件变更。

## 阶段编排

默认执行顺序：

1. server 契约与核心域。
2. server 平台适配器。
3. server 事件、AI、Obsidian、邮件和数据治理。
4. web 工作台。
5. miniapp 只读查看。
6. 跨端回归。

每一阶段都必须先完成对应 PRD 的 SDD/TDD/E2E 审核，再进入下一阶段。
