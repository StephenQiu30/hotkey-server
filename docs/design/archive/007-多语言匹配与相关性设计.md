# 多语言匹配与相关性设计

匹配模块判断标准化内容是否与用户监控意图相关。它组合核心词、审核后的扩展词、实体和多语言向量，并优先排除明显冲突，避免只靠语义相似度产生主题漂移。

## 匹配主链路

```text
Content -> 排除词与冲突实体
-> 候选Monitor召回 -> 关键词评分
-> 实体评分 -> 向量评分 -> 偏好评分
-> 阈值判定 -> 保存匹配解释
```

排除条件在昂贵计算之前执行。匹配失败不能影响Content保存，也不能阻塞其他Monitor。

## 监控意图

Monitor 的检索意图由以下内容组成：

- 用户核心词
- 用户批准的中英文翻译、别名和相关词
- 人物、组织、地点、产品等实体
- 排除词和冲突实体
- 用户填写的目标、地区和语言偏好

系统把上述内容生成稳定 `query_text`，再生成 `query_embedding`。`query_text`、模型名、模型版本和生成时间必须保存，词项变化后重新生成。

AI 生成的词项默认低于用户核心词权重，未经用户批准不能进入正式查询和匹配。

## 候选召回

禁止对全部内容和全部 Monitor 做笛卡尔积比较。候选来自两条路径：

1. 来源查询路径：内容由某个`monitor_source`查询获得后，优先召回该查询关联的Monitor
2. 宽泛趋势路径：按核心词、实体倒排和 Monitor 向量近邻召回有限候选

同一候选只计算一次。每条内容设置最大候选数，并记录召回路径，避免50个监控主题产生不必要的全量比较。

## 词法规范化

英文匹配执行大小写折叠、Unicode 规范化和词边界判断；中文匹配执行 Unicode 规范化、简繁映射和短语匹配。专有名词、产品型号和缩写保留原始形式作为别名。

词项权重建议：

| 词项 | 相对权重 |
|---|---|
| 用户核心词 | 1.00 |
| 用户确认实体 | 1.00 |
| 翻译和别名 | 0.85 |
| AI 相关词 | 0.60 |
| 排除词 | 直接抑制或拒绝 |

同一含义的多次命中不能无限累加。

## Embedding 模型

V1默认使用[`BAAI/bge-m3`](https://huggingface.co/BAAI/bge-m3)的1024维多语言向量，并通过[pgvector](https://github.com/pgvector/pgvector)的`halfvec(1024)`保存和检索。Provider可替换模型，但活动模型必须保持相同维度和向量空间契约。

选择原因：

- 同一模型覆盖中文和英文语义空间
- 1024维契约与内容、Monitor、事件和主题向量表一致
- PostgreSQL 可同时完成业务过滤和近邻检索
- 无需引入独立向量数据库

内容向量基于原语言标题、正文摘要和规范实体生成，不先翻译全文。Monitor 向量基于结构化 `query_text` 生成。

每条向量分别保存到`content_embeddings`和`monitor_embeddings`，至少包含：

```text
target_id, model_profile_id, model_profile_version, model_version
input_hash, query_text(仅Monitor), embedding, active, created_at
```

写库前必须校验向量恰为 1024 个有限值；Repository 在同一个 `ai-embedding:<target-type>:<target-id>:<profile-id>` advisory-lock transaction 内停用该 target/profile 的旧 active 行，再插入新 active 行。每张表以 `(target_id, model_profile_id) WHERE active` 保证至多一个 active 行。

不同 profile 或 model version 的向量不能直接比较。读取必须同时过滤 `active=true`、`model_profile_id` 与 `model_version`，以 `embedding <=> query::halfvec` 排序；HNSW 仅服务 active 集，验收使用 `EXPLAIN (COSTS OFF)` 验证相应 `*_active_hnsw_idx`。模型升级创建新 profile，先重算活跃 Monitor 与当前事件窗口内容、再在后续匹配计划切换读取 profile；历史冷数据按需重算，绝不把新旧空间混排。

## 混合相关性评分

最终相关性统一为 `0-100`：

```text
relevance_score =
  semantic_score * 0.35
  + keyword_score * 0.25
  + entity_score * 0.20
  + title_score * 0.10
  + preference_score * 0.10
```

各分项含义：

- `semantic_score`：内容和 Monitor 向量余弦相似度归一化结果
- `keyword_score`：核心词、翻译、别名和相关词的加权命中
- `entity_score`：关键实体重合度及置信度
- `title_score`：高价值词项是否出现在标题或视频名
- `preference_score`：用户语言、地区和内容类型偏好

默认阈值：

| 分数 | 结果 |
|---|---|
| `>= 75` | 强相关，进入主要事件候选 |
| `60-74.99` | 一般相关，可进入聚类但降低排序权重 |
| `< 60` | 不建立正式匹配 |

Monitor 可以提高阈值，但不能低于平台安全下限。阈值调整属于配置，不改变评分公式版本。

## 排除与语义漂移控制

出现高置信排除词时直接拒绝匹配。排除词存在歧义时，结合词边界、上下文和实体判断，避免误伤。

以下情况降低或拒绝相关性：

- 核心词相同但实体指向不同对象
- 产品名相同但版本、公司或行业冲突
- 语义相似但事件时间明显不在监控范围
- 只命中低权重 AI 相关词，没有核心词或实体支持

高向量相似度不能覆盖明确的实体冲突。

## 匹配解释

`monitor_matches`保存总分、各分项、决策、原因码、召回路径和`algorithm_version`。`explanation JSONB`只保存解释所需结构：

```json
{
  "matched_terms": ["OpenAI", "GPT"],
  "matched_entities": ["OpenAI"],
  "excluded_terms": [],
  "recall_path": "source_query",
  "semantic_score": 84.2,
  "keyword_score": 90.0
}
```

解释必须能回答“为什么命中”和“为什么被拒绝”，但不保存模型思维过程或敏感提示词。

## 降级策略

- Embedding 生成失败：使用关键词、实体、标题和偏好评分，并标记降级
- pgvector查询失败：只使用monitor_source关联、关键词和实体倒排候选
- LLM 不可用：不影响匹配主链路
- 单条内容异常：记录失败原因后继续其他内容

降级结果使用独立 `scoring_version`，避免与完整评分直接混排而不知情。

## 用户反馈

用户可以将内容或事件标记为 `relevant` 或 `irrelevant`。MVP 只用于：

- 离线评测准确率
- 建议增加排除词、别名或实体
- 调整该用户的轻量偏好分

系统不做在线模型训练，不让单次反馈直接修改全局模型或其他用户结果。

## 质量评测

建立覆盖中英文、跨语言同事件、同名异义和排除词的固定标注集。每次修改模型、权重、阈值或词法规则都执行回归评测。

MVP 门槛：

- Top 20 内容相关性精确率 `P@20 >= 80%`
- 明确排除词的误命中率低于 `2%`
- 评分结果可以由保存的版本和证据重现
- Embedding 不可用时核心词和实体匹配仍可完成

## 验收门禁

- 相同内容不会与所有 Monitor 做全量比较
- 中英文内容可以匹配到同一监控意图
- 用户核心词权重高于 AI 扩展词
- 排除词和冲突实体可以抑制高语义相似度误报
- 每个匹配保存分项、证据和评分版本
- 模型升级不会混用不同向量空间
- 降级链路不依赖 LLM
- 固定评测集达到 `P@20 >= 80%`
