# 数据来源、查询规划与采集设计

采集层把用户审核后的词项转换为来源查询表达式，并通过合规接口获取标准化来源数据。V1只接入官方API、RSS、Atom或明确授权的Feed；不为50个监控主题提前建立独立QueryPlan业务模型。P0 配置发布、来源执行语义、canonical 签名输入和纯配置预览以 Design-014 为准；本设计只定义后续 Connector 与采集运行如何消费已发布配置。

## 来源范围

MVP 来源分为三档：

| 接入级别 | 来源 | 条件 |
|---|---|---|
| 必选 | 中英文 RSS、Atom、授权新闻 Feed | 来源允许聚合，并遵守其保留和展示条款 |
| 必选 | 新闻聚合或新闻数据库官方 API | 已取得有效密钥和对应使用权限 |
| 必选 | YouTube Data API | 使用官方搜索接口，保存视频元数据和指标 |
| 条件接入 | X API | 账户具备可用搜索权限和足够配额 |
| 条件接入 | Reddit Data API | 商业用途、内容展示和数据保留已获许可 |
| 条件接入 | 微博等境内平台 | 仅使用官方开放接口或授权数据服务 |

不得绕过登录、验证码、反爬机制或平台访问限制。条件不满足的来源必须保持禁用，不能用非授权网页抓取补位。

来源能力和条款以官方文档为准：

- [X API 搜索说明](https://docs.x.com/x-api/posts/search/introduction)
- [YouTube Data API search.list](https://developers.google.com/youtube/v3/docs/search/list)
- [Reddit Data API Terms](https://redditinc.com/policies/data-api-terms)

## Connector 边界

每个来源实现一个小型接口：

```go
type Connector interface {
	Validate(ctx context.Context, connection SourceConnection) error
	Fetch(ctx context.Context, request FetchRequest) (FetchResult, error)
	Health(ctx context.Context, connection SourceConnection) HealthResult
}
```

`FetchRequest` 是共享来源查询请求，不携带单个 Monitor target。它包含：

```text
collection_run_id, source_connection_id, query_signature, query
language, region, window_start, window_end
request_cursor, etag, last_modified, limit
```

`request_cursor` 和条件请求字段属于这一次 shared run 的来源请求状态；单个 immutable MonitorSource 只作为 `collection_run_targets` 的消费 target，不能改变同一外部请求的身份或 River 幂等键。

`FetchResult` 包含：

```text
items, next_cursor, etag, last_modified
has_more, rate_limit, server_time, diagnostics
```

Connector 只负责：

- 来源认证、请求和响应解析
- 分页、增量游标和配额读取
- 将来源数据转换为 `SourceItem`
- 把来源错误分类为认证、限流、服务端、解析或永久错误

Connector 不负责匹配、聚类、热度和 AI 摘要。项目使用显式 `map[sourceCode]Connector` 注册实现，不建设动态插件、反射注册或通用采集 DSL。

## 统一来源项

`SourceItem` 是 Connector 与内容模块之间的内部契约：

```text
source_code, external_id, content_type
title, body, language, url, author
published_at, observed_at
metrics, source_entities, raw_payload(transient)
```

`external_id` 必须来自来源稳定标识。来源没有稳定 ID 时，使用规范化 URL 或来源规定字段生成可重复哈希，并记录生成方式。

## 词项到来源查询

对 active Monitor 的 immutable published config，系统执行：

1. 根据核心词生成中英文翻译、别名、实体名和相关词
2. 用户审核新增词项并配置排除词
3. 对每个 published、enabled `monitor_sources` 按来源语法、有效语言/地区、query override 和窗口生成查询表达式
4. 对表达式生成稳定 `query_signature`
5. 相同调度轮次内按 `source_connection_id + query_signature` 合并外部请求，结果分别写入关联监控

稳定签名的完整 canonical JSON、算法版本、排序、query override、语言/地区归一化和来源安全语义由 Design-014 第 6 节固定。签名不包含 Monitor ID，因此同一轮相同查询只调用一次外部接口；数据库以 immutable `monitor_sources` 保存独立配置事实，以 `collection_runs` 保存共享 source/query/window 运行，并以 `collection_run_targets` 和 `source_checkpoints` 保存每个配置目标的处理与增量状态。不同来源语法在 Connector 内显式转换，不引入通用查询语言或 QueryPlan 表。

## 周期调度

Cron按监控配置检查到期来源，默认周期不低于5分钟。小时级监控的单轮目标是45分钟内完成采集，为标准化、匹配、聚类和摘要保留至少15分钟。

调度步骤：

1. 读取 active Monitor 的已发布 enabled `monitor_sources`、enabled/non-deleted SourceConnection 与到期 `source_checkpoints.next_poll_at`，作为候选 targets。
2. 生成查询表达式和稳定签名，按 `source_connection_id + query_signature + window_start + window_end` 分组并加入稳定抖动。
3. 创建或复用该唯一 `collection_run`，为每个候选写入 immutable `collection_run_target`，并以同一四元组提交唯一 River `collect_source` 任务。
4. Worker 按来源并发上限、剩余配额和 shared-run 请求状态只执行一次 Connector Fetch；它不得以 `monitor_source_id` 分拆外部请求。
5. 保存 shared `collection_runs`、逐项结果、请求游标、数量和错误分类，再把结果分发给各 target。
6. 每个 target 的来源页解析完成且全部 `SourceItem` 已幂等写入 `collection_run_items` 后才推进该 target 的 `source_checkpoint`；PLAN-007 以这些持久化项异步创建 Content、证据和下游任务，不得重新调用来源或改变 fetch checkpoint。某一 target 的捕获失败不推进它的 checkpoint，也不使已成功 target 倒退。

并发必须有界。每个来源设置独立并发上限、超时和每轮最大页数，不能为每个Monitor创建无限goroutine。进程重启后根据`collection_runs`和`source_checkpoints`继续执行。

## 幂等与增量

采集捕获以 `(source_connection_id, external_id)` 为第一层幂等键。PLAN-006 将版本化、脱敏的 `SourceItem` envelope 写入 `collection_run_items.captured_item`，并以 `raw_payload_disposition` 记录原始第三方响应是明确丢弃还是仅允许从 capture envelope 生成证据；原始响应字节不写 PostgreSQL。PLAN-007 从 captured item 幂等创建 Content，并按来源策略将安全的证据 envelope 写入 MinIO，不重新调用 Connector。

每次运行保存：

- 查询窗口和实际完成时间
- 请求游标与返回游标
- 请求数、成功数、跳过数和失败数
- 消耗配额和来源返回的剩余配额
- 最后成功检查点

重试不能推进未成功保存的游标。来源只支持时间窗口时，相邻运行保留小范围重叠，并依赖幂等键消除重复。

## 错误、限流与熔断

| 错误类型 | 处理 |
|---|---|
| `401/403` | 标记凭据或来源不可用，停止自动重试并告警 |
| `429` | 尊重 `Retry-After`，暂停该来源并延后计划 |
| `5xx` 或网络错误 | 指数退避加抖动，单轮最多重试 3 次 |
| 单条解析错误 | 跳过该条，记录计数和脱敏样本 |
| 协议或字段永久变化 | 打开来源熔断，等待人工修复 |

连续失败超过来源阈值后打开熔断；健康检查成功或冷却时间结束后半开探测。一个来源失败不能阻塞其他来源。

配额不足时按以下顺序保留任务：

1. 新建或近期有用户活动的 Monitor
2. 处于快速增长事件关联的监控来源
3. 普通周期查询
4. 宽泛趋势补充查询

## 凭据与合规

`source_connections`保存来源类型、能力、状态、协议引用和非秘密配置，不保存明文密钥。密钥从环境变量或本地秘密引用读取，不得进入数据库、日志、错误响应或原始响应归档。

每个来源必须能配置：

- 是否允许正文保存和二次展示
- 原始响应、标准化内容和指标的保留期限
- 是否要求来源署名、回链或删除同步
- 允许的语言、地区、查询方式和速率

来源协议优先于系统默认策略。

## 验收门禁

- 相同来源和查询签名在同一调度轮次只产生一次外部请求
- 同一来源内容重试采集不会重复入库
- 每个来源的并发、超时、分页和配额可独立控制
- 认证失败、限流和服务端失败具有不同处理路径
- 单一来源失败不影响其他来源完成本轮采集
- 条件未满足的来源保持禁用，不存在绕过式抓取
- 日志、数据库和 API 响应不包含来源密钥
- 单轮采集能在小时流水线预算内结束或安全续跑
