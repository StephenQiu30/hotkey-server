# 内容标准化、去重与证据设计

内容模块把不同来源的新闻、帖子和视频转换为统一`Content`，持久化到`contents`。它只折叠同一内容的重复副本，不会把同一事件的独立报道误删，从而保留事件热度和可信度所需证据。

## 处理流程

PLAN-007 从 Source-owned `collection_run_items` 中读取已持久化的 `CapturedItem`，不重新请求来源。每个捕获项按固定顺序处理：

```text
字段校验 -> 文本清洗 -> URL规范化 -> 语言识别
-> 指标映射 -> 三层去重 -> 内容入库 -> 绑定 capture
```

任何一步失败都必须产生明确状态或错误计数，不能静默丢弃。

## 标准化规则

### 字段校验

- `source_code`、`external_id`、`content_type` 和 `observed_at` 必填
- 标题和正文至少存在一个
- 时间无法解析时使用 `observed_at` 作为排序兜底，并标记时间质量
- URL 必须是允许的 `http` 或 `https` 地址
- 来源指标只能映射到已知指标名，未知字段保留在受控扩展区域

### 文本和 URL

- Unicode 统一为 NFC
- 删除脚本、样式、跟踪像素和不可见控制字符
- HTML 转换为纯文本，保留合理段落
- 合并连续空白，不改变有语义的标点和大小写原文
- URL 统一协议、主机大小写和常见尾部形式
- 删除已知追踪参数，保留会改变内容含义的参数
- 标题、正文和规范化 URL 分别生成稳定哈希

系统保留清洗后的原文，不强制把英文翻译成中文。跨语言关联由匹配和聚类模块完成。

### 语言和实体

语言识别输出 BCP 47 主语言和置信度。短文本置信度不足时，可以结合来源区域和 Monitor 查询语言判断，但必须保存判断来源。

标准化前的来源实体不属于 PLAN-007 输入契约；实体、别名与事件关系由后续事件治理任务建立，不得在本计划创建空实体模型。

Hacker News `comment` 在 Content 归一化时映射为受控 `post`；`article/post/video/podcast/bulletin` 之外的来源类型写入 capture 诊断但不创建 Content。来源指标使用可空值：`null` 表示来源未提供或能力未知，只有来源明确返回 `0` 才持久化零。

## 内容状态

`content_status` 只有以下状态：

```text
active | invalid | duplicate | deleted | expired
```

- `active`：可参与匹配、聚类、热度和摘要
- `invalid`：缺少必要内容或违反解析约束
- `duplicate`：确认是另一Content的重复副本
- `deleted`：来源撤回或要求删除
- `expired`：超过当前来源允许的保留期限

只有 `active` 内容可成为新的事件证据。

## 三层去重

### 第一层：来源幂等

使用`(source_connection_id, external_id)`唯一键。再次采集相同来源项时更新采集时间和指标，不新建记录。

### 第二层：精确重复

按以下顺序判断：

1. 同一规范化 URL
2. 标题和正文的标准化哈希相同
3. 来源提供的转载或引用原文 ID 相同

精确重复记录设置`duplicate_of_id`。优先保留信息更完整、发布时间更早且来源标识更稳定的记录。

### 第三层：近似重复

仅在相近发布时间窗口内生成候选，并组合：

- 标题和允许持久化正文的确定性 token 指纹相似度
- 规范化 URL 和作者关系

PLAN-007 只使用固定文本规则 `near_text-v1`，且候选必须属于同一 `source_connection_id`：发布时间相差不超过 24 小时、规范化标题 token 完全一致、两个非空正文 token 的 Jaccard 相似度至少 0.98，才可标为近似重复。跨来源只允许第一、二层的 exact URL/hash 判定，绝不因近似文本自动折叠；因此即使两个媒体的 100 个 token 相同、其中一篇另有两个新事实，也必须各自保留为 active。Embedding、实体重合和跨语言语义判断留给 PLAN-008/009，不能在 PLAN-007 自动折叠。每个 duplicate 都保存目标 Content、`dedupe_reason` 和 `dedupe_version`。

## 重复内容与事件证据

以下情况不是重复：

- 不同媒体对同一发布会的独立报道
- 中英文媒体对同一事件的独立采访或分析
- 原帖、引用帖和包含新信息的跟进帖
- 同一主题在不同时间发生的独立事件

这些内容应分别保留，并通过 `event_contents` 关联到同一 Event。来源数量和独立证据数量是事件可信度及热度的输入。

## 代表内容选择

事件展示需要代表内容，但代表选择不能删除其他证据。优先级为：

1. 更接近原始发布者或一手信源
2. 来源可信度更高
3. 标题、正文、时间和作者更完整
4. 发布时间更早
5. 互动指标更充分

事件可以分别选择中文和英文代表内容。

## 指标处理

`contents`使用`view_count`、`like_count`、`comment_count`和`share_count`保存统一当前指标，来源特有指标保存到`extra_metrics`。只有仍活跃或正在增长的内容写入`content_metric_snapshots`，避免为全部历史内容无限保存快照。当前值和每一条快照的四个统一指标都采用相同语义：`null` 表示来源未知，非 null 必须非负，显式 `0` 才表示零；快照不得把未知值压成 `0`。

统一指标包括：

```text
views, likes, comments, shares, reposts, favorites
```

来源不支持的指标保持缺失，不写成零。热度模块根据来源能力处理缺失值，不能直接比较不同平台的原始数值。

PLAN-006 留下的 CapturedItem v1 使用非指针计数，历史 `0` 无法证明是来源显式返回还是字段缺失；PLAN-007 读取 v1 时必须把 `0` 保守映射为未知，正值保持原值，绝不重抓来源。新 capture 使用 v2，以 nullable 指标编码缺失与显式 `0`，并由 schema/record/normalizer/snapshot 保持到查询 DTO。

## 可追溯证据

每个Content至少保存：

- 来源 ID、外部 ID 和原始链接
- 作者、发布时间、首次与最近观察时间
- 对应采集运行、monitor_source和query_signature
- 已持久化 CapturedItem 的 `payload_hash` 与标准化内容哈希
- 标准化与去重版本；Embedding 版本仅在其拥有模块实施后才出现
- 命中的 Monitor、Event 和后续摘要引用关系

完整正文、截图和附件只在来源协议允许时写入 MinIO，`content_assets` 保存对象键、MIME、大小和 SHA-256。PLAN-006 已永久丢弃原始 Connector response，PLAN-007 绝不重新请求来源补取它；本计划仅将 `allow_body_storage=true` 时已持久化的捕获正文以 `text` asset 写入 MinIO。正文未获许可时只保存 title、URL、哈希、指标与来源链接，不创建文本对象。日志只记录哈希、标识和计数，不打印完整正文或原始响应。

## 来源删除同步

收到删除、撤回或合规清除要求后：

1. 把Content标记为`deleted`，立即停止读取
2. 删除或标记删除 MinIO 正文，并按保留策略删除 Content 指标快照
3. 将 MinIO asset 标为 `delete_pending`，成功删除后转为 `deleted`
4. 本计划的 Content 查询和新的下游输入一律只读取 active Content；Embedding、匹配、Event、热度和摘要等未实施模块必须由各自拥有模块在后续 Plan 中实现关系清理与重算，ingestion 不得读写其表
5. 按来源协议保留最小 tombstone，或彻底清除证据对象

删除流程必须幂等。同一来源的重复删除通知不能产生错误，也不能让已删除内容恢复。

## 实现边界

内容模块按实际职责拆分标准化、去重和 Repository 文件。不同内容类型使用显式函数和小型策略，不引入通用规则引擎、内容工作流引擎或来源专属内容表。

## 验收门禁

- 所有Connector输出都能转换为统一Content
- 同一`(source_connection_id, external_id)`重试采集只更新一条记录
- 精确重复和近似重复均能输出目标记录及判断版本
- 同一事件的独立报道不会被当作重复内容删除
- 缺失来源指标不会被错误写为零
- 任何展示、匹配、事件和摘要都能回溯到原始链接
- 来源删除后，内容不再参与任何计算或展示
- 删除重试幂等；未实施下游模块在各自 Plan 中完成其证据清理与重算
