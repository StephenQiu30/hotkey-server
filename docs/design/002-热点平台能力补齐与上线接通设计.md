# 002 热点平台能力补齐与上线接通设计

## 背景

`hotkey-server` 已经具备配置、采集、标准化、去重、embedding、热点聚类、评分、日报、邮件、调度和管理员接口等多处单点能力，但当前状态更接近“样机能力 + 文档和 schema 完整”，还没有形成一条真实可运行、可恢复、可观测的上线主链路。

项目当前的核心问题不是缺少概念，而是以下几类断点仍然存在：

1. 配置和运行时依赖没有统一接线。
2. Redis 队列和 job 状态没有成为所有后台任务的统一底座。
3. X 采集、标准化、去重、embedding、聚类、评分之间仍然存在运行时断层。
4. 日报生成、邮件发送和管理员观测还没有收口到同一套任务和状态模型上。

因此，本设计不重做平台，而是以“补齐已有能力并接通主链路”为目标，约束后续任务拆分和 Linear issue 生成。

## 目标

1. 建立从配置加载到热点输出、日报发送和管理员观测的统一主链路。
2. 优先复用现有 `internal/config`、`internal/app`、`internal/worker`、`internal/scheduler`、`internal/service/*`、`internal/platform/*`、`internal/repository/postgres/*` 能力。
3. 将任务拆成可独立执行、可做红绿测试、可写入 Linear 的单 issue。
4. 保持 `server-only` 边界，不将本阶段工作扩散到 `hotkey-web` 或 `hotkey-miniapp`。

## 非目标

1. 不重写整体架构，不拆分为多服务。
2. 不引入新的编排系统替换现有 Symphony + Linear 工作流。
3. 不在本阶段建设 Web 管理页面或复杂前端体验。
4. 不把 issue 草稿当作长期项目文档存入 `docs/`；`docs/` 只保留设计和计划。

## 范围与模块拆分

本阶段上线接通范围拆成 3 个层次：

### A. 配置与运行时层

- env 文件与统一配置加载
- runtime/bootstrap 依赖注入
- Redis job/runtime 接线

### B. 数据与热点主链路层

- X 来源采集接线
- 标准化、过滤、去重
- embedding 与热点聚类
- 评分与热点读取 API

### C. 输出与运维层

- 日报生成与邮件发送
- 管理员观测、依赖状态与任务状态读取

## 主链路

本阶段唯一事实主链路定义为：

`配置加载 -> runtime 装配 -> Redis/job 调度 -> X 采集 -> 标准化/过滤/去重 -> embedding -> 热点聚类 -> 评分 -> 热点读取 -> 日报生成 -> 邮件发送 -> 管理员观测`

约束：

1. 后续 issue 必须沿着这条链路补齐，不新增并行事实链。
2. 告警、日报和管理员观测都消费已有热点/任务结果，不重复发明状态。
3. 所有后台步骤都应最终收口到 job/run 状态可追踪的模型。

## 配置与依赖

本阶段必须统一接线的配置包括：

- app/http
- PostgreSQL
- Redis
- MinIO
- SMTP
- DashScope
- X

配置规则：

1. 统一通过 `internal/config` 读取。
2. 支持默认值、env 文件和环境变量覆盖。
3. 对关键依赖执行启动前校验。
4. 对可选能力明确 `disabled` 或 `degraded`，避免静默失败。

## 文档与任务规范

`docs/` 目录只承担项目文档职责：

1. `docs/design/`：长期设计方案和架构边界。
2. `docs/plans/`：与具体任务一一对应的实现计划。

Linear issue 不是项目文档，不应长期存放在 `docs/` 中。后续由 `create-spec` skill 基于 PRD、设计和 plan 直接生成 issue 内容并写入 Linear。

任务规范：

1. 一个 Linear issue 对应一个 `docs/plans/N-能力名称实现计划.md`，编号从 13 起全局连续递增。
2. Plan 必须遵循 `docs/TEMPLATE.md` 的 frontmatter 与必含章节；Linear issue 正文不作为 docs 文件长期保留。
3. issue 内容引用 PRD、Design 和对应 Plan，并包含任务范围、TDD 验收命令与回写要求。

## 测试与验收

本阶段所有 issue 必须遵守：

1. 先写失败测试，再写最小实现。
2. 所有 Go 代码改动在交付前至少执行：

```bash
gofmt -w cmd internal
go test ./...
python3 -m unittest discover -s tests
```

3. 按任务类型追加验证：
   - HTTP/API：handler 或 contract test
   - 数据库：migration 与 repository test
   - Redis：幂等、重试、队列或降级测试
   - DashScope/SMTP：fake/mock 测试
   - 主链路：e2e 或 smoke 覆盖

## 风险与边界

1. 如果继续把 issue 草稿混入 `docs/`，会导致项目文档和任务载体混杂，后续维护成本上升。
2. 如果绕过 Redis/job 统一底座直接补业务链路，会再次形成不可恢复的并行流程。
3. 如果不先统一配置与 runtime 装配，后续每个业务任务都会重复处理 bootstrap 问题。
4. 当前设计默认单服务部署边界有效；如果未来改成多服务，需要新增新的架构设计文档而不是在本设计内硬扩展。
