# AI 热点事件监控平台需求分析

## 1. 文档地位

本文定义 HotKey 首个完整可交付版本。目标方案不以当前代码结构和未完成实现为设计依据；当前代码只作为后续迁移时需要处理的遗留系统。

本文与旧决策冲突时，以本文、`002` 和 `003` 为目标基线。目标能力在实施和验收完成前不得描述为已经实现。

## 2. 产品定位

HotKey 是本地优先、可开源自部署的 AI 热点事件监控与 Obsidian 知识库治理平台。它把多来源新闻、帖子和视频转化为可追溯事件、长期主题知识、日报和周报，并通过邮件及 RSS/Atom 交付。

产品闭环：

1. 用户创建监控主题和匹配规则。
2. 系统从合规稳定源增量采集候选内容。
3. 系统执行标准化、去重、相关性判断、事件聚类和趋势计算。
4. PostgreSQL 保存结构化事实，MinIO 保存原始证据和附件。
5. 系统把事件笔记、日报和周报写入本地 Obsidian Vault。
6. AI 对长期主题、实体关系、合并和归档只生成变更提案，由用户审核后应用。
7. 已发布报告通过邮件和 RSS/Atom 提供给订阅者。

## 3. 目标用户和容量

### 3.1 目标用户

- 个人研究者和内容创作者
- 产品经理、新媒体编辑和技术观察者
- 共享监控主题和审核知识变更的小团队

### 3.2 V1 容量基线

| 指标 | 设计上限 |
|---|---:|
| 团队成员 | 5—10 人 |
| 启用监控主题 | 50 个 |
| 候选内容 | 约 50,000 条/日 |
| 知识库 | 一个本地 Obsidian Vault |
| 原始证据存储 | 一个 MinIO 实例 |
| 周期报告 | 日报、周报 |

从 Connector 可读取内容到事件更新完成，正常来源 P95 不超过 60 分钟。该规模不需要微服务、Kafka、Elasticsearch或独立向量数据库。

## 4. 术语

| 术语 | 定义 |
|---|---|
| 监控主题 | 用户设置的检索意图、规则、来源、语言、地区、阈值和调度周期 |
| 候选内容 | 新闻、帖子、视频或 Feed Item 的统一记录 |
| 原始证据 | 来源原始响应、正文、截图、图片、附件及其校验信息 |
| 热点事件 | 多条候选内容描述的同一现实事件 |
| 长期主题 | 跨多个事件持续积累的知识主题 |
| 知识投影 | 从数据库事实生成到本地 Vault 的 Markdown 文件 |
| 变更提案 | AI 对长期知识生成的待审核差异 |

## 5. 功能需求

### 5.1 用户和权限

| 编号 | 需求 | 验收规则 |
|---|---|---|
| FR-IAM-001 | 支持本地用户和角色 | 角色固定为 `admin`、`editor`、`viewer` |
| FR-IAM-002 | 支持停用成员 | 停用后不能登录，历史审计仍保留 |
| FR-IAM-003 | 保护管理操作 | 成员、来源凭据和保留策略只允许管理员修改 |
| FR-IAM-004 | 记录人工决策 | 事件合并、知识审核、归档和发布记录操作者 |

### 5.2 监控主题

| 编号 | 需求 | 验收规则 |
|---|---|---|
| FR-MON-001 | 创建和维护监控主题 | 名称、意图、语言、地区、频率、阈值和状态可配置 |
| FR-MON-002 | 配置匹配规则 | 支持关键词、短语、排除词、域名、作者、实体和受控正则 |
| FR-MON-003 | 关联数据源 | 一个主题可关联多个来源并设置优先级和查询覆盖 |
| FR-MON-004 | 审核 AI 扩展词 | 未确认的翻译、别名和相关词不能扩大正式采集范围 |
| FR-MON-005 | 启停监控 | 停用后不产生新采集任务，历史事件继续可读 |
| FR-MON-006 | 预览规则 | 配置阶段预览只验证版本、签名、规则和来源，不读取候选内容或产生正式事件；基于最近候选内容的命中、拒绝和解释预览属于相关性匹配阶段 |

### 5.3 数据来源

V1 按以下顺序接入：

1. RSS/Atom、Hacker News。
2. GDELT、Reddit、YouTube。
3. X 官方 API。
4. 国内平台只在存在可用、稳定和合规官方接口时逐步增加。

| 编号 | 需求 | 验收规则 |
|---|---|---|
| FR-SRC-001 | 统一 Connector | 支持配置校验、增量读取、健康状态和速率信息 |
| FR-SRC-002 | 增量采集 | 按来源使用 cursor、ETag、Last-Modified 或时间高水位 |
| FR-SRC-003 | 合规接入 | 只使用官方 API、RSS/Atom 或获得授权的 Feed |
| FR-SRC-004 | 凭据保护 | 数据库只保存凭据引用，不保存明文 Token 或密钥 |
| FR-SRC-005 | 限流退避 | 识别 429 和服务端退避，不进行无界重试 |
| FR-SRC-006 | 来源追踪 | 每条内容保存来源、外部 ID、URL、发布时间和采集时间 |

禁止绕过登录、验证码、反爬或平台访问限制。每个来源必须记录授权方式、商业使用限制、保留期限和删除要求。

### 5.4 候选内容和证据

| 编号 | 需求 | 验收规则 |
|---|---|---|
| FR-ING-001 | 标准化内容 | 统一标题、摘要、正文引用、作者、URL、语言、发布时间和指标 |
| FR-ING-002 | 来源内幂等 | 同一来源和外部 ID 只能形成一条内容记录 |
| FR-ING-003 | 跨来源去重 | 保留独立来源证据，通过重复关系折叠转载和镜像 |
| FR-ING-004 | 保存原始证据 | 原始响应、正文和附件以确定性对象键写入 MinIO |
| FR-ING-005 | 校验证据 | 数据库保存对象键、MIME、大小和 SHA-256 |
| FR-ING-006 | 分阶段判断 | 依次执行规则过滤、语义召回和边界样本 LLM 复核 |
| FR-ING-007 | 隔离坏数据 | 单条内容失败不回滚整个采集批次 |

### 5.5 事件、实体和事实主张

事件卡片必须包含：

- 中英文标题和事件摘要
- 相关性分数、命中原因和模型版本
- 当前热度、热度增速和趋势状态
- 首次发现、最近活跃和峰值时间
- 新闻、社交和视频来源分布
- 代表内容、全部有效证据链接和证据更新时间
- 单一来源、多源印证或争议信息标识

| 编号 | 需求 | 验收规则 |
|---|---|---|
| FR-EVT-001 | 聚合相同事件 | 多来源相同事件共享一个稳定事件键 |
| FR-EVT-002 | 跨监控复用 | 一个事件可匹配多个监控主题 |
| FR-EVT-003 | 跨语言聚类 | 综合实体、动作、时间和多语言语义，不只比较标题 |
| FR-EVT-004 | 计算热度趋势 | 结合来源数量、内容和互动增长、时间衰减及来源权重 |
| FR-EVT-005 | 管理生命周期 | 支持发现、活跃、降温、关闭、合并和归档 |
| FR-EVT-006 | 抽取实体关系 | 保存实体类型、角色、置信度和人工确认状态 |
| FR-EVT-007 | 抽取事实主张 | 标记未验证、单源、多源印证、争议和撤回 |
| FR-EVT-008 | 保存证据立场 | 证据标记支持、反驳或仅提及 |
| FR-EVT-009 | 人工修正优先 | 人工锁定字段和关系不得被模型覆盖 |
| FR-EVT-010 | 合并和拆分 | 操作事务化，保留原事件和完整审计 |

### 5.6 AI 能力

AI 只执行：

- 双语扩词、翻译和实体辅助
- 相关性边界样本复核
- 复杂聚类边界辅助
- 事件标题、摘要和时间线生成
- 实体、关系和事实主张抽取
- 日报、周报草稿和长期知识变更提案

| 编号 | 需求 | 验收规则 |
|---|---|---|
| FR-AI-001 | 结构化输出 | 每类任务具有版本化 JSON Schema |
| FR-AI-002 | 可重现 | 保存模型、参数、提示词版本、输入哈希和 Schema 版本 |
| FR-AI-003 | 引用证据 | 摘要、主张和报告条目能定位到候选内容 |
| FR-AI-004 | 降级 | Schema 修复一次仍失败时使用规则结果或进入人工审核 |
| FR-AI-005 | 幂等复用 | 相同输入、模型和提示词版本复用成功结果 |
| FR-AI-006 | 成本控制 | 模型配置支持超时、预算和回退优先级 |

LLM 不负责采集、基础去重、热度计算或最终事实裁决。LLM 或 Embedding 不可用时，关键词、实体匹配和原始证据查询仍需工作。

### 5.7 本地 Obsidian 知识库

系统管理的 Vault 相对目录固定为：

```text
HotKey/
  Inbox/
  Events/
  Topics/
  Reports/
    Daily/
    Weekly/
  Archive/
```

| 编号 | 需求 | 验收规则 |
|---|---|---|
| FR-KB-001 | 自动事件笔记 | 已接收事件自动创建或更新 `Events` Markdown |
| FR-KB-002 | 自动报告笔记 | 已发布日报、周报写入对应目录 |
| FR-KB-003 | 审核长期知识 | 主题、实体关系、合并和归档先生成变更提案 |
| FR-KB-004 | 原子写入 | 使用同目录临时文件、刷盘和原子重命名 |
| FR-KB-005 | 路径锁 | 同一路径同一时刻只有一个写入者 |
| FR-KB-006 | 检测人工编辑 | 使用修订号、生成哈希和当前哈希识别变化 |
| FR-KB-007 | 冲突保护 | 哈希不一致时进入人工冲突处理，不静默覆盖 |
| FR-KB-008 | 历史恢复 | 每次写入形成不可变修订，恢复旧版也创建新修订 |
| FR-KB-009 | 稳定 Frontmatter | 保存文档 ID、业务 ID、修订、状态、时间和证据引用 |
| FR-KB-010 | 不依赖 Git | 运行流程不执行 Git commit、branch、push 或 pull request |

### 5.8 日报、周报和订阅

| 编号 | 需求 | 验收规则 |
|---|---|---|
| FR-RPT-001 | 生成周期报告 | 支持全局或单监控主题的日报、周报 |
| FR-RPT-002 | 冻结报告条目 | 保存发布时的事件标题、摘要、排名和入选原因 |
| FR-RPT-003 | 发布校验 | 没有证据、结构错误或空报告不能发布 |
| FR-RPT-004 | 邮件订阅 | 按用户、主题、周期和时区投递，失败可追踪和重试 |
| FR-RPT-005 | RSS/Atom | 已发布报告形成支持 ETag 和 Last-Modified 的 Feed |
| FR-RPT-006 | 私有 Feed | 数据库只保存订阅令牌哈希，轮换后旧地址失效 |
| FR-RPT-007 | 防重复投递 | 同一报告和订阅只能存在一个投递实例 |

### 5.9 运行治理

| 编号 | 需求 | 验收规则 |
|---|---|---|
| FR-OPS-001 | 运行可观察 | 能查询采集、AI、Vault、报告和投递状态 |
| FR-OPS-002 | 安全重试 | 只有临时错误重试，达到上限后进入明确失败状态 |
| FR-OPS-003 | 操作审计 | 记录操作者、动作、资源、结果和请求链路 |
| FR-OPS-004 | 数据保留 | 各数据类别具有默认保留期和引用保护 |
| FR-OPS-005 | 跨存储对账 | 识别 MinIO 孤儿对象、缺失对象和 Vault 冲突 |

## 6. 非功能需求

| 编号 | 要求 | 目标 |
|---|---|---|
| NFR-001 | 本地可运行 | 默认一个命令启动 API、调度和 Worker |
| NFR-002 | 幂等 | 采集、AI、聚类、报告、Vault 和投递都可安全重试 |
| NFR-003 | 一致性 | PostgreSQL 内事务；跨存储使用状态机、哈希和对账 |
| NFR-004 | 查询性能 | 目标容量下核心列表 P95 不高于 500 ms，不允许无索引全表扫描 |
| NFR-005 | 安全 | 密钥不明文入库、不进日志、不进入响应 |
| NFR-006 | 可测试 | 时间、ID、模型、对象存储、文件系统、邮件和任务调度可替换 |
| NFR-007 | 可维护 | 模块边界、命名、错误、事务和 Repository 风格统一 |
| NFR-008 | 可重建 | 完整 `db/schema.sql` 可在空库幂等建立目标结构，结构变更必须同步数据处理说明 |
| NFR-009 | 可追踪 | HTTP、任务、AI 和投递共享 request/trace 关联信息 |
| NFR-010 | 合规 | 来源协议、速率、保留和删除要求优先于产品默认值 |

## 7. 非目标

V1 不包含：

- 多租户 SaaS、计费和公共注册运营
- 秒级实时流处理和金融交易决策
- 无限制全网抓取或规避平台限制
- AI 自动发布完整文章或视频
- AI 未审核直接修改长期知识
- Git 作为知识库运行存储或同步机制
- Docker Compose、Kubernetes、云资源和生产部署拓扑设计
- Obsidian 插件、移动端和 Miniapp 页面
- 在线训练推荐模型
- 用户自定义通用规则引擎或工作流引擎

## 8. 关键场景

```gherkin
Scenario: 从多来源热点形成本地知识和日报
  Given 用户创建“AI Agent”监控主题并确认中英文规则
  When 系统完成一次增量采集、匹配和事件聚类
  Then 相同现实事件只生成一个事件
  And 事件保留新闻、社交和视频来源证据
  And 事件笔记写入本地 Obsidian Vault
  And 日报发布后可以通过邮件和 RSS/Atom 获取
```

```gherkin
Scenario: 人工编辑与 AI 提案冲突
  Given 用户已经人工修改长期主题 Markdown
  And AI 提案基于更早的文档修订
  When 用户尝试批准该提案
  Then 系统检测到内容哈希不一致
  And 提案进入 conflict 状态
  And 系统不覆盖人工编辑内容
```

```gherkin
Scenario: 单一来源不可用
  Given 一个来源返回限流或临时服务错误
  When 调度流水线继续执行
  Then 其他来源仍完成采集和事件更新
  And 失败来源记录退避时间和可重试状态
```

## 9. 验收标准

- 人工标注集热点结果 `Precision@20 >= 80%`
- 跨语言事件聚类 `F1 >= 75%`
- 95% 的正常可采集内容在 60 分钟内形成或更新事件
- 重跑同一采集窗口不产生重复内容、事件证据、报告或邮件投递
- 单一来源失败不阻塞其他来源
- 所有 AI 摘要、主张和报告核心结论能追溯到有效内容
- 来源删除内容后不再参与展示、热度、摘要和报告生成
- 事件笔记、日报和周报能写入本地 Vault 并通过稳定 Frontmatter 查询
- 人工修改 Vault 文件后，过期 AI 提案不能静默覆盖
- 邮件失败可追踪重试，RSS/Atom 支持条件请求和令牌轮换
- 所有业务表都有统一 Repository CRUD 测试
- 公共 API 只暴露经过授权和业务校验的操作
- 核心工作流在没有 Git、Kafka 和 Redis 的情况下可以运行

## 10. 风险与边界

| 风险 | 处理 |
|---|---|
| 平台 API 政策和配额变化 | Connector 隔离、来源健康状态、按来源降级 |
| 跨语言聚类误差 | 标注集回归、版本化阈值和人工修正锁 |
| LLM 输出不稳定 | JSON Schema、证据约束、一次修复和规则降级 |
| Vault 被其他工具并发修改 | 路径锁、内容哈希、修订和冲突队列 |
| 本地数据增长 | 分阶段过滤、保留策略、批量清理和容量阈值 |
| 内容版权与平台条款 | 按来源配置保留期，报告只引用必要短片段 |
| MinIO 发行策略变化 | 领域层依赖 S3 语义端口，保留对象存储替换能力 |
