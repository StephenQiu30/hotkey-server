# 数据库与数据生命周期设计

本文定义 PostgreSQL 目标模型。MVP 使用 18 张核心表，统一内容和事件概念，并根据数据恢复价值选择软删除或硬删除。

## 建模原则

- PostgreSQL 是唯一业务事实源
- `db/schema.sql` 是唯一结构事实源
- 不创建 `db/migrations/`
- 全平台只有一个 `events` 概念
- 不为每个数据来源创建独立内容表
- 核心查询字段使用独立列
- JSONB 只保存来源扩展数据、指标和解释
- 新增表必须对应明确查询、约束或生命周期需求

## 核心数据关系

```text
User -> Monitor
Monitor -> MonitorContentMatch -> ContentItem
Monitor -> MonitorEventMatch -> Event
Event -> EventContent -> ContentItem
```

`Event` 与具体用户无关。`MonitorEventMatch` 保存事件在某个用户监控任务中的相关性、排序和反馈。

## 核心表

| 领域 | 表 | 职责 |
|---|---|---|
| 用户 | `users` | 用户身份、状态和角色 |
| 用户 | `auth_sessions` | 登录会话和撤销状态 |
| 监控 | `monitors` | 用户监控意图和配置 |
| 监控 | `monitor_terms` | 核心词、扩展词和排除词 |
| 来源 | `data_sources` | 来源、授权、配额和保留策略 |
| 查询 | `query_plans` | 可跨用户复用的来源查询 |
| 查询 | `monitor_query_plans` | Monitor 与 QueryPlan 关联 |
| 采集 | `collection_runs` | 来源采集窗口、游标和结果 |
| 内容 | `content_items` | 统一新闻、帖子和视频 |
| 内容 | `content_metric_snapshots` | 活跃内容指标历史 |
| 匹配 | `monitor_content_matches` | 内容相关性和命中解释 |
| 事件 | `events` | 全平台唯一热点事件 |
| 事件 | `event_contents` | 事件证据关系 |
| 事件 | `monitor_event_matches` | 用户上下文评分和排序 |
| 事件 | `event_snapshots` | 热度和趋势历史 |
| AI | `event_summaries` | 可追溯事件摘要 |
| 运行 | `pipeline_runs` | 小时级流水线状态 |
| 通知 | `notifications` | 用户提醒和投递状态 |

## 监控与查询

`monitors` 关键字段：

```text
id, user_id, name, query_intent
language_mode, regions, schedule_minutes
relevance_threshold, alert_config, status
query_embedding vector(384), embedding_model, embedding_version
deleted_at, created_at, updated_at
```

`monitor_terms` 关键字段：

```text
monitor_id, term, normalized_term, language
term_type, weight, origin, confidence
approval_status, deleted_at
```

`term_type` 限定为：

```text
core | translation | alias | related | entity | excluded
```

`query_plans.signature` 对来源、语言、地区、查询词、排除词和窗口生成稳定哈希。签名唯一，多个 Monitor 通过 `monitor_query_plans` 复用同一计划。

## 统一内容

`content_items` 关键字段：

```text
source_id, external_id, content_type
title, body, language, url, canonical_url
author_external_id, author_name
published_at, first_seen_at, last_seen_at
metrics, entities, raw_payload
normalized_hash, duplicate_of_content_id
embedding vector(384), embedding_model, embedding_version
content_status, deleted_at
```

唯一约束：

```text
(source_id, external_id)
```

`content_type` 限定为：

```text
article | post | video | podcast | bulletin
```

`content_status` 限定为：

```text
active | invalid | duplicate | deleted | expired
```

来源扩展指标保存在 `metrics JSONB`。标题、正文、语言、时间、状态和分数不能只保存到 JSONB。

## 全局事件

`events` 关键字段：

```text
fingerprint, title_zh, title_en, category
lifecycle_status, event_time
first_seen_at, last_seen_at, peak_at
heat_score, momentum_score, quality_score
source_count, content_count
merged_into_event_id
created_at, updated_at
```

生命周期：

```text
emerging -> active -> peaked -> declining -> dormant
                     \-> merged
                     \-> rejected
```

`event_contents` 保存聚类关系、相似度和证据角色。`monitor_event_matches` 保存相关性、热度、增速、可信度、最终分数、解释和用户反馈。

## AI 摘要

`event_summaries` 保存：

```text
event_id, language, summary, creator_note
cited_content_ids, provider, model
prompt_version, generated_at, is_current
```

摘要引用只能指向有效的 `content_items`。来源删除后必须重建受影响摘要。

## 索引与约束

必须包含：

- `content_items(source_id, external_id)` 唯一索引
- `monitor_terms(monitor_id, normalized_term, language, term_type)` 活跃记录唯一索引
- `query_plans(signature)` 唯一索引
- `monitor_content_matches(monitor_id, content_id)` 唯一索引
- `event_contents(event_id, content_id)` 唯一索引
- `monitor_event_matches(monitor_id, event_id)` 唯一索引
- `event_snapshots(event_id, captured_at)` 唯一索引
- `content_items(source_id, published_at desc)` 查询索引
- `monitor_event_matches(monitor_id, final_score desc)` 排序索引
- Embedding 余弦 HNSW 索引

所有 `0-100` 分数增加 CHECK 约束。外键删除行为必须在 `db/schema.sql` 中显式声明。

MVP 初期不做表分区。`content_items` 超过 10,000,000 行，或保留期清理影响正常查询后，再评估按月分区。

## 软删除

使用 `deleted_at timestamptz null`：

- `users`
- `monitors`
- `monitor_terms`
- `notifications`

Repository 默认只查询 `deleted_at is null` 的数据。包含软删除字段的唯一约束使用部分唯一索引。

## 状态替代删除

以下数据通过状态表达业务生命周期：

- Event 的活跃、峰值、休眠、合并和拒绝
- ContentItem 的有效、重复、删除、过期和无效
- QueryPlan 的活跃、暂停、耗尽和无效
- PipelineRun 的运行、成功、失败和取消

事件合并保留 `merged_into_event_id`，不能静默删除旧事件。

## 硬删除

可以按保留周期硬删除：

- 过期或撤销的会话
- 超期内容指标快照
- 超期采集和流水线运行记录
- 超期原始响应
- 无 Monitor 引用的失效查询计划
- 已替换的旧摘要
- 重复内容的正文、Embedding 和原始响应
- 超过保存期限的通知

清理任务必须分批执行、记录数量和耗时，并支持幂等重试。

## 来源删除同步

来源要求删除内容时：

1. 将内容状态改为 `deleted`
2. 立即停止展示、匹配、热度和摘要使用
3. 删除正文、Embedding、指标和原始响应
4. 移除事件证据关系并重算事件
5. 根据来源协议决定是否保留最小 tombstone

最小 tombstone 只能包含来源 ID、外部 ID 哈希、删除时间和原因。来源要求彻底清除时不保留 tombstone。

## 数据保留

| 数据 | 默认期限 |
|---|---|
| 原始响应 | 30 天 |
| 标准化内容 | 90 天 |
| 内容指标快照 | 30 天 |
| 采集和流水线运行 | 90 天 |
| 事件、事件快照和当前摘要 | 长期保留 |

具体来源协议优先于默认期限。

## Schema 管理

- 所有表、索引、约束和扩展位于 `db/schema.sql`
- GORM Entity 必须与 Schema 一致
- 应用不得使用 GORM AutoMigrate 修改生产结构
- 架构校验必须阻止 `db/migrations/` 回归
- Schema 变更必须通过本地 PostgreSQL 验证

## 验收门禁

- 18 张核心表覆盖 MVP 主链路
- 不存在重复 Topic、Event、HotEvent 模型
- 核心唯一键和外键删除行为明确
- 软删除查询默认排除删除数据
- 硬删除清理可以分批重试
- 来源删除后事件和摘要不再引用该内容
- Schema、Entity 和设计文档保持一致
