# 数据库与数据生命周期设计

## 1. 文档地位

本文定义 PostgreSQL 目标模型，不复用当前数据库表和 GORM Entity。实施时必须用单一完整 `db/schema.sql` 建立本文模型；当前分表Schema目录不是目标事实源。

PostgreSQL 保存结构化业务事实和运行状态，MinIO 保存原始正文、响应、截图、附件和大体积历史快照，本地 Obsidian Vault 保存面向人的 Markdown 知识投影。

## 2. 建模原则

1. 全平台只保留一个全局 `events` 概念。
2. 不为 RSS、Reddit、YouTube 等来源建立独立内容表。
3. 核心查询、排序、状态和约束字段使用独立列。
4. JSONB 只保存来源扩展配置、非统一指标和模型扩展解释。
5. 原始大正文和二进制对象不进入 PostgreSQL。
6. 每张业务表具有统一 Repository CRUD 和明确所有者模块。
7. 运行、修订、快照和审计表按生命周期提供受限 Repository。
8. 事务、唯一约束和幂等键共同保证至少一次任务执行不会重复业务事实。
9. 所有时间使用 `timestamptz` 并以 UTC 保存；用户时区只在配置和展示层使用。
10. 数据库结构只在完整 `db/schema.sql` 中定义，不使用 GORM AutoMigrate，不维护并行Migration体系。

## 3. 通用字段规范

### 3.1 业务表

除特别说明外，业务表包含：

| 字段 | 类型 | 规则 |
|---|---|---|
| `id` | `bigint generated by default as identity` | 主键 |
| `version` | `bigint not null default 1` | 乐观锁，每次更新加一 |
| `created_by` | `bigint null` | 关联用户，系统创建可空 |
| `updated_by` | `bigint null` | 关联用户，系统更新可空 |
| `created_at` | `timestamptz not null default now()` | 创建时间 |
| `updated_at` | `timestamptz not null default now()` | 更新时间 |
| `deleted_at` | `timestamptz null` | 仅软删除表使用 |

### 3.2 数据类型

| 数据 | 类型和规则 |
|---|---|
| 邮箱 | `varchar(254)`，写入前 trim 和小写，使用 `lower(email)` 唯一索引 |
| 名称 | 默认 `varchar(120)`，说明使用 `text` |
| 状态/类型 | `varchar(32)` 加 `CHECK`，Go 侧使用具名类型 |
| SHA-256 | `char(64)` 小写十六进制 |
| 评分 | `numeric(5,2)`，范围 `0-100` |
| 计数 | 有值时为非负 `bigint`；来源能力未知的 Content 指标必须为 `null`，不得以 `0` 代替未知 |
| 配置扩展 | `jsonb not null default '{}'::jsonb` |
| 字符串集合 | 优先关系表；固定低复杂度筛选可使用 `text[]` |
| 向量 | V1 使用 `halfvec(1024)`，余弦距离 |

## 4. 关系总览

核心事实关系：

```text
User -> Monitor -> MonitorConfigVersion -> MonitorSource -> SourceConnection
Monitor -> MonitorMatch -> Content -> ContentAsset -> MinIO
Monitor -> MonitorEvent -> Event -> EventContent -> Content
Event -> Claim -> ClaimEvidence -> Content
Event -> EventEntity -> Entity
Topic -> TopicEvent -> Event
Topic -> KnowledgeDocument -> KnowledgeChangeProposal -> KnowledgeRevision
Report -> ReportItem -> Event
ReportSubscription -> ReportDelivery -> DeliveryAttempt
```

该文本只表达关系，不是架构图。

#### `monitors`

| 字段 | 类型 | 约束与说明 |
|---|---|---|
| `name` | `varchar(120)` | 活动监控中唯一 |
| `description` | `text` | 监控意图 |
| `status` | `varchar(16)` | `draft/active/paused/archived` |
| `draft_config_version_id` | `bigint null` | 指向唯一可编辑 draft，属于本 Monitor |
| `published_config_version_id` | `bigint null` | 指向当前 immutable published 配置，属于本 Monitor |

#### `monitor_config_versions`

| 字段 | 类型 | 约束与说明 |
|---|---|---|
| `monitor_id` | `bigint` | 外键，逻辑 Monitor |
| `revision` | `bigint` | 永不复用的配置序号；`UNIQUE(monitor_id, revision)` 与 `UNIQUE(id, monitor_id)`，供历史事实验证归属 |
| `version` | `bigint` | 仅 draft 编辑乐观锁版本 |
| `state` | `varchar(16)` | `draft/published/superseded`；每 Monitor 分别最多一个 draft 和 published |
| `timezone` | `varchar(64)` | IANA 时区配置事实 |
| `languages` / `regions` | `text[]` | 规范化 BCP-47 / ISO alpha-2 配置集合 |
| `collection_interval_seconds` | `integer` | 300–86,400，60 秒倍数 |
| `relevance_threshold` / `event_threshold` | `numeric(5,2)` | 分别 60–100 / 0–100 |
| `retention_days` | `integer` | 1–3,650 |
| `config_hash` | `char(64) null` | 仅 published 版本的 canonical SHA-256 |
| `published_at` | `timestamptz null` | 仅 `draft -> published` 时赋值，之后不可修改 |

published/superseded 配置及其子记录不可修改或删除；数据库触发器只允许旧发布版本的 `published -> superseded` 状态变化。编辑已发布 Monitor 时创建新 draft revision，历史运行、匹配和审计都保留其配置版本事实。

### 5.8 AI与保留配置

#### `ai_model_profiles`

完整 Schema 与 Application 均只允许 `embedding` 与 `term_expansion` profile；其他任务类型必须由其拥有 PRD 先扩展 Schema、静态 Schema 与测试。字段为：`name`、`task_type embedding/term_expansion`、`provider openai/onnx`、`model_name`、`model_version`、`credential_ref`、`embedding_dimensions`、`timeout_seconds`、`max_attempts 1..3`、`max_cost numeric(12,4) > 0`、`daily_budget numeric(12,4) null`、`fallback_priority`、`enabled`、`created_by`、`updated_by`、`deleted_at`。

`provider`、任务类型、模型名/版本、凭据引用和向量维度是不可变的语义字段；修改它们必须 archive 原 profile 后新建，不能 update 后让历史运行或向量失去出处。首版不接受 endpoint 或自由 `parameters`：endpoint 必为 NULL、parameters 为空 JSON。`credential_ref` 只允许 OpenAI 的 `env:OPENAI_API_KEY`，ONNX 必为 NULL；它是仅 POST Create 可写的配置引用，PATCH 出现它必须为 `70000`，任何读 API、日志、审计详情与指标均不返回它。OpenAI 的 `model_name` 是 SDK request/response 必须精确相等的 provider model ID；`model_version` 仅为本地不可变的语义/reuse 元数据，响应 ID 不符以 `70000` 拒绝。Embedding profile 的维度必须为 `1024`，非 embedding profile 维度为 NULL。活动名称唯一。

管理员可用乐观 `version` 修改的只有 enabled、timeout、max_attempts、max_cost、daily_budget 和 fallback priority。每个 profile 均须有 max_cost；daily_budget 为 NULL 时无每日上限但仍记录 ledger，为非 NULL 时必须不小于 max_cost。选择顺序是匹配任务、未删除、已启用、凭据/构建可用、预算可预留、`fallback_priority ASC, id ASC`；无候选是显式降级，不建立隐式默认模型。

#### `retention_policies`

字段：`data_class varchar(64) unique`、`retention_days integer > 0`、`action archive/delete`、`enabled`、`description`。公共 API 只允许管理员修改并校验最小保留期。

`ai_runs.target_type + target_id` 是跨领域运行引用，Application 在创建时验证目标存在；核心业务外键仍使用真实数据库外键。`reuse_key` 是 task、target、profile/version、Prompt、输入/输出 Schema、parameters、input 与 evidence hash 的 SHA-256；只有 `succeeded` 运行可复用。每个 claim、retry、settle、cancel 与 reclaim 事务都先取得 `ai-budget:<profile>:<UTC-day>`，再取得 `ai-run:<reuse_key>` advisory lock，绝不反向；成功运行直接复用、in-flight 运行不重复外呼、其余才创建 queued 运行。

预算 reserve/settle/release 与运行状态在 profile+UTC-day advisory lock 的同一事务内完成。profile 必有 `max_cost`；reserve 必须满足 `overage_blocked=false` 且（daily cap 为空或 `settled_cost + reserved_cost + max_cost <= daily_budget`）。reserve 加 max，failed/cancelled/lease reclaim 减该 run 的 reserved，success 减 reserved 后将实际 cost 加 settled。actual cost 大于 reserve 时仍精确记录 cost、标记 `failed/70002`、把真实 cost 加 settled 并将该 profile+UTC-day ledger 置 `overage_blocked=true`；这会封锁当天后续 reserve，即使 daily cap 为空或仍有余额，唯一自动重置是下一 UTC 日的新 ledger；不得伪造较小 cost。

运行只能 `queued -> running -> validating -> succeeded`，或经 `retry_wait` 重试，最终失败/取消；429、超时和临时 5xx 以 `min(2^(attempt-1),4)` 秒退避且在 max attempts 内重试，配置/预算/输入/Schema/证据错误不重试。每个 in-flight 状态有 lease：create/queued→running/running→validating/retry_wait→running 都在同一锁定事务刷新为 `now+timeout+30s`，running|validating→retry_wait 则刷新为 `retry_after+timeout+30s`。worker-only reclaimer 每 30 秒以 `ai-budget(profile,day) -> ai-run(reuse_key)` 固定锁序标记**过期** run 为 `failed/70009` 并释放 reservation，不重放 Provider 调用；测试同时证明有效 retry 不会被回收。Provider 外呼永远在已提交的 claim/reserve 事务外执行。

River 自有表属于第三方基础设施元数据，不实现项目 Repository，不对公共 API 暴露。

## 7. CRUD与Repository规则

### 7.1 业务Repository

所有业务 Repository 嵌入：

```go
type CRUDRepository[T any, ID comparable] interface {
	Create(ctx context.Context, entity *T) error
	GetByID(ctx context.Context, id ID) (*T, error)
	List(ctx context.Context, query PageQuery) (PageResult[T], error)
	Update(ctx context.Context, entity *T) error
	Delete(ctx context.Context, id ID) error
}
```

统一语义：

- `Create` 返回唯一冲突、外键冲突和校验错误的稳定领域错误。
- `GetByID` 默认排除软删除，找不到返回 `ErrNotFound`。
- `List` 使用游标、固定排序允许列表和类型化筛选对象。
- `Update` 条件包含 `id + version`，成功后版本加一。
- `Delete` 对核心表执行软删除或归档，对纯关系表执行硬删除。
- 批量写入、聚合、向量搜索和锁定查询使用显式专用方法。

### 7.2 删除分类

软删除：`users`、`monitors`、`source_connections`、`contents`、`events`、`entities`、`topics`、`knowledge_annotations`、`reports`、`report_subscriptions`、`ai_model_profiles`。

状态归档且不普通删除：`knowledge_documents`、`knowledge_change_proposals`、已发布报告历史。

仅 draft 的 `monitor_rules`、`monitor_sources` 可在完整替换中删除；published/superseded 配置与其子记录不删除。纯关系表硬删除：`event_contents`、`monitor_events`、`entity_aliases`、`event_entities`、`claim_evidences`、`topic_events`、`topic_entities`、`topic_relations`、`entity_relations`、`report_items`。

运行历史按保留策略分批硬删除。

### 7.3 外键行为

- 核心事实默认 `ON DELETE RESTRICT`。
- 纯关系随拥有者删除时可 `ON DELETE CASCADE`。
- 审计、报告快照和修订禁止级联丢失。
- 可选代表内容、作者等引用使用 `ON DELETE SET NULL`。
- 任何外键行为必须在 `db/schema.sql` 中显式声明。

## 8. 关键事务

### 8.1 内容入库

单条内容的数据库事务由 ingestion 用同一个 `database.Runtime.WithinTransaction` callback 创建；Content PostgreSQL adapter 和 Source PostgreSQL adapter 都只复用 callback context 中的 transaction，绝不自行开启嵌套事务。事务包含：

1. Upsert 作者和内容。
2. 写入 ContentAsset 索引。
3. 以 Source-owned compare-and-set 绑定 `collection_run_items.content_id`，并从 `pending/failed` 转为 `succeeded`、清除 `ingestion_error_code`。

该 Source 更新必须以 `id + run_id + source_connection_id + content_id is null + ingestion_status in (pending, failed)` 作为谓词；依靠上述复合外键验证 Content 与 run 同属一个来源，Source adapter 不查询 `contents`。`ListUnboundCaptured` 固定按 `collection_run_items.id ASC` 做 keyset cursor，默认只返回 pending；明确的重试输入才包含 failed。

允许保存的正文证据先以确定性对象键写入 MinIO 并校验 SHA-256，再在同一数据库事务创建 `content_assets` 引用并执行上述绑定；事务失败后立即删除对象。删除失败不会创建数据库引用，随后由对象键前缀与 `content_assets` 的差集对账清理。该事务不重新请求来源、也不提交 River 任务。

MinIO对象在事务前按确定性键幂等写入；数据库失败产生的孤儿对象由对账清理。

### 8.2 事件合并

1. 锁定源事件和目标事件。
2. 迁移或合并 EventContent、MonitorEvent、Entity、Claim 和 Topic 关系。
3. 将源事件状态设为 `merged` 并设置 `merged_into_id`。
4. 重算目标事件指标。
5. 创建知识变更提案和审计记录。

### 8.3 知识提案应用

1. 锁定状态为`approved`的提案记录并校验审核权限，防止重复应用。
2. 检查文档 `version`、`revision_no`、`base_hash` 和当前文件哈希。
3. 创建不可变 KnowledgeRevision。
4. 原子写入Vault。
5. 更新文档哈希、修订和提案状态。

文件写入后数据库提交失败时，对账任务使用修订和文件哈希完成恢复，不覆盖未知人工内容。

### 8.4 报告发布

1. 校验报告结构、证据和条目。
2. 冻结ReportItem快照。
3. 标记报告published。
4. 为每个活动邮件订阅创建唯一ReportDelivery。
5. 提交Vault写入和邮件River任务。

- `reports(report_type, monitor_id, period_start desc, id desc)`
- `knowledge_change_proposals(status, created_at, id)`
- `report_deliveries(status, next_attempt_at, id)`
- `source_checkpoints(next_poll_at, monitor_source_id)`；是否启用由已发布关联、active Monitor 与 enabled/non-deleted SourceConnection 联合判断

软删除常用查询使用部分索引。外键列必须建立普通索引。

### 10.3 文本与向量

- 启用 `pg_trgm`，标题、规范化别名和短摘要按实际查询建立 `gin_trgm_ops`。
- 启用 `vector` 扩展。
- 活动 `content_embeddings`、`monitor_embeddings`、`event_embeddings` 和 `topic_embeddings` 使用 HNSW `halfvec_cosine_ops` 部分索引。
- 只有通过基础规则的候选内容生成向量。
- 活动模型必须输出1024个有限值；Repository 在写入前拒绝长度不等于 1024、`NaN` 与 `Inf`，再以 target/profile advisory lock 停用旧 active 行并插入新行。
- 近邻查询必须过滤 `active=true`、`model_profile_id` 和 `model_version`，使用余弦 `<=>`；不同 profile/version 永不混合。每个 target/profile 的 partial unique index 防止双 active 行，HNSW `EXPLAIN` 是验收证据。
- 改变维度或向量空间需要新的完整 Schema、profile、数据重建/切换计划和全量验证；不得 update 旧 profile 或混合旧向量。

### 10.4 分页

所有列表使用稳定游标，例如 `(published_at,id)` 或 `(final_score,id)`。禁止在大表中使用无限增长的 `OFFSET`。

## 11. 分区与容量

V1不提前对业务表分区：

- 180天候选内容约900万条。
- 30天采集明细约150万条。
- 正确索引、游标分页和批量清理足以覆盖目标容量。

满足任一条件后再评估追加型历史表按月分区：

- 单表活动数据超过2000万行。
- 单表超过100GB。
- 保留期清理持续超过5分钟并影响正常查询。
- 索引维护显著阻塞采集。

优先分区 `collection_run_items`、指标快照和 `audit_logs`。`contents` 涉及跨周期唯一键和大量外键，不作为首批分区对象。

## 12. 数据保留

| 数据类别 | 默认期限 |
|---|---:|
| 规则拒绝候选内容元数据 | 30天 |
| 被拒绝内容的脱敏 CapturedItem 元数据 | 30天 |
| 已接收但未被引用的内容 | 180天 |
| 已接收内容原始证据 | 365天，可按来源缩短 |
| 被事件、主张、报告或知识修订引用的元数据 | 引用存在期间保留 |
| `collection_runs` | 90天 |
| `collection_run_items` | 30天 |
| `collection_run_target_items` | 30天 |
| 内容和事件指标快照 | 90天 |
| AI原始请求与响应对象 | 30天 |
| AI结构化结果与调用元数据 | 180天 |
| 投递记录 | 180天 |
| 审计日志 | 365天 |
| 报告、主题、实体和知识修订 | 管理员明确清理前长期保留 |

来源协议优先于默认策略。引用保护优先于时间策略；仍被引用的数据只能归档或脱敏，不能清理。

## 13. 清理与对账

数据库和MinIO不使用跨系统级联删除：

1. 数据库先标记 `delete_pending`。
2. River任务幂等删除MinIO对象。
3. 成功后更新对象状态并按策略删除元数据。
4. 定期扫描数据库缺失对象和MinIO孤儿对象。
5. Vault扫描比较路径、文档ID、修订和哈希。

批量清理每批默认5000行，记录扫描数、删除数、跳过引用数、耗时和失败原因。请求正确性不能依赖清理任务及时运行。

## 14. 来源删除同步

来源要求删除内容时：

1. 内容状态改为 `deleted`，立即停止展示和匹配。
2. ingestion 仅删除或标记删除其拥有的正文、资产和指标快照。
3. Embedding、事件证据、热度、主张和摘要由各自拥有模块在其实施 Plan 中清理和重算。
4. 报告或知识模块在自身实施后使引用该内容的文档进入证据缺失检查。
5. 按来源协议决定是否保留最小Tombstone。

最小Tombstone只允许来源ID、外部ID哈希、删除时间和原因。来源要求彻底清除时不保留Tombstone。

## 15. 完整 Schema 事实源

`db/schema.sql` 是唯一数据库结构事实源。

规则：

1. Schema 必须是一个可审查、可重入、可在空库执行的完整 SQL 文件。
2. 不维护 `db/migrations/`、`db/schema/` 分表目录、手工快照或第二套建表资源。
3. SQL 顺序固定为：扩展 -> 核心表 -> 关系表 -> 运行表 -> 约束 -> 索引 -> River基础设施。
4. 初始化命令只对空库或明确允许重建的本地库执行，禁止服务启动时静默改表。
5. 已有数据库的结构变更必须在实施计划中附带备份、数据转换、校验和回退步骤，但不把一次性转换脚本变成并行事实源。

PLAN-007 对既有 PLAN-006 数据执行受控升级，不由服务启动或普通测试自动修改任意开发/生产库：先用 `pg_dump --format=custom` 生成并验证可列出的备份；在维护窗口内将 `collection_run_items.source_connection_id` 从其 `collection_runs.source_connection_id` 回填，先验证零空值/零错属后再加非空与复合键；旧 `contents`、`content_metric_snapshots` 和 CapturedItem v1 的 `0` 不具备“来源未知/显式零”存在性信息，保守转换为 `NULL`/nil，正值原样保留。新 CapturedItem v2 必须以 nullable metric 表示存在性，才可保留显式 `0`。升级后执行 `db verify`、schema/upgrade integration、行数/正值聚合对账；任何失败在写入前中止，或从已验证备份恢复到升级前状态。运行手册只记录从本 Schema 导出的受控操作，不成为第二份 Schema 事实源。
6. 大数据回填拆成可重入、可分批的维护任务。
7. Schema 测试必须检查空库建立、二次执行、关键约束、索引和与记录模型的一致性。

### 15.1 F0 原子交付顺序

为避免出现“空 Schema 已交付、完整模型仍在下游”的伪完成状态，首个 F0 任务必须原子交付完整 `db/schema.sql`、全部数据库记录模型、统一 Repository CRUD **契约**，以及空库二次执行、关键约束、幂等键和模型一致性测试。该任务不得创建业务用例或公共 CRUD API。

第二个 F0 任务只在已验证的 Schema 基线上交付 pgx/GORM 运行时、连接池、显式事务、分页、错误映射、Repository **实现**和 `db init`/`db verify` 命令。它可为后续明确批准的模型演进修改完整 Schema，但不得重新分片 Schema、重新定义表目录，或把 Schema 基线验收下放到未来任务。

### 15.2 运行时连接与兼容性边界

- 每个进程只创建一个 `*pgxpool.Pool`；GORM 只能消费由该 pool 导出的 `database/sql` facade，不得用第二个 DSN 或连接池。
- Application 事务以单个 `*sql.Tx` 为边界，同时绑定 GORM 会话和参数化原生 SQL；不得在同一闭包混用独立 pgx 事务。
- `db/schema.sql` 以相邻 `go:embed` 形式随二进制发布。`db init --empty-only --confirm-empty` 只对空库执行，`db verify` 只读检查 PostgreSQL 版本、扩展、嵌入 Schema 派生的对象集、主键和 catalog 指纹。
- 服务启动只执行兼容性检查，不执行 AutoMigrate 或建表。River 在后续 PLAN-013 消费同一 pool，不在 F0 数据库平台创建第二套运行时。

## 16. 数据库验收门禁

- 32张业务表和19张运行表覆盖完整需求闭环。
- 每张业务表有统一Repository CRUD和并发更新测试。
- 运行历史表不能通过普通Repository篡改不可变事实。
- 所有状态、分数、“恰好一个目标”和 Monitor/Config/Source 历史归属约束在数据库中可验证。
- 核心唯一键和外键删除行为明确。
- 重跑采集、AI、报告和投递不会创建重复事实。
- 来源删除后事件、摘要、报告和知识引用得到同步处理。
- MinIO对象、数据库记录和Vault文件可以完成双向对账。
- 在900万内容规模测试数据上验证核心查询计划和游标分页。
- 完整 `db/schema.sql` 可从空库幂等建立目标结构，应用启动不会执行AutoMigrate。
- 设计、`db/schema.sql`、GORM记录模型和Repository测试保持一致。

## 17. 待确认问题
