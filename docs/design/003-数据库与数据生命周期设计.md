# 数据库与数据生命周期设计

## 1. 文档地位

本文定义 PostgreSQL 目标模型，不复用当前数据库表和 GORM Entity。当前 Schema 只作为迁移输入；实施时必须通过版本化 Migration 建立本文模型。

PostgreSQL 保存结构化业务事实和运行状态，MinIO 保存原始正文、响应、截图、附件和大体积历史快照，本地 Obsidian Vault 保存面向人的 Markdown 知识投影。

## 2. 建模原则

1. 全平台只保留一个全局 `events` 概念。
2. 不为 RSS、Reddit、YouTube 等来源建立独立内容表。
3. 核心查询、排序、状态和约束字段使用独立列。
4. JSONB 只保存来源扩展配置、非统一指标和模型扩展解释。
5. 原始大正文和二进制对象不进入 PostgreSQL。
6. 每张业务表具有统一 Repository CRUD 和明确所有者模块。
7. 运行、修订、快照和审计表按生命周期提供受限 Repository。
8. 事务、唯一约束和幂等键共同保证至少一次任务执行不会重复业务事实。
9. 所有时间使用 `timestamptz` 并以 UTC 保存；用户时区只在配置和展示层使用。
10. 数据库结构只通过 Goose SQL Migration 改变，不使用 GORM AutoMigrate。

## 3. 通用字段规范

### 3.1 业务表

除特别说明外，业务表包含：

| 字段 | 类型 | 规则 |
|---|---|---|
| `id` | `bigint generated by default as identity` | 主键 |
| `version` | `bigint not null default 1` | 乐观锁，每次更新加一 |
| `created_by` | `bigint null` | 关联用户，系统创建可空 |
| `updated_by` | `bigint null` | 关联用户，系统更新可空 |
| `created_at` | `timestamptz not null default now()` | 创建时间 |
| `updated_at` | `timestamptz not null default now()` | 更新时间 |
| `deleted_at` | `timestamptz null` | 仅软删除表使用 |

### 3.2 数据类型

| 数据 | 类型和规则 |
|---|---|
| 邮箱 | `varchar(254)`，写入前 trim 和小写，使用 `lower(email)` 唯一索引 |
| 名称 | 默认 `varchar(120)`，说明使用 `text` |
| 状态/类型 | `varchar(32)` 加 `CHECK`，Go 侧使用具名类型 |
| SHA-256 | `char(64)` 小写十六进制 |
| 评分 | `numeric(5,2)`，范围 `0-100` |
| 计数 | `bigint not null default 0`，禁止负数 |
| 配置扩展 | `jsonb not null default '{}'::jsonb` |
| 字符串集合 | 优先关系表；固定低复杂度筛选可使用 `text[]` |
| 向量 | V1 使用 `halfvec(1024)`，余弦距离 |

## 4. 关系总览

核心事实关系：

```text
User -> Monitor -> MonitorSource -> SourceConnection
Monitor -> MonitorMatch -> Content -> ContentAsset -> MinIO
Monitor -> MonitorEvent -> Event -> EventContent -> Content
Event -> Claim -> ClaimEvidence -> Content
Event -> EventEntity -> Entity
Topic -> TopicEvent -> Event
Topic -> KnowledgeDocument -> KnowledgeChangeProposal -> KnowledgeRevision
Report -> ReportItem -> Event
ReportSubscription -> ReportDelivery -> DeliveryAttempt
```

该文本只表达关系，不是架构图。

### 5.7 报告与订阅

#### `reports`

字段：`report_type daily/weekly`、`monitor_id bigint null`、`period_start timestamptz`、`period_end timestamptz`、`timezone varchar(64)`、`title text`、`summary text`、`body text`、`status draft/published/failed/archived`、`generated_at`、`published_at`、`version_no integer`、`deleted_at`。

使用表达式唯一索引保证同一 `report_type + coalesce(monitor_id,0) + period_start + period_end + version_no` 唯一。

#### `report_items`

字段：`report_id`、`event_id`、`rank integer > 0`、`section varchar(64)`、`inclusion_reason text`、`title_snapshot text`、`summary_snapshot text`、`heat_score_snapshot numeric(7,2)`。唯一约束：`(report_id, event_id)` 和 `(report_id, rank)`。

#### `report_subscriptions`

字段：`user_id`、`monitor_id null`、`report_type daily/weekly`、`channel email/rss`、`recipient varchar(254) null`、`rss_token_hash char(64) null`、`timezone varchar(64)`、`schedule varchar(64)`、`enabled boolean`、`deleted_at`。

### 5.8 AI与保留配置

#### `ai_model_profiles`

字段：`name`、`task_type classification/clustering/extraction/summarization/report/embedding`、`provider`、`model_name`、`endpoint`、`credential_ref`、`parameters jsonb`、`timeout_seconds`、`daily_budget numeric(12,4) null`、`fallback_priority smallint`、`enabled`、`deleted_at`。活动名称唯一。

#### `retention_policies`

字段：`data_class varchar(64) unique`、`retention_days integer > 0`、`action archive/delete`、`enabled`、`description`。公共 API 只允许管理员修改并校验最小保留期。

## 7. CRUD与Repository规则

### 7.1 业务Repository

所有业务 Repository 嵌入：

```go
type CRUDRepository[T any, ID comparable] interface {
	Create(ctx context.Context, entity *T) error
	GetByID(ctx context.Context, id ID) (*T, error)
	List(ctx context.Context, query PageQuery) (PageResult[T], error)
	Update(ctx context.Context, entity *T) error
	Delete(ctx context.Context, id ID) error
}
```

统一语义：

- `Create` 返回唯一冲突、外键冲突和校验错误的稳定领域错误。
- `GetByID` 默认排除软删除，找不到返回 `ErrNotFound`。
- `List` 使用游标、固定排序允许列表和类型化筛选对象。
- `Update` 条件包含 `id + version`，成功后版本加一。
- `Delete` 对核心表执行软删除或归档，对纯关系表执行硬删除。
- 批量写入、聚合、向量搜索和锁定查询使用显式专用方法。

### 7.2 删除分类

软删除：`users`、`monitors`、`monitor_rules`、`source_connections`、`contents`、`events`、`entities`、`topics`、`knowledge_annotations`、`reports`、`report_subscriptions`、`ai_model_profiles`。

状态归档且不普通删除：`knowledge_documents`、`knowledge_change_proposals`、已发布报告历史。

纯关系表硬删除：`monitor_sources`、`event_contents`、`monitor_events`、`entity_aliases`、`event_entities`、`claim_evidences`、`topic_events`、`topic_entities`、`topic_relations`、`entity_relations`、`report_items`。

运行历史按保留策略分批硬删除。

### 7.3 外键行为

- 核心事实默认 `ON DELETE RESTRICT`。
- 纯关系随拥有者删除时可 `ON DELETE CASCADE`。
- 审计、报告快照和修订禁止级联丢失。
- 可选代表内容、作者等引用使用 `ON DELETE SET NULL`。
- 任何外键行为必须在 Migration 中显式声明。

## 8. 关键事务

### 8.1 内容入库

单条内容事务包含：

1. Upsert 作者和内容。
2. 写入 ContentAsset 索引。
3. 写入 MonitorMatch。
4. 提交后续AI或聚类River任务。

MinIO对象在事务前按确定性键幂等写入；数据库失败产生的孤儿对象由对账清理。

### 8.2 事件合并

1. 锁定源事件和目标事件。
2. 迁移或合并 EventContent、MonitorEvent、Entity、Claim 和 Topic 关系。
3. 将源事件状态设为 `merged` 并设置 `merged_into_id`。
4. 重算目标事件指标。
5. 创建知识变更提案和审计记录。

### 8.3 知识提案应用

1. 锁定状态为`approved`的提案记录并校验审核权限，防止重复应用。
2. 检查文档 `version`、`revision_no`、`base_hash` 和当前文件哈希。
3. 创建不可变 KnowledgeRevision。
4. 原子写入Vault。
5. 更新文档哈希、修订和提案状态。

文件写入后数据库提交失败时，对账任务使用修订和文件哈希完成恢复，不覆盖未知人工内容。

### 8.4 报告发布

1. 校验报告结构、证据和条目。
2. 冻结ReportItem快照。
3. 标记报告published。
4. 为每个活动邮件订阅创建唯一ReportDelivery。
5. 提交Vault写入和邮件River任务。

软删除常用查询使用部分索引。外键列必须建立普通索引。

### 10.3 文本与向量

- 启用 `pg_trgm`，标题、规范化别名和短摘要按实际查询建立 `gin_trgm_ops`。
- 启用 `vector` 扩展。
- 活动 `content_embeddings`、`monitor_embeddings`、`event_embeddings` 和 `topic_embeddings` 使用 HNSW `halfvec_cosine_ops` 部分索引。
- 只有通过基础规则的候选内容生成向量。
- 活动模型必须输出1024维；改变维度需要新表或新列Migration和全量重建。

### 10.4 分页

所有列表使用稳定游标，例如 `(published_at,id)` 或 `(final_score,id)`。禁止在大表中使用无限增长的 `OFFSET`。

## 11. 分区与容量

V1不提前对业务表分区：

- 180天候选内容约900万条。
- 30天采集明细约150万条。
- 正确索引、游标分页和批量清理足以覆盖目标容量。

满足任一条件后再评估追加型历史表按月分区：

- 单表活动数据超过2000万行。
- 单表超过100GB。
- 保留期清理持续超过5分钟并影响正常查询。
- 索引维护显著阻塞采集。

优先分区 `collection_run_items`、指标快照和 `audit_logs`。`contents` 涉及跨周期唯一键和大量外键，不作为首批分区对象。

## 12. 数据保留

| 数据类别 | 默认期限 |
|---|---:|
| 规则拒绝候选内容元数据 | 30天 |
| 被拒绝内容的MinIO原始响应 | 7天 |
| 已接收但未被引用的内容 | 180天 |
| 已接收内容原始证据 | 365天，可按来源缩短 |
| 被事件、主张、报告或知识修订引用的元数据 | 引用存在期间保留 |
| `collection_runs` | 90天 |
| `collection_run_items` | 30天 |
| 内容和事件指标快照 | 90天 |
| AI原始请求与响应对象 | 30天 |
| AI结构化结果与调用元数据 | 180天 |
| 投递记录 | 180天 |
| 审计日志 | 365天 |
| 报告、主题、实体和知识修订 | 管理员明确清理前长期保留 |

来源协议优先于默认策略。引用保护优先于时间策略；仍被引用的数据只能归档或脱敏，不能清理。

## 13. 清理与对账

数据库和MinIO不使用跨系统级联删除：

1. 数据库先标记 `delete_pending`。
2. River任务幂等删除MinIO对象。
3. 成功后更新对象状态并按策略删除元数据。
4. 定期扫描数据库缺失对象和MinIO孤儿对象。
5. Vault扫描比较路径、文档ID、修订和哈希。

批量清理每批默认5000行，记录扫描数、删除数、跳过引用数、耗时和失败原因。请求正确性不能依赖清理任务及时运行。

## 14. 来源删除同步

来源要求删除内容时：

1. 内容状态改为 `deleted`，立即停止展示和匹配。
2. 删除或标记删除正文、Embedding、指标和原始证据。
3. 移除事件证据并重算事件热度、主张和摘要。
4. 使引用该内容的报告或知识文档进入证据缺失检查。
5. 按来源协议决定是否保留最小Tombstone。

最小Tombstone只允许来源ID、外部ID哈希、删除时间和原因。来源要求彻底清除时不保留Tombstone。

## 15. Schema Migration

使用 [pressly/goose](https://github.com/pressly/goose) 管理 `db/migrations/` 中的版本化SQL。

规则：

1. Migration是数据库结构唯一事实源。
2. `db/schema.sql`如保留，只能由Migration生成作为快照，禁止手工修改。
3. 文件名使用UTC时间戳，例如 `20260715143000_create_events.sql`。
4. 已发布Migration永不修改；错误通过新Migration修正。
5. Migration嵌入二进制并由 `hotkey db migrate` 显式执行。
6. 服务启动只检查Schema兼容性，不静默执行破坏性升级。
7. 可逆DDL提供Down；发布后的数据破坏操作采用向前修复，不承诺无损Down。
8. `CREATE INDEX CONCURRENTLY`放入明确的非事务Migration。
9. 大数据回填拆成可重入、可分批运行的后台任务。
10. 删除列采用新增、双读或回填、切换、删除的渐进步骤。

首批Migration顺序：

1. 启用 `pg_trgm` 和 `vector` 扩展。
2. 创建用户、来源和监控表。
3. 创建内容、证据和采集运行表。
4. 创建事件、实体、主张和关系表。
5. 创建AI、知识、报告、订阅、投递和审计表。
6. 创建普通、部分、GIN、BRIN和HNSW索引。
7. 安装River所需Schema。

## 16. 数据库验收门禁

- 31张业务表和17张运行表覆盖完整需求闭环。
- 每张业务表有统一Repository CRUD和并发更新测试。
- 运行历史表不能通过普通Repository篡改不可变事实。
- 所有状态、分数和“恰好一个目标”约束在数据库中可验证。
- 核心唯一键和外键删除行为明确。
- 重跑采集、AI、报告和投递不会创建重复事实。
- 来源删除后事件、摘要、报告和知识引用得到同步处理。
- MinIO对象、数据库记录和Vault文件可以完成双向对账。
- 在900万内容规模测试数据上验证核心查询计划和游标分页。
- Migration可从空库建立完整Schema，应用启动不会执行AutoMigrate。
- 设计、Migration、GORM记录模型和Repository测试保持一致。
