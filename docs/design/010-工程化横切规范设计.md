# 背景

`009` 已经冻结了 HotKey 后端的总体架构主线，但真正决定工程一致性的能力大多是横切的：错误处理、响应包络、请求上下文、全局状态、枚举、配置、日志和测试门禁。如果这些能力不先统一，后续模块实现会迅速分叉。

# 目标

1. 冻结一套统一错误模型和 HTTP 错误映射机制。
2. 冻结统一成功响应、分页响应和请求上下文注入规则。
3. 冻结全局状态、枚举常量、配置和结构化日志的边界。
4. 形成所有业务模块共享的工程能力基线。

# 非目标

1. 本文档不定义具体业务字段和数据库表。
2. 本文档不决定 OpenAPI 维护方式与 Worker 调度主线。
3. 本文档不替代各业务模块的局部设计。

# 统一错误模型

系统必须采用统一错误模型，而不是让 handler、service、repository 各自拼接错误。

建议能力如下：

1. `ErrorCode`
2. `AppError`
3. `RecoverMiddleware`
4. `ErrorResponder`
5. 错误码注册表

推荐结构：

```go
type ErrorCode string

type AppError struct {
    Code       ErrorCode
    Message    string
    HTTPStatus int
    Cause      error
}
```

约束如下：

1. 业务错误由 service 构造。
2. 基础设施错误由 repository / platform 包装后上抛。
3. handler 不自行判断字符串错误并拼响应。
4. panic 必须在 middleware 层 recover，并映射为统一内部错误。

# 错误码注册表

错误码必须集中定义，不能散落魔法字符串。

建议按模块分组：

1. `AUTH_*`
2. `MONITOR_*`
3. `CONTENT_*`
4. `TOPIC_*`
5. `TREND_*`
6. `NOTIFY_*`
7. `DIGEST_*`
8. `COMMON_*`
9. `INFRA_*`

规则如下：

1. 每个错误码必须有稳定语义。
2. 同一错误码不可跨语义复用。
3. HTTP status 由错误码映射规则统一决定，不由 handler 临时决定。

# 统一响应设计

HTTP 成功与失败响应必须统一包络。

成功响应建议：

```json
{
  "data": {},
  "request_id": "req_xxx"
}
```

失败响应建议：

```json
{
  "code": "MONITOR_NOT_FOUND",
  "message": "monitor not found",
  "request_id": "req_xxx"
}
```

分页响应建议：

```json
{
  "data": [],
  "page": 1,
  "page_size": 20,
  "total": 100,
  "request_id": "req_xxx"
}
```

统一响应规则：

1. 所有 handler 都通过统一 responder 输出。
2. 不同模块不能各自定义成功/失败 JSON 风格。
3. `request_id` 必须在所有响应中可见。

# 请求上下文设计

系统必须将请求级元信息统一注入上下文。

最低字段：

1. `request_id`
2. `trace_id`
3. `user_id`
4. `module`
5. `operator`

规则如下：

1. 上下文字段在 middleware 中注入。
2. service、日志、审计记录统一从上下文读取。
3. 禁止通过全局变量或 goroutine 共享状态传递请求信息。

# 全局状态边界

允许的应用级全局状态：

1. 配置对象
2. logger
3. db client
4. redis client
5. queue client
6. 应用装配容器

禁止的全局状态：

1. 业务可变 map
2. 跨模块共享写入的状态对象
3. 隐式单例 service
4. 请求级数据挂在应用级全局对象上

原则如下：

1. 允许受控应用状态。
2. 禁止无边界业务状态。

# 枚举与常量规范

关键状态值必须使用强类型枚举或常量承载。

推荐形式：

```go
type NotifyChannel string

const (
    NotifyChannelEmail NotifyChannel = "email"
    NotifyChannelInApp NotifyChannel = "in_app"
)
```

规则如下：

1. 枚举放在所属业务模块。
2. 跨模块公共协议常量放在统一公共位置。
3. 必须提供 `IsValid()` 或等价校验能力。
4. OpenAPI 与数据库约束中必须同步反映枚举语义。

# 配置体系

配置必须通过统一入口加载和校验。

正式主线：

1. 环境变量
2. `.env` 仅开发辅助

启动规则：

1. 应用启动时一次性加载配置。
2. 缺失关键配置立即失败。
3. 业务代码不得直接 `os.Getenv`。

配置域至少包括：

1. app
2. db
3. redis
4. auth
5. x-platform
6. queue
7. observability

# 日志与可观测性

系统默认采用结构化日志。

统一字段：

1. `request_id`
2. `trace_id`
3. `user_id`
4. `module`
5. `action`
6. `error`
7. `duration_ms`

最低能力：

1. request logging
2. error logging
3. panic logging
4. health check
5. 慢请求/慢查询观测
6. worker 执行日志

# 测试门禁

横切工程能力必须先有公共测试门禁。

至少包括：

1. 配置缺失时启动失败测试
2. 统一错误响应测试
3. request id / trace id 注入测试
4. recover middleware 测试
5. 枚举校验测试

# 验收门禁

1. 本文档明确了统一错误、响应、上下文、状态、枚举、配置和日志规范。
2. 所有横切能力都能映射成可执行任务，而不是停留在抽象口号。
3. 文档中的工程能力没有互相冲突。

# 风险与边界

1. 如果先做业务模块、后补横切能力，后续收敛成本会很高。
2. 如果错误码和响应不先统一，Web 和 Miniapp 契约很快会漂移。

# 待确认问题

1. 是否需要在第一阶段引入更完整的 metrics/tracing 栈，还是先冻结日志和 request context。
