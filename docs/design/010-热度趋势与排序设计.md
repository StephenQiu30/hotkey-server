# 热度、趋势与排序设计

## 1. 目标

热度回答“当前有多重要”，趋势回答“变化有多快”，Monitor 排序回答“对该用户的监控意图有多值得看”。三者必须分开建模，不得用单一平台的原始点赞数代替。

## 2. 边界

- 热度计算是确定性业务规则，不调用 LLM。
- 只使用 `active` 的独立证据，重复、删除、过期和无效内容不计入。
- 热度是全局 Event 属性；Monitor 下的排序分是热度与相关性的组合。
- 来源不支持的指标是缺失值，不伪造为零。

## 3. 来源能力档案

每类 Connector 提供版本化 `metric_capability_profile`：

```text
source_code, profile_version
supported_metrics
independence_strategy
normalization_window
credibility_weight
maximum_single_item_contribution
```

`credibility_weight` 只表示来源的可验证性和一手程度，不表示政治或内容立场。调整必须由管理员审核并保存原因。

## 4. 指标归一化

对每个可用指标计算窗口内增量，使用同来源、同内容类型和相近账号量级的滚动分布归一化：

```text
normalized_metric = clamp(percentile_rank(log1p(delta)), 0, 1)
```

分布样本不足时使用版本化分段函数降级，并记录 `normalization_mode=fallback`。单条内容、单一作者或单一来源的贡献必须封顶，避免一个爆款或刷量账号主导整个 Event。

## 5. 计算窗口

V1 保留三个滚动窗口：

| 窗口 | 用途 |
|---|---|
| 1小时 | 检测快速爆发 |
| 6小时 | 平滑短期波动 |
| 24小时 | 全局热度和日报排序 |

原始指标快照写入 `content_metric_snapshots`，事件计算结果写入 `event_metric_snapshots`。相同事件、窗口结束时间、窗口长度、算法、证据集合和能力档案集合只产生一份有效快照。

## 6. 热度公式

各分量均归一化到 `0-100`：

```text
heat_score =
  independent_source_score * 0.25
  + content_velocity_score * 0.25
  + source_breadth_score * 0.15
  + engagement_velocity_score * 0.15
  + recency_score * 0.15
  + credibility_score * 0.05
```

- `independent_source_score`：独立发布组织或作者数及增长。
- `content_velocity_score`：有效独立内容的新增速度。
- `source_breadth_score`：新闻、社交、视频等来源类型覆盖。
- `engagement_velocity_score`：已归一化互动增量；缺失时剩余分量按原权重比例重新归一。
- `recency_score`：按事件类型半衰期计算，新证据到来时回升。
- `credibility_score`：高可验证独立来源的封顶加分。

权重、半衰期和分段函数属于 `heat_algorithm_version`。报告和 API 必须同时输出分数版本和简化原因码。

## 7. 趋势

```text
short_ema = EMA(heat_score, 1h)
long_ema  = EMA(heat_score, 6h)
trend_score = clamp((short_ema - long_ema) + normalized_acceleration, -100, 100)
```

API 展示状态由版本化阈值派生，不另建事实源：

| 状态 | 默认规则 |
|---|---|
| `emerging` | 事件较新且短窗增速高 |
| `rising` | `trend_score >= 15` |
| `stable` | `-15 < trend_score < 15` |
| `falling` | `trend_score <= -15` |
| `dormant` | 热度低且长时间无新证据 |

趋势状态不直接改写事件生命周期，只作为生命周期规则的输入。

## 8. Monitor 排序

```text
monitor_event.final_score =
  relevance_score * 0.55
  + heat_score * 0.25
  + trend_component * 0.15
  + freshness_component * 0.05
```

强相关但尚未成为全局热点的事件仍可进入 Monitor 视图。全局热度不得使低相关事件突破 Monitor 的最低相关性阈值。

## 9. 异常与防刷

- 过大、负数或回退的原始指标标记异常，不直接计入增量。
- 单一来源超过封顶后只增加证据数，不继续线性提高热度。
- 转载、机器人镜像和同一作者批量转发按 `source_independence_key` 折叠。
- 异常检测只降低计分权重并进入复核，不自动删除证据。

## 10. 重算与幂等

以下变更触发事件重算：新证据、指标快照、内容删除、重复关系修正、事件合并拆分和算法版本切换。

幂等键：

```text
event_id + window_end + window_hours + heat_algorithm_version
+ evidence_set_hash + capability_profile_set_hash
```

`capability_profile_set_hash` 使档案升级追加新历史快照而不是改写旧快照。重算使用快照式输入；计算期间新证据到来时再提交下一个稳定任务，不在当前计算中混入不同时点的集合。

## 11. 解释与 API

事件详情和排序响应至少包含：

```text
heat_score, trend_score, trend_status
calculated_at, window, algorithm_version
independent_source_count, active_content_count
top_reason_codes
```

不返回内部完整权重配置或反刷阈值，但必须向用户解释主要增长原因。

## 12. 验收门禁

- 不同来源的原始互动数不会直接横向比较。
- 缺失指标不当作零，也不阻断热度计算。
- 重复、删除和过期内容不参与计分。
- 同一输入、窗口和版本必须产生相同结果。
- 单一来源或单一内容无法无界推高事件热度。
- 移除证据后热度、趋势、排序和报告候选可幂等重算。
- 固定样本中人工标注的“上升事件”在 Top 20 的召回率和误报率有版本化基线。
