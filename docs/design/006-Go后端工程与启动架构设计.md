# 背景

旧工程方案中存在明显脱节：

1. 设计文档声明的工程主线与实际运行代码不一致。
2. HTTP、启动、配置、持久化存在多套并行路径。
3. 部分依赖已经写入 `go.mod` 与设计文档，但没有形成真实、完整、统一的实现主线。

因此，本设计不再延续“复杂工程拼装再逐步补齐”的思路，而是重新定义一条团队易理解、社区常见、可直接落地的 Go 后端工程主线。

# 目标

1. 明确 `hotkey-server` 的唯一 Go 工程主线。
2. 明确唯一正式启动方式与统一生命周期管理。
3. 明确 HTTP 与 ORM 的职责边界，避免框架渗透到业务层。
4. 明确 Docker 是正式部署基线，避免多套正式运行文档并存。

# 非目标

1. 本文档不定义具体业务表结构。
2. 本文档不直接拆实施步骤。
3. 本文档不保留旧 Huma / legacy server 路由为目标架构组成部分。

# 技术选型结论

本轮工程主线选型冻结如下：

1. HTTP 主框架：`Gin`
2. ORM 主框架：`GORM`
3. 启动结构：`cmd/hotkey/main.go + internal/app`
4. 配置加载：保留统一配置模块，按新主线收敛为应用唯一配置入口
5. 部署方式：`Dockerfile + docker-compose.yml`

本设计同时冻结一个工程约束：

1. 本轮不再以 `fx`、`wire`、`cobra`、`sqlc`、`Huma` 作为未来目标架构继续推进。
2. 若仓库中仍保留这些遗留依赖或代码路径，仅视为迁移期历史负担，而非新主线组成部分。

# 选型理由

## Gin

选择 `Gin` 的原因：

1. 是 Go 社区最常见、最成熟的通用 Web 框架之一。
2. 路由分组、中间件、参数绑定与 JSON 响应模式直接、稳定、易维护。
3. 对团队成员的理解门槛低，适合纯后端业务 API 长期维护。

不再延续旧 HTTP 方案的原因：

1. 旧方案已经出现设计与实现脱节。
2. 新旧路由并存会持续放大维护成本。
3. 当前阶段更需要简单、稳定、统一的主线，而不是继续背负一套半迁移状态。

## GORM

选择 `GORM` 的原因：

1. 是 Go 生态中最常见、最成熟的 ORM 方案之一。
2. 适合团队统一 CRUD、事务、关联和模型约定。
3. 能在 repository 内部兼顾常规 ORM 访问与复杂查询封装。

不再延续旧数据库主线的原因：

1. 当前持久化路径没有统一收敛为一条清晰主线。
2. 面向团队长期维护时，统一 ORM 路径更符合当前用户目标。
3. 工程现实不支持继续维护多条同等级持久化主链路。

# 启动架构

唯一正式应用入口为：

```text
cmd/hotkey/main.go
```

启动主流程定义为：

```text
main -> app.Run() ->
load config ->
init logger ->
init database ->
build gin router ->
build worker runtime ->
start API + worker concurrently ->
wait signal ->
graceful shutdown
```

该流程下的规则如下：

1. `main.go` 只做入口调用，不写业务逻辑。
2. `internal/app` 负责应用装配、生命周期和优雅退出。
3. `API + worker` 由同一应用进程启动，并共享核心依赖。
4. 当前阶段不提供“只启动 API”或“只启动 worker”的第二正式启动模式；如需未来拆分，必须新增设计文档重新评审。

# 目录结构规范

目标目录结构建议如下：

```text
cmd/
  hotkey/
    main.go

internal/
  app/                    # 应用装配、生命周期、启动与关闭
  config/                 # 配置定义与加载
  platform/
    http/                 # Gin router、middleware、errors、response
    database/             # GORM init、pool、migration helpers
    cache/                # Redis client 与缓存辅助
    x/                    # X 平台接入
  domain/
    auth/
    monitor/
    content/
    topic/
    trend/
    alert/
    notify/
    digest/
  repository/             # 基于 GORM 的 repository 实现
  worker/                 # 任务注册、调度器、后台作业

db/
  migrations/
```

该目录结构的原则是：

1. `cmd` 只负责入口。
2. `app` 只负责装配。
3. `platform` 只负责技术适配。
4. `domain` 只负责业务规则。
5. `repository` 只负责数据访问实现。
6. `worker` 负责后台任务运行时，不自行复制一套业务层。

# HTTP 组织方式

`Gin` 只存在于 HTTP 适配层，推荐组织方式如下：

1. `router.go`
   - 创建 `gin.Engine`
   - 注册全局中间件
   - 建立路由组
2. `middleware.go`
   - request id、recover、auth、logging 等
3. `handlers/*.go`
   - DTO 绑定
   - service 调用
   - 响应封装

必须遵循以下边界：

1. 业务层不得依赖 `gin.Context`。
2. handler 不承载业务规则。
3. 错误响应格式必须统一，避免各 handler 自行定义风格。

# 数据访问组织方式

`GORM` 只存在于 repository 实现层，推荐组织方式如下：

1. `platform/database`
   - 初始化 `*gorm.DB`
   - 连接池配置
   - migration 入口
2. `repository/*`
   - 实现各业务域 repository
   - 统一事务、查询和模型映射

必须遵循以下边界：

1. service 层不得直接依赖 `*gorm.DB`。
2. handler 层不得直接做数据库读写。
3. worker 作业通过 service / repository 边界访问数据库。

复杂查询策略：

1. 常规 CRUD 使用标准 GORM 能力。
2. 聚合、统计、趋势、日报导出等复杂查询允许在 repository 内部使用 `GORM Raw` 或 query builder。
3. 不允许在业务层、HTTP 层或 worker 层散落原生 SQL。

# 接口契约策略

新架构下仍保留机器可读的接口契约产物，但契约维护方式必须与 `Gin` 主线一致，并满足：

1. 契约生成机制与运行代码保持同一事实源。
2. 不允许出现“代码一套、文档一套”的双维护模式。
3. 契约产物是否继续采用 `docs/openapi.json` 路径，应在后续实施计划中一并冻结。

本设计在此只冻结原则，不冻结具体生成工具。

# 生命周期管理

应用生命周期由 `internal/app` 统一管理，必须覆盖：

1. 配置加载失败即启动失败。
2. 数据库连接失败即启动失败。
3. API 与 worker 共享退出信号。
4. 应用收到终止信号后，先停止接收新请求，再收敛后台任务，最后释放资源。

# 启动与部署约定

对外只保留以下两类运行约定：

1. 标准 Go 应用运行
   - 面向本地开发、调试和构建验证。
2. Docker Compose 一键部署
   - 面向本地联调、验收与部署基线。

README、运维文档和后续计划文档只能有一套推荐路径：

1. 部署与联调优先使用 Docker Compose。
2. 二进制入口用于本地开发与构建验证，不作为第二套长期部署叙事。

以下内容不再作为目标架构的一部分：

1. 多套并行启动脚本作为正式运行入口。
2. 多套入口文档并存。
3. 旧路由层和旧工程拼装方案继续被描述为目标态。
