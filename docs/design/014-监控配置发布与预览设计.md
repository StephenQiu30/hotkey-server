# 监控配置发布与预览设计

## 1. 目标与边界

本设计交付 P0 的 Monitor 配置闭环：团队成员可以创建和编辑草稿，管理员可以原子发布、暂停、恢复和归档；后续采集只能读取不可变的已发布配置。它同时定义安全的 SourceConnection 引用、稳定查询签名和不读取 Content 的配置预览。

本设计不实现 Connector、外部请求、River Job、Content 匹配、Embedding、健康探测或基于 Content 的命中预览。这些分别由 PLAN-006、PLAN-007、PLAN-009 和后续运行计划实现。`preview` 在本任务中只验证和解释配置；命中 Content、拒绝原因和相关性预览明确属于 PRD-009。

Monitor 是全团队共享资源，不引入 owner、可见性或成员表。`created_by` 和 `updated_by` 只用于审计，绝不作为资源授权依据。

## 2. 不可变发布版本

`monitors` 是逻辑 Monitor 身份和当前运行状态。它保存不影响采集语义的 `name`、`description`、状态、指针和 aggregate version；名称在未归档 Monitor 内唯一。它不直接保存可编辑的采集规则、语言、地区、频率或阈值。每个逻辑 Monitor 有零或一个可编辑草稿，以及零或一个当前已发布版本：

```text
Monitor
  ├── draft_config_version (mutable, optional)
  └── published_config_version (immutable, optional)
        ├── rules
        └── source associations
```

`monitor_config_versions` 保存一份完整配置：`revision`、规范化后的语言/地区、IANA 时区、频率、阈值、保留期、`config_hash`、`state` 和审计字段。其子表 `monitor_rules`、`monitor_sources` 通过 `config_version_id` 关联，而不再直接关联逻辑 Monitor。

目标关系事实固定如下。`monitors.version` 是 aggregate version；`monitor_config_versions.revision` 是永不复用的配置序号；`monitor_config_versions.version` 是仅草稿编辑使用的 draft version。因此 `expected_monitor_version` 比较前者，`expected_draft_version` 比较后者，二者绝不混用。

| 表 | 必要字段与约束 |
|---|---|
| `monitors` | `id`、`version`、`name`、`description`、`status`、`draft_config_version_id null`、`published_config_version_id null`、审计/软删除字段；`lower(name)` 在 `deleted_at is null and status <> 'archived'` 内唯一。两个 pointer 均以 `ON DELETE RESTRICT` 外键引用 config version，应用在同一事务校验它们属于本 Monitor。 |
| `monitor_config_versions` | `id`、`monitor_id`、`revision`、`version`、`state(draft/published/superseded)`、时区/语言/地区/频率/阈值/保留期、`config_hash null`、`published_at timestamptz null` 和审计字段；`UNIQUE(monitor_id, revision)`，每 Monitor 仅一个 draft 的部分唯一索引，每 Monitor 仅一个 published 的部分唯一索引。`published_at` 只可在 `draft -> published` 时从 `null` 赋为该事务的 UTC 时刻，之后不可修改。 |
| `monitor_rules` | `id`、`version`、`config_version_id`、既有规则字段与审计字段；不再保存 `monitor_id`。 |
| `monitor_sources` | `id`、`version`、`config_version_id`、`source_connection_id`、`query_override`、`query_signature null`、`source_options`、`priority`、`enabled`；`UNIQUE(config_version_id, source_connection_id)` 与 `UNIQUE(id, config_version_id)`。 |

数据库触发器拒绝修改或删除 published/superseded config version 及其 rules/sources；唯一允许的发布期变更是旧版本 `published -> superseded` 的 `state` 切换。草稿可修改，其 `version` 必须随完整替换递增。发布时持久化 `config_hash` 和每个 enabled source 的 `query_signature`；预览只计算，绝不持久化。目标数据模型还必须为 `monitor_config_versions(id, monitor_id)` 建联合唯一约束；后续 `monitor_matches` 以 `(monitor_config_version_id, monitor_id)` 复合外键引用它，`collection_run_targets` 以 `(monitor_source_id, monitor_config_version_id)` 复合外键引用 `monitor_sources(id, config_version_id)`，禁止写入错属 Monitor/config 的历史事实。

- 新建 Monitor 会创建 `draft` 配置版本，逻辑状态为 `draft`。
- 编辑只修改草稿。若当前不存在草稿，`PUT /draft` 必须传当前 `expected_monitor_version` 与显式 `expected_draft_version: null`；应用锁定 Monitor 后仅在 draft pointer 仍为 null 时，复制已发布版本为 `revision=max(revision)+1`、`version=1` 的新草稿，并在同一事务应用本次完整用户可编辑配置替换。AI/system 规则会被复制且只能经专用审批命令变化；绝不修改已发布版本及其子记录。新草稿 pointer 与 Monitor aggregate version 同时推进并返回。若 pointer 已非空、Monitor version 不匹配或调用方错误提供非 null draft version，均以 `30001` 拒绝；因此两个并发的首次编辑恰有一个成功。
- 发布要求 `expected_monitor_version` 和 `expected_draft_version` 同时匹配。应用锁定 Monitor 与草稿，验证完整配置，计算 `config_hash`，将草稿标为 `published`，将旧发布版本标为 `superseded`，推进 Monitor 的 published pointer 和 aggregate version，并把状态置为 `active`。任一子写入、指针更新或审计失败都会回滚整个事务。
- 已发布和 superseded 版本及其子记录不可修改、不可删除；历史采集、匹配、审计和报告通过配置版本或其 `monitor_sources` 事实保持可追溯。
- 已有草稿的完整替换必须同时传当前 `expected_monitor_version` 与非 null 的当前 `expected_draft_version`；成功后草稿 `version` 和 Monitor aggregate version 都递增。草稿读取与发布均使用明确乐观锁，不使用最后写入者覆盖。
- `name` 与 `description` 随草稿替换命令使用 `expected_monitor_version` 更新；它们不进入 `config_hash`，因为不影响查询或调度。名称变更仍会推进 aggregate version 并与草稿写入、审计在同一事务中完成。

归档会释放名称唯一约束；restore 前若同名未归档 Monitor 已存在，操作以 `30004` / HTTP 409 拒绝。核心规则精确定义为至少一条 `origin in (user, system)`、`approval_status=approved`、`enabled=true` 且 `rule_type in (keyword, phrase, entity)` 的规则。

计划 006 的调度器只读取 `monitors.status = active` 且 `published_config_version_id` 非空的配置。它不得读取 draft，也不得重新解释编辑中的规则。

## 3. 状态机与调度谓词

| 当前状态 | 命令 | 目标状态 | 角色 | 结果 |
|---|---|---|---|---|
| 无 | create | draft | editor/admin | 创建逻辑 Monitor 与草稿版本。 |
| draft | publish | active | admin | 原子发布当前有效草稿。 |
| active | publish | active | admin | 发布一个新草稿；旧发布版本保留。 |
| active | pause | paused | admin | 不改配置；后续调度不再选择它。 |
| paused | resume | active | admin | 复用当前已发布版本，不重放历史任务。 |
| draft/active/paused | archive | archived | admin | 保留所有历史，不再调度或匹配。 |
| archived | restore | paused | admin | 恢复为不调度的 paused；须由单独 resume 明确启用。 |

未列出的转换一律以 `30000` / HTTP 409 拒绝。每个状态命令都比较当前 `expected_monitor_version`；版本过期时一律以 `30001` / HTTP 409 拒绝，即使目标状态已经出现。只有调用方提供当前版本且资源已处于目标状态时，pause、archive、restore、enable、disable 才返回无副作用成功。`resume` 要求当前有有效的已发布版本；draft 不能 resume。

一个 Monitor Source 可被计划 006 选中，当且仅当：逻辑 Monitor 为 active、已发布配置包含该关联、关联 enabled、SourceConnection 未删除且 enabled。paused、archived、draft、草稿来源、已 superseded 版本和 disabled/deleted SourceConnection 都不得进入到期查询。暂停或归档不会删除 MonitorEvent、Event、Content、审计或历史 checkpoint。

## 4. P0 授权矩阵

| 操作 | viewer | editor | admin |
|---|---:|---:|---:|
| 已发布 Monitor/安全 SourceConnection 列表与详情 | 是 | 是 | 是 |
| 草稿详情 | 否 | 是 | 是 |
| 创建 Monitor、编辑/替换草稿、管理规则和来源关联、配置预览 | 否 | 是 | 是 |
| 发布、暂停、恢复、归档、恢复归档、批准/拒绝 AI 词项 | 否 | 否 | 是 |
| 创建、更新、启用/禁用、归档/恢复 SourceConnection 和轮换凭据引用 | 否 | 否 | 是 |

所有动作均要求有效身份会话。资源无 owner，因此 editor 可协作编辑任何未归档 Monitor 草稿；admin 可操作全部资源。HTTP transport 只读取 `Subject` 并使用显式角色中间件；授权和状态规则留在 Monitor/Source application，不扩展 identity 模块的资源策略。

## 5. 配置、规则和发布校验

### 5.1 Monitor 配置

每个逻辑 Monitor 的元数据和每个配置版本必须满足：

- `name` 为 1–120 个 Unicode code point，`description` 最多 2,000 个 code point；活动逻辑 Monitor 名称唯一。
- `timezone` 为有效 IANA 名称；默认从创建者偏好复制，但随后属于 Monitor 配置事实。
- `languages` 为 1–8 个规范化 BCP-47 标签；`regions` 为 0–8 个大写 ISO 3166-1 alpha-2 代码。
- `collection_interval_seconds` 为 300–86,400 的 60 秒倍数；`relevance_threshold` 为 60–100，`event_threshold` 为 0–100；`retention_days` 为 1–3,650。
- 每份配置最多 100 条规则和 10 个来源关联；发布至少要求一个 enabled SourceConnection 和一个 approved、enabled、非 AI 的 keyword/phrase/entity 核心规则。

`monitor_rules` 的 P0 兼容矩阵如下；其他组合以 `30002` / HTTP 400 拒绝：

| rule_type | operator | value | weight | 审批 |
|---|---|---|---|---|
| keyword、phrase、entity | contains、equals | 1–160 字符 | 0–100 | user/system 自动 approved；AI 初始 pending。 |
| exclude_keyword | contains、equals | 1–160 字符 | 固定 0 | 同上。 |
| domain | equals、not_equals | 1–253 字符的规范化域名 | 固定 0 | 同上。 |
| author | equals、not_equals | 1–128 字符 | 固定 0 | 同上。 |
| language | equals、not_equals | BCP-47 标签 | 固定 0 | 同上。 |
| region | equals、not_equals | ISO alpha-2 代码 | 固定 0 | 同上。 |
| regex | matches | 1–256 UTF-8 字节的 Go RE2 | 0–100 | 同上。 |

服务必须编译 regex，并拒绝无效语法。`origin=ai` 的规则只能由 admin 改为 approved 或 rejected；pending/rejected/disabled 规则不进入 `query_signature`，但完整 `config_hash` 保留它们的安全状态以审计草稿和发布事实。`origin=user` 与 `origin=system` 初始为 approved。语言/地区规则与配置数组相交：语言或地区的正向规则缩小配置数组，否定规则从数组排除；空交集使发布失败。

### 5.2 SourceConnection

PLAN-005 只允许 `rss`（RSS 或 Atom Feed）和 `hacker_news` 两种 SourceConnection；其他预留 source_type 由实现 Connector 的后续 Plan 开放。P0 不发起 Validate、Fetch 或 Health 外部调用，连接创建和任意配置/凭据变更后的 `health_status` 一律为 `unknown`。`source_connections.version` 是所有 Source 写命令的 `expected_source_version`。enable/disable 改变 `enabled`；archive 写 `deleted_at` 并强制 `enabled=false`；restore 清除 `deleted_at` 但保留 `enabled=false`；每一步都推进 version。

`endpoint` 是 SourceConnection 的执行语义：一旦被任一 published config version 引用，`source_type`、`endpoint` 和 `config` 不可原地修改；管理员必须创建新的 SourceConnection、在草稿中替换关联并重新发布。即使尚无 published 引用，修改这三个字段的 PATCH 也必须取得第 5.2 节的同一配置 advisory lock、锁定 SourceConnection 并在锁后重新检查不存在 published 引用；数据库触发器以同一谓词拒绝任何绕过 application 的更新。只有 `name`、credential reference、enabled、health 和归档状态可在既有连接上变化。这样已发布配置通过 stable `source_connection_id` 保持不可变执行语义。

RSS endpoint 必须为 HTTPS URI、不得带 userinfo、fragment、非 443 端口或 `token`、`key`、`secret`、`password`、`authorization` 查询参数；host 必须是 DNS 名称而非字面 IP。Hacker News endpoint 固定为 `https://hacker-news.firebaseio.com/v0`。计划 006 的 Connector 每次连接和每次重定向都必须重新解析并拒绝 loopback、private、link-local、multicast、unspecified、reserved IP；最多跟随 3 次 HTTPS/443 且仍符合 RSS/官方 HN host 策略的重定向。`auth_type=none` 时 credential reference 必须为空；`api_key`、`oauth2` 或 `bearer` 时只能使用 `env:NAME`，其中 `NAME` 匹配 `[A-Z_][A-Z0-9_]{0,127}`。数据库、日志、审计和任何 HTTP DTO 都不返回原始 credential reference，只返回 `credential_configured` 布尔值。

`config` 只保存来源扩展的非秘密键。P0 白名单及类型为：`allow_body_storage`、`requires_attribution`、`requires_deletion_sync`（boolean）；`content_retention_days`、`metrics_retention_days`（1–3,650 integer）；`allowed_languages`（0–8 个 BCP-47 string）；`allowed_regions`（0–8 个 ISO alpha-2 string）；`rate_limit_per_minute`（1–600 integer）；`request_timeout_seconds`（1–120 integer）；`max_pages_per_run`（1–20 integer）。键可省略，服务在创建、更新、读取、发布和预览时都使用同一组稳定默认值：三个 boolean 均为 `false`，两个 retention 均为 `30`，allowed arrays 均为 `[]`（不额外限制），rate limit 为 `60`，timeout 为 `30` 秒，`max_pages_per_run` 为 `1`。因此 P0 的预览始终可计算，PLAN-006 也可直接使用 rate limit、timeout 和 page 上限；实现 Connector 前不得另行引入隐式默认值。未知键、嵌套 secret 字段或凭据样式字符串一律拒绝，不能把 JSONB 作为任意配置或密钥旁路。

发布、resume、修改 SourceConnection 执行语义的 PATCH、SourceConnection disable 和 archive 都先获取相同的事务级 PostgreSQL advisory lock `hotkey.monitor_source_configuration`，然后锁定受影响的 Monitor、draft/published config version、关联行和 SourceConnection 行，并在锁定后重新读取/校验所有谓词。这个低频全局配置锁消除 publish、恢复和来源生命周期操作的竞态而不引入多表锁顺序死锁。禁用或归档 SourceConnection 时，应用锁定所有引用它的 active 或 paused Monitor；如果该连接是其中任一 active Monitor 唯一的可调度来源，操作以 `40001` / HTTP 409 拒绝；否则关联保持历史事实但被调度谓词排除。resume 在同一锁内验证当前 published config 至少存在一个 enabled、未删除的 SourceConnection；否则以 `40001` / HTTP 409 拒绝，不得把零可调度来源的 Monitor 置为 active。恢复后的连接保持 disabled 与 unknown，须由 admin 显式启用。发布草稿时，任何 enabled 关联指向 disabled/deleted SourceConnection 都使发布失败。

## 6. 查询签名与纯配置预览

每个 enabled Monitor Source 在发布或预览时计算 SHA-256 `query_signature`。哈希输入是 UTF-8 canonical JSON，固定 `signature_version: 1`，包含：

```text
source_connection_id
source_type
normalized_endpoint
normalized_query_override
sorted languages and regions
sorted approved enabled include rules (type, operator, normalized value, weight, priority)
sorted approved enabled exclude rules (type, operator, normalized value)
query_window_kind = scheduled_interval
```

P0 的 `source_options` 固定为空对象；任何会改变查询语法的来源选项必须先修改本设计，不能静默加入签名。字符串在哈希前执行 trim 和 Unicode NFC；域名、语言、地区转小写后分别规范为 domain/BCP-47/ISO 格式；keyword、phrase、entity、author、regex 和 query override 保留大小写。规则按 `(rule_type, operator, normalized_value, weight, priority, id)` 排序，语言/地区数组按规范值排序去重；JSON 使用固定 struct 字段顺序、全部字段显式编码（空数组为 `[]`、空 override 为 `""`），绝不序列化 map。有效语言/地区先应用正/负 language/region 规则，再与非空的 SourceConnection allowed arrays 相交；结果为空即发布失败。

`config_hash` 是 SHA-256 canonical JSON，字段为 `hash_version=1`、`monitor_id`、`revision`、时区、有效语言/地区、interval、两个 threshold、retention，以及按 `(rule_type, operator, normalized_value, weight, priority, origin, approval_status, enabled, id)` 排序的规则数组和按 `(source_connection_id, query_override, priority, enabled, id)` 排序的来源数组。`query_override` 必须为空，或为 trim/NFC 后 1–2,048 UTF-8 字节且不含 NUL 的字符串；其来源专用语法仍由 PLAN-006 定义。每条规则只编码 `id`、`rule_type`、`operator`、规范 value、`weight`、`priority`、`origin`、`approval_status`、`enabled`；每个来源只编码 `id`、`source_connection_id`、规范 query override、`priority`、`enabled`、固定空 `source_options`。它不包含规则/来源行 version、名字、审计主体或创建/更新时间；它持久化在 published config version。`query_signature` 是上述签名输入的 SHA-256，持久化在 published monitor source；它不含逻辑 Monitor ID，但必须包含 `query_override`，不能把不同查询错误合并。

`POST /api/v1/monitors/{id}/preview` 只针对草稿运行，返回 `eligible`、`config_hash`、每个 enabled Source 的 `source_connection_id`/`query_signature`/纳入规则 ID/排除规则 ID/未批准规则 ID、`estimated_requests` 和稳定警告码。估算为每个 enabled Source 的一轮 `max_pages_per_run` 之和，不主张真实配额或命中数量。预览绝不读取 Content、Connector、Redis 或 River，且不写 monitors、版本、规则、来源、checkpoint、运行、River 或 audit 表。

计划 006 会以 `collection_runs` 表示唯一的 source/query/window 执行，必要字段为 `source_connection_id`、`query_signature`、`window_start`、`window_end`、运行状态和唯一 `(source_connection_id, query_signature, window_start, window_end)`；`collection_run_targets` 保存 `collection_run_id`、不可变 `monitor_source_id`、`monitor_config_version_id` 和唯一 target 约束。`source_checkpoints` 继续按不可变 `monitor_source_id` 保存；新发布 config 的每个 source 生成新的 checkpoint、`query_hash=query_signature`、空 cursor 和 `next_poll_at=published_at`，不继承旧配置 cursor。计划 006 只有在共享 run 成功且该 target 的处理成功时才推进其 checkpoint。它不得把当前单个 `monitor_source_id` 的 checkpoint/run 行错误地当成共享请求的唯一事实。

## 7. HTTP 与错误契约

所有接口位于 `/api/v1`，使用 `Result{code,message,data}` 和现有 Bearer 身份中间件。不存在通用 `expected_version` 字段：Monitor 状态命令只传 `expected_monitor_version`，有草稿的编辑和 publish 传 `expected_monitor_version` 加 `expected_draft_version`，首次从已发布版本创建草稿时 `expected_draft_version` 显式为 null，Source 写命令只传 `expected_source_version`；创建命令与无副作用 preview 不传预期版本。服务以 409 稳定冲突而非静默覆盖。列表使用现有 cursor 分页与固定 `id ASC` 默认排序，筛选字段采用白名单。

| 路由 | 角色 | 语义 |
|---|---|---|
| `GET /monitors`、`GET /monitors/{id}` | viewer+ | viewer 只见已发布安全视图；editor/admin 可见草稿元数据。 |
| `POST /monitors` | editor+ | 创建 Monitor 与完整草稿，HTTP 201。创建/替换 payload 只能写 user rules；服务端强制 `origin=user` 和 `approval_status=approved`。 |
| `PUT /monitors/{id}/draft` | editor+ | 用完整草稿替换当前草稿。已有草稿时校验 expected monitor/draft version；首次创建草稿时校验 expected monitor version 和 null draft version；不能伪造或修改 AI/system rule 的 origin 或审批。 |
| `POST /monitors/{id}/draft/ai-candidates`、`POST /monitors/{id}/draft/rules/{rule_id}/approval` | admin | 显式创建 pending AI 候选和批准/拒绝候选；每次必须提供 expected monitor/draft version，推进二者并写审计。 |
| `POST /monitors/{id}/preview` | editor+ | 返回纯配置预览，无持久化副作用。 |
| `POST /monitors/{id}/publish` | admin | 传 `expected_monitor_version` 与 `expected_draft_version` 原子发布草稿。 |
| `POST /monitors/{id}/pause`、`/resume`、`/archive`、`/restore` | admin | 只传 `expected_monitor_version` 执行状态命令。 |
| `GET /source-connections`、`GET /source-connections/{id}` | viewer+ | 仅安全 DTO。 |
| `POST /source-connections`、`PATCH /source-connections/{id}`、`POST /source-connections/{id}/enable`、`/disable`、`/archive`、`/restore` | admin | 管理全局来源与安全凭据引用。 |

Monitor viewer DTO 只含 `id`、`name`、`description`、`status`、published revision、时区/语言/地区/频率/阈值/保留期、以及 published rules（ID/type/operator/value/weight/priority/origin/approval/enabled）和 sources（ID/source connection ID/name/type/query override/priority/enabled）；它不含 endpoint、Source config、credential reference、health diagnostic 或 draft。editor/admin 额外可读 draft 的同一安全字段和 draft version。viewer/editor 的 Source DTO 只含 `id`、`version`、`name`、`source_type`、`enabled`、`health_status`、`terms_policy_url`、`credential_configured` 和 `deleted`。admin 的 Source management DTO 额外包含 endpoint 和经过白名单验证的非秘密 config，供管理和编辑来源执行语义；两个 DTO 都不含 credential reference 或 health diagnostic。Preview 仅返回第 6 节所列规则 ID 和哈希，不返回额外的敏感来源字段。

监控错误码固定为：`30000` invalid monitor state（409）、`30001` monitor version conflict（409）、`30002` invalid monitor configuration（400）、`30003` monitor draft unavailable（409）、`30004` monitor name conflict（409）。来源错误码固定为：`40000` invalid source configuration（400）、`40001` source connection required（409）、`40002` unsupported source type（400）、`40003` source connection unavailable（409）。响应中的 `data` 在错误时一律为 `null`，不得泄露数据库错误、内部 URI 或 credential reference。

## 8. 审计、事务和验收边界

`operations` 模块拥有用于新业务模块的窄 `AuditWriter` application port 与 PostgreSQL adapter。Monitor/Source application 在同一个 `database.Runtime.WithinTransaction` 内写业务事实和一条经过白名单过滤的 audit entry；不得调用 identity 私有 audit adapter，也不得启动隐藏嵌套事务。允许动作包括 `monitor.created`、`monitor.draft_updated`、`monitor.ai_candidate_created`、`monitor.ai_candidate_approved`、`monitor.ai_candidate_rejected`、`monitor.published`、`monitor.paused`、`monitor.resumed`、`monitor.archived`、`monitor.restored`、`source.created`、`source.updated`、`source.enabled`、`source.disabled`、`source.archived`、`source.restored`。审计只记录版本、状态、规则/来源计数、hash、审批结果和安全布尔值，绝不记录规则原文、endpoint、credential reference、config JSON 或密钥。

P0 验收必须以真实 PostgreSQL 证明：并发发布恰有一个成功，两个首次创建草稿请求也恰有一个成功；发布与审计完整回滚；完整状态机和 active-only 查询正确，resume 会拒绝零可调度来源；SourceConnection 语义 PATCH 与 publish 串行化且 published 引用后被拒绝；monitor/config 与 source/config 错配的历史写入均被复合外键拒绝；viewer/editor/admin 权限矩阵正确；未批准 AI 词项不会进入 `query_signature`；相同规范配置的签名稳定而 override 改变签名；省略 Source config 字段仍有稳定默认值和可计算预览；`published_at` 只在发布时写入且 checkpoint 使用该值；预览前后所有业务、审计和 River 表行数/内容不变；安全 DTO、日志和审计不泄露 endpoint、config 或 credential reference。Schema、记录模型、OpenAPI、架构门禁与全量 `make ci` 同样是发布门槛。

## 9. 后续同步

本设计获接受后，Design-003 必须同步版本化 Monitor 数据模型；PRD-005、PLAN-005 与 Acceptance-005 必须以本设计为准。Design-005、007 与 012 继续处于 review，它们关于 Connector、Content 匹配与调度的宽泛描述不能覆盖或冲突本设计的 P0 配置边界。PLAN-006 在开工前必须使用本设计的已发布版本与共享 query-run 事实模型。
