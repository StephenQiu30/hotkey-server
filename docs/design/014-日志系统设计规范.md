# 日志系统设计规范

> **日期：** 2026-07-09
> **状态：** 已实现
> **关联：** [012-LLM集成与内容聚合设计](012-LLM集成与内容聚合设计.md)、[013-后台任务与消息队列基础设施设计](013-后台任务与消息队列基础设施设计.md)

---

## 1. 背景

改造前存在以下问题：

- 仅使用 Go 标准库 `log.Printf` 输出纯文本日志，无法被日志收集系统解析
- 无日志级别，无法按 debug/info/warn/error 过滤
- 无上下文关联（RequestID / UserID 已存入 context 但从未被日志消费）
- 无 HTTP 访问日志中间件
- GORM SQL 日志静默，不利于排查慢查询
- 后台任务零运行时日志

## 2. 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 日志内核 | uber-go/zap v1.27+ | Fx 已引入，无需新依赖；Uber 生产验证；强类型 API |
| Gin 中间件 | 自实现（ginzap 适配） | zap 官方 Gin 适配器 |
| GORM 日志 | 自实现 ZapWriter | GORM logger.Interface 适配，< 50 行 |

## 3. 架构与数据流

```
Config → Init → global zap.Logger
                     │
      ┌──────────────┼──────────────┐
      │              │              │
  Gin 中间件     Controller/     GORM Logger
 (AccessLog)   Service/Worker   (SQL + 慢查询)
      │              │              │
      └──────────────┼──────────────┘
                     ▼
            os.Stdout (JSON Lines)
```

Gin 中间件解析请求后将元数据注入 context → 各业务层通过 `logging.Ctx(ctx)` 获取带上下文的 Logger → 输出 JSON Lines 到 stdout。

## 4. 日志配置

| 环境变量 | 默认值 | 说明 |
|----------|--------|------|
| LOG_LEVEL | info | debug / info / warn / error |
| LOG_FORMAT | json | json / console（console 仅用于本地开发） |
| LOG_OUTPUT | stdout | stdout / stderr |

Console format 仅在 `make dev` 时使用，生产环境始终用 JSON Lines。

## 5. 组件设计

### 5.1 Logger 工厂 — zap 初始化

提供三个顶层函数：

- `Init(level, format string) error` — 根据 level 设定 zap.AtomicLevel，根据 format 选择 JSONEncoder 或 ConsoleEncoder。启用 `zap.AddCaller()` 和 `zap.AddStacktrace(zapcore.ErrorLevel)`。
- `L() *zap.Logger` — 返回全局 Logger
- `S() *zap.SugaredLogger` — 返回全局 SugaredLogger（Printf 风格兼容）

### 5.2 Context 字段提取

函数 `FieldsFromContext(ctx)` 从 runtime context 提取非空字段：

- request_id
- trace_id（预留）
- user_id
- module

函数 `Ctx(ctx)` 等价于 `L().With(FieldsFromContext(ctx)...)`，业务代码一行获取带上下文的 Logger。

### 5.3 Gin AccessLog 中间件

每个 HTTP 请求结束时记录结构化日志：

```json
{
    "level": "info",
    "ts": "2026-07-08T08:30:00+08:00",
    "method": "GET",
    "path": "/api/v1/monitors",
    "status": 200,
    "latency": "45ms",
    "request_id": "req-a1b2c3d4",
    "user_id": 42,
    "module": "http"
}
```

不包含请求体（避免敏感数据泄露），不记录静态文件（/swagger/*）。

### 5.4 GORM Logger 适配器

实现 `gorm.io/gorm/logger.Interface`：

- 成功 SQL → Debug 级别
- 慢查询（> 200ms）→ Warn 级别
- 错误 SQL → Error 级别
- 使用 `logging.Ctx(ctx)` 拾取 HTTP 上下文，慢 SQL 可关联到具体请求 ID

## 6. 替换规则

| 原写法 | 替换为 |
|--------|--------|
| `log.Printf("...: %v", err)` | `logging.L().Error(...)` |
| `log.Printf("warning: ...")` | `logging.L().Warn(...)` |
| `log.Printf("...: starting")` | `logging.L().Info(...)` |
| `log.Panicf(...)` | `logging.L().Panic(...)` |
| worker 中零日志 | 新增 `logging.L().Info/Warn` |

## 7. 文件结构（已实现）

```
internal/platform/
├── logging/
│   ├── zap.go         ← zap Logger 初始化工厂
│   └── context.go     ← Context 字段提取 Helper
│
├── http/
│   └── accesslog.go   ← Gin 请求日志中间件

internal/platform/database/
└── logger.go          ← GORM 日志适配器

internal/config/
└── config.go          ← 追加 LogLevel / LogFormat / LogOutput
```

## 8. 设计决策

| 决策 | 理由 |
|------|------|
| 仅输出 JSON Lines 到 stdout | 容器化环境由日志收集器处理轮转与持久化 |
| Init 失败时 panic（不静默降级） | 日志配置错误应在启动时快速失败 |
| 不记录请求体 | 避免敏感数据泄露；由业务层在必要时自行记录 |
