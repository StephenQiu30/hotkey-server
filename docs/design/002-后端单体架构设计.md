# 后端单体架构设计

## 1. 文档地位

本文描述 HotKey Server 的目标架构，不代表当前代码已经完成迁移。设计不复制当前 Controller、Service、Repository、Kafka 或重复模型结构；实施必须按后续计划逐步迁移并同步测试、Schema 和 OpenAPI。

本文按用户要求不提供架构图。运行角色、模块、依赖方向和数据流使用文字、表格与目录契约表达。

## 2. 架构决策

| 决策 | 结论 |
|---|---|
| 代码组织 | 单仓库、单 Go Module、单二进制 |
| 架构形态 | 模块化单体，不拆微服务 |
| 本地运行 | 默认 `all` 角色，在一个进程同时启动 API、调度和 Worker |
| 可扩展运行 | 同一二进制支持 `api` 或 `worker` 角色，未来可独立启动 |
| 业务事实源 | PostgreSQL |
| 原始证据 | MinIO |
| 人类知识投影 | 本地 Obsidian Vault |
| 可靠后台任务 | PostgreSQL + River |
| 向量检索 | PostgreSQL + pgvector |
| 数据库结构 | 单一完整 `db/schema.sql` |
| 知识库版本控制 | 不使用 Git；使用数据库修订、哈希和审核状态 |

V1 不使用 Kafka、微服务、分布式事务、Elasticsearch或独立向量数据库。Redis只能作为未来可选缓存和限流组件，核心工作流在没有 Redis 时必须运行。

## 3. 设计原则

1. 本地默认简单运行，但 API 和 Worker 不依赖进程内共享业务状态。
2. PostgreSQL 保存业务事实、检查点、任务和审核状态。
3. MinIO 和 Vault 都通过领域端口接入，第三方 SDK 不进入领域层。
4. 每个模块职责单一，跨模块只调用应用端口。
5. 每张业务表提供统一 Repository CRUD；公共 API 只开放安全业务操作。
6. 复杂协议和可靠基础设施优先使用成熟库，简单稳定逻辑优先标准库。
7. 所有外部调用具有超时、限流、重试边界和可观测状态。
8. 不为未来可能性提前建立微服务、插件系统、规则引擎或工作流引擎。

## 4. 运行角色

同一个 `hotkey` 二进制支持：

```text
hotkey serve --role=all
hotkey serve --role=api
hotkey serve --role=worker
hotkey db verify
hotkey db init --empty-only
```

### 4.1 `all`

本地默认模式，启动：

- Gin HTTP Server
- 调度器
- River Worker
- 来源 Connector
- Vault 对账和文件写入
- 报告生成及邮件投递
- 日志、指标和优雅停机

### 4.2 `api`

只启动 HTTP API、鉴权、查询和业务命令入口，不执行采集和长任务。需要异步处理的应用用例在数据库事务内提交 River 任务。

### 4.3 `worker`

只启动调度、River Worker、Connector、AI、聚类、Vault、报告、投递和清理任务，不开放业务 HTTP API；可保留独立健康检查端点。

本期不设计具体部署方式。运行角色只是代码和生命周期边界，确保未来上线时不需要重写业务模块。

| 领域 | 选型 | 使用边界 |
|---|---|---|
| Language | Go 1.26 | 后端实现 |

| ORM | GORM v2 | 业务 Repository CRUD 和事务内持久化 |

| DI | go.uber.org/fx | 装配和生命周期 |
| Config | Viper | 本地文件、环境变量和配置校验 |
| Database | PostgreSQL 16+ | 业务事实、运行状态和分析查询 |
| Vector | pgvector + pgvector-go | 多语言向量及近邻检索 |
| Job Queue | River | 事务入队、唯一任务、租约和重试 |
| Scheduler | robfig/cron | 计算到期任务并提交 River |
| Object Storage | minio-go | ObjectStore 的 MinIO 适配器 |
| Logging | Zap | 结构化日志 |
| Observability | OpenTelemetry, Prometheus | 链路和指标 |
| Validation | validator/v10 | HTTP DTO 和配置校验 |
| API Docs | Swaggo, Gin Swagger | OpenAPI 生成 |
| Security | JWT v5, bcrypt | 会话令牌和密码安全 |
| AI | 官方 SDK + Provider 端口 | LLM 和 Embedding |

项目预选并在实际模块需要时引入：

- [mmcdole/gofeed](https://github.com/mmcdole/gofeed)：RSS/Atom 解析
- [google-api-go-client](https://github.com/googleapis/google-api-go-client)：YouTube 官方 API
- [golang/oauth2](https://github.com/golang/oauth2)：OAuth2 授权
- [minio-go](https://github.com/minio/minio-go)：MinIO 客户端
- [riverqueue/river](https://github.com/riverqueue/river)：PostgreSQL 后台任务
- [pgvector-go](https://github.com/pgvector/pgvector-go)：向量类型和查询适配

不预先引入 Git SDK、GitHub SDK、通用 HTTP 包装器、Markdown AST 或动态插件框架。

## 7. 分层职责

| 层 | 职责 | 禁止事项 |
|---|---|---|
| `domain` | 实体、值对象、领域规则、Repository和领域端口 | Gin、GORM、River、MinIO和具体SDK |
| `application` | 用例、权限、事务边界、跨领域编排 | 拼装HTTP响应、直接调用平台SDK |
| `infrastructure` | PostgreSQL、River、MinIO、Vault、邮件和外部API适配 | 决定业务权限和状态规则 |
| `transport/http` | DTO绑定、认证上下文、调用用例和响应转换 | 直接访问数据库、MinIO或Connector |
| `platform` | 与业务无关的连接、协议和生命周期 | 保存监控、事件、知识等业务规则 |

依赖方向：

```text
transport/http -> application -> domain
infrastructure -> domain
bootstrap -> all adapters
```

跨模块访问必须调用目标模块应用端口或明确只读查询端口，不能导入目标模块的基础设施 Repository。

## 9. 核心端口

### 9.1 Connector

```go
type Connector interface {
	Validate(ctx context.Context, connection SourceConnection) error
	Fetch(ctx context.Context, request FetchRequest) (FetchResult, error)
	Health(ctx context.Context, connection SourceConnection) HealthResult
}
```

Connector 只负责外部数据读取和统一 SourceItem 映射，不负责 LLM、事件聚类、报告或 Vault 写入。

### 9.2 基础设施端口

| 端口 | 责任 |
|---|---|
| `ObjectStore` | 原始证据和修订快照的 Put、Get、Stat、Delete |
| `VaultStore` | 扫描、读取、加锁、原子写入和归档 Markdown |
| `ModelProvider` | 结构化生成和 Embedding，不保存业务状态 |
| `JobDispatcher` | 事务内入队、唯一任务、延迟任务和取消 |
| `MailSender` | 发送 HTML/纯文本邮件并返回消息 ID |
| `Clock` | 提供可测试的时间 |
| `IDGenerator` | 生成领域稳定键 |

第三方 SDK 类型必须在 Infrastructure 内转换，不能穿透这些端口。

## 10. Repository 契约

每张业务表必须有对应 Repository，并嵌入统一 CRUD：

```go
type CRUDRepository[T any, ID comparable] interface {
	Create(ctx context.Context, entity *T) error
	GetByID(ctx context.Context, id ID) (*T, error)
	List(ctx context.Context, query PageQuery) (PageResult[T], error)
	Update(ctx context.Context, entity *T) error
	Delete(ctx context.Context, id ID) error
}
```

领域 Repository 增加明确的业务查询，例如 `FindDue`、`FindSimilar`、`ListByEvent`。不得把所有查询塞入通用过滤 JSON，也不得使用反射生成跨领域万能 Repository。

规则：

1. `Create` 执行持久化约束映射，不替代应用层业务校验。
2. `GetByID` 统一返回可识别的 `ErrNotFound`。
3. `List` 使用游标分页、排序允许列表和类型化筛选。
4. `Update` 使用 `version` 乐观锁，受影响行数为零返回并发冲突。
5. 核心业务表默认软删除或状态归档；纯关系表可硬删除。
6. 批量写入、搜索和聚合使用独立方法，不扭曲单实体 CRUD。
7. Repository 不接收 HTTP DTO，不返回 Gin 类型，不自行开启隐藏事务。

运行日志、修订、快照、审计和第三方队列表按其生命周期使用受限 Repository，不提供篡改历史的任意 Update/Delete。

## 11. 公共 API 边界

公共资源：

```text
/api/v1/users
/api/v1/sources
/api/v1/monitors
/api/v1/contents
/api/v1/events
/api/v1/entities
/api/v1/topics
/api/v1/knowledge
/api/v1/reports
/api/v1/subscriptions
/api/v1/operations
```

即使业务表有 CRUD Repository，公共 API 也只开放安全用例：

- 停用用户和监控主题，而不是任意硬删除
- 合并、拆分和归档事件，而不是直接修改外键
- 审批或拒绝知识提案，而不是直接覆盖文件
- 发布和重新生成报告，而不是修改已发布历史
- 测试、暂停和轮换订阅，而不是返回私有令牌哈希

禁止 `/api/v1/tables/{table}` 一类通用表操作接口。

## 12. 完整数据流

### 12.1 采集

1. 调度器查询已启用且到期的监控来源。
2. 使用“监控来源 ID + 调度时间片”提交唯一 River 任务。
3. Worker 读取 checkpoint 并调用 Connector。
4. 对候选内容进行字段校验、URL规范化、语言识别和哈希。
5. 原始响应、正文和附件使用确定性对象键写入 MinIO。
6. 单条内容事务写入内容、证据索引、匹配结果和后续任务。
7. 批次成功持久化后推进 checkpoint。

### 12.2 相关性和事件

1. 基础规则过滤明显无关内容。
2. 通过规则的内容生成向量并进行语义召回。
3. 只把阈值附近的边界样本交给 LLM 复核。
4. 使用时间、实体、关键词和向量近邻召回候选事件。
5. 更新事件成员、监控关系、热度和趋势快照。
6. 抽取实体、事实主张和证据立场。
7. 跳过人工锁定字段和关系。

### 12.3 知识和交付

1. 已接收事件创建或更新事件知识文档。
2. 周期任务选择事件并冻结报告条目快照。
3. 模型从结构化证据生成报告草稿。
4. 报告校验通过后发布并原子写入 Vault。
5. 发布事务提交邮件任务，报告同时进入 RSS/Atom Feed。
6. 长期知识修改生成提案，经审核和哈希检查后写入。

## 13. 事务与跨存储一致性

### 13.1 PostgreSQL 事务

以下操作必须事务化：

- 监控主题与规则整体更新
- 内容、资产索引、匹配和后续任务入队
- 事件合并、拆分和成员迁移
- 报告发布和投递实例创建
- 知识提案批准、修订创建和文档状态更新

事务由 Application 声明，Repository 使用传入事务上下文。

### 13.2 PostgreSQL、MinIO 和文件系统

不实现伪分布式事务：

1. MinIO 对象键由内容哈希确定，重复 Put 安全。
2. 数据库保存对象状态和校验哈希。
3. Vault 写入使用同目录临时文件、刷盘和原子重命名。
4. 失败操作保持明确状态并由 River 重试。
5. 对账任务识别孤儿对象、缺失对象和文件冲突。

## 14. 错误和重试

| 错误 | 行为 |
|---|---|
| 配置错误 | 禁用相关任务并提示管理员，不自动重试 |
| 授权失败 | 标记来源不可用，等待凭据更新 |
| 429或临时网络错误 | 使用服务端建议或指数退避 |
| 数据格式错误 | 隔离单条内容，批次继续 |
| LLM Schema 错误 | 修复一次，仍失败则规则降级或人工审核 |
| MinIO临时错误 | 保持待处理状态并重试 |
| Vault哈希冲突 | 进入人工冲突处理，不重试覆盖 |
| 邮件永久拒收 | 停止自动重试并保留失败原因 |

所有重试必须有最大次数、最大延迟和稳定幂等键。

## 15. 生命周期

启动顺序：

1. 加载并校验配置。
2. 初始化日志、指标和追踪。
3. 连接 PostgreSQL 并检查 Schema 版本。
4. 按运行角色构造 HTTP 或 Worker 组件。
5. `api/all` 启动 Gin；`worker/all` 启动调度和 River Worker。
6. 启动健康检查和跨存储对账计划。

停止顺序：

1. 停止接受新 HTTP 请求和新调度任务。
2. 停止领取新的 River Job。
3. 等待运行任务在超时内完成或释放租约。
4. 刷新日志与遥测。
5. 关闭数据库、MinIO和其他客户端。

单一来源、单条内容或单个邮件错误不能导致进程退出；数据库或配置不可恢复错误应使对应角色启动失败。

## 16. 配置规则

- 本地配置文件和环境变量使用同一结构。
- 启动时校验 PostgreSQL、MinIO、Vault 路径和启用能力的必需配置。
- API Key、密码和 Token 不进入仓库或数据库明文。
- 日志只记录配置名和脱敏标识，不记录秘密值。
- Vault 路径必须解析为允许根目录内的绝对路径。
- 未启用的可选能力显式返回 unavailable，不使用伪数据。

## 17. 工程风格

1. 包名使用小写单数，文件名使用 `snake_case.go`。
2. 禁止 `utils`、`helpers`、`common_service` 等无边界包。
3. 状态和类型在 Go 使用具名类型，数据库使用 `CHECK`。
4. `context.Context` 是 Repository、端口和长任务方法的第一个参数。
5. HTTP DTO、领域实体和数据库记录模型不得混用。
6. 不使用可变全局状态，依赖通过 Fx 明确装配。
7. Handler 只绑定、鉴权、调用用例和转换响应。
8. 错误使用统一领域错误和稳定业务码，禁止返回 `err.Error()`。
9. 列表接口使用游标分页和排序允许列表。
10. 时间、ID、模型、对象存储、文件系统和邮件可注入测试替身。

## 18. 未来上线边界

本期不设计部署，但必须保留：

- `api` 和 `worker` 角色不依赖进程内共享状态
- ObjectStore 使用 S3 兼容语义
- VaultStore 可增加网络文件系统或同步适配器
- JobDispatcher 可增加其他实现，但V1只实现River
- 配置不硬编码本地端口、主机名或绝对路径
- 多个Worker依靠任务租约和幂等键安全协作

上述边界不授权提前加入 Docker、Kubernetes、Kafka 或服务发现。

## 19. 验收门禁

- 一个命令以 `all` 角色启动完整本地后端
- 相同二进制可分别以 `api` 和 `worker` 角色启动
- 模块依赖符合 Transport、Application、Domain 和 Infrastructure 方向
- 每张业务表具有统一 Repository CRUD 和测试
- 公共 API 不暴露危险通用 CRUD
- PostgreSQL 是任务和业务状态事实源
- MinIO和Vault失败不会破坏已提交数据库事实
- 新代码不引入Kafka、微服务、独立向量数据库或Git知识库流程
- 优雅停机不领取新任务，并能恢复未完成任务
- 架构校验、Lint、测试和构建通过
