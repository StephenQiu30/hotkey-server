# 后端单体架构设计

本文定义 HotKey Server 的目标架构。系统使用一个 Go 二进制和一个进程，同时运行 HTTP API、定时调度和后台流水线，避免在小团队阶段引入分布式系统成本。

## 架构原则

- 单仓库、单二进制、单进程
- 模块化单体，不拆分 API 和 Worker
- PostgreSQL 是业务事实源
- 使用成熟 Go 框架和显式分层
- 只为真实替换边界定义接口
- 允许后续扩展，不提前建设分布式能力
- 当前代码现实与目标架构必须明确区分

## 运行模型

```text
cmd/hotkey/main.go
        |
      Fx App
        |
  +-----+-------------------+
  |     |                   |
 Gin   Cron          Pipeline Executor
 API  Scheduler       bounded concurrency
  |     |                   |
  +-----+-------------------+
        |
PostgreSQL + pgvector
        |
optional Redis cache
```

Fx 统一管理组件构造、启动、停止和资源释放。应用启动后同时提供 API 和小时级任务能力，不提供 `api`、`worker` 等运行角色。

本地和生产均使用同一入口：

```bash
go run ./cmd/hotkey
```

## 技术栈

| 领域 | 选型 | 用途 |
|---|---|---|
| Language | Go 1.26 | 后端实现 |
| HTTP | Gin | 路由、中间件和请求处理 |
| ORM | GORM v2 | CRUD、事务和仓储实现 |
| DI | go.uber.org/fx | 依赖注入和生命周期 |
| Config | Viper | 环境变量和配置加载 |
| Database | PostgreSQL 16+ | 业务事实和分析查询 |
| Vector | pgvector | 多语言向量相似度 |
| Scheduler | robfig/cron | 小时级调度 |
| Logging | Zap | 结构化日志 |
| Observability | OpenTelemetry, Prometheus | 链路和指标 |
| Validation | validator/v10 | 请求校验 |
| API Docs | Swaggo, Gin Swagger | OpenAPI 生成 |
| Security | JWT v5, bcrypt | 会话令牌和密码安全 |
| AI | 官方 SDK, Provider 接口 | LLM 和 Embedding 接入 |

Redis 只用于缓存、短期会话或限流。业务状态、任务检查点和配额事实必须保存到 PostgreSQL。

MVP 不使用 Kafka、微服务、分布式事务、Elasticsearch 或独立向量数据库。

每个模块只创建当前需要的文件。常用文件包括：

```text

```

没有对应职责时不得创建空文件或空接口。

## 模块职责

| 模块 | 职责 |
|---|---|
| identity | 用户、认证、会话和权限 |
| monitor | 监控任务和用户检索意图 |
| source | 来源注册、Connector、配额和采集运行 |
| queryplan | 扩展词、查询签名、共享查询和调度计划 |
| content | 统一内容、标准化、去重和证据 |
| matching | 关键词、实体和向量相关性 |
| event | 跨来源事件聚类、合并和生命周期 |
| ranking | 热度、增速、可信度和最终排序 |
| notification | 事件提醒和投递状态 |
| report | 基于事件生成日报和导出 |
| operations | 流水线运行、来源健康和人工重跑 |

## 依赖边界

模块内部依赖固定为：

```text
Controller -> Service -> Repository -> PostgreSQL
```

约束如下：

- Controller 只处理协议、参数、认证上下文和 Result 输出
- Service 保存业务规则、事务边界和用例编排
- Repository 只处理数据读写
- 跨模块调用通过目标模块 Service
- `platform/` 不包含业务规则
- 业务模块不得直接读取其他模块拥有的表
- 禁止 Controller 直接访问 GORM
- 禁止 Repository 反向调用 Service
- 禁止包级可变业务状态

接口只用于：

- 跨模块 Service 边界
- 数据来源 Connector
- LLM、Embedding、邮件等外部服务
- 需要替换的时钟和测试依赖

单一实现的模块内部结构使用具体类型。

## 应用生命周期

启动顺序：

1. 加载并校验配置
2. 初始化日志和可观测性
3. 连接 PostgreSQL 和可选 Redis
4. 构造 Repository、Service 和 Controller
5. 注册路由和中间件
6. 启动 Gin Server
7. 启动 Cron 和 Pipeline Executor

停止顺序：

1. 停止接受新 HTTP 请求
2. 停止调度新任务
3. 等待运行中任务在超时内完成
4. 刷新日志和遥测
5. 关闭数据库和缓存连接

HTTP Server、Cron 或数据库出现不可恢复启动错误时，应用整体启动失败。单个来源或单条内容错误不能导致进程退出。

## 配置规则

- 配置通过环境变量和 `.env` 加载
- 启动时校验必需配置
- API Key、密码和令牌不得进入仓库或数据库明文
- 日志只能记录配置项名称，不能记录秘密值
- 开发、测试和生产使用相同配置结构
- 可选能力未启用时必须显式降级，不得静默使用伪数据

## 复杂度边界

禁止引入：

- 微服务和服务发现
- Kafka 或内部事件总线
- 通用 `Repository[T]`
- 通用规则引擎和工作流引擎
- 动态插件框架
- 无当前用例的数据表和配置项

出现第二个真实实现或明确替换需求后才能提取抽象。

## 当前代码迁移边界

本文描述目标状态，不代表当前代码已经完成迁移。实施时必须通过独立架构基础计划逐步替换旧目录、Kafka 任务和重复数据模型。

每个迁移任务必须同步更新：

- 代码和测试
- `db/schema.sql`
- OpenAPI
- `AGENTS.md`
- 架构校验脚本
- 相关设计文档

## 验收门禁

- 一个命令启动完整后端进程
- API 和定时任务共享同一 Fx 生命周期
- 模块依赖符合 Controller、Service、Repository 方向
- PostgreSQL 是任务状态和业务状态事实源
- 新代码未引入 Kafka、微服务或通用框架式抽象
- 优雅停机不会接受新任务或丢失已提交状态
- `make lint`、`make test`、`make build` 和架构校验通过
