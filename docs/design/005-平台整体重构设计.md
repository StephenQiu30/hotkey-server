# 背景

`hotkey-server` 当前已具备热点监控平台的业务雏形，但工程主链路已经发生明显漂移：

1. 存在多套启动方式并存，正式入口、脚本入口与容器入口边界不清。
2. HTTP 层存在新旧并存状态，实际运行路径与历史设计文档脱节。
3. 数据持久化路径没有统一收敛到一条团队易维护的主线。
4. 旧设计文档中的工程选型与当前代码现实不一致，难以继续作为新一轮实现依据。

当前用户目标不是对旧方案做局部修补，而是保留既有业务目标，重新定义一条简单、成熟、常用、可维护的 Go 后端主架构。

# 目标

本轮重构后的 `hotkey-server` 应满足以下目标：

1. 明确收敛为纯后端单体应用，不承担前端页面交付职责。
2. 只保留一套正式运行架构：单二进制、单入口、单进程同时运行 `API + worker`。
3. 使用团队更通用、更易维护的 Go 技术主线，降低工程理解和接手成本。
4. 保留 Docker 一键部署能力，形成统一的本地联调与部署基线。
5. 为后续计划拆解、代码迁移、验收和运维提供新的上游权威文档。

可用以下 BDD 场景表达目标状态：

```gherkin
Scenario: 运维启动新的 hotkey-server
  Given 仓库已经完成新架构重构
  When 开发者通过标准入口启动应用或执行 docker compose up
  Then 系统会在同一应用进程内同时启动 API 和 worker
  And 系统只依赖一套明确的 HTTP、ORM、配置和部署主链路
  And 团队无需在多套并行设计之间做选择
```

# 非目标

1. 本轮不做微服务拆分。
2. 本轮不引入前端页面、SSR、Next.js 或独立 Web 交付职责。
3. 本轮不保留双 HTTP 框架并存的目标架构。
4. 本轮不保留 ORM 与原生 SQL 作为双主链路长期并存。
5. 本轮不为了假设中的未来复杂规模提前引入重型基础设施。

# 系统边界

`hotkey-server` 在产品整体中的职责限定为：

1. 认证与账号能力。
2. 关键词监控任务管理。
3. 平台内容采集与标准化。
4. 热点聚合、趋势分析与提醒。
5. 通知投递与日报导出。
6. 运维、任务运行与后台管理接口。

`hotkey-server` 不承担：

1. 前端页面渲染。
2. 独立 Web 工作台实现。
3. 跨仓 UI 组件职责。

# 目标运行架构

目标运行形态采用 `纯后端单体 + 内嵌 worker`：

```text
+-----------------------------------------------------------+
| hotkey-server process                                     |
|                                                           |
|  +-------------------+     +---------------------------+  |
|  | Gin HTTP API      |     | Worker Runtime           |  |
|  | auth/monitor/...  |     | poll/aggregate/notify... |  |
|  +-------------------+     +---------------------------+  |
|                 shared config / logger / services / repos |
+-----------------------------------------------------------+
                 |                               |
                 v                               v
             PostgreSQL                       Redis
```

该架构下只有一个正式应用程序入口，对外只暴露一套运行主线：

1. 标准 Go 应用入口运行。
2. Docker Compose 一键部署运行。

# 模块边界

建议的新系统模块边界如下：

1. `Auth`
   - 注册、登录、身份校验、用户偏好。
2. `Monitor`
   - 关键词监控任务的创建、更新、启停和配置。
3. `Content Ingestion`
   - 平台内容采集、标准化、去重、命中关联。
4. `Topic`
   - 热点主题聚合与主题事实维护。
5. `Trend`
   - 趋势快照、窗口统计和趋势查询。
6. `Alert`
   - 升温规则判定与提醒事件生成。
7. `Notify`
   - 站内通知、邮件或其他投递渠道。
8. `Daily Digest`
   - 日报导出、摘要生成和知识库沉淀。
9. `Admin/Ops`
   - 任务状态、连接器健康、异常审计与人工重跑接口。

模块之间遵循以下规则：

1. `API` 与 `worker` 共用同一套业务服务层。
2. 平台适配、HTTP 适配和持久化适配不得直接污染业务规则。
3. 所有持久化经由统一 repository 边界。

# 架构原则

本轮重构采用以下架构原则：

1. 一套正式启动架构优于多套平行启动方式。
2. 一套正式工程主线优于“文档一套、代码一套、脚本一套”的漂移状态。
3. 常用、成熟、团队可快速理解的技术栈优于半落地的复杂工程拼装。
4. 在当前阶段，单体收敛优于过早服务拆分。
5. API 与 worker 必须共享配置、日志、数据库和业务核心，避免形成一个仓库中的两套后端。

# 部署边界

当前阶段默认部署基线为单副本 Docker Compose：

1. 一个 `hotkey-server` 应用实例。
2. 一个 PostgreSQL。
3. 一个 Redis。

该边界意味着：

1. 本轮设计优先保证单副本下逻辑正确、部署简单、运维清晰。
2. 多副本扩展不是本轮目标。
3. 若未来需要多副本部署，必须补充 worker 幂等、竞争控制或独立 worker 运行策略设计。

# 迁移原则

本轮重构不是“推倒重写业务”，而是“保留业务目标、重建工程主线”。迁移顺序必须遵循：

1. 先建立新权威设计文档。
2. 再重建新的实施计划，并明确旧计划降级为历史方案。
3. 再进行应用骨架收敛。
4. 再进行 HTTP、ORM 和 worker 主链路迁移。
5. 最后删除遗留路径与过渡实现。

迁移过程中允许短期兼容层存在，但不允许长期双主线共存。

# 风险与边界

1. 将 `API + worker` 合并为单进程运行会简化当前架构，但也限制未来横向扩容方式。
2. 旧实现与新设计之间存在过渡期，需要通过清晰的删除清单控制技术债膨胀。
3. 若缺少新的实施计划承接，本设计容易再次沦为空泛描述。
4. 若不及时将旧文档降级为历史方案，后续实现会出现设计输入分叉。

# 待确认问题

1. 新接口契约产物的正式生成和维护机制由哪份工程设计文档落定。
2. Redis 在新架构中的职责边界是否只保留缓存与 worker 辅助状态，还是继续承担更多运行时职责。
3. 日报、通知和人工重跑等运维能力是否全部纳入本轮主迁移范围，还是作为后续阶段分批收敛。
