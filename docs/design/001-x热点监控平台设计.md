# 背景

当前目标不是构建一个简单的热点时间告警工具，而是构建一个面向内容创作者的完整热点发现平台。用户通过配置关键词，从海外社交平台优先获取最相关、最新、最热的内容，系统再将内容组织为可读的主题、趋势和提醒结果。

首版聚焦 `X` 平台，原因是：

1. 内容传播快，适合作为热点首发信号源。
2. 文本内容结构相对统一，适合先验证采集、评分、聚合与提醒闭环。
3. 后续扩展到 YouTube、Reddit 等平台时，可以复用同一套接入抽象与分析链路。

# 目标

首版必须支撑以下闭环：

1. 用户注册登录后创建关键词监控任务。
2. 系统定时从 `X` 采集相关内容并标准化入库。
3. 平台向用户展示热点内容流、热点主题、趋势变化。
4. 当某个关键词或主题明显升温时，生成站内提醒并支持邮件通知。
5. 管理后台能够查看任务运行、连接器健康、异常和人工重跑。

可以用以下 BDD 场景表述：

```gherkin
Scenario: 创作者监控 AI 关键词热点
  Given 用户创建一个 "openai agent" 关键词监控任务
  When 系统按 10 分钟频率从 X 拉取相关内容
  Then 用户可以看到相关推文列表、聚合后的主题、最近 24 小时趋势
  And 当某个主题热度在 1 小时窗口内显著上涨时收到提醒
```

# 非目标

1. 首版不支持多平台采集，`X` 是唯一内容源。
2. 首版不做自动内容创作、自动发帖或脚本生成。
3. 首版不做复杂个性化推荐，只围绕用户显式配置的关键词工作。
4. 首版不追求秒级实时流处理，只提供分钟级更新。
5. 首版不引入重型大数据基础设施，如 Kafka、Flink、ES 集群。

# 技术选型

1. `Go`：作为后端 API 和 worker 语言，复用当前 `hotkey-server` 技术方向。
2. `PostgreSQL`：存储用户、监控任务、内容、主题、趋势快照和提醒事实数据。
3. `Redis`：用于缓存、去重键、速率限制和异步任务辅助状态。
4. `Docker`：作为本地和部署的基础容器运行方式。
5. `X API Connector`：通过统一的平台接入接口实现首版内容拉取。

# 系统架构

系统采用 `分层单体 + 异步任务`：

1. `API 层`
   - 提供认证、监控任务、内容查询、主题查询、趋势查询、提醒中心和后台管理接口。
2. `Worker 层`
   - 执行采集、清洗、去重、打分、主题聚合、趋势快照和通知发送。
3. `Store 层`
   - PostgreSQL 持久化业务事实。
   - Redis 存放缓存、幂等键、速率限制和短期队列辅助信息。

建议的仓库结构如下：

```text
cmd/api/
cmd/worker/
internal/config/
internal/server/
internal/auth/
internal/monitor/
internal/platform/x/
internal/content/
internal/topic/
internal/trend/
internal/alert/
internal/notify/
internal/store/postgres/
internal/store/redis/
db/schema.sql
```

# 核心模块

## 1. Auth & User

负责用户注册、登录、资料、通知偏好和订阅方案信息。

## 2. Keyword Monitor

负责关键词监控任务生命周期：

1. 创建任务。
2. 管理关键词、排除词和频率。
3. 控制启停状态。
4. 暴露用户可见的监控详情和统计。

## 3. Platform Connector

定义跨平台接口，但首版仅实现 `X`：

1. `SearchPosts`
2. `FetchAuthorProfile`
3. `NormalizePost`
4. `ReportRateLimit`

业务层不得直接依赖 X 的原始响应结构。

## 4. Content Ingestion

负责将 X 原始内容变为平台统一内容对象，完成：

1. 原始响应落盘。
2. 结构标准化。
3. 去重。
4. 与监控任务建立命中关系。

## 5. Scoring Engine

对命中的内容计算多个分值：

1. 关键词相关性。
2. 互动热度。
3. 发布时间新鲜度。
4. 作者影响力。

最终得到 `final_score`，用于内容流排序和主题代表内容选择。

## 6. Topic Aggregator

在单个监控任务范围内，将相似内容聚成主题，并计算：

1. 主题标题。
2. 主题摘要。
3. 代表内容。
4. 当前热度。

## 7. Trend Analyzer

按固定时间窗口沉淀快照：

1. `15m`
2. `1h`
3. `24h`

支持监控级和主题级趋势曲线。

## 8. Alert Engine

根据规则判断是否触发提醒，首版规则可包含：

1. 主题 1 小时热度增幅超过阈值。
2. 某关键词在短时间内新增高热内容数超过阈值。
3. 首次出现高传播内容。

## 9. Notification Center

负责提醒事件转为用户通知，支持：

1. 站内消息。
2. 邮件发送记录。

# 数据流

主链路如下：

1. 用户创建关键词监控任务。
2. Scheduler 触发 `monitor_poll_job`。
3. `X Connector` 拉取内容。
4. `Content Ingestion` 清洗、标准化、去重并建立命中关系。
5. `Scoring Engine` 计算分值并更新 `monitor_post_hits`。
6. `Topic Aggregator` 形成主题并选择代表内容。
7. `Trend Analyzer` 生成监控级与主题级快照。
8. `Alert Engine` 判断是否生成提醒。
9. `Notification Center` 投递站内消息和邮件。
10. API 层提供统一查询结果。

# 数据模型

## 用户域

1. `users`
2. `user_profiles`

## 监控域

1. `keyword_monitors`
2. `keyword_monitor_keywords`
3. `monitor_runs`

## 内容域

1. `platform_posts`
2. `platform_authors`
3. `monitor_post_hits`

## 分析域

1. `topics`
2. `topic_posts`
3. `topic_snapshots`
4. `monitor_snapshots`

## 通知域

1. `alerts`
2. `user_notifications`
3. `email_deliveries`

## 关键约束

1. `platform_posts` 唯一键：`(platform, platform_post_id)`。
2. `monitor_post_hits` 唯一键：`(monitor_id, post_id)`。
3. `topics` 建议唯一键：`(monitor_id, topic_key)`。
4. 所有异步链路必须支持幂等重跑。

# API 边界

## 业务 API 接口

1. 认证：`/api/v1/auth/*`
2. 当前用户：`/api/v1/me`
3. 监控任务：`/api/v1/monitors`
4. 内容流：`/api/v1/monitors/{id}/posts`
5. 主题：`/api/v1/monitors/{id}/topics`
6. 趋势：`/api/v1/monitors/{id}/trends`、`/api/v1/topics/{id}/trends`
7. 通知：`/api/v1/notifications`

## 管理后台接口

1. `GET /api/v1/admin/users`
2. `GET /api/v1/admin/monitors`
3. `GET /api/v1/admin/runs`
4. `GET /api/v1/admin/connectors/x/status`
5. `POST /api/v1/admin/monitors/{id}/rerun`

# 非功能设计

## 调度

1. 轮询频率仅支持固定档位：`5/10/15/30` 分钟。
2. 默认调度采用应用内 scheduler + worker 任务分发。

## 限流与风控

1. 平台连接器必须维护独立速率限制器。
2. 请求日志需记录状态码、剩余额度、重试次数和错误类型。
3. 持续失败任务必须进入降级态，避免无限重试。

## 可观测性

至少要观测：

1. 任务执行成功率。
2. 平台采集延迟。
3. 主题聚合和趋势快照耗时。
4. 通知发送结果。

## 降级策略

1. 主题聚合失败时，内容流仍然可查询。
2. 趋势快照失败时，API 返回上一次有效快照。
3. 邮件发送失败时，站内提醒仍然落库可见。

# 验收门禁

设计文档完整的判定标准：

1. 能支撑单平台首版 MVP 端到端实现。
2. 能明确区分 API、Worker 和数据层边界。
3. 数据表已覆盖监控、内容、主题、趋势、提醒主链路。
4. 下游计划文档可以直接基于本设计进行拆分，不需要重新发明模块边界。

# 风险与边界

1. `X` 平台额度和稳定性是首版最大外部风险。
2. 主题聚合和趋势判断不能设计成强依赖 LLM，否则成本和可控性都过高。
3. 前后端仓库当前为空壳，首版计划必须包含项目脚手架与验证链路，不可默认已有基础设施。

# 待确认问题

1. 首版登录方式是邮箱密码优先，还是同时支持第三方登录。
2. X 接入的实际鉴权方式和额度方案。
3. 邮件服务商选择。
