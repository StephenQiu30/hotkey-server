# 001 V1 数据库设计

## 背景

HotKey V1 的 PRD 已经定义了账号、授权、主题、采集、内容处理、事件聚类、AI 摘要、Obsidian Git 同步、邮件、MinIO 留存、审计和删除等产品能力。进入 API、Repository、Worker 或 OpenAPI 实现前，必须先把这些能力落到数据库表结构中，避免端侧、接口和任务队列各自发明字段。

## 目标

1. `hotkey-server/db/schema.sql` 是 V1 数据库结构事实源。
2. 数据库设计覆盖所有 server V1 PRD 的核心状态模型。
3. 原始内容和派生结果分离，支持最小化留存。
4. 授权凭据只保存密文，不允许明文 token 入库。
5. 内容向量和事件向量使用 `pgvector`，维度固定为 `1536`。
6. 所有用户数据按 `user_id` 隔离；V1 不引入团队或企业 RBAC。

## 非目标

1. 不设计团队、组织、多租户成员体系。
2. 不设计计费、账单、套餐和企业权限。
3. 不把 MinIO 对象二进制存入 PostgreSQL。
4. 不把真实 token、cookie、API key 明文写入任何表。
5. 不把正式迁移工具绑定到当前阶段；后续可以从 `schema.sql` 拆分 migration。

## 核心设计

### 账号与授权

`users` 只表达个人账号，状态与 PRD 保持一致：`pending_email_verification`、`active`、`locked`、`deleting`、`deleted`。平台授权使用 `authorization_connections` 表达连接状态，密文凭据进入 `credential_secrets`。业务代码只能通过凭据服务解密，不能直接在日志、审计或 MinIO 对象里保存明文。

### 主题与来源

`monitored_topics` 保存关键词、排除词、语言、相似度阈值、采集频率和 Obsidian 输出目录。`topic_sources` 绑定主题与平台连接。RSS、微信公众号和新闻源作为用户可管理 feed，统一进入 `feed_sources`。

### 采集与内容

`collection_jobs` 记录采集窗口、幂等键、失败分类和重试策略。平台返回的原始响应进入 MinIO，并由 `storage_objects` 保存对象索引。标准化后的内容进入 `source_items`，再由 `topic_content_matches` 记录主题匹配、过滤原因和相似度分数。

### 去重与向量

去重分为 exact source id、canonical URL、text hash 和 near duplicate。去重组由 `content_dedupe_groups` 和 `content_dedupe_members` 保存。内容语义向量写入 `content_embeddings`，事件语义向量写入 `event_embeddings`，二者都使用 `vector(1536)`。

### 热点事件与 AI 摘要

`hotspot_events` 保存事件生命周期、热度、相关性、可信度、新鲜度和趋势分。`event_sources` 保存事件证据链。`ai_event_summaries` 保存结构化摘要 JSON、模型版本、prompt 版本、schema 版本和失败状态。

### Obsidian 与邮件

`obsidian_vaults` 保存 Git Vault 配置。`git_sync_jobs` 保存 Markdown 渲染、提交、冲突和 commit URL。`report_subscriptions` 保存日报/周报订阅，`report_runs` 保存每次生成记录，`email_deliveries` 保存邮件发送、重试和失败状态。

### 留存、审计与删除

`storage_objects.retention_policy` 控制 MinIO 生命周期：`raw_30_days`、`short_cache_7_days`、`derived_long_term`、`audit_required`。用户撤权、删除主题和删除账户通过 `deletion_requests` 与 `cleanup_tasks` 驱动，所有关键动作写入 `audit_events`。审计事件只保存必要元数据，不保存明文凭据。

## 验收门禁

1. `hotkey-server/db/schema.sql` 存在并包含 `CREATE EXTENSION IF NOT EXISTS vector`。
2. `schema.sql` 至少包含账号、授权、主题、采集、内容、embedding、事件、摘要、Git 同步、邮件、MinIO、审计和删除清理表。
3. 每个核心状态模型都有 CHECK 约束或等价字段约束。
4. 敏感凭据只出现在 `credential_secrets.encrypted_payload`。
5. 原始内容只通过 `storage_objects` 引用 MinIO 对象，不在 PostgreSQL 保存二进制文件。
6. 后续 API/Repository/Worker ticket 必须先引用本设计和 `schema.sql`。

## 风险与边界

1. `pgvector` 需要数据库环境安装扩展；本地和 CI 必须明确安装步骤。
2. 当前为完整 schema 文件，不是正式 migration 序列；引入迁移工具后要保持 `schema.sql` 与 migration 输出一致。
3. 国内平台授权和采集字段可能因接入方式变化而调整，变更前必须先更新本设计和 SQL。
4. V1 不支持团队协作。如果后续加入团队，需要新增组织、成员和资源权限模型，不能复用 `user_id` 直接冒充租户。
