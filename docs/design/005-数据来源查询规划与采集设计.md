# 数据来源、查询规划与采集设计

采集层把用户审核后的双语词项转换为可复用查询计划，并通过合规接口获取标准化来源数据。MVP 采用分层混合采集，但只接入官方 API、RSS、Atom 或明确授权的 Feed。

## 来源范围

MVP 来源分为三档：

| 接入级别 | 来源 | 条件 |
|---|---|---|
| 必选 | 中英文 RSS、Atom、授权新闻 Feed | 来源允许聚合，并遵守其保留和展示条款 |
| 必选 | 新闻聚合或新闻数据库官方 API | 已取得有效密钥和对应使用权限 |
| 必选 | YouTube Data API | 使用官方搜索接口，保存视频元数据和指标 |
| 条件接入 | X API | 账户具备可用搜索权限和足够配额 |
| 条件接入 | Reddit Data API | 商业用途、内容展示和数据保留已获许可 |
| 条件接入 | 微博等境内平台 | 仅使用官方开放接口或授权数据服务 |

不得绕过登录、验证码、反爬机制或平台访问限制。条件不满足的来源必须保持禁用，不能用非授权网页抓取补位。

来源能力和条款以官方文档为准：

- [X API 搜索说明](https://docs.x.com/x-api/posts/search/introduction)
- [YouTube Data API search.list](https://developers.google.com/youtube/v3/docs/search/list)
- [Reddit Data API Terms](https://redditinc.com/policies/data-api-terms)

## Connector 边界

每个来源实现一个小型接口：

```go
type Connector interface {
	Code() string
	Collect(ctx context.Context, request CollectRequest) (CollectResult, error)
	Health(ctx context.Context) HealthStatus
}
```

`CollectRequest` 包含：

```text
query_plan_id, query, language, region
start_time, end_time, cursor, limit
```

`CollectResult` 包含：

```text
items, next_cursor, has_more, quota
```

Connector 只负责：

- 来源认证、请求和响应解析
- 分页、增量游标和配额读取
- 将来源数据转换为 `SourceItem`
- 把来源错误分类为认证、限流、服务端、解析或永久错误

Connector 不负责匹配、聚类、热度和 AI 摘要。项目使用显式 `map[sourceCode]Connector` 注册实现，不建设动态插件、反射注册或通用采集 DSL。

## 统一来源项

`SourceItem` 是 Connector 与内容模块之间的内部契约：

```text
source_code, external_id, content_type
title, body, language, url, author
published_at, observed_at
metrics, source_entities, raw_payload
```

`external_id` 必须来自来源稳定标识。来源没有稳定 ID 时，使用规范化 URL 或来源规定字段生成可重复哈希，并记录生成方式。

## 词项到查询计划

用户创建 Monitor 后，系统执行：

1. 根据核心词生成中英文翻译、别名、实体名和相关词
2. 用户审核新增词项并配置排除词
3. 按来源支持的语法、语言、地区和窗口生成查询表达式
4. 对表达式生成稳定签名
5. 复用已有 QueryPlan，或创建新的 QueryPlan

稳定签名输入为：

```text
source + language + region + sorted_terms + sorted_excludes + time_window
```

词项必须先规范化、去重和稳定排序。签名不包含 Monitor ID，因此多个用户的相同查询可以共享采集结果。不同来源的查询语法在对应 Connector 内显式转换，不引入通用查询语言。

## 小时级调度

Cron 每小时触发查询规划和采集。单轮默认目标是 45 分钟内完成采集，为标准化、匹配、聚类和摘要保留至少 15 分钟。

调度步骤：

1. 读取当前有效且到期的 QueryPlan
2. 按来源分组，并在组内加入稳定抖动
3. 按来源并发上限和剩余配额执行
4. 为 QueryPlan 和时间窗口获取 PostgreSQL advisory lock
5. 保存 `collection_runs`、游标、数量和错误分类
6. 将 SourceItem 交给内容标准化模块

并发必须有界。每个来源设置独立并发上限、超时和每轮最大页数，不能为每个 Monitor 创建无限 goroutine。进程重启后根据 `collection_runs` 和 QueryPlan 游标继续执行。

## 幂等与增量

采集写入以 `(source_id, external_id)` 为第一层幂等键。同一内容再次出现时更新 `last_seen_at` 和最新指标，不重复创建内容。

每次运行保存：

- 查询窗口和实际完成时间
- 请求游标与返回游标
- 请求数、成功数、跳过数和失败数
- 消耗配额和来源返回的剩余配额
- 最后成功检查点

重试不能推进未成功保存的游标。来源只支持时间窗口时，相邻运行保留小范围重叠，并依赖幂等键消除重复。

## 错误、限流与熔断

| 错误类型 | 处理 |
|---|---|
| `401/403` | 标记凭据或来源不可用，停止自动重试并告警 |
| `429` | 尊重 `Retry-After`，暂停该来源并延后计划 |
| `5xx` 或网络错误 | 指数退避加抖动，单轮最多重试 3 次 |
| 单条解析错误 | 跳过该条，记录计数和脱敏样本 |
| 协议或字段永久变化 | 打开来源熔断，等待人工修复 |

连续失败超过来源阈值后打开熔断；健康检查成功或冷却时间结束后半开探测。一个来源失败不能阻塞其他来源。

配额不足时按以下顺序保留任务：

1. 新建或近期有用户活动的 Monitor
2. 处于快速增长事件关联的 QueryPlan
3. 普通周期查询
4. 宽泛趋势补充查询

## 凭据与合规

`data_sources` 保存来源代码、能力、状态、配额、协议版本、允许用途和保留期限，不保存明文密钥。密钥从环境变量或密钥管理服务读取，不得进入数据库、日志、错误响应或原始响应归档。

每个来源必须能配置：

- 是否允许正文保存和二次展示
- 原始响应、标准化内容和指标的保留期限
- 是否要求来源署名、回链或删除同步
- 允许的语言、地区、查询方式和速率

来源协议优先于系统默认策略。

## 验收门禁

- 相同查询签名只产生一个共享 QueryPlan
- 同一来源内容重试采集不会重复入库
- 每个来源的并发、超时、分页和配额可独立控制
- 认证失败、限流和服务端失败具有不同处理路径
- 单一来源失败不影响其他来源完成本轮采集
- 条件未满足的来源保持禁用，不存在绕过式抓取
- 日志、数据库和 API 响应不包含来源密钥
- 单轮采集能在小时流水线预算内结束或安全续跑
