# 后台任务与消息队列基础设施设计

> **日期：** 2026-07-09
> **状态：** 已实现
> **关联：** [014-日志系统设计规范](014-日志系统设计规范.md)

---

## 1. 背景

### 1.1 现状（改造前）

- 定时任务为单进程手写调度器——`time.Ticker` 每分钟轮询 + `ShouldRun()` 逻辑判断
- 无 cron 表达式支持，每个新定时任务都需要手写轮询逻辑
- 无异步任务队列，耗时操作直接阻塞调度循环
- 无失败重试和死信机制，任务失败仅有日志

### 1.2 目标

- 引入标准 cron 调度库，支持 cron 表达式配置
- 引入 Kafka 异步消息队列，实现生产-消费解耦
- 实现死信队列机制，失败消息可追溯
- 单实例部署，无需分布式协调
- 首批落地 Obsidian 日报生成 + 每小时热点聚合

## 2. 整体架构

```
robfig/cron/v3（定时触发）
    │  0 8 * * * → 日报 / 0 * * * * → 热点聚合
    │  每条 cron 触发 → 生产消息到 Kafka
    ▼
Kafka Producer → topic: hotkey.digest.run / hotkey.hourly.run
    │
    ▼
Kafka Consumer（ConsumerGroup: "hotkey-{domain}-worker"）
    │  Auto Commit（At-Least-Once）
    ▼
Dispatcher（按 msg.Type 路由到具体 Handler）
    ├── "digest.run"  → DailyObsidianPublishJob
    └── "hourly.run"  → HourlyAggregateJob
```

### 2.1 组件职责

| 组件 | 职责 |
|------|------|
| **robfig/cron/v3** | 只负责"到点了，生产一条消息"——不做具体业务 |
| **Kafka Producer** | 只负责"把消息写到 topic"——不决定何时触发 |
| **Kafka Consumer** | 只负责"收到消息 → dispatcher → handler" |
| **Dispatcher** | 按 `msg.Type` 路由到对应的 Handler 实现 |
| **Handler** | 执行业务逻辑（生成日报、采集数据等） |
| **Dead Letter（DLQ）** | 多次重试失败后，消息投递到 DLQ topic + 数据库记录 |

## 3. 消息模型

```go
type Message struct {
    ID         string    `json:"id"`          // UUID v7
    Type       string    `json:"type"`        // "digest.run", "hourly.run"
    Payload    []byte    `json:"payload"`     // 业务数据
    CreatedAt  time.Time `json:"created_at"`
    RetryCount int       `json:"retry_count"`
}
```

### 3.1 Topic 命名规范

`hotkey.{domain}.{action}`

| Topic | DLQ Topic | 用途 |
|-------|-----------|------|
| `hotkey.digest.run` | `hotkey.digest.run.dlq` | 日报生成 |
| `hotkey.hourly.run` | `hotkey.hourly.run.dlq` | 每小时热点聚合 |

### 3.2 Handler 接口

```go
type Handler interface {
    Type() string
    Handle(ctx context.Context, msg Message) error
}
```

## 4. 死信队列（DLQ）

Consumer 收到消息后：
- ✅ 成功处理 → commit offset
- ❌ 可重试错误 → retry_count < maxRetry（默认 3）→ 投递回原 topic（retry_count+1）→ commit offset
- ❌ 超过 maxRetry → 投递到 DLQ topic + 写入 `dead_letter_records` 表
- ❌ 不可恢复错误 → 直接投递 DLQ + commit offset
- 💥 系统 panic → 不 commit offset，重启重新消费

DLQ 记录表：

```sql
CREATE TABLE dead_letter_records (
    id              BIGSERIAL PRIMARY KEY,
    topic           VARCHAR(255) NOT NULL,
    message_id      VARCHAR(64)  NOT NULL,
    message_type    VARCHAR(64)  NOT NULL,
    payload         TEXT,
    error_message   TEXT,
    retry_count     INT          NOT NULL DEFAULT 0,
    created_at      TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);
```

## 5. 去重机制

利用 Redis SETNX 实现消息去重锁（TTL 24 小时），避免同一条消息被重复消费。

```go
type Dedupe struct {
    rdb *redis.Client
}
func (d *Dedupe) Seen(ctx context.Context, msgID string) (bool, error)
```

Handler 通过 `DedupeEnabled() bool` 声明是否需要去重。

## 6. Cron 调度配置

```go
// 日报：每天 08:00 Asia/Shanghai
c.AddFunc("0 8 * * *", func() {
    producer.Publish(ctx, TopicDigestRun, ...)
})

// 热点聚合：每小时整点
c.AddFunc("0 * * * *", func() {
    producer.Publish(ctx, TopicHourlyRun, ...)
})
```

配置通过环境变量控制：
- `DAILY_DIGEST_TIME`（默认 08:00）
- `DAILY_DIGEST_TIMEZONE`（默认 Asia/Shanghai）
- `KAFKA_BROKERS`（默认 localhost:9092）
- `KAFKA_CONSUMER_GROUP`（默认 hotkey-workers）

## 8. 设计决策

| 决策 | 理由 |
|------|------|
| Kafka 而非内存队列 | 持久化、可重试、不随进程重启丢失消息 |
| robfig/cron 而非手写 ticker | 标准 cron 表达式，零重复实现 |
| At-Least-Once + 去重锁 | 保证不丢消息；去重锁保证幂等 |
| DLQ 写数据库+独立 topic | 数据库便于管理查询；topic 可用于后续重放 |
