# 背景

`hotkey-server` 当前的数据层已经承载了用户、监控任务、采集内容、主题趋势、通知和日报等多个能力方向，但缺少一套面向新工程主线的数据设计权威文档。

在新架构下，数据库设计不仅要支撑现有业务目标，还要满足：

1. 统一的 ORM 主链路。
2. 清晰的业务实体边界。
3. 幂等、审计、趋势与导出等场景的长期可维护性。

# 目标

1. 明确系统的核心实体与表职责。
2. 明确实体关系、唯一约束、状态流转和审计字段要求。
3. 明确 `GORM` 主链路下的模型映射原则。
4. 明确复杂查询与 schema 迁移的设计边界。

# 非目标

1. 本文档不直接写出每一张表的最终 SQL 细节。
2. 本文档不定义具体 migration 文件步骤。
3. 本文档不将接口响应模型等同于数据库模型。

# 核心数据设计原则

新数据模型必须满足以下原则：

1. 以业务实体边界建模，而不是以当前代码文件结构建模。
2. 所有核心业务表必须具备统一审计字段。
3. 所有幂等、去重和重复执行保护场景必须通过数据库约束显式建模。
4. 平台原始采集事实、聚合结果、通知投递结果和日报导出结果必须分层存储。
5. ORM 是唯一主链路，但复杂查询封装能力必须被保留。

# 核心实体

建议冻结以下核心实体作为新主线的一等公民：

1. `User`
   - 用户账号、身份、偏好和通知相关基础信息。
2. `KeywordMonitor`
   - 用户创建的关键词监控任务及其运行配置。
3. `MonitorRun`
   - 每次采集或处理任务的运行记录与审计状态。
4. `PlatformPost`
   - 外部平台采集到的原始内容事实。
5. `MonitorPostHit`
   - 某条内容与某个监控任务的命中关系。
6. `Topic`
   - 聚合后的热点主题实体。
7. `TrendSnapshot`
   - 某主题或某监控对象在特定时间窗口下的趋势快照。
8. `Alert`
   - 规则判断后生成的提醒事件。
9. `Notification`
   - 对用户可见或可投递的通知记录。
10. `TopicDailyExport`
   - 日报导出与知识库沉淀结果。

# 实体关系

推荐的高层关系如下：

1. `User 1:N KeywordMonitor`
2. `KeywordMonitor 1:N MonitorRun`
3. `KeywordMonitor N:M PlatformPost`，通过 `MonitorPostHit` 建模命中关系
4. `KeywordMonitor 1:N Topic`
5. `Topic 1:N TrendSnapshot`
6. `Topic 1:N Alert`
7. `User 1:N Notification`
8. `Topic 1:N TopicDailyExport`

关系设计必须坚持：

1. 命中关系使用显式关联表，不将多职责强塞进单表。
2. 聚合结果不回写覆盖原始采集事实。
3. 导出结果和通知结果保持独立审计链。

# 审计与状态字段

所有核心表应遵循统一审计约定：

1. `created_at`
2. `updated_at`
3. 在需要时增加 `deleted_at`、`published_at`、`processed_at` 等阶段性时间字段

状态流转要求：

1. 有业务状态的表必须有显式状态字段。
2. 状态值必须是受控枚举，不允许由应用层随意拼写。
3. 与外部流程有关的结果表，如 `MonitorRun`、`Notification`、`TopicDailyExport`，必须包含失败状态和错误信息承载能力。

# 幂等与唯一约束

以下类型的场景必须通过数据库约束保证幂等或去重：

1. 用户邮箱等天然唯一身份标识。
2. 同一监控下的重复运行标识。
3. 同一内容与同一监控的重复命中关系。
4. 同一主题、同一日期的重复导出。
5. 其他重复执行会造成副作用放大的任务记录。

原则是：

1. 不能只依赖应用层 if-check 保证幂等。
2. 必须优先通过唯一键、组合唯一键或唯一索引建模。

# GORM 模型映射原则

`GORM` 是唯一 ORM 主链路，但模型职责必须分层：

1. ORM 模型
   - 用于表映射、字段标签、关联与数据库行为。
2. 领域模型
   - 用于业务规则、状态流转和服务层语义。
3. 接口响应模型
   - 用于 API 出参与前端契约。
4. 查询结果模型
   - 用于聚合、统计、趋势或日报等读模型查询结果承载。

不允许强行使用一个 struct 同时承担多种冲突职责。

推荐规则：

1. ORM 模型集中定义表名、索引、唯一键和关联关系。
2. 领域服务通过 repository 获取领域可用对象，而不是直接操作 ORM 细节。
3. HTTP 层不得直接序列化 ORM 模型作为最终对外响应。
4. 聚合型查询结果优先使用专门的查询结果模型，而不是滥用 ORM 实体。

# Repository 规则

数据访问统一通过 repository 边界暴露，对业务层提供稳定接口。

必须遵循：

1. service 层只依赖 repository interface，不直接依赖 `*gorm.DB`。
2. handler 和 worker 不直接拼接查询语句。
3. repository 内部负责事务与查询封装。

# 复杂查询策略

并非所有查询都应被简化为纯 CRUD ORM 调用。对于以下场景：

1. 热点统计
2. 趋势窗口聚合
3. 主题排序
4. 日报导出
5. 运维查询与审计列表

允许在 repository 内部使用：

1. `GORM` 的链式查询能力
2. `GORM Raw`
3. 必要的聚合 SQL 封装

但必须满足：

1. 原生 SQL 不得泄漏到业务层。
2. 复杂查询必须被封装为清晰的 repository 行为。
3. 读模型查询与写模型更新不能无边界混合。

# 迁移原则

数据库 schema 迁移采用正式 migration 主线，不将 `AutoMigrate` 作为生产唯一事实源。

迁移原则：

1. 以 migration 文件管理结构变化。
2. 以保留数据为优先。
3. 先兼容后收敛，避免一次性破坏旧数据路径。
4. ORM 模型必须服务于正式 schema，而不是反过来让 schema 漂移。
5. 在迁移完成前，旧 `db/schema.sql` 仅作为历史兼容材料存在，不再视为未来唯一 schema 权威来源。

# 未来扩展边界

本轮设计允许未来扩展，但必须避免过度预埋：

1. 不为尚未确认的平台来源预建大量空表。
2. 不为尚未确认的多租户模式提前复杂化主模型。
3. 不为尚未确认的 BI 或大数据能力提前引入重型建模。

未来扩展应优先基于：

1. 现有实体的字段扩展。
2. 新增清晰职责表。
3. 通过 migration 演进，而不是在首版模型中无限预留。

# 风险与边界

1. 如果复杂分析查询边界不清，`GORM` 迁移后可能出现性能退化和可读性下降。
2. 如果没有统一的 ORM 模型与领域模型分层，后续业务代码会被数据库字段细节污染。
3. 如果 migration 策略不先冻结，后续实现容易再次退回“代码里隐式改表”的混乱状态。

# 待确认问题

1. 现有 `db/schema.sql` 是否在过渡期继续保留为历史产物，还是迁移后完全由 `db/migrations/` 接替。
2. 软删除是否需要进入核心业务主线，还是只对个别表按需启用。
3. 趋势快照和日报导出在新模型中是否需要额外的物化读模型表。
