# 背景

`009` 已经把 OpenAPI 和 Worker 都定义成正式主线，但如果不明确契约维护方式和任务运行机制，后续执行计划会天然带着返工风险：API DTO 归位会反复改，Worker 的调度与部署边界也会失控。

# 目标

1. 冻结 OpenAPI 的唯一维护方式与产物路径。
2. 冻结 DTO、鉴权表达、错误码在契约中的表达规则。
3. 冻结 Worker 的运行主线、调度模型、单实例约束和幂等策略。
4. 为后续计划提供不会来回摇摆的关键技术决定。

# 非目标

1. 本文档不定义每个业务接口的字段细节。
2. 本文档不定义具体队列库或具体 cron 表达式实现代码。
3. 本文档不引入分布式多副本调度体系。

# OpenAPI 主线冻结

正式契约主线采用：

1. Gin handler 使用显式 DTO struct。
2. OpenAPI schema 与 route metadata 与这些 DTO 强绑定维护。
3. 生成产物统一落盘到 `docs/swagger.json`。

原则如下：

1. 服务端代码是契约事实源。
2. DTO、错误码、鉴权方案必须与运行代码同源维护。
3. Web 和 Miniapp 只从 `docs/swagger.json` 生成客户端，不手写服务端类型。

# DTO 归位规则

DTO 必须贴近业务模块归位，不集中堆在巨大公共包中。

推荐方式：

1. 模块内 `dto.go` 或 `http/dto.go`
2. 统一错误响应、分页响应可放在 `platform/http`

规则如下：

1. DTO 不直接复用 ORM 模型。
2. DTO 不直接等于领域模型。
3. 枚举字段必须与模块枚举保持一致。

# 鉴权与错误码在契约中的表达

OpenAPI 中必须显式表达：

1. Bearer/JWT 鉴权方式
2. 统一错误响应结构
3. 关键枚举约束

规则如下：

1. 所有需要登录的接口必须在契约中标记鉴权要求。
2. 公共错误响应结构保持统一。
3. 业务错误码不能仅出现在文档描述文字中，必须在接口规范中可追溯。

# Worker 运行主线冻结

当前阶段冻结为：

1. 单体应用内的正式 Worker 运行面
2. 单实例部署前提下的调度与消费机制
3. 任务通过 service 调用业务核心

不冻结为：

1. 分布式多节点调度系统
2. 独立微服务化 Worker 集群

这意味着：

1. 第一阶段默认单实例 `API + Worker` 运行。
2. 若未来进入多副本部署，必须新增专门设计文档处理 leader election、锁和竞争问题。

# 调度与执行模型

当前阶段采用“轻量调度器 + 统一任务执行入口”的方案，而不是先引入独立复杂队列平台。

推荐模型：

1. 定时任务由统一 scheduler 注册与触发。
2. 任务执行入口统一进入 job runner。
3. job runner 调用对应 service。
4. Redis 作为缓存、轻量队列辅助或短期互斥辅助，而不是当前阶段的分布式调度真相源。

选择理由：

1. 符合当前单实例主线。
2. 实现与运维复杂度更低。
3. 满足当前工程化要求，不提前引入过重基础设施。

# 单实例边界

当前阶段必须明确写死：

1. 生产与联调目标架构默认单实例。
2. 同一套 scheduler 不允许在多实例无保护并发运行。
3. 所有文档、计划和验收都按单实例边界设计。

如果部署层未来出现多副本需求，必须先补：

1. leader election 或互斥锁策略
2. 重复消费保护
3. 幂等执行验证

# 任务幂等与重试

即使在单实例下，任务仍需具备幂等与失败处理能力。

最低要求：

1. 关键任务有唯一运行标识或去重键。
2. 通知、日报导出、聚合写入等副作用任务必须有数据库级或业务级幂等保护。
3. 任务失败必须记录原因。
4. 可配置重试次数和退避策略。

# 运行与审计要求

Worker 必须具备：

1. 启动日志
2. 每类任务的执行日志
3. 失败日志
4. 人工重跑入口
5. 运行结果审计记录

# 验收门禁

1. OpenAPI 维护方式被明确冻结，不再处于“注释/显式 schema/混合”未决状态。
2. Worker 运行机制被明确冻结为单实例轻量调度主线。
3. 多副本未覆盖能力被明确列为未来设计边界，而不是隐含假设。

# 风险与边界

1. 如果后续执行偏离单实例边界，重复调度和重复通知会成为高风险问题。
2. 如果 DTO 与 OpenAPI 维护不统一，Web 和 Miniapp 客户端会迅速失配。

# 待确认问题

1. 是否需要在第一阶段即开放人工重跑 API，还是先保留内部运维入口。
