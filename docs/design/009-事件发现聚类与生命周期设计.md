# 事件发现、聚类与生命周期设计

## 1. 目标

把已通过 Monitor 相关性判断的多来源 Content 稳定地归入同一现实事件，同时保留独立证据、人工修正和可重现的决策过程。

V1 必须满足：

1. 全平台只存在一个全局 `Event` 概念。
2. 事件先由确定性规则和向量候选决策，LLM 只处理边界样本。
3. 同一事件的独立报道作为证据保留，不作内容去重。
4. 每次成员变更、合并和拆分都可追溯、可重试、可人工覆盖。

## 2. 非目标

- 不使用 LLM 对全量内容两两比较。
- 不让热度或用户关注度反向改变事件事实边界。
- 不在 V1 建设通用知识图谱推理或无界社区发现。
- 不自动覆盖人工锁定的事件字段、成员和合并关系。

## 3. 输入与前置条件

只有满足以下条件的 Content 可进入正式聚类：

- `content_status = active`
- 至少一个 `monitor_matches.decision = accepted`
- 已有规范化时间、主语言和内容哈希
- 具有可用的标题、摘要或正文片段
- 来源删除、无效和确认重复的内容不参与新聚类

输入特征由确定性处理产生：

```text
published_at, language, normalized_title
confirmed_entities, normalized_actions, locations
content_embedding, source_independence_key
monitor_ids, duplicate_of_id, algorithm_versions
```

## 4. 事件候选召回

禁止将新 Content 与全部 Event 全量比较。候选 Event 由下列通道取并集：

1. 共享高置信实体和关键动作的活动事件。
2. 同地区且事件向量近邻的活动事件。
3. 共享实体、时间和稳定事实指纹的已关闭事件。

默认时间窗口：

| 事件类型 | 默认窗口 | 说明 |
|---|---:|---|
| 快讯、发布、故障 | 72小时 | 强调时间边界 |
| 政策、诉讼、产品演进 | 30天 | 允许持续进展 |
| 人工指定长期事件 | 90天 | 必须有明确分类或人工设置 |

窗口和候选数属于版本化 `clustering_profile`，不在业务代码中散落硬编码。V1 每条 Content 最多进入 50 个候选事件的精排。

## 5. 事件指纹与稳定键

`event_key` 是事件创建时由 `IDGenerator` 生成的不可变业务ID，不从标题或模型输出推导。事实指纹只用于候选召回和并发创建防重，不作为永久业务ID。

```text
event_key = "evt_" + sortable_unique_id

fingerprint_input =
  sorted(high_confidence_entity_keys)
  + normalized_primary_action
  + normalized_location
  + event_time_bucket

event_fingerprint = sha256(fingerprint_version + fingerprint_input)
```

规则：

- 缺少实体或动作时不生成过度宽泛的指纹，直接依赖其他候选通道。
- 后续标题、摘要或代表内容变化不改写 `event_key`。
- 指纹不是唯一约束；同指纹可以对应不同现实事件，必须继续执行成员决策。
- 合并后保留原事件键并指向 `merged_into_id`。
- 拆分后新事件使用新键，审计记录保留派生关系。

## 6. 成员决策

候选精排统一输出 `membership_score 0-100`：

```text
membership_score =
  entity_action_score * 0.30
  + semantic_score * 0.30
  + temporal_score * 0.20
  + location_score * 0.10
  + source_context_score * 0.10
```

默认决策：

| 分数 | 行为 |
|---|---|
| `>= 80` | 直接加入最佳候选事件 |
| `65-79.99` | 规则信息不足时交给 LLM 边界复核 |
| `< 65` | 不加入已有事件，必要时创建新事件 |

如多个候选都达到阈值，只加入最高分事件；前两名差值小于 5 时进入人工或 LLM 边界复核。明确的核心实体、动作或时间冲突具有一票否决权，不得被高向量相似度覆盖。

## 7. 新事件创建

无候选通过时创建 `detected` 事件，首条 Content 为临时代表证据。

新事件不因单条内容立即变为高可信；以下任一条件满足后进入 `active`：

- 至少两个独立来源支持相同核心事实。
- 单一权威一手来源且相关性达到高置信阈值。
- 经人工确认。

`source_independence_key` 使用真实发布组织或原始作者识别，转载网站不计为新的独立来源。

## 8. 生命周期

```text
detected -> active -> cooling -> closed -> archived
    |          |          |         |
    +-------> rejected    +-------> active
active/cooling/closed -> merged
```

| 状态 | 进入条件 | 允许行为 |
|---|---|---|
| `detected` | 首次发现 | 继续收集证据，不默认进入高可信报告 |
| `active` | 证据或人工确认达标 | 更新摘要、热度、实体和主张 |
| `cooling` | 新证据和热度持续下降 | 新证据可重新激活 |
| `closed` | 超过类型窗口且无新信息 | 保留查询，高质量后续可重新激活 |
| `merged` | 人工或高置信合并 | 只读并重定向目标事件 |
| `archived` | 人工归档或保留策略 | 不参与新候选召回 |
| `rejected` | 误报、无事件性或证据失效 | 不展示，保留审计 |

自动迁移由版本化规则和 `Clock` 驱动。所有迁移必须比较 `version`，并在运行或审计记录中保存原状态、新状态和原因码。

## 9. 合并与拆分

### 9.1 合并

自动合并只对高置信且没有人工锁的事件开放；V1 默认使用人工确认。合并在一个 PostgreSQL 事务内：

1. 锁定源事件和目标事件。
2. 以唯一约束幂等迁移成员、Monitor、实体和主张关系。
3. 保留原事件，设置 `merged_into_id` 和 `merged`。
4. 重算目标事件代表内容、热度、摘要和知识投影。
5. 写入审计记录和后续幂等任务。

### 9.2 拆分

拆分必须由人工发起或确认，选定成员迁移到新事件。原事件与新事件都重算派生数据，已发布报告保留冻结快照，不改写历史。

## 10. 人工修正优先级

- `event_contents.manual_locked = true` 时，自动任务不得移除或改变成员归属。
- `events.manual_locked = true` 保护人工标题、摘要、代表内容和生命周期。
- 人工拒绝的成员对写入稳定负样本，相同算法版本不反复建议。
- 人工调整不删除模型决策证据，只新增最终决策和审计记录。

## 11. 迟到、删除与重算

- 迟到内容使用原始 `published_at` 进入候选，但记录实际入库时间。
- 已关闭事件遇到同一事实的高质量后续时可重新激活；独立新阶段应创建新事件并通过 Topic 关联。
- Content 删除后移除其活动证据关系，重算事件可信度、代表内容、热度、主张和摘要。
- 事件失去全部有效证据后进入 `rejected` 或人工复核，不硬删除审计历史。

## 12. 幂等、并发与事务

- 成员决策幂等键：`content_id + clustering_profile_version + feature_input_hash`。
- `event_contents(event_id, content_id)` 和 `monitor_events(monitor_id, event_id)` 使用唯一约束。
- 创建事件、加入成员、关联 Monitor 和提交后续任务使用一个数据库事务。
- 事件更新使用 `version` 乐观锁；合并和拆分使用稳定顺序行锁。
- 重试不得创建重复事件、成员关系或下游任务。

## 13. 评测

固定标注集至少覆盖：

- 中英文对同一事件的独立报道
- 同一主题不同时间的独立事件
- 同名实体、转载、更正、撤回和迟到内容
- 需要合并和拆分的边界案例

数据集按现实事件分组切分，防止同一事件同时进入调参集和验收集。保存数据集版本、时间范围和标注冲突决议。

## 14. 验收门禁

- 验收集跨语言聚类 `F1 >= 75%`。
- 确定性规则和向量可处理主链路，LLM 不可用时仍能创建和更新事件。
- 相同输入重跑不产生重复 Event 或成员关系。
- 合并、拆分、删除证据和状态迁移具有事务与故障测试。
- 人工锁定字段和成员不会被后续模型任务覆盖。
- 每个决策能回溯到特征输入、算法版本、证据和最终决策者。
