# 身份认证、会话与权限设计

## 1. 目标与范围

本设计为所有后续业务 API 提供一个可验证、可撤销且不泄露凭据的身份边界。`identity` 模块拥有 `users`、`user_preferences`、`auth_sessions` 和新增的 `auth_refresh_tokens`；其他模块只能消费认证主体和角色检查接口，不得直接读取这些表。

首版只支持邮箱密码认证：邮箱验证码后注册、密码登录、刷新、退出、当前主体、修改密码、重置密码以及管理员用户生命周期管理。它不实现 OAuth、SSO、MFA、多租户、通用策略语言或任意用户 CRUD。

## 2. 关键决定

| 决定 | 契约 |
|---|---|
| 注册 | 用户必须先证明邮箱控制权，再设置密码；公开注册一律创建 `viewer`。 |
| 密码 | 只保存 `bcrypt` 哈希；明文密码、验证码和刷新凭据绝不写日志、审计 `before_data/after_data`、数据库或 HTTP 响应。 |
| Access Token | 15 分钟 HS256 JWT，经 `Authorization: Bearer` 传入；只含 `sub`、`sid`、`iss`、`aud`、`iat`、`nbf`、`exp` 与 `jti`，角色从当前数据库用户记录读取。 |
| Refresh Token | 32 字节以上密码学随机不透明值；浏览器仅以 `HttpOnly` Cookie 传输，数据库仅保存 SHA-256 哈希。 |
| 会话事实 | PostgreSQL 是用户状态、会话撤销、刷新轮换和重放处置的唯一事实源；JWT 不能单独证明会话仍有效。 |
| 时限 | 每次成功刷新将有效期滑动至 `min(now + 7d, session.absolute_expires_at)`；会话绝对上限为首次登录后 30 天。 |
| Redis | 仅存验证码、单用途验证票据、发送频率和短锁。Redis 故障不能使既有登录、Access Token 校验、刷新、退出或撤销产生错误的认证结论。 |
| 初始管理员 | 仅 `hotkey user bootstrap-admin` 可以在没有任何未删除用户时创建一个管理员；它只读取 `HOTKEY_BOOTSTRAP_ADMIN_EMAIL` 和 `HOTKEY_BOOTSTRAP_ADMIN_PASSWORD`，事务中串行检查，不能被 HTTP 调用。 |

JWT 使用 `github.com/golang-jwt/jwt/v5` 的 `RegisteredClaims`，解析时固定只接受 HS256 并校验 issuer、audience、签名和时间声明。bcrypt 使用 `GenerateFromPassword` 和 `CompareHashAndPassword`；密码输入限制为 UTF-8 且不超过 bcrypt 的 72 字节上限，避免哈希函数截断不同密码。

## 3. 模块与请求流

```text
HTTP transport -> identity application -> identity domain <- postgres / redis / SMTP adapters
                         |
                         +-> shared authentication subject for platform HTTP middleware
```

`transport/http` 只绑定 DTO、设置 `module=identity`、读取认证主体并通过现有 `Wrap`/Result 输出。`application` 拥有事务、验证票据消费、会话轮换、审计和角色用例；`domain` 定义用户、角色、会话、端口和错误；PostgreSQL、Redis、SMTP、bcrypt 与 JWT 均为 infrastructure 适配器。

平台 HTTP 不导入 identity 的具体类型。它依赖一个窄认证端口：输入 Bearer Token，输出 `Subject{UserID, SessionID, Role}` 或已注册的 `AppError`。该端口在验证 JWT 后读取 PostgreSQL 的会话和用户状态；停用、软删除、退出、改密或重放撤销会在下一次请求立即生效。

中间件顺序保持设计 004：请求 ID、trace、访问日志、恢复、CORS、超时、认证、授权、Handler、`Wrap`。公开认证端点跳过认证；认证后路由把 `Subject` 放入请求上下文。`RequireRoles(admin)` 是 identity transport 的显式 Gin group/handler 中间件，不建立通用策略引擎。

## 4. 用户与角色生命周期

`users.email` 在 trim 并转为小写后写入，且仅在 `deleted_at IS NULL` 内唯一。`active` 用户可以登录；`disabled` 与软删除用户不能获得或刷新会话。`user_preferences` 在创建用户的同一事务中创建，默认 `UTC`/`zh-CN`。

角色是有限枚举：

| 角色 | 首版权限 |
|---|---|
| `viewer` | 登录、刷新、退出、查看自身、修改自己的密码；后续模块的只读业务接口。 |
| `editor` | `viewer` 权限及后续模块的内容维护接口。 |
| `admin` | `editor` 权限及用户列表、角色、停用、软删除和恢复。 |

首版不把未来业务资源的授权规则提前写入 identity。管理员只能修改角色和状态，不能修改他人邮箱、密码或偏好。任何停用、软删除和恢复均写入 `audit_logs`，并撤销受影响用户的所有会话；恢复后的用户保持 `disabled`，必须由管理员显式启用。

系统始终保留至少一个未删除、`active` 的管理员。管理员对自己或他人的角色、状态、删除操作在同一 PostgreSQL 事务内锁定相关用户并检查该不变量；试图移除最后一个活动管理员返回稳定冲突错误。`bootstrap-admin` 使用同一约束，且只能在用户表为空时运行。

## 5. 密码、邮箱验证和恢复

注册和密码重置共用目的受限的邮箱验证流程：

1. 客户端向 `POST /api/v1/auth/email-verifications` 提交规范化邮箱与 `purpose`（`registration` 或 `password_reset`）。响应固定为成功受理，避免暴露邮箱是否已注册。
2. identity 在 Redis 中保存验证码哈希、用途、邮箱哈希、失败次数、过期时间和发送冷却期；163 SMTP 适配器仅发送验证码。真实密码与验证码均不进入 Zap、trace、metrics 或审计字段。
3. 客户端向 `POST /api/v1/auth/email-verifications/confirm` 提交邮箱、用途和验证码。原子消费成功后返回短期、单用途、不透明的 `verification_ticket`。
4. `POST /api/v1/auth/registrations` 仅接受 `registration` 票据并创建 `viewer`；`POST /api/v1/auth/password-resets/confirm` 仅接受 `password_reset` 票据、更新密码哈希并撤销该用户所有会话。

票据、验证码和频率状态只在 Redis 存在。Redis 或 SMTP 不可用时，这些新建邮箱流程返回 `90001`/HTTP 503；不会用内存降级、不会把验证码落库，也不会影响已建立会话的认证正确性。163 是首个 SMTP 配置示例，但 SMTP 主机、端口、TLS 模式、账号、授权码和发件人均从运行配置读取，密钥不进入仓库。

修改密码要求已认证主体提供当前密码；成功后更新 bcrypt 哈希、撤销该用户全部会话、清除当前 Refresh Cookie，并写审计记录。登录对不存在邮箱、错误密码、禁用或已删除账户统一返回 `invalid credentials`，不区分原因。

## 6. 会话与刷新轮换

一个 `auth_sessions` 记录代表一个可撤销登录会话（也是一个刷新族），而不是单一 Refresh Token。其目标字段为：

| 表 | 字段与约束 |
|---|---|
| `auth_sessions` | `id`、`user_id`、`family_id uuid unique`、`absolute_expires_at`、`last_seen_at`、`revoked_at null`、`revoke_reason null`、`created_at`；`absolute_expires_at > created_at`。 |
| `auth_refresh_tokens` | `id`、`session_id`、`token_hash char(64) unique`、`expires_at`、`used_at null`、`revoked_at null`、`created_at`；令牌有效期不得超过所属会话绝对上限。 |

登录在一个事务内确认活动用户和密码、创建会话、创建一个未使用 Refresh Token，并更新 `last_login_at`。刷新按以下顺序在一个事务中进行：

1. 对 Refresh Token 的 SHA-256 哈希行使用 `FOR UPDATE`；令牌不存在、过期或已撤销时返回 `session invalid`。
2. 读取并锁定父会话与用户，检查会话未撤销、未超过绝对上限且用户仍为 active、未删除。
3. 已被使用的令牌视为重放：撤销整个会话、撤销该会话所有仍有效 Refresh Token、写失败审计并返回 `session invalid`。
4. 对首次使用令牌写入 `used_at`，创建一个新的唯一 Refresh Token，更新 `last_seen_at`，签发新 Access Token 并设置新 Cookie。

因此并发两个刷新请求只能有一个成功；旧 Refresh Token 不能重复使用；发现重放即使攻击者已取得后续 Token，也会撤销整条会话。每次受保护 API 请求还会验证 JWT 的 `sid` 对应未撤销会话及活动用户，因此退出、改密、停用、删除和重放处置立即失效既有 Access Token。

`POST /api/v1/auth/logout` 优先依据有效 Access Token 撤销指定会话；Access Token 已过期时可依据有效 Refresh Cookie 撤销同一会话。无有效凭据时仍清除 Cookie 并返回空成功，避免泄露会话存在性。

Refresh Cookie 名为 `hotkey_refresh`，限定 `Path=/api/v1/auth`、`HttpOnly`、`SameSite=Strict`，生产环境必须 `Secure`。Access Token 只在 Result `data` 返回，Web 客户端只存于内存。Cookie 认证端点校验允许的 Origin；CORS 不得再以 `*` 和凭据组合。浏览器跨源使用必须显式配置允许源和凭据，且不得放宽 Cookie 属性。

## 7. HTTP 契约

所有接口使用设计 004 的 `Result{code,message,data}`。请求和响应 DTO 与领域实体分离；用户响应永不含 `password_hash`、Refresh Token、验证码、验证票据、审计原文或数据库错误。

| 路由 | 认证 | 结果 |
|---|---|---|
| `POST /auth/email-verifications` | 公开 | 受理验证码发送，固定成功语义。 |
| `POST /auth/email-verifications/confirm` | 公开 | 返回单用途验证票据。 |
| `POST /auth/registrations` | 公开 | 验证票据创建 viewer 与偏好，HTTP 201。 |
| `POST /auth/login` | 公开 | 返回 Access Token、用户 DTO，并设置 Refresh Cookie。 |
| `POST /auth/refresh` | Refresh Cookie | 轮换凭据，返回新 Access Token 和用户 DTO。 |
| `POST /auth/logout` | 可选 Access/Refresh | 撤销当前会话并清除 Cookie。 |
| `GET /auth/me` | 任意活动角色 | 返回当前用户 DTO。 |
| `POST /auth/password` | 任意活动角色 | 改密、撤销所有会话、清除 Cookie。 |
| `POST /auth/password-resets/confirm` | 公开 | 用 reset 票据改密并撤销所有会话。 |
| `GET /users` | admin | 仅列出安全用户 DTO。 |
| `PATCH /users/{id}` | admin | 仅修改 `role` 或 `status`。 |
| `DELETE /users/{id}` | admin | 软删除并撤销会话。 |
| `POST /users/{id}/restore` | admin | 恢复为 disabled，不恢复会话。 |

身份领域新增稳定业务码，不改变既有 `20000` 未认证、`20001` 禁止、通用校验和限流码的语义：

| 业务码 | HTTP | 安全消息 |
|---|---:|---|
| `20002` | 401 | `invalid credentials` |
| `20003` | 401 | `session invalid` |
| `20004` | 400 | `verification invalid` |
| `20005` | 409 | `last active admin` |

验证码过频使用既有 `10004`/429；外部邮件、Redis 或数据库临时不可用使用既有 `90001`/503。内部 Cause 只留在受控日志。

## 8. 审计、可观测性与安全边界

成功和失败的登录、刷新重放、退出、改密、重置密码、管理员角色/状态/删除/恢复操作均写 `audit_logs`。审计仅记录 actor、资源、动作、结果、request ID、trace ID 及经过白名单过滤的状态变更；绝不记录邮件验证码、密码、Token、Cookie 或哈希。

HTTP 日志、指标和 trace 保持设计 004 的 `request_id`、route、module、status 与耗时，并可添加不含个人数据的认证结果标签。认证失败不得记录完整 Authorization、Cookie、邮箱、JWT、Refresh Token 或 bcrypt 错误。

## 9. 测试和验收边界

实施前 Plan 必须为以下每项定义红灯、绿灯和长期验收证据：

- 邮箱规范化、验证码限流、票据原子消费、注册和重置的正常/错误/过期路径。
- bcrypt 72 字节边界、错误凭据不可枚举、密码修改与重置撤销所有会话。
- JWT 签名算法拒绝、issuer/audience/时间声明、撤销/禁用/软删除后 Access Token 立即失败。
- 两个并发刷新只有一个成功，旧 Token 重放撤销整条会话，7 天滑动与 30 天绝对上限准确。
- 最后活动管理员保护、管理员生命周期审计、软删除后的重注册和恢复冲突。
- Cookie 的 HttpOnly/SameSite/Secure/Path、Result/HTTP/业务码、OpenAPI、日志与审计脱敏。
- PostgreSQL 真实事务测试；Redis 可用与不可用时分别验证新邮箱流程的明确 503 和既有认证流程正确性。

## 10. 后续同步

本设计获接受后，必须同步修订设计 003 的 `auth_sessions` 数据目录、完整 `db/schema.sql`、数据库记录模型、错误码、运行配置、OpenAPI、PRD-004、PLAN-004 与验收文档。没有这些可执行事实和通过的架构/契约校验，不得把身份能力描述为已交付。
