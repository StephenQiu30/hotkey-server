# 20-日报邮件输出管线实现计划

## 1. 目标

让热点结果可以稳定生成日报并进入邮件发送闭环，形成可追踪、可失败恢复的输出链路。

## 2. 文件清单

- 修改：`internal/service/report/`
- 修改：`internal/service/mail/`
- 修改：`internal/scheduler/`
- 修改：`internal/worker/`

## 3. 任务拆解

1. 为日报生成、邮件发送成功/失败和降级行为写失败测试。
2. 串联 `generate_report` 与 `send_daily_email` 任务。
3. 接通日报保存和投递记录。
4. 增加失败重试和降级输出。
5. 运行验证并回写结果。

## 4. TDD 与验证

- 日报和邮件主链路可运行。
- 发送结果和失败状态可追踪。
- AI 与 SMTP 失败行为有明确降级策略。

## 5. 执行顺序

1. `test:` 日报生成、邮件成功/失败、降级行为测试。
2. `impl:` 任务串联、投递记录与重试逻辑。
3. `refactor:` 统一 report 与 mail job 状态模型。

## 6. 回滚策略

回滚任务串联，保留 report 与 mail service 独立能力；已生成日报保留，未发送邮件可重试。

## 7. 验收命令

```bash
go test ./internal/service/report/...
go test ./internal/service/mail/...
go test ./internal/scheduler/...
gofmt -w cmd internal
go test ./...
python3 -m unittest discover -s tests
```

## 8. Symphony / Linear 要求

任务状态、标签和流转规则完全以本仓库 `WORKFLOW.md` 和本地 Symphony 实现为准。Plan 不定义额外状态、不发明额外标签、不覆盖 Symphony 的状态机。

Linear issue 只承载任务内容：PRD 路径、Plan 路径、任务范围、禁止范围、TDD 验收命令和回写要求。Symphony 负责监听 active states、创建 workspace、运行 Codex，并按 `WORKFLOW.md` prompt 驱动执行。
