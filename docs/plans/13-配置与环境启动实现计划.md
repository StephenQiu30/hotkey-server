# 13-配置与环境启动实现计划

## 1. 目标

建立统一的 env 配置加载与校验能力，为后续 runtime、worker、scheduler 和外部依赖接线提供稳定输入。

## 2. 文件清单

- 修改：`internal/config/config.go`
- 修改：`.env.example`
- 修改：`README.md`
- 创建：`internal/config/config_test.go`（如尚不存在或需扩展）

## 3. 任务拆解

1. 为配置缺失、格式错误和默认值回退写失败测试。
2. 扩展 `internal/config` 的配置结构和解析逻辑。
3. 补齐 `.env.example` 以及最小使用说明。
4. 增加启动前配置校验和降级标识。
5. 运行完整验证并整理回写信息。

## 4. TDD 与验证

- 关键配置缺失时启动应快速失败并给出明确提示。
- 默认值回退行为由测试覆盖。
- 可选能力应明确 `disabled` 或 `degraded`，避免静默失败。

## 5. 执行顺序

1. `test:` 配置缺失、格式错误、默认值回退失败测试。
2. `impl:` 扩展 config 结构、解析与启动校验。
3. `docs:` 更新 `.env.example` 与 README 说明。

## 6. 回滚策略

回滚 config 结构扩展与校验逻辑，保留原有最小配置字段；不影响已持久化数据。

## 7. 验收命令

```bash
go test ./internal/config/...
gofmt -w cmd internal
go test ./...
python3 -m unittest discover -s tests
```

## 8. Symphony / Linear 要求

任务状态、标签和流转规则完全以本仓库 `WORKFLOW.md` 和本地 Symphony 实现为准。Plan 不定义额外状态、不发明额外标签、不覆盖 Symphony 的状态机。

Linear issue 只承载任务内容：PRD 路径、Plan 路径、任务范围、禁止范围、TDD 验收命令和回写要求。Symphony 负责监听 active states、创建 workspace、运行 Codex，并按 `WORKFLOW.md` prompt 驱动执行。
