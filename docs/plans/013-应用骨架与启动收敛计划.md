# 背景

当前仓库存在多套启动叙事与装配路径。新主线必须先收敛应用骨架，才能安全迁移 HTTP、ORM 和 worker。

# 目标

1. 建立 `cmd/hotkey/main.go` 唯一正式入口。
2. 建立 `internal/app` 统一装配与生命周期管理。
3. 将正式启动叙事收敛为标准 Go 入口与 Docker Compose。
4. 为后续 Gin、GORM 与 worker 收敛提供稳定骨架。

# 非目标

1. 本计划不直接完成 Gin 路由迁移。
2. 本计划不直接完成 GORM repository 迁移。
3. 本计划不直接删除所有遗留实现，但会切断其正式入口身份。

# Task 1: 建立唯一入口与应用装配骨架

目标：

1. `cmd/hotkey/main.go` 成为唯一正式入口。
2. `internal/app` 承担配置加载、资源初始化、启动和关闭。

关键交付：

1. 唯一入口文件。
2. 应用装配骨架。
3. 生命周期关闭流程。

验证门禁：

```bash
go build ./cmd/hotkey
```

# Task 2: 收敛配置、日志与资源初始化

目标：

1. 配置加载只保留一条正式入口。
2. 日志、数据库、Redis 等基础依赖由 app 统一初始化。
3. worker 与 API 共用同一套核心依赖。

验证门禁：

```bash
go test ./internal/config ./internal/app ./internal/observability ./...
```

# Task 3: 收敛正式启动方式与 Docker 叙事

目标：

1. README 只保留一套推荐运行路径。
2. Docker Compose 成为联调与部署基线。
3. 冗余脚本降级为迁移期工具，而非正式入口。

验证门禁：

```bash
docker compose config
rg -n "make dev|scripts/dev.sh|scripts/start-local.sh" README.md docs/operations scripts
```

# Task 4: 为后续迁移准备应用接口

目标：

1. 为 Gin router 注入保留明确装配点。
2. 为 worker runtime 注入保留明确装配点。
3. 为 GORM database init 保留明确装配点。

验证门禁：

1. app 层接口稳定。
2. 后续 `014/015/016` 不需要重写入口骨架。

# 风险与边界

1. 若本阶段不先完成，后续 HTTP 与 ORM 迁移会反复改入口。
2. 若保留多套正式启动叙事，用户目标会再次被破坏。
