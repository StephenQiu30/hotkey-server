# 事件聚类、生命周期与人工治理计划

## 计划目标

把已经通过 Monitor 相关性判断的 active Content 聚合为可追溯的全局 Event，并保证重跑、迟到证据、生命周期迁移、合并、拆分和人工锁在 PostgreSQL 单体架构内保持幂等和事务一致。Plan 010 只消费 PLAN-009 的 accepted 匹配，不重新实现相关性或 AI Provider。

## 开工条件

- 本 Plan 为 `status: accepted`、`review_status: approved`、`execution_status: ready`。
- PRD-010 为 `status: accepted`、`execution_status: ready`。
- PLAN-009 为 `archived/done`，工作区干净且当前分支已同步主线基线。
- PostgreSQL/Redis 只使用可丢弃的 `.env` 验收配置；生产配置仅由 `.env.prod` 提供，不新增第三套环境或运行时 DDL。
- 每个任务开始前先执行 `git status --short`、`git diff --check`，staging 只允许当前任务文件。

## 不可突破的边界

- 候选召回最多 50 个 Event；不得做 Content×Event 全表比较。PLAN-010 只使用自己的确定性输入：Content 的规范化标题/摘要/语言/发布时间/来源、PLAN-009 accepted Monitor match 的规则/语义分项、现有同 profile 的 content/event embedding（可选）和版本化事实指纹；不读取 PLAN-012 的 entities、relations 或 claims 表。
- `event_key` 由 `IDGenerator` 创建且不可变；`event_fingerprint` 只用于召回，不能替代业务唯一键。
- membership score 固定为 entity/action 0.30、semantic 0.30、temporal 0.20、location 0.10、source/context 0.10；`>=80` 接受、`65–79.99` 灰区、`<65` 新建或拒绝，硬冲突一票否决。
- LLM 仅可处理灰区，必须可降级为确定性决策；不新增 `task_type`，不把 Provider SDK 类型带入 domain/application。
- 事件、成员、Monitor 关系和审计写入同一 PostgreSQL 事务；合并/拆分按稳定事件键顺序加行锁。
- `manual_locked` 的事件字段和成员关系不能被自动重算覆盖；删除证据后必须重算，不物理删除审计或合并历史。
- 本计划只迁移 Event 自有的 `event_contents`、`monitor_events` 和 `events.merged_into_id`；entities、claims、topics、reports 等关系由 PLAN-012/014/015 通过本计划的 append-only `downstream_reconcile_required` 审计事件消费，不在本计划直接写入。
- 本计划不实现热度趋势、实体/Claim、任务队列/调度、摘要、知识库和投递。

## 固定契约与测试夹具

### 需要落地的持久化契约

`event_clustering_decisions` 为 append-only business table，列顺序固定为 `id, content_id, candidate_event_id, candidate_event_key, clustering_version, feature_input_hash, channel, candidate_rank, entity_action_score, semantic_score, temporal_score, location_score, source_context_score, membership_score, decision, decision_origin, reason_codes, feature_snapshot, evidence_content_ids, actor_user_id, created_at`；`feature_snapshot` 只保存有限数值/版本/来源，不保存完整正文、Prompt 或 Provider 原文。对 `(content_id, clustering_version, feature_input_hash, candidate_event_key)` 建唯一索引，`candidate_event_key` 对新事件使用固定值 `__new_event__`，从而让重跑和拒绝结果可审计且幂等。

`event_governance_audits` 为 append-only retained table，列为 `id, event_id, action, actor_user_id, reason_code, from_status, to_status, source_event_id, target_event_id, expected_version, metadata, created_at`；`action` 至少包含 `lifecycle_transition`、`merge`、`split`、`member_move`、`manual_lock`、`manual_unlock`、`evidence_recompute`、`downstream_reconcile_required`、`deduplicated`。`metadata` 只允许关系 ID、版本、计数和安全原因，不允许正文/密钥。`event_contents` 增加 `CREATE UNIQUE INDEX ... ON event_contents(content_id) WHERE evidence_role <> 'duplicate'`，保证当前非 duplicate Content 只有一个归属；唯一冲突由应用按锁定和最高分规则解决，不用 `ON CONFLICT DO NOTHING` 静默丢事实。

固定评测集至少包含中英文同一事件、同名实体不同事件、跨时段事件、转载/更正/撤回、迟到内容、硬冲突和需要拆分的成员；数据按现实事件分组切分，保存 fixture 版本和标注冲突决议。每条候选决策必须写入 `event_clustering_decisions`，保存 `clustering_version`、`feature_input_hash`、四通道候选来源/排序、特征分项、证据 ID、最终决策者和 decision reason。`event_contents` 以部分唯一索引保证一个 active content 只有一个非 duplicate 当前归属；被拒绝候选只留在 append-only 决策表，不写成员关系。

候选通道的输入和资格固定如下：词法/Monitor 规则从 accepted PLAN-009 match 的规则词和 Content 标题摘要取 ≤16；时间/来源从 `published_at`、`source_connection_id` 和 Event 时间窗口取 ≤12；事实指纹从版本化 token/action/time bucket 取 ≤10；向量仅在同一 `model_profile_id/model_version` 存在 active `content_embeddings` 与 `event_embeddings` 时取 ≤12，否则记录 `vector_unavailable` 并继续规则链路。`detected`、`active`、`cooling`、`closed` 可成为候选，`merged`、`archived`、`rejected` 不进入新召回；每个通道 SQL 必须有候选计数测试和 `EXPLAIN (COSTS OFF)`，确认索引/partial HNSW 被使用或在无向量时显式降级。

合并先读取目标链，再将 source 与完整链按稳定 `event_key` 顺序锁定；若加锁期间链发生变化，返回 409 且不写入。随后在同一事务内比较 source `expected_version` 与最终 canonical target 的 `expected_version`；请求中 target 已 merged 时，`target_expected_version` 必须是 root 当前版本，不比较历史中间 Event 的版本。只允许非 merged/archived/rejected 源指向非自身 canonical 目标。任何环、锁定成员冲突、源/root 版本过期或请求重复均返回 409 且不写入。每个普通成员迁移写 `member_move` audit；重复 event_contents/monitor_events 保留目标最高分和最早关系，并以含 Content/Monitor ID 的 `deduplicated` audit 记录丢弃关系。合并后的 source 清除 `representative_content_id`；target 在同一事务内重算。人工锁定成员不自动迁移，必须由管理员先通过成员锁 endpoint 显式解锁后重试。拆分 body 必须携带源事件版本及每个选中成员的 `event_contents.version`；事务按 content_id 升序锁定并逐一重验，任一成员陈旧、锁定或已迁移均返回 409，不产生新 Event 或部分迁移。拆分只从一个非 merged 事件选择未锁定成员，新 Event 使用新 event_key；每个迁移成员写 `member_move` audit，源/新事件及其 Monitor 关系和审计在同一事务内写入，失败完全回滚。实体、Claim、Topic、Report 不在本事务迁移，统一写 `downstream_reconcile_required` audit 供后续 Plan 消费。

必须覆盖以下负向/并发红灯：重复重跑不新增 Event/成员；超过 50 个候选被截断；人工锁后自动重算不得覆盖；删除最后一条有效证据进入 rejected 或人工复核；合并/拆分故障不留下半迁移关系；并发相同输入只有一个创建者；API 非管理员、过期 version、非法状态迁移和越权事件 ID 均返回稳定 numeric code。
