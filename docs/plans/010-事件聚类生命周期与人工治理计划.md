# 事件聚类、生命周期与人工治理计划

## 计划目标

把已经通过 Monitor 相关性判断的 active Content 聚合为可追溯的全局 Event，并保证重跑、迟到证据、生命周期迁移、合并、拆分和人工锁在 PostgreSQL 单体架构内保持幂等和事务一致。Plan 010 只消费 PLAN-009 的 accepted 匹配，不重新实现相关性或 AI Provider。

## 开工条件

- 本 Plan 为 `status: accepted`、`review_status: approved`、`execution_status: ready`。
- PRD-010 为 `status: accepted`、`execution_status: ready`。
- PLAN-009 为 `archived/done`，工作区干净且当前分支已同步主线基线。
- PostgreSQL/Redis 只使用可丢弃的 `.env` 验收配置；生产配置仅由 `.env.prod` 提供，不新增第三套环境或运行时 DDL。
- 每个任务开始前先执行 `git status --short`、`git diff --check`，staging 只允许当前任务文件。

## 不可突破的边界

- 候选召回最多 50 个 Event；不得做 Content×Event 全表比较。
- `event_key` 由 `IDGenerator` 创建且不可变；`event_fingerprint` 只用于召回，不能替代业务唯一键。
- membership score 固定为 entity/action 0.30、semantic 0.30、temporal 0.20、location 0.10、source/context 0.10；`>=80` 接受、`65–79.99` 灰区、`<65` 新建或拒绝，硬冲突一票否决。
- LLM 仅可处理灰区，必须可降级为确定性决策；不新增 `task_type`，不把 Provider SDK 类型带入 domain/application。
- 事件、成员、Monitor 关系和审计写入同一 PostgreSQL 事务；合并/拆分按稳定事件键顺序加行锁。
- `manual_locked` 的事件字段和成员关系不能被自动重算覆盖；删除证据后必须重算，不物理删除审计或合并历史。
- 本计划不实现热度趋势、实体/Claim、任务队列/调度、摘要、知识库和投递。

## 固定契约与测试夹具

固定评测集至少包含中英文同一事件、同名实体不同事件、跨时段事件、转载/更正/撤回、迟到内容、硬冲突和需要拆分的成员；数据按现实事件分组切分，保存 fixture 版本和标注冲突决议。每条决策必须保存 `clustering_version`、`feature_input_hash`、特征分项、候选排序、证据 ID 和最终决策者。

必须覆盖以下负向/并发红灯：重复重跑不新增 Event/成员；超过 50 个候选被截断；人工锁后自动重算不得覆盖；删除最后一条有效证据进入 rejected 或人工复核；合并/拆分故障不留下半迁移关系；并发相同输入只有一个创建者；API 非管理员、过期 version、非法状态迁移和越权事件 ID 均返回稳定 numeric code。

## 文档和回滚要求
