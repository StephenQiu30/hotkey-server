# 模块化单体启动与工程门禁计划

## 计划目标

交付一个可编译的 hotkey 二进制，使 all、api、worker 三种角色具有可测试的启动、就绪和优雅停机行为。

## 开工条件

- 当前 Plan 的 status 为 accepted、review_status 为 approved、execution_status 为 ready
- PRD-001 的 status 为 accepted，execution_status 为 ready
- 当前 main 已同步，工作区只包含本任务文件
- Go 1.26 工具链可用

## 执行文件

| 动作 | 路径 | 目的 |
|---|---|---|
| 创建 | cmd/hotkey/main.go | 统一二进制入口和退出码 |
| 修改 | internal/bootstrap/app.go | Fx 装配与生命周期 |
| 修改 | internal/bootstrap/role.go | all、api、worker 角色解析 |
| 创建 | internal/bootstrap/lifecycle.go | 启动、停止和超时边界 |
| 修改 | internal/platform/config/config.go | 角色与停机配置 |
| 修改 | internal/platform/http/server.go | HTTP 启停和就绪 |
| 修改 | internal/platform/http/router.go | healthz 与 readyz |
| 修改 | internal/platform/logging/logger.go | 启动日志与脱敏 |
| 修改 | tests/architecture/layout_test.go | 目标入口和禁止目录门禁 |
| 修改 | scripts/validate_architecture.ps1 | 目标目录与依赖校验 |
| 修改 | Makefile、README.md、.env.example | 构建、启动和配置说明 |

## 执行步骤

1. 先扩展角色、布局、退出和优雅停机测试。
2. 运行红灯命令，确认失败原因是 cmd/hotkey 或生命周期缺失。
3. 创建最小 main 和 lifecycle，实现三种角色装配。
4. 接入 HTTP 健康与就绪状态，不引入业务模块实现。
5. 更新 Makefile、环境变量说明和架构校验。
6. 运行范围测试、全量测试、构建和验证。

## 验收命令

| 阶段 | 命令 | 通过标准 |
|---|---|---|
| 红灯 | go test ./internal/bootstrap ./tests/architecture -count=1 | 新增测试因入口或生命周期缺失失败 |
| 绿灯 | go test ./internal/bootstrap ./internal/platform/... ./tests/architecture -count=1 | 全部通过 |
| 构建 | go build -o hotkey ./cmd/hotkey | 生成二进制且退出码为 0 |
| 门禁 | make lint && make test && make validate | 全部通过 |
| 清理 | make clean && git status --short | 无 hotkey 构建产物 |

## 验收清单

- 非法角色启动失败
- api 不启动 Worker，worker 不监听业务 HTTP
- all 同时装配 API 与 Worker 生命周期
- readyz 能反映必要依赖未就绪
- 取消信号停止接收新工作并在超时内退出
- 架构门禁阻止旧顶层目录与基础设施反向依赖

## 提交边界

- test: 定义启动角色与工程门禁
- impl: 建立模块化单体启动入口
- docs: 同步启动与配置说明

## 风险与回滚

- 实现需要改变 Design 或 PRD 契约时立即停止，先更新正式文档并重新复核
- 下游 Plan 开工前，可整体回退本任务提交并同步恢复 Schema、OpenAPI 和测试
- 下游 Plan 已开工后，使用新的前向修复 Plan，不恢复旧双轨或兼容实现

不得在本计划实现数据库或业务模块。
