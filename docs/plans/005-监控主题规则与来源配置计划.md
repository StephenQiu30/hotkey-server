# 监控主题、规则与来源配置实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` and `superpowers:subagent-driven-development` to execute Tasks 1–6 in order. Each task has a fresh implementer and an independent reviewer; the next task may not start until the previous review has no Critical/Important finding.

**Goal:** 交付团队共享的、可安全发布的 Monitor 控制平面。编辑永远发生在 draft，发布产生 immutable config version；来源连接和审计处于受控事务中，HTTP 仅暴露安全 DTO，preview 完全无副作用。

**Architecture:** `monitor` 拥有逻辑 Monitor、config version、rule 和 MonitorSource；`source` 拥有 SourceConnection；`operations` 拥有新模块可用的 AuditWriter；identity 仅提供 `Subject` 和角色中间件。Monitor/Source application 在一个 `database.Runtime.WithinTransaction` 内编排 Repository、状态、乐观锁、PostgreSQL advisory lock 和 AuditWriter。PLAN-006 所需 Connector、shared run、River 与 checkpoint 只保留 schema-compatible 输入，不在本计划实现。

**Tech Stack:** Go 1.26、Gin、GORM v2、pgx/PostgreSQL 18、Fx、validator/v10、Zap、Swaggo、现有 identity HTTP Subject/Result/Error 机制；不新增外部依赖。

## 全局约束

- 仅修改 `hotkey-server`。不创建 migration 目录、不使用 AutoMigrate、不改 Web/Miniapp、不连接真实来源或真实秘密。
- 所有数据库改动同时修改 `db/schema.sql`、`internal/platform/database/model/model.go`、模型/Schema/集成测试和 `scripts/verify-schema.sh` 的表数/约束断言。验证只使用 `HOTKEY_TEST_DSN='postgres:///hotkey_plan005_test?sslmode=disable'` 这个可丢弃库。
- 成功 API 使用 `Result{code:0,message,data}`；所有错误使用已注册稳定业务码，`data:null`，禁止 HTTP handler 直接 `c.JSON`。
- Application/Repository 绝不导入 Gin；HTTP 不导入 PostgreSQL adapter；`platform` 不导入 monitor/source/operations application 或 domain。跨模块只经窄应用端口或只读查询端口。
- 所有修改既有 Monitor/Source 资源的写命令均执行 caller role、Design-014 指定的期望版本和 transaction；create 命令不传期望版本，preview 不写任何事实。不得静默最后写入者覆盖；不得用 identity 私有 audit repository 记录 Monitor/Source 事实。
- credential reference、secret-shaped config、endpoint 中受禁查询参数、完整 config、规则原文、内部 SQL/URI/错误原因不得泄露到 viewer/editor DTO、错误、audit 或日志。admin management DTO 仅返回经验证的 endpoint 与非秘密 config，从不返回 credential reference。
- 每个 Task 先留下可复现 RED 证据，写最小实现得到 GREEN，提交当前 Task 文件，独立审核，再继续。执行状态、审核结论与命令输出写入 `.superpowers/sdd/progress.md`（忽略文件）；最终稳定结论写入 Acceptance-005。
