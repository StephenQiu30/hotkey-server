# 多语言相关性匹配与反馈计划

**目标：** 只让 active Content 与有限的已发布 Monitor 候选比较，保存可解释、版本化、可降级的相关性快照；灰区可以由受控 AI 复核，用户反馈只进入离线评测和待审核建议。

**边界：** 本计划是 relevance_review 的唯一任务类型扩展，但不引入新 Provider、endpoint、通用 Prompt、后台编排、在线训练、自动修改 Monitor draft、Event、热度或报告。所有跨模块访问通过 ingestion 和 intelligence 的 Application 端口或只读端口完成；ingestion 不直接查询 ai_* 表，intelligence 不直接写 monitor_matches。

## 固定行为契约

### 1. 候选、输入和评分

- 只处理 content.status=active、monitor.status=active 和 published monitor_config_version。Content 删除、过期、Monitor pause/archive 或 config supersede 后，读取和写入都拒绝该事实；不删除历史快照。
- 三条候选路径分别最多 source=8、lexical=12、vector=12。先按 monitor_id 去重并合并 recall_paths，最终最多 20 个；排序固定为 source 优先、词法分数降序、向量距离升序、monitor_id 升序。任何路径不足或不可用只减少候选，不扩大到全量扫描。
- source 只从同 source_connection 的已发布 monitor_sources 读取；lexical 只从已批准且 enabled 的 keyword、phrase、entity、exclude_keyword 规则倒排读取；vector 只读取同一 model_profile_id、model_profile_version、model_version 的 active Content/Monitor 1024 维向量。没有严格同空间向量时跳过 vector 路径。
- 候选输入哈希是 canonical SHA-256：content.id、现有 content.dedupe_key、language、title/excerpt 的 NFC 规范化值、monitor_config_version.config_hash、已批准规则、scoring_version、embedding 模型三元组、选定 relevance_review profile 三元组或空值和 degraded 标记。dedupe_key 变化必须产生新快照；不得新增 contents.content_hash，也不得把正文、对象键、Prompt、API key 或 Provider 原文放入哈希或 explanation。
- 完整评分 relevance-v1 为 semantic×0.35 + lexical×0.25 + entity×0.20 + title×0.10 + preference×0.10。无向量或向量查询失败时使用 relevance-v1-degraded：lexical×0.40 + entity×0.30 + title×0.15 + preference×0.15，并写 degraded=true。
- 规则来源 user 的有效词权重基准为 1.00；已批准的 AI rule 最高为 0.60，且任何单词项重复命中只计一次。exclude_keyword、language/region/domain/author 不满足和实体冲突是硬否决，必须在向量和 AI 前产生 rejected；它们不能被高语义相似度覆盖。
- 高阈值是 max(75, monitor_config.relevance_threshold)，低阈值固定 60。分数大于等于高阈值为 rule accepted，小于 60 为 rule rejected，闭区间 [60, high) 为 rule review。manual_locked 快照不再自动重算或请求 AI。

### 2. 受控 relevance_review

- PLAN-009 扩展 domain.TaskType、ai_model_profiles task_type CHECK、静态 input/output JSON Schema、嵌入 instruction、profile 校验、RunService 结构化执行器和管理员 profile HTTP DTO/handler/routes/OpenAPI，使 relevance_review 成为第三个且唯一新增值。OpenAI 可用于该任务，ONNX 不可用于该任务，embedding_dimensions 必须为 NULL；管理员 Create 只能按既有 write-only credential_ref 规则创建该 OpenAI profile，GET/PATCH/Swagger 不得泄露凭据。
- 输入 Schema v1 只允许 content_excerpt（最大 1200 Unicode 字符）、content_language、monitor_intent（最大 500）、scoring_version、五个 0–100 分项、recall_paths、最多 12 个受控原因码和 16 个证据词项。输出 Schema v1 只允许 decision=accepted|rejected|review、score=0–100、最多 8 个受控原因码。instruction 固定在仓库中；未知字段、长文本和自由 reason 均被拒绝。
- rule_score 始终保存确定性五分项评分，初始 final_score 等于 rule_score。仅对未 manual_locked、无硬否决且初始 rule review 的 monitor_match 调用一次 relevance_review。先持久化精确快照，target_type 固定 monitor_match、target_id 为该 match ID；相同 input_hash 重试复用该 run。AI 成功时 llm_score 与 final_score 都写入同一个经 Schema 校验的 AI score；其 decision 必须严格符合 score>=high 为 accepted、score<60 为 rejected、其余为 review，否则安全保留 rule review。列表与下游按 final_score DESC、id DESC，解释保留 rule_score、llm_score、阈值、reason_codes 和 review_ai_run_id。
- 无 profile、预算不足、Provider 错误、schema 错误、in-flight 或 lease 回收时，保留 rule review、reason_codes 增加 ai_unavailable 或 ai_in_progress、degraded=true；不得再次外呼、不得阻塞其他候选或改变硬否决。AI 绝不改写 Rule、反馈、manual_locked、阈值或历史快照。

### 3. Schema、历史和 Repository

monitor_matches 从原来的一条当前关系升级为可审计的输入快照。完整 db/schema.sql、PostgreSQL record、Repository 和测试必须同步：

- 新增非空 input_hash char(64)、scoring_version varchar(64)、recall_paths text[]、degraded boolean、decision_origin varchar(16)；新增可空 embedding_model_profile_id、embedding_model_profile_version、embedding_model_version、review_ai_run_id。decision_origin 只允许 rule、ai；embedding 三元组必须同存同空，并以 FK 指向 ai_model_profiles，Repository 验证保存的版本和 model_name 与 profile 精确相符。review_ai_run_id 只在 AI 成功时存在，以 FK 指向 ai_runs；Repository 验证其 task_type=relevance_review、target_type=monitor_match、target_id=本行 ID 和成功 provenance。向量空间出处与结构化 review run 永远独立。
- explanation 是受限 JSONB：matched_terms、matched_entities、excluded_terms、recall_paths、五个数值分项、reason_codes 与 provenance；Domain 验证拒绝 Prompt、完整正文、对象键、endpoint、credential、raw response 和未知顶层字段。
- 把原 unique(monitor_config_version_id,content_id) 替换为 unique(monitor_config_version_id,content_id,input_hash,scoring_version)。精确重试使用 INSERT ON CONFLICT 返回既有快照；不同 input_hash 生成新事实，不覆写历史。新增 current/list 索引支持每个 config/content 的最新快照、monitor+decision+final_score 的 cursor 列表和 active Content join。
- 新建 monitor_match_feedbacks：id、version、monitor_id、monitor_config_version_id、content_id、monitor_match_id 可空、actor_user_id、feedback_type、created_at、updated_at。反馈类型只允许 relevant、irrelevant、false_positive、false_negative；unique(monitor_config_version_id,content_id,actor_user_id)。Repository 验证 match（若给出）属于同 config/content，actor 是 active 用户且 Content active。首次创建必须显式 expected_feedback_version=null；更新必须传入正数且匹配 feedback.version，否则返回 10002；match.version 从不参与 feedback 锁。
- 新建 monitor_feedback_suggestions：id、version、monitor_id、monitor_config_version_id、suggestion_type、value、support_count、status、reviewed_by_user_id 可空、created_at、updated_at。suggestion_type 只允许 add_term、add_exclude、add_entity，status 只允许 pending、approved、rejected；pending 行对同 config/type/value 唯一，support_count 至少 2。审核只改变 suggestion 状态，绝不写 monitor_rules。
- 所有 FK 使用 ON DELETE RESTRICT；纯建议/反馈不会在 Content 删除时重连到其他 Content。读取只显示 active Content 的相关快照；历史 FK 仍用于审计。

## HTTP 契约与授权

所有路由均在 /api/v1/monitors/:id 下并使用统一 Result。没有接口返回 Prompt、完整正文、object key、Provider raw response、credential_ref、endpoint、内部 SQL 或其他用户的身份信息。

| 路径与方法 | 角色 | 行为 |
|---|---|---|
| GET /matches | viewer、editor、admin | 只列 active Content 的最新匹配快照；decision、cursor、limit 过滤，limit 1–100。 |
| GET /matches/:match_id | viewer、editor、admin | 返回属于该 Monitor 的单个解释快照和安全 Content 摘要。 |
| POST /relevance-preview | editor、admin | 固定读取最近 20 条 active Content，使用同一评分器但零写入、零 AI 调用、零任务入队。 |
| PUT /matches/:match_id/feedback | editor、admin | 用 expected_feedback_version=null 创建或用匹配的正数 version 更新本人对既有快照的反馈。 |
| PUT /contents/:content_id/feedback | editor、admin | 为没有既有 Match 的 false_negative 创建或更新本人反馈；使用相同 expected_feedback_version 规则。 |
| GET /feedback/evaluation | admin | 返回固定标注/用户反馈的 P@20、排除误命中率和按 scoring_version 分组的安全聚合。 |
| POST /feedback/suggestions/refresh | admin | 从 feedback 生成或更新 pending 建议；只读规则，不改 draft。 |
| GET /feedback/suggestions | admin | 分页列出 pending/已审核建议。 |
| POST /feedback/suggestions/:suggestion_id/review | admin | 按 expected_version 标记 approved/rejected，不创建或修改 Monitor rule。 |

无认证为 401/20000，角色不足为 403/20001，格式/limit/filter 非法为 400/10000，Monitor、Match、Content、Suggestion 归属不符或非 active 为 404/10003，stale feedback/suggestion version 为 409/10002。所有路径、DTO、分页 Cursor、错误 Result 和 Swagger 注解必须由 HTTP/OpenAPI tests 覆盖。
