- 当前 Plan 的 status 为 accepted、review_status 为 approved、execution_status 为 ready

- frontmatter 中 depends_on 列出的 Plan 全部为 done

## 执行步骤

1. 确认 PLAN-001 至 PLAN-016 均有对应 Acceptance 且 done。
2. 先写运行查询、清理幂等、删除级联和对账红灯测试。
3. 实现 operations 模块、受限 API、清理与对账 Job。
4. 生成目标容量数据，记录查询计划、分页、向量和队列基线。
5. 注入 PostgreSQL 重连、MinIO、Provider、Vault 和 SMTP 故障。
6. 跑通 P0/P1、备份恢复、无可选依赖和最终 make ci。
7. 形成 Acceptance 与 Operations 文档。

## 验收清单

- Precision@20 不低于 80%
- 跨语言聚类 F1 不低于 75%
- 95% 正常内容在 60 分钟内形成或更新 Event
- 重跑不重复内容、匹配、Event、AI、报告或投递
- 单一来源失败不阻塞其他来源
- 无 LLM、Vault、SMTP、Git、Kafka 和 Redis 时 P0 可用
- DB、MinIO、Vault 对账和备份恢复有证据
- 所有 Plan、Acceptance、Schema、OpenAPI 和代码状态一致

## 提交边界

- test: 定义系统收口与故障恢复门禁
- impl: 实现 operations、清理和对账
- docs: 归档 Acceptance 与 Operations

## 风险与回滚

- 实现需要改变 Design 或 PRD 契约时立即停止，先更新正式文档并重新复核
- 下游 Plan 开工前，可整体回退本任务提交并同步恢复 Schema、OpenAPI 和测试
- 下游 Plan 已开工后，使用新的前向修复 Plan，不恢复旧双轨或兼容实现

不得在本计划新增来源、模型任务或产品能力。
