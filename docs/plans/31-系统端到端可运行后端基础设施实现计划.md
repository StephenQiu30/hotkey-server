# 31-系统端到端可运行后端基础设施 实现计划

## 1. 目标

重构 hotkey-server 为标准 Go 分层项目结构，接通 PostgreSQL 和 Redis，为 4 个核心服务实现 PG repository，加入认证、CORS、日志中间件和优雅关闭。

## 2. 文件清单

- PRD：`docs/product/prd/26-系统端到端可运行与基础设施对接PRD.md`
- Plan：`docs/plans/31-系统端到端可运行后端基础设施实现计划.md`
- 设计输入：`docs/superpowers/specs/2026-05-28-system-running-design.md`
- 数据库 Schema：`db/schema.sql`
- 迁移目录：`db/migrations/`

## 3. 任务拆解

### 3.1 项目结构重构

**Task 31-1：拆分 model 层**
- 从各领域包提取实体定义到 `internal/model/`
- 涉及文件：keyword、source、content、hotspot 的实体类型
- 验收：`internal/model/` 包含领域实体，编译通过

**Task 31-2：拆分 repo 接口层**
- 在 `internal/repo/` 定义仓储接口（KeywordRepo、SourceRepo、ContentRepo、HotspotRepo）
- 验收：接口定义完整，覆盖现有内存实现的全部方法

**Task 31-3：拆分 service 层**
- 将业务逻辑迁移到 `internal/service/keyword/`、`source/`、`content/`、`hotspot/`
- service 依赖 repo 接口，不依赖具体实现
- 验收：service 层编译通过，现有测试适配后通过

**Task 31-4：拆分 handler 层**
- 从 `internal/httpapi/` 提取 handler 到 `internal/handler/`
- router.go 简化为路由注册和依赖组装
- 验收：所有 49 个端点正常响应

**Task 31-5：迁移内存实现到 store/memory/**
- 现有内存 repository 迁移到 `internal/store/memory/`
- 验收：`go test ./...` 全绿，行为不变

### 3.2 基础设施层

**Task 31-6：PostgreSQL 连接池**
- `internal/infrastructure/postgres/pool.go`：pgx/v5 连接池初始化
- 从 `config.DatabaseURL` 读取连接串
- 启动时健康检查，失败拒绝启动
- 验收：连接池创建成功，健康检查通过

**Task 31-7：Redis 客户端**
- `internal/infrastructure/redis/client.go`：go-redis/v9 客户端初始化
- 从 `config.RedisURL` 读取连接串
- 验收：Redis 连接成功，ping 通过

**Task 31-8：数据库迁移工具**
- 集成 golang-migrate/migrate
- `db/migrations/000001_init.up.sql` 从现有 schema.sql 派生
- `db/migrations/000001_init.down.sql` 提供回滚
- 启动时自动运行待执行迁移
- 验收：迁移执行成功，表结构正确

### 3.3 核心 PG Repository

**Task 31-9：keyword PG repository**
- `internal/store/postgres/keyword.go`
- 实现 KeywordRepo 接口全部方法
- 验收：keyword service 测试使用 PG 实现通过

**Task 31-10：source PG repository**
- `internal/store/postgres/source.go`
- 验收：source service 测试使用 PG 实现通过

**Task 31-11：content PG repository**
- `internal/store/postgres/content.go`
- 包含三层去重逻辑的 SQL 实现
- 验收：content service 测试使用 PG 实现通过

**Task 31-12：hotspot PG repository**
- `internal/store/postgres/hotspot.go`
- 验收：hotspot service 测试使用 PG 实现通过

### 3.4 中间件与服务治理

**Task 31-13：认证中间件**
- `internal/middleware/auth.go`
- JWT BearerAuth 验证，user_id/tenant_id 注入 Gin Context
- 白名单路径跳过：/healthz、/openapi.json、/api/v1/auth/*
- 验收：无 token 返回 401，有效 token 放行

**Task 31-14：CORS 中间件**
- `internal/middleware/cors.go`
- 支持 HOTKEY_CORS_ORIGINS 配置
- 验收：前端跨域请求成功

**Task 31-15：请求日志中间件**
- `internal/middleware/logging.go`
- 使用 slog，记录方法、路径、状态码、耗时、请求 ID
- 验收：请求日志输出正确

**Task 31-16：优雅关闭**
- 监听 SIGINT/SIGTERM
- 停止接受新连接，等待进行中请求完成（最多 10 秒）
- 关闭 PG 连接池和 Redis 客户端
- 验收：Ctrl+C 后服务优雅退出

**Task 31-17：main.go 依赖注入重构**
- 初始化 PG pool、Redis client
- 运行迁移
- 选择 PG 或 memory repository 实现注入 service
- 注册中间件
- 验收：`go run ./cmd/server` 启动成功，所有端点正常

## 4. TDD 与验证

- 项目结构重构：每个 Task 完成后运行 `go test ./...` 确保不破坏现有测试
- PG Repository：使用 testcontainers-go 编写集成测试
- 中间件：编写单元测试验证认证、CORS、日志行为
- 迁移：验证 up/down 可逆
- 全部完成后：`go test ./...` + `curl http://127.0.0.1:18080/healthz`

## 5. 执行顺序

```
31-1 → 31-2 → 31-3 → 31-4 → 31-5（结构重构，顺序执行）
31-6 → 31-7 → 31-8（基础设施，并行）
31-9 → 31-10 → 31-11 → 31-12（PG Repository，顺序执行）
31-13 → 31-14 → 31-15（中间件，并行）
31-16 → 31-17（关闭与注入，最后执行）
```

## 6. 回滚策略

- 项目结构重构：每个 Task 可独立 revert，保留旧结构
- PG Repository：通过配置切换回 memory 实现
- 中间件：可通过配置禁用
- 迁移：提供 down.sql 回滚

## 7. 验收标准

- `go run ./cmd/server` 启动成功，`/healthz` 返回 200
- PG 和 Redis 连接正常
- 核心数据（keyword、source、content、hotspot）持久化到 PG
- 认证中间件正常工作
- `go test ./...` 全绿
- 项目结构符合标准 Go 分层
