# 任务执行与计划归档治理实施计划

**Goal:** 把 Design-015 的唯一任务闭环变为仓库内可重复验证的能力，消除 006–017 的已知计划缺口；它作为产品主链并行的治理支线，不能阻塞 006–017 的排期。

**Architecture:** `internal/documentation` 只读取仓库文件与 Git 对象，解析正式文档为 `Document`、`Reference` 与 `Issue`，再由纯校验规则生成稳定错误。`tests/documentation` 构造最小仓库 Fixture 和真实当前仓库回归，不依赖运行中的 HotKey 服务。`scripts/smoke.sh` 是显式提供 URL 的只读运行时探针；它不参与无服务进程的 `make ci`。文档仍在各自 canonical path 中通过状态归档。

**Tech Stack:** Go、`go.yaml.in/yaml/v3`、标准库 `os/exec`/`net/url`/`path/filepath`、现有 Make、POSIX `sh` 和 GitHub Actions；不新增业务服务或数据库依赖。

## 全局约束

- 仅修改 `hotkey-server`；不改 Web/Miniapp，不创建 migration，不改变业务数据库结构、HTTP 路由或认证行为。
- 所有文档路径均为仓库相对路径；校验器拒绝目录逃逸、绝对本地路径和第二套 archive 目录。
- `internal/documentation` 不导入 Gin、GORM、业务模块或 `internal/platform`；它只依赖标准库和 YAML 解析。
- 治理规则以 Design-015 与 `docs/README.md` 为准。已归档 001–005 的状态不因本计划被重写；只补充可复核元数据和索引。
- 每个 Task 先留下可复现 RED，完成最小实现得到 GREEN，再重构并运行指定回归。每个提交只包含该 Task 的文件；下一个 Task 必须等前一个 Task 的独立 Reviewer 无 Critical/Important 发现。
- 本计划首次获独立审核前必须保持 `status: review`、`review_status: pending`、`execution_status: backlog`。实施开始前按 Design-015 更新到 `accepted/approved/ready`，然后以独立提交激活为 `in_progress`。

2026-07-16，非本计划主要编写者的独立 Reviewer 审核 v1.3，确认状态门禁、任务 DAG、future Acceptance 受控例外、Task 1–17 的路径/红绿/回归/提交边界、可丢弃测试服务命令，以及 013/017 的 opt-in smoke 边界。`make validate` 和 `git diff --check` 均通过；结论为 **approved**，无 Critical 或 Important 发现。本文档曾以独立提交进入 `in_progress`；按产品 006–017 优先级，现恢复为 `accepted/approved/ready` 的非阻塞支线。

## 风险与回滚

- YAML 兼容性或历史提交缺失阻塞验收时，保持 PLAN-018 `in_progress` 或 `blocked`，先修复证据或校验器；不得降低 SHA/状态规则。
- 若治理校验发现早期已归档文档矛盾，优先以长期 Acceptance、提交和现有 Schema/OpenAPI 事实修正文档；不通过移动文件或双份归档掩盖问题。
- 006–017 的范围、接口或依赖若在逐份拆解时与 Design 冲突，停止该份 Plan 的改写，先修正 Design/PRD 并重新独立审核；不得为了使校验通过而虚构文件或命令。
- 归档后发现治理回归时，新建前向修复 PRD/Plan，保留 Acceptance-018 和归档提交，不恢复旧状态或删除历史证据。
