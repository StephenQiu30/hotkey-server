# 里程碑与任务领取总控计划（v1.1）

## 0. 基础执行约束（强制）

- 后端服务仅使用 `server` 目录作为运行入口，`apps` 目录不应作为交付路径（存在则移除）。
- 除 `openspec/changes/archive` 历史归档外，不允许仓库中出现运行时 `apps/` 目录或 `apps/api` 后端入口引用。
- 入口约束：接口、模型接入、评分与源路由相关实现统一挂到 `server/app/...`。
- 除 archive 外，任何任务周期执行前后均需扫描一次 `apps` 目录或文件残留，禁止将 `apps` 作为后端交付运行路径。若检测到残留，优先生成 P0 cleanup task 并完成后再进入新任务。
- 每个 Issue 完成前按 TDD 门禁执行：先补测试并执行 → 再提交代码；默认要求执行 `python3 -m unittest discover -s tests -p 'test_repository_governance.py'`（或对应组件测试）作为本次执行最低验收。
- 每次实现执行必须严格遵循顺序：
  - 先完成测试（或最小相关测试集）
  - 若是基础约束类任务（如 #54），必须先提交仓库扫描命令输出：
    - `python3 -m unittest discover -s tests -p 'test_repository_governance.py'`
    - `find . -type d -name apps -print`
    - `find . -type f | rg '(^|/)apps/'`
  - 用 `git status --short` 确认无中间文件
  - 工作区无未提交内容后再执行 commit
  - 无未提交中间产物；
  - 工作区 `git status --short` 为空后再执行 commit；
  - commit message 需要提及关联 issue 与验收项。
- 每次执行必须以 `git status --short` 为空状态结束；未通过测试或有中间文件不得提交。

- 执行收口前提：
  - 本次执行的 commit 中不得出现未归档的 `apps/` 文件变更；
  - 若出现 `apps` 相关变更，需先确认是否历史归档，否则阻断并转为 P0 cleanup；
  - `git diff --name-only HEAD~1` 与 `git status --short` 均需与收口结论一致。

本执行周期复核结论：
- 本轮扫描确认仓库中无运行时 `apps/` 目录或 `apps` 文件残留，可执行 `find . -type d -name apps -print` 与 `find . -type f | rg '(^|/)apps/'` 两条命令复核。
- 因仓库无 apps 运行时目录，P0 清理任务不涉及代码文件删改，可直接将任务归档为“完成待验”；随后进行一次 issue #54 的最终闭环。
- 本次执行按“`server` 单一后端入口 + 可移除历史 `apps` 文件 + test-first + commit 后保持工作区清洁”执行，无中间产物留存。
