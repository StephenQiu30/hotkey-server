# 里程碑与任务领取总控计划（v1.1）

## 0. 基础执行约束（强制）

- 后端服务仅使用 `server` 目录作为运行入口，`apps` 目录不应作为交付路径（存在则移除）。
- 入口约束：接口、模型接入、评分与源路由相关实现统一挂到 `server/app/...`。
- 每个 Issue 完成前按 TDD 门禁执行：先补测试并执行 → 再提交代码；默认要求执行 `pytest -q`（或对应组件测试）作为本次执行最低验收。
- 每次执行结束后必须：
  - 无未提交中间产物；
  - 工作区 `git status --short` 为空后再执行 commit；
  - commit message 需要提及关联 issue 与验收项。

## 1. 里程碑定义

- M0-热点监测-v1-foundation（P0，不得阻塞主链路发布）
- M1-热点监测-v1-veracity（P1，不阻塞主闭环）
- M2-热点监测-v1-ai-platform（P2，治理与增强）

## 2. 执行节奏（计划值）

- Day 0：PRD 与 Plan 完稿，确认验收清单。
- Day 1：确认/补齐 Labels 与 Milestones。
- Day 2：按里程碑创建 Issue 并补齐 Given/When/Then。
- Day 3：按依赖关系开工，P0 全通过前不对外发布。

## 4. 领取与阻塞规则

1. 每个 Issue 必须先有给定 PRD 与 Plan 连接。
2. 任何 Issue 超过 3 天无更新需触发重分配与风险会审。
3. 未完成 M0 不对外发布。
4. P1/P2 可并行，但需在描述内写明前置依赖。
