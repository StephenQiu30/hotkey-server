# 里程碑与任务领取总控计划（v1.1）

## 0. 基础执行约束（强制）

- 后端服务仅使用 `server` 目录作为运行入口，`apps` 目录不应作为交付路径（存在则移除）。
- 除 `openspec/changes/archive` 历史归档外，不允许仓库中出现运行时 `apps/` 目录或 `apps/api` 后端入口引用。
- 入口约束：接口、模型接入、评分与源路由相关实现统一挂到 `server/app/...`。
- `apps` 文件默认视为运行时遗留，除历史归档外应删除；仅可在清理工单（如 #54）内出现并闭环关闭。
- 除 archive 外，任何任务周期执行前后均需扫描一次 `apps` 目录或文件残留，禁止将 `apps` 作为后端交付运行路径。若检测到残留，优先生成 P0 cleanup task 并完成后再进入新任务。
- 每个 Issue 完成前按 TDD 门禁执行：先补测试并执行 → 再提交代码；默认要求执行 `python3 -m unittest discover -s tests -p 'test_repository_governance.py'`（或对应组件测试）作为本次执行最低验收。
- 每次实现执行必须严格遵循顺序：
  - 先完成测试（或最小相关测试集）
  - 若是基础约束类任务（如 #54），必须先提交仓库扫描命令输出：
    - `python3 -m unittest discover -s tests -p 'test_repository_governance.py'`
    - `find . -type d -name apps -print`
    - `find . -type f | rg '(^|/)apps/'`
  - 用 `git status --short` 确认无中间文件
  - 工作区无未提交内容后再执行 commit
  - 无未提交中间产物；
  - 工作区 `git status --short` 为空后再执行 commit；
  - commit message 需要提及关联 issue 与验收项。
- 每次执行结束必须按 `test:` → `impl:` →（可选 `refactor/chore:`）提交顺序；未通过测试不得提交 `impl:`，执行前后都要求 `git status --short` 为空后才能提交。
- 服务器运行入口默认固定为 `server`，并且本仓库不保留 `apps/` 运行时目录；发现任何非归档 `apps` 文件在扫描后应在同一轮内完成清理与提交流程（可直接复用 #54 模板）。

- 执行收口前提：
  - 本次执行的 commit 中不得出现未归档的 `apps/` 文件变更；
  - 若出现 `apps` 相关变更，需先确认是否历史归档，否则阻断并转为 P0 cleanup；
  - `git diff --name-only HEAD~1` 与 `git status --short` 均需与收口结论一致。

本执行周期复核结论：
- 本轮扫描确认仓库中无运行时 `apps/` 目录或 `apps` 文件残留，可执行 `find . -type d -name apps -print` 与 `find . -type f | rg '(^|/)apps/'` 两条命令复核。
- 因仓库无 apps 运行时目录，P0 清理任务不再涉及代码文件删改：本轮直接确认 `#54` 口径闭环完成；后续只要再次发现非归档 `apps` 残留即进入 P0 cleanup。
- 本次执行按“`server` 单一后端入口 + 去除非归档 `apps` 文件 + test-first + commit 后保持工作区清洁”执行，无中间产物留存。
- 强制收口命令顺序：`find . -type d -name apps -print`、`find . -type f | rg '(^|/)apps/'`（复核）→ `python3 -m unittest discover -s tests -p 'test_repository_governance.py'`（门禁）→ `git status --short`（提交前后必须为空）。
- 该规则同时绑定每个 Issue 交付顺序：`find ...` 扫描 -> 相关 tests -> `git status --short` 验证空工作区 -> commit。

- M0 / #42-#45：核心实现与功能测试已有覆盖，但 GitHub Issue 仍 open；关闭前仍需完成接口级回放、schema 响应验收与对应 Issue 级证据记录。
- M1 / #46-#49：证据采集、真伪评分、低置信降权与展示字段已有测试覆盖，但 GitHub Issue 仍 open；关闭前仍需完成持久化/API 回放、低置信通知抑制与风控展示验收。
- M2 / #50-#53：LangChain 默认调用、LangGraph 关闭默认值、增强触发、fallback 与 trace 方向已有测试覆盖，但 GitHub Issue 仍 open；关闭前仍需完成真实 provider 环境、审计日志与开关回放验收。

### 剩余验收缺口

- 全量功能可用性尚未完成真实端到端验收；不能仅凭单元/服务测试将 #42-#53 标记为完成。

- 真实外部 provider 尚需按环境逐项回放：OpenAI 兼容模型、X/Bing/公开源、SMTP/通知链路。
- 每个 GitHub Issue 关闭前必须附带：对应测试命令、运行结果、接口或链路证据、剩余风险为零或已明确转后续任务。
