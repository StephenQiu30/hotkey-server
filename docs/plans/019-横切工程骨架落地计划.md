# 背景

横切工程能力如果不先建公共骨架，后续各业务模块会迅速出现多套错误响应、多套上下文传递和多套日志风格，后续收敛成本极高。

# 目标

1. 落地统一错误模型、错误码注册表与 recover middleware。
2. 落地统一成功/失败响应器与分页响应结构。
3. 落地 request context 注入、结构化日志字段和配置加载校验骨架。
4. 建立枚举与常量的统一约束方式。

# 非目标

1. 本计划不迁移具体业务模块。
2. 本计划不定义 OpenAPI 生成实现细节。
3. 本计划不处理 Worker 调度运行逻辑。

# Task 1: 统一错误处理骨架

目标：

1. 建立 `ErrorCode`、`AppError` 和错误码注册表。
2. 建立 `RecoverMiddleware`。
3. 建立统一 HTTP 错误映射。

验证门禁：

```bash
go test ./internal/platform/http/... ./...
```

# Task 2: 统一响应与请求上下文骨架

目标：

1. 建立统一成功响应、失败响应和分页响应包络。
2. 注入 `request_id`、`trace_id`、`user_id` 等请求上下文字段。
3. 建立公共 responder。

验证门禁：

```bash
go test ./internal/platform/http/... ./internal/platform/runtime/... ./...
```

# Task 3: 配置、枚举与日志骨架

目标：

1. 收敛统一配置对象与启动校验。
2. 建立枚举与常量的落位规则和基础 helper。
3. 统一结构化日志字段与日志入口。

验证门禁：

```bash
go test ./internal/config/... ./internal/platform/logging/... ./...
```

# Task 4: 横切测试门禁

目标：

1. 建立配置缺失失败测试。
2. 建立 recover middleware 测试。
3. 建立 request context 注入测试。
4. 建立错误响应一致性测试。

验证门禁：

```bash
go test ./...
```

# 风险与边界

1. 如果这一阶段被跳过，后续模块接入会分叉。
2. 如果错误码注册表不先冻结，契约和前端调用会很快漂移。
