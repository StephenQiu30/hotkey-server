# 21-管理员观测运行时接通实现计划

## 1. 目标

提供最小可用的管理员观测能力，帮助查看任务状态、采集运行状态和关键外部依赖配置/健康。

## 2. 文件清单

- 修改：`internal/service/admin/`
- 修改：`internal/transport/http/handlers/admin/`
- 修改：`internal/service/source/`
- 修改：`internal/queue/`
- 修改：`internal/transport/http/router.go`

## 3. 任务拆解

1. 为管理员权限、依赖 degraded 和状态读取写失败测试。
2. 增加管理员读取 jobs、collection runs 和依赖状态的服务能力。
3. 接通对应的 HTTP handler 和必要 OpenAPI 更新。
4. 确认输出对排障足够，但不泄漏 secret。
5. 运行验证并回写结果。

## 4. TDD 与验证

- 管理员可查看关键运行状态。
- 非管理员访问被正确拒绝。
- 输出足够排障且不暴露敏感配置。

## 5. 执行顺序

1. `test:` 权限、degraded 状态、读取 API 失败测试。
2. `impl:` admin service、handler 与路由接线。
3. `docs:` 更新 OpenAPI 契约（如有字段变更）。

## 6. 回滚策略

回滚运行时观测接线，保留 admin service 基础能力；不影响已有审计日志数据。

## 7. 验收命令

```bash
go test ./internal/service/admin/...
go test ./internal/transport/http/...
gofmt -w cmd internal
go test ./...
python3 -m unittest discover -s tests
```

## 8. Symphony / Linear 要求

任务状态、标签和流转规则完全以本仓库 `WORKFLOW.md` 和本地 Symphony 实现为准。Plan 不定义额外状态、不发明额外标签、不覆盖 Symphony 的状态机。

Linear issue 只承载任务内容：PRD 路径、Plan 路径、任务范围、禁止范围、TDD 验收命令和回写要求。Symphony 负责监听 active states、创建 workspace、运行 Codex，并按 `WORKFLOW.md` prompt 驱动执行。
