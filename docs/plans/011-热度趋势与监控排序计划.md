## 执行步骤

1. 在 source 模块先固定 `metric_capability_profiles` 的列、draft/published/archived 状态机、已发布不可变约束、管理员发布理由/审计和 Event 只读查询端口；写 schema、Repository、RBAC、并发和 OpenAPI 红灯测试。
2. 固定 `testdata/ranking/v1`：同源同类型的指标窗口、缺失指标、异常/回退指标、独立性折叠、单项封顶、证据删除、merge/split 和人工 rising/非 rising Top-20 标签；测试保存期望排名、recall 与 false-positive 阈值。
3. 用真实 active、非 duplicate、未删除/过期/无效 EventContent 与 `content_metric_snapshots` 构造 1/6/24 小时快照输入和稳定 `evidence_set_hash`/`capability_profile_set_hash`；先写相同输入幂等、删除后新快照、历史快照不变和投影原子性红灯测试。同一证据、窗口、算法下切换 profile vN→vN+1 必须保留两条快照，并将 Event/Monitor 当前投影更新为新档案的计算结果。
4. 重写现有 `metrics.go` 的脚手架以符合 Design-010：同来源同内容类型的窗口化归一化与版本化 fallback；independent source、content velocity、source breadth、engagement velocity、recency、credibility 六分量热度；short/long EMA 加 acceleration 的 emerging/rising/stable/falling/dormant 趋势；relevance 0.55、heat 0.25、trend 0.15、freshness 0.05 排序并拒绝低于 Monitor 最低相关性阈值的热度穿透。
5. 实现无调度依赖的 `RecomputeEventMetrics`：删除、duplicate 修正、merge/split、指标更新和算法/档案切换均同步调用；它在单个事务内追加版本化 `event_metric_snapshots`、更新 Event 当前投影和相关 `monitor_events.final_score`。PLAN-013 只负责未来的触发、去重和重试。
6. 更新 Event/Monitor 查询和管理员 capability profile API、OpenAPI、架构测试与 Acceptance-011；所有响应只输出版本、窗口和原因码，不输出完整权重或反刷阈值。

## 验收清单

- 缺失指标不当作零
- 不直接比较不同来源原始互动数
- 单来源或单内容不能无界推高热度
- 相同输入、窗口和版本结果一致
- 历史快照不被算法升级改写
- 删除证据后热度与排序可幂等重算
- capability profile 仅管理员能以理由发布，已发布版本不可原地改写，Event 快照引用所用版本
- 同步重算不依赖 Cron/River；调度、队列和长期线上质量分别留给 PLAN-013/017
- Top-20 fixture 同时回归 rising recall 与 false-positive 基线
