# 16-X来源采集主线接通实现计划

## 1. 目标

让 X 来源采集真正进入系统主链路，并为后续内容处理和热点识别提供稳定输入。

## 2. 文件清单

- 修改：`internal/platform/fetcher/`
- 修改：`internal/worker/`
- 修改：`internal/service/source/`

## 3. 任务拆解

1. 为 X 成功采集、鉴权失败、限流失败写失败测试。
2. 串联采集任务输入与 fetcher 调用。
3. 增加运行记录、错误分类和统一输出结构。
4. 确认成功结果可交给后续标准化入口。
5. 运行验证并回写结果。

## 4. TDD 与验证

- X 采集可以被统一调度和记录。
- 采集失败原因清晰可追踪。
- 采集结果可以稳定交给后续内容处理。

## 5. 执行顺序

1. `test:` 采集成功、鉴权失败、限流失败测试。
2. `impl:` 任务接线、运行记录与错误分类。
3. `docs:` 更新采集运行状态说明（如有 API 变更）。

## 6. 回滚策略

回滚 worker 采集任务接线，保留 fetcher 独立能力；已采集数据保留在库中。

## 7. 验收命令

```bash
go test ./internal/platform/fetcher/...
go test ./internal/worker/...
gofmt -w cmd internal
go test ./...
python3 -m unittest discover -s tests
```

## 8. Symphony / Linear 要求

任务状态、标签和流转规则完全以本仓库 `WORKFLOW.md` 和本地 Symphony 实现为准。Plan 不定义额外状态、不发明额外标签、不覆盖 Symphony 的状态机。

Linear issue 只承载任务内容：PRD 路径、Plan 路径、任务范围、禁止范围、TDD 验收命令和回写要求。Symphony 负责监听 active states、创建 workspace、运行 Codex，并按 `WORKFLOW.md` prompt 驱动执行。
