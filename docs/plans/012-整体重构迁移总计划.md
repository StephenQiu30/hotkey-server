# 背景

旧计划链 `001` 到 `011` 建立在旧设计与旧工程主线之上，已经不能直接作为新一轮重构执行基线。当前需要一份新的总计划，将执行顺序、验证门禁和新旧文档关系收敛到单一路径。

# 目标

1. 以 `005/006/007` 为唯一上游输入重建实施链。
2. 定义整体重构的阶段顺序与依赖。
3. 明确每个阶段的验证门禁，避免一次性大爆炸迁移。
4. 明确旧计划文档的历史化边界，防止执行输入分叉。

# 非目标

1. 本计划不替代各阶段详细实现计划。
2. 本计划不直接定义具体代码改动。
3. 本计划不复用旧计划中的旧技术主线。

# 计划权威关系

自本计划生效起：

1. `012` 是新一轮整体重构的总入口。
2. `013` 到 `016` 是新主线的详细执行计划。
3. 旧计划 `001` 到 `011` 保留为历史材料，不再作为重构执行依据。

# 阶段顺序

## Phase 0: 文档权威切换

目标：

1. 冻结 `005/006/007` 为新设计权威。
2. 建立 `012` 到 `016` 新计划链。
3. 在后续 README、计划和实现说明中显式降低旧计划优先级。

验证门禁：

```bash
rg -n "005-|006-|007-|012-|013-|014-|015-|016-" docs
```

## Phase 1: 应用骨架与启动收敛

对应计划：[`013-应用骨架与启动收敛计划.md`](013-应用骨架与启动收敛计划.md)

目标：

1. 统一 `cmd/hotkey/main.go` 唯一入口。
2. 收敛 `internal/app` 生命周期管理。
3. 收敛正式启动叙事到二进制入口与 Docker Compose。

门禁：

1. 单一入口可构建。
2. API 与 worker 可由同一进程拉起。
3. README 与部署文档不再推荐多套正式启动方式。

## Phase 2: Gin HTTP 主线迁移

对应计划：[`014-Gin-HTTP迁移计划.md`](014-Gin-HTTP迁移计划.md)

目标：

1. 建立 `Gin` HTTP 主线。
2. 替换旧运行主路由。
3. 冻结新的错误响应、中间件和接口分组规范。

门禁：

1. 健康检查与主要 API 端点跑通。
2. 新接口契约产物可生成并校验。
3. 旧 Huma / legacy server 不再是主运行路径。

## Phase 3: GORM 与数据库迁移

对应计划：[`015-GORM与数据库迁移计划.md`](015-GORM与数据库迁移计划.md)

目标：

1. 建立 `GORM` 主持久化链路。
2. 建立 migration 主线。
3. 按业务域迁移 repository，并约束复杂查询封装。

门禁：

1. 主要业务表和核心域完成 ORM 化。
2. 复杂查询保留在 repository 内部。
3. `db/schema.sql` 不再被当作未来唯一 schema 权威。

## Phase 4: worker 收敛与遗留清理

对应计划：[`016-worker与遗留清理计划.md`](016-worker与遗留清理计划.md)

目标：

1. 收敛 worker 运行时、调度和退出流程。
2. 删除旧路由主线、旧持久化主线和冗余启动叙事。
3. 清理无效依赖与过渡目录。

门禁：

1. 单一正式运行路径成立。
2. 冗余旧入口、旧路由、旧计划引用得到清理或降级。
3. 验证、构建和部署文档一致。

# 执行纪律

1. 每个阶段完成前不得提前宣称整体迁移完成。
2. 如某阶段发现新设计不成立，必须回到对应设计文档修订，而不是在实现层临时绕过。
3. 阶段间可以短期兼容，但不得长期保留双主线。

# 风险与边界

1. 若不先切换计划权威，团队仍会回到旧 `001/002/009` 主线。
2. 若阶段边界不清，重构容易重新滑向大爆炸式一次切换。
3. 若新旧计划并行叙事持续存在，后续实现验收会失焦。
