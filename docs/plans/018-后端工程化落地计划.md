# 背景

`009` 提供了目标架构总纲，`010` 和 `011` 补齐了横切规范与契约/Worker 决策。当前需要一份新的执行计划，将这些设计拆成可以逐阶段落地和验收的任务顺序。

# 目标

1. 冻结实现前必须先定的关键决策。
2. 将横切工程能力前置，而不是留到各业务模块里零散补齐。
3. 将 API、数据库、Worker 和契约建设拆成可验证阶段。
4. 避免“业务先跑起来，工程规范后补”的返工路径。

# 非目标

1. 本计划不替代具体 issue/task 级拆分。
2. 本计划不描述当前实现已经完成的历史重构。
3. 本计划不覆盖多副本部署演进。

# 执行前冻结项

在进入阶段实施前，必须先视为已冻结：

1. OpenAPI 采用“服务端 DTO 与 route metadata 同源维护，产物落盘 `docs/openapi.json`”主线。
2. Worker 采用“单实例轻量调度器 + 统一任务执行入口”主线。
3. 横切工程能力必须先行建设，不允许留到业务模块中零散实现。

# 阶段顺序

## Phase 0: 文档与设计冻结

目标：

1. 确认 `009/010/011` 成为本轮工程化唯一设计输入。
2. 明确 `018` 为执行入口。

验证门禁：

```bash
rg -n "009-|010-|011-|018-" docs
```

## Phase 1: 横切工程骨架

目标：

1. 建立统一错误模型、错误码注册表和 recover middleware。
2. 建立统一成功/失败响应器。
3. 建立 request context 注入和统一日志字段。
4. 建立枚举、配置和基础测试门禁骨架。

交付：

1. `platform/http/errors`
2. `platform/http/response`
3. `platform/runtime`
4. `platform/logging`
5. 基础横切测试

验证门禁：

```bash
go test ./internal/platform/... ./internal/config/... ./...
```

## Phase 2: 契约与 HTTP 规范收敛

目标：

1. 为 Gin handler 建立统一 DTO 和 route metadata 归位方式。
2. 生成并校验 `docs/openapi.json`。
3. 统一鉴权、错误响应和分页响应在契约中的表达。

验证门禁：

```bash
test -f docs/openapi.json
go test ./internal/platform/http/... ./...
```

## Phase 3: 数据访问与迁移规范收敛

目标：

1. 统一 repository 边界。
2. 建立 `db/migrations` 为 schema 主线。
3. 明确复杂查询只允许收敛在 repository 内部。

验证门禁：

```bash
go test ./internal/... ./...
rg -n "os.Getenv|gorm.DB" internal | cat
```

说明：

1. 第二条门禁需要结合白名单解释，重点是防止业务层越界引用。

## Phase 4: Worker 运行面收敛

目标：

1. 建立统一 scheduler / job runner 入口。
2. 任务统一调用 service。
3. 建立单实例运行边界、任务幂等和失败记录能力。

验证门禁：

```bash
go test ./internal/jobs/... ./internal/app/... ./...
```

## Phase 5: 业务模块按规范迁移

目标：

1. 按 `auth/monitor/content/topic/trend/notify/digest/ops` 顺序分批迁移。
2. 每个模块接入统一错误、响应、上下文、日志和契约规范。

验证门禁：

1. 模块单测通过
2. 模块 API 测试通过
3. 契约变更同步更新

## Phase 6: 综合验收

目标：

1. 契约、数据库、Worker、横切规范和业务模块全部按新主线运行。
2. README、design、plans、acceptance 的叙事一致。

验证门禁：

```bash
go test ./...
go build ./...
docker compose config
```

# 执行纪律

1. 不允许跳过 Phase 1 直接做业务模块。
2. 若 Phase 2 或 Phase 4 的冻结项被推翻，必须先回到设计文档修订。
3. 每个阶段结束前必须留下对应验证证据。

# 风险与边界

1. 如果横切骨架不先建，模块实现会迅速分叉。
2. 如果单实例 Worker 边界在计划中不写死，部署时会出现重复调度风险。
3. 如果契约主线不先冻结，Web 与 Miniapp 的生成客户端会反复返工。

# 待确认问题

1. 模块迁移的第一批优先顺序是否以 `auth + monitor` 为主，还是先收敛 `monitor + content` 主链路。
