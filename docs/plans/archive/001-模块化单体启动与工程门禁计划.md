# 模块化单体、Schema基线与工程门禁计划

## 计划目标

交付可编译的 `hotkey` 二进制与可复核的数据库事实基线。`cmd/hotkey` 支持 all、api、worker 三种角色；同一任务同时交付设计 003 所定义的完整 `db/schema.sql`、全部记录模型和 Repository CRUD **接口契约**。Schema、模型、契约和测试必须在同一实现边界内完成，不能以空 Schema 或下游占位替代。

本计划不接入数据库运行时，也不实现业务用例、业务模块或公共 CRUD API。pgx/GORM 连接池、事务、分页、错误映射、Repository 具体实现和 `db init`/`db verify` 命令由 PLAN-002 负责。

## 开工条件

- 当前 Plan 的 `status` 为 accepted、`review_status` 为 approved、`execution_status` 为 ready。
- PRD-001 的 `status` 为 accepted、`execution_status` 为 ready；设计 001–004 均已 accepted。
- 当前 `main` 已同步，工作区只包含本任务文件，Go 1.26 工具链可用。
- `HOTKEY_TEST_DSN` 指向可丢弃的空 PostgreSQL 16+ 与 pgvector 测试库；测试前记录其实例和数据库名，禁止指向开发或生产库。
- `make validate` 使用仓库提供的 POSIX shell 门禁，不依赖本机 PowerShell。

## 审核记录

- 2026-07-16：独立 Reviewer 审核通过。确认完整 Schema、记录模型、Repository CRUD 接口契约和空库/约束/幂等验证均由本计划原子交付；PLAN-002 只负责数据库运行时与具体实现；运维探针的 Result 与 OpenAPI 排除边界在 AGENTS、Design、PRD 和 Plan 中一致。审核结论：`approved`。

## 执行步骤

1. 先新增角色、配置失败、端口竞争、生命周期失败、取消清理、healthz/readyz、模型目录、Repository 接口和 Schema 一致性红灯测试。测试必须把“缺少入口、生命周期、完整 Schema、模型或契约”与“尚未实现数据库运行时”区分开。
2. 在可丢弃空库运行 Schema 红灯：测试应先因不存在 `db/schema.sql`、完整表目录、关键约束或模型映射失败；不得把缺少 pgx/GORM/事务实现误列为失败原因。
3. 创建最小 `main` 和可注入 lifecycle，实现三种角色、启动失败回滚、端口互斥、信号取消与超时停机。worker 不监听业务 HTTP；api 不启动 Worker；all 装配两者。
4. 实现 `healthz` 和聚合必要检查的 `readyz`：成功为 200、未就绪为安全 503 Result；失败、取消和再启动不保留监听器或 goroutine。探针不暴露依赖地址、SQL、密钥或内部原因。
5. 依照设计 003 的固定顺序一次性写入完整 `db/schema.sql`：扩展、核心表、关系表、运行表、约束、索引、River 基础设施。删除 `db/schema/`；不引入 migrations、AutoMigrate 或第二套建表资源。
6. 为全部业务与运行表创建分组记录模型，并在 `internal/shared/repository` 定义通用 CRUD 与不可变运行历史的受限接口。模型和接口只能依赖 Go 基础类型与 shared 契约；不得引入 GORM、pgx、Gin 或业务用例。
7. 让架构测试从 `db/schema.sql` 解析表、关键列、约束和索引，再与模型声明和设计目录比对；在空测试库连续执行 Schema 两次，验证非法状态/分数、错误外键、唯一键和幂等键被拒绝。
8. 用 POSIX shell 重写架构与仓库校验，修正 Makefile 构建产物与清理路径，并同步 `.env.example` 和 README。`make validate` 必须能在没有 PowerShell 的 macOS/Linux 环境执行。
9. 运行范围、Schema、全量、构建、门禁和清理验证；审阅 diff，确保本任务没有接入 pgx/GORM 运行时、事务、分页、数据库命令、业务模块或 OpenAPI。

## 验收清单

- 非法角色、无效配置、端口占用和必要 lifecycle 失败均以非零退出并清理已启动资源。
- api 不启动 Worker，worker 不监听业务 HTTP，all 同时装配 API 与 Worker 生命周期。
- 两个 API 实例竞争端口时只有一个成功；取消、半途失败和再次启动不泄漏监听器、goroutine 或全局状态。
- readyz 对失败必要检查返回非敏感 503 Result、对全部成功返回 200；healthz 和 readyz 不属于 `/api/v1`，且不泄露内部依赖信息。
- 完整 `db/schema.sql` 在空 PostgreSQL 数据库连续执行两次成功，包含设计 003 的全部业务/运行表、扩展、约束、索引和 River 基础设施；分片 Schema、Migration 和 AutoMigrate 无残留。
- 非法状态和分数、错误外键、重复唯一键与幂等键均被数据库拒绝；Schema 所定义的软删除、关系硬删除和不可变运行历史保护可复核。
- 每张表均有记录模型；模型表名、关键列和生命周期分类与 Schema 自动一致。通用与受限 Repository 接口明确 CRUD、乐观锁和不可变边界，但没有数据库具体实现。
- 架构门禁阻止旧顶层目录、旧分表 Schema、PowerShell-only 校验和基础设施反向依赖；仓库根目录无构建产物。

## 提交边界

- `test:` 定义角色、生命周期、探针、Schema、模型和 Repository 契约红灯。
- `impl:` 原子提交启动骨架、完整 Schema、记录模型和纯接口契约；不得把 Schema 与模型拆到后续提交或任务。
- `build:` 切换 POSIX 门禁、Makefile、环境样例与 README。
- `docs:` 写入准确的验收证据；公共 OpenAPI 保持由 PLAN-003 负责。

## 风险与回滚

- 完整 Schema、模型或 Repository 契约如需改变设计 003，立即停止实现，先更新 Design、PRD、Plan 并重新独立审核。
- 空测试库必须可丢弃；不得对开发或生产数据库运行 Schema 验收。未完成前可整体回退本任务提交；已被下游运行时使用后只能以前向修复 Plan 演进 Schema，不恢复分片或双轨事实源。
- 如果 pgvector/River 扩展在本机不可用，使用受控 PostgreSQL+pgvector 测试实例复现；不得降低 Schema、模型或约束范围以适配本机环境。
