# AI 热点信息监控平台 PRD（企业级 0→1）

版本：1.0.0（企业级重建草案）
更新：2026-05-14

## 1. 背景

现有版本已实现轻量 MVP 闭环，但未覆盖企业级可扩展要求。该 PRD 面向下一阶段建设目标，定义 `AI Hot Monitor Platform` 的 0→1 生产导向版本：支持多平台热点聚合、AI 分析、RSS、日报/周报、权限体系、后台运营、异步任务与可观测部署。

## 2. 目标与范围

### 2.1 业务目标

1. 通过关键词订阅自动化监控全网热点，减少人工检索与筛选成本。
2. 通过 AI 完成摘要、分类、情绪、风险识别、聚类与报告生成，形成可执行洞察。
3. 支持 Web + RSS + 邮件 + API 的多通道分发。
4. 支持单用户可用到小组级可扩展的迭代路径。

### 2.2 目标用户

- 个人创作者 / 内容运营 / 产品经理
- 小型内容团队
- 具备自部署能力的企业与团队

### 2.3 交付范围（按优先级）

- P0：可运行闭环（关键词配置、异步抓取、AI 分析、热点列表、去重、RSS、日报/周报、JWT 鉴权、后台基础管理）。
- P1：多平台来源扩展、权限与审计体系增强、趋势可视化、AI 运营报表、系统监控。
- P2：Bot/消息生态扩展、预测分析、RAG/向量检索与协作化功能。

## 3. 主要用例与用户故事

### 3.1 普通用户

- 作为用户，我可以创建关键词并设置阈值/频率/包含/排除词，得到持续监控能力。
- 作为用户，我可以查看实时热点排行、分类统计和趋势图。
- 作为用户，我可以订阅关键词 / 分类 / 热门 RSS，接收结构化订阅内容。
- 作为用户，我可以生成并下载日报/周报（Markdown / HTML / PDF）并接收邮件。

### 3.2 管理员

- 作为管理员，我可以管理所有来源、关键词、用户角色与权限。
- 作为管理员，我可以管理 AI Prompt、模型供应商、任务调度与告警策略。
- 作为管理员，我可以查看操作日志、任务日志与错误日志，支持问题定位。

### 3.3 超级管理员

- 作为超级管理员，我可以管理系统级配置（密钥、限流、重试、缓存策略）和部署参数模板。

## 4. 功能需求

### 4.1 关键词管理

- 字段：名称、描述、匹配模式（完全/模糊/语义）、热度阈值、来源、是否启用、抓取频率、排除词、必含词、分类标签。
- 支持批量导入 / 批量停用。
- 支持订阅与收藏映射。

### 4.2 数据源与抓取（Provider）

要求：
- 统一基类：
  - `class BaseProvider: async def fetch_hot_topics(self): ...`
- 工厂模式注册与发现，支持插件式 Provider 变更和热插拔。
- 支持的来源：微博热搜、知乎热榜、B站热门、今日头条、百度热搜、抖音热点、小红书、GitHub Trending、Hacker News、Reddit、36kr、财联社、RSS Feed、自定义 RSS。
- 失败重试、熔断、限流、代理池、缓存、失败隔离。
- 每条抓取任务可并发执行，失败不影响主任务链路。

### 4.3 AI 分析能力

统一抽象：
- `class BaseLLMProvider: async def summary(self, content: str): ...`

能力清单：
- 热点摘要、分类、情绪分析、趋势分析、聚类、风险识别、报告生成、标题优化。
- 提供商可切换：OpenAI、DeepSeek、Gemini。
- 支持 prompt 模板版本化管理、token 用量统计、结果缓存、输出结构化 schema。

### 4.4 去重与聚类

- URL/标题归一化去重。
- 基于文本与语义的相似内容聚类；同一热点事件聚合为主题组，减少重复通知。
- `hot_topics` 记录 `cluster_id` 与 `dedupe_key`。

### 4.5 热点展示与检索

- 列表支持分页、排序（热度/相关性/时间）、筛选（关键词/来源/状态/时间）。
- 全文检索 + 过滤组合查询。
- 热点详情页展示 AI 结果、版本、来源历史。

### 4.6 RSS 系统

支持生成 RSS：
- `/rss/keyword/{keyword_slug}`
- `/rss/user/{user_id}`
- `/rss/trending`
- `/rss/summary/{date}`

每条条目包含：标题、摘要、热度、来源、发布时间、AI 分析结果、原文链接。

### 4.7 报告系统

- 周期：日报、周报、趋势报告。
- 输出：Markdown、HTML、PDF。
- 邮件发送可配置失败回退。
- 报告内容至少包含：TopN 热点、趋势图、AI 摘要、风险事件、分类统计、用户订阅摘要。

### 4.8 用户与权限（RBAC）

- 角色：超级管理员、管理员、普通用户。
- 能力：
  - JWT 登录、Token 续期、失效策略
  - API Key 管理（可选）用于外部集成
  - 权限点：`keyword.manage`、`source.manage`、`report.manage`、`user.manage`、`audit.view`、`task.manage`、`system.configure`
  - 操作日志与登录日志

### 4.9 后台运营

- 用户管理、关键词管理、数据源管理、RSS 管理、AI 配置、Prompt 管理、报表管理、系统监控、日志中心、任务管理。

### 4.10 可视化

- 热点趋势、热度排行、来源占比、情绪分布、订阅用户统计、词频词云、时间线分析。
- 支持刷新频率配置与导出。

### 4.11 部署与运维

- Docker、Docker Compose、Nginx 反向代理、环境变量注入、CI/CD、日志与告警、数据库与缓存分离。
- 支持部署目标：Ubuntu/CentOS/VPS/Railway/阿里云/腾讯云。

## 5. 系统架构（目标）

### 5.1 目标分层

- API 接入层（FastAPI）
- 服务层（领域服务）
- 仓储层（Repository）
- 工作流层（任务队列 + 调度）
- 集成层（Provider、LLM Provider）
- 数据层（PostgreSQL + Redis + 共享文件）
- 前端层（React + Vite + TanStack Query）

### 5.2 异步与并发

- Celery/Dramatiq/RQ 之一作为任务框架（后续评估）
- APScheduler 负责轻量定时编排与心跳任务
- 抓取、AI 分析、报告生成可并行执行

### 5.3 数据流

`Keyword -> Check Schedule -> Source Fetch -> Normalize -> Dedup -> AI Analyze -> Rank -> Persist -> Notify/RSS/Reports -> Monitor`

## 6. 数据库设计（实体）

### 6.1 关键表

- `users`：用户与认证信息、角色、状态、最后登录
- `keywords`：关键词与告警阈值、匹配策略、频率、来源、标签、启停
- `hot_topics`：来源热点原始元数据、标准化字段、聚类信息、热度指标、时间维度
- `subscriptions`：用户-关键词/分类/来源订阅关系
- `rss_feeds`：RSS 配置与发布记录（签名、令牌、刷新粒度）
- `ai_reports`：日报/周报/趋势报告元数据与产物地址
- `ai_logs`：分析调用日志、token 成本、provider、耗时、状态
- `provider_logs`：抓取来源调用日志、状态码、重试计数、限流/失败
- `notifications`：邮件/任务通知结果与重试
- `system_configs`：系统级配置（只读密钥引用、开关、配额、限流）

### 6.2 存储要求

- 使用 `JSONB` 存放扩展配置和分析输出。
- `hot_topics` 增加 `created_at/updated_at/collected_at/published_at` 与索引。
- 支持 `GIN`（JSONB）与 B-Tree 复合索引以支持热点检索。
- 外键关系完整：`hot_topics` 关联 `keywords`、`users`、`ai_reports` 可追溯。
- 时间序列查询优化：按 `published_at`/`created_at` 分区或索引策略。

## 7. 功能性验收标准（P0）

### 7.1 平台基础

- 用户登录与 JWT 校验通过，RBAC 生效。
- 关键词增删改查可用，包含字段与参数完整生效。
- 关键词和来源触发抓取任务成功入库。

### 7.2 数据采集与分析

- 至少完成 3 个来源抓取并入库。
- 抓取失败不阻塞其他来源。
- AI 分析返回结构化字段且可追踪版本。
- URL + 来源去重成功，重复结果不重复入库。
- 聚类 ID 可用于事件级显示与订阅推送去重。

### 7.3 分发与报告

- RSS 正常输出且支持按关键词 / 热门 / 用户。
- 日报/周报可生成 Markdown 并发送邮件（SMTP 配置存在时）。
- 报告图表与数据与源数据一致。

### 7.4 可视化与运维

- 仪表盘关键图表加载正常。
- 日志中心可查询抓取失败和 AI 失败明细。
- 任务监控可查看失败率与平均耗时。

## 8. 非功能性需求

- **性能**：热点列表查询 P95 响应 < 1.2s（按常规规模）；抓取并发可扩展。
- **可用性**：核心链路冗余降级，单点组件故障不影响全链路。
- **安全**：JWT、RBAC、API 限流、XSS/SQL 注入/CSRF 防护、敏感信息密钥外置。
- **可观测性**：关键步骤日志、任务 trace、失败告警可视化。
- **可维护性**：Provider/LLM 插件化，支持独立扩展。
- **合规**：不持久化外部敏感账号凭据，遵循 robots 与站点抓取规范。

## 9. 风险与边界

- 第三方来源波动性导致采集率不稳定（需重试 + 降级来源）。
- AI 成本波动（需模型路由、缓存、调用上限控制）。
- 抓取合法性与版权合规风险（必须保留来源元信息与最小化采集）。
- 多租户和计费能力在本阶段不做，避免范围扩张。

## 10. 里程碑（草案）

1. P0：平台基底 + 关键词闭环 + RSS + 报告 + 鉴权（目标 4~8 周）
2. P1：权限增强 + 多源扩展 + 可视化 + 后台治理（目标 6~8 周）
3. P2：消息生态、Bot、预测能力与 RAG（目标 8~12 周）

## 11. 开发执行约束（与仓库规范对齐）

- 优先实现 MVP，避免过度设计。
- 新增能力先更新 PRD/技术方案/验收项再编码。
- 与现有仓库事实源对齐后再改造：`server`、`apps/web`、`packages/core`、`sql`、`infra`。
- 任务提交建议沿用 `test/impl/refactor/chore` 与中文提交信息。
