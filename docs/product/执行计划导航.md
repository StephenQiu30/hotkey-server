# AI 热点监控工具执行计划导航

本项目从零重建。旧实现、旧目录结构、旧数据库结构、旧 OpenAPI 契约和旧示例数据均不保留。

后续实现以 `docs/plans/` 下的计划文件为准，每个计划文件都是一个可独立执行和验收的任务单。
本轮新增企业级 P0 补充计划由 OpenSpec 任务清单补充执行，可参考：`openspec/changes/enterprise-p0-hotspot-platform/`。

执行规则（版本一）：
- 需求对齐顺序：`docs/product/产品需求文档.md` -> `docs/engineering/技术方案.md` -> `docs/engineering/验收标准.md` -> `docs/plans/*` -> `openspec/changes/enterprise-p0-hotspot-platform/tasks.md`。
- 所有企业级 P0 任务按“先本地可验证模块，再联调”的方式并行推进。
- 每个任务必须至少包含一条可重复执行的验证证据（单元测试/集成测试或人工验收步骤）。
- 新任务默认以低耦合方式拆分，优先 `service` 与 `provider` 边界独立执行。

## 总体目标

- 完成可自部署 AI 热点监控 MVP。
- 支持关键词管理、多源抓取、AI 查询扩展、AI 分析、热点展示、全网搜索、SMTP 事件邮件、日报/周报邮件、手动触发和简单定时触发。
- 当前执行重点已从后端检测、即时搜索、日报/周报闭环推进到单用户私有部署 SaaS 前端平台。
- 功能闭环围绕本项目 AI 热点监控 MVP 自主设计。
- PostgreSQL 作为唯一数据库。
- SQLAlchemy 作为 ORM。
- `sql/001_init_schema.sql` 作为数据库表结构事实源。
- 不使用 Alembic 或迁移文件，采用重建库策略。
- SaaS 前端首版不做多用户、租户隔离、真实认证和真实计费。

## 执行约束

- 每个阶段完成后再进入下一阶段。
- 不为了兼容旧实现增加额外复杂度。
- 每个阶段必须补齐对应测试或可验收步骤。
- 新增敏感配置只允许通过环境变量注入。
- 企业级 P0 采用“低耦合边界切分”执行：
  - 平台骨架、数据模型、鉴权/权限与任务框架为关键路径；
  - 数据源接入与 AI 分析采用独立模块并行；
  - 前端可视化与运营报表与核心链路解耦，先行补齐 API 后再联调。

## 当前实现状态

- 已实现并有自动化测试或类型检查覆盖：后端检测主链路、查询扩展降级、AI 分析降级、阈值过滤、即时搜索、统一报告 API、SMTP 缺失降级、前端工作台主要页面。
- 需配置后实测：X/Twitter、Bing、SMTP 真实发送和 OpenAI 兼容模型真实调用。
- 暂不做：多用户、多租户、真实认证、真实计费、实时推送、复杂任务队列、向量库和复杂报告模板系统。
