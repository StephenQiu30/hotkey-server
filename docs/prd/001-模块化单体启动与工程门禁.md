# 模块化单体启动与工程门禁

## 目标

把当前最小骨架补成可编译、可启动、可验证的模块化单体基线，为后续任务提供稳定入口和依赖边界。

## 当前差距

- Makefile 指向 cmd/hotkey，但该入口尚不存在。
- all、api、worker 角色只有局部骨架，没有完整生命周期和优雅停机契约。
- 结构校验仍要求旧的分目录 Schema，和目标设计不一致。

## 范围

- 建立 cmd/hotkey/main.go 和统一命令入口。
- 完成 all、api、worker 三种运行角色的 Fx 装配、启动失败处理和优雅停机。
- 保留 healthz，增加能反映必要依赖状态的 readyz。
- 固化 config、logging、clock、ID 和错误基础设施的装配边界。
- 修订 Makefile 和架构校验，使当前最小基线可以执行 lint、test、build、validate。
- 创建十个业务模块的真实目录入口，但不创建没有职责的空业务实现。

## 非范围

- 不实现数据库表、认证、采集或业务 API。
- 不引入 Kafka、微服务、动态插件和部署拓扑。
- 不为 api 与 worker 建立两套代码。

## 功能要求

1. 同一个二进制支持 all、api、worker，非法角色必须启动失败。
2. 配置加载失败、端口占用或必要生命周期钩子失败必须返回非零退出。
3. api 只启动 HTTP 所需组件；worker 不监听业务 HTTP 端口。
4. all 在一个进程启动 API、调度和 Worker 的生命周期占位装配。
5. 收到取消信号后停止接收新请求和新任务，再在超时内释放资源。
6. 业务模块不得依赖 Gin、GORM、River、MinIO 等具体基础设施类型。

## 交付物

- 可执行启动入口、角色解析和生命周期测试。
- 更新后的 Makefile、环境变量样例和架构验证脚本。
- 目标目录边界测试与禁止依赖测试。
- README 中准确的启动命令和当前实现状态。

## 验收标准

- 三种合法角色均可在测试装配中启动并停止。
- healthz 返回统一 Result；readyz 在必要依赖不可用时不报告就绪。
- go test ./...、go build ./cmd/hotkey 和架构校验通过。
- 仓库根目录没有被提交的构建产物。
- 新增代码不包含业务全局单例或旧 controller/service/repository 顶层目录。

## 完成定义

本任务完成后，PRD-002 可以在不改写启动骨架的前提下接入 PostgreSQL、GORM 和事务基础设施。
