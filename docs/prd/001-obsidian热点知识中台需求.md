# 背景

`hotkey-server` 当前已经具备 `X -> post -> topic -> trend` 的热点采集与聚合能力，并且已经存在一版面向 Obsidian 的热点日报导出设计与实现雏形。但这条能力当前仍然偏“日报写文件”，还不是一套完整的知识中台。

当前存在的主要问题：

1. 热点事实、主题聚合、日报产物和导出结果仍然混杂，难以形成长期知识结构。
2. Obsidian 目前更像导出终点，而不是可统计、可查看、可导出、可回写的知识工作台。
3. 现有 `topic_daily_exports` 能表达“某个 topic 某日已导出”，但不足以承载事件级沉淀、专题聚合、周报月报和人工研判闭环。
4. 如果继续沿用“只导出 topics/*.md”思路，后续很容易出现目录失控、事实重复、导出模板不统一和回写污染机器事实的问题。

因此需要将现有能力升级为：以 `HotKey` 作为机器事实源，以 `Obsidian` 作为知识沉淀与消费工作台，通过明确同步契约连接两侧，形成长期可维护的热点知识中台。

# 产品目标

本需求的目标不是再新增一个 Markdown 导出功能，而是构建一个围绕热点事件研究、主题追踪和稳定导出的知识中台。

首版目标：

1. 按北京时间自然日持续沉淀热点日报到 Obsidian。
2. 在知识层明确区分 `Event` 与 `Topic` 两种对象，支持按时间、监控任务、主题和专题查看。
3. 支持在 Obsidian 中统一查看、统计与组织热点知识，而不是只阅读单篇日报。
4. 支持从同一套知识对象中导出 `日报/周报/月报`、`专题报告` 和 `素材清单`。
5. 正式纳入 `Obsidian -> HotKey` 的结构化回写能力，但只能回写白名单知识字段，不能污染机器事实字段。

可用以下 BDD 场景描述：

```gherkin
Scenario: 研究者沉淀并导出 AI 监管热点
  Given 用户已经为 "AI监管" 创建关键词监控任务
  And 系统已经完成每日热点采集、主题聚合和趋势快照
  When 每日知识中台任务运行
  Then 系统会把昨日热点沉淀为日报、事件、主题与专题索引写入 Obsidian Vault
  And 用户可以在 Obsidian 中按日期、主题与专题查看热点结构
  And 用户可以导出日报、周报、专题报告或内容素材清单
  And 用户对专题标签或人工结论的结构化修改可以按白名单字段回写 HotKey
```

# 非目标

1. 首版不做 Obsidian 插件开发或双向实时编辑器。
2. 首版不做多用户、多 Vault、多租户隔离设计。
3. 首版不做复杂协同审批、评论线程或多人冲突编辑。
4. 首版不把 Obsidian 文本内容整体回灌数据库，只回写明确的结构化字段。
5. 首版不替代 `hotkey-web` 的在线运营界面，Web 与 Miniapp 仍通过后端契约消费数据。

# 核心用户故事

## 用户故事 1：每日沉淀

作为热点研究者，我希望每天固定时间自动收到昨日热点知识沉淀，以便持续积累长期可检索的研究材料。

验收标准：

1. 系统每日按北京时间固定时刻生成昨日知识沉淀。
2. 沉淀结果不只有日报，还包括可被索引的事件和主题对象。
3. 同一日期重复运行不会生成重复知识对象。

## 用户故事 2：主题研究

作为内容研究者，我希望围绕某个主题连续查看多日热点、关联事件和趋势变化，以便判断哪些主题值得继续跟踪。

验收标准：

1. 可从 Topic 进入其关联 Event 列表。
2. 可查看 Topic 的热度、趋势与代表性内容。
3. 可将 Topic 归入长期专题，而不影响机器事实层。

## 用户故事 3：统一导出

作为创作者或分析者，我希望基于同一套知识对象导出不同形态的结果，以便直接用于汇报、复盘或内容生产。

验收标准：

1. 可导出日报、周报、月报。
2. 可导出专题报告。
3. 可导出素材清单，至少含事实摘要、引用来源与主题归属。

## 用户故事 4：结构化回写

作为知识整理者，我希望在 Obsidian 中补充专题标签、人工结论和导出编排信息，并将这些结构化字段回写到 HotKey，避免两边长期漂移。

验收标准：

1. 回写只允许白名单字段进入 HotKey。
2. 机器事实字段不能被自由覆盖。
3. 回写必须有审计记录、冲突检测和失败结果。

# 用户体验原则

1. Obsidian 中的知识结构必须稳定，不因单次任务运行而频繁改目录。
2. 同一个知识对象必须有稳定 ID，避免链接频繁失效。
3. 机器事实与人工知识要有明确可见边界，避免用户误以为可任意修改事实源。
4. 导出入口必须基于统一知识对象，而不是为每种导出单独维护一套数据。
5. 同步失败、回写失败和冲突状态必须可见，不能静默吞掉。

# 数据与权限边界

## 默认收集与存储

1. 平台帖子原始事实由 `PlatformPost` 及其关联表保存。
2. 聚合后的机器结果由 `Event`、`Topic`、`TrendSnapshot` 和导出运行记录保存。
3. Obsidian 中保存知识编排结果、专题组织结果和允许回写的结构化人工字段。

## 默认不收集

1. 不采集用户在 Obsidian 中的任意自由文本历史作为数据库事实。
2. 不把整篇 Markdown 正文无选择地回写到 HotKey。
3. 不默认保存用户未显式映射到白名单字段的私有笔记内容。

## 权限边界

1. `HotKey -> Obsidian` 同步默认由后端 worker 执行。
2. `Obsidian -> HotKey` 回写只处理白名单字段，如专题归属、人工标签、人工结论和导出编排状态。
3. 机器事实字段如原始帖来源、采集时间、趋势快照、命中关系、热度计算结果不得被自由回写覆盖。

# 状态模型

## 知识同步状态

1. `pending`：待生成或待同步。
2. `published`：已生成并写入 Vault。
3. `failed`：生成或写入失败。
4. `stale`：上游事实变化后需重新同步。

## 回写状态

1. `detected`：已检测到可回写变更。
2. `validated`：通过白名单与格式校验。
3. `applied`：已回写数据库。
4. `conflicted`：与 HotKey 当前状态冲突。
5. `rejected`：超出白名单或校验失败。

# 技术方案

本 PRD 只保留影响验收的技术边界：

1. `HotKey` 继续作为机器事实层。
2. `Obsidian` 承担知识沉淀、组织、查看与导出消费层。
3. 新增 `Event` 与 `Topic` 双层对象结构，避免继续把所有知识语义塞进 `topic_daily_exports`。
4. 同步能力分为三条链路：
   - `事实沉淀链路`：HotKey 生成 Event / Topic / DailyDigest 等知识对象并写入 Vault。
   - `导出编排链路`：基于知识对象生成日报、周报、月报、专题报告和素材清单。
   - `结构化回写链路`：扫描 Vault 中白名单字段并回写 HotKey。
5. 人工知识不能直接落回 `events`、`topics` 等机器事实表，必须落在独立 sidecar 模型中，例如：
   - `event_annotations`：人工标签、人工结论、人工优先级
   - `topic_annotations`：人工主题判断、素材状态、人工补充摘要
   - `theme_memberships`：Topic / Event 到长期专题的人工归属
   - `knowledge_object_revisions`：Vault 对象 revision / digest，用于冲突检测
6. 回写白名单必须先冻结对象归属、字段类型和冲突规则，再进入实现：
   - `event_annotations.manual_tags`：字符串数组，仅人工标签
   - `event_annotations.analyst_conclusion`：字符串，仅人工结论
   - `topic_annotations.material_status`：枚举值，仅素材状态
   - `theme_memberships.theme_ref`：结构化专题归属，仅人工策展关系
5. 现有 `docs/design/004` 与 `docs/plans/011` 视为“日报导出能力”历史输入，新主线不再把它们视为完整知识中台权威。

# 首版验收门禁

1. 存在新的 `PRD + Plan` 文档链路，且明确 `004/011` 为历史输入。
2. 后端数据模型中存在 `Event` 相关主对象，而不是继续只依赖 `TopicDailyExport`。
3. Obsidian Vault 目录至少区分 `events/`、`topics/`、`digests/`、`themes/`、`exports/`。
4. 可按统一知识对象导出日报/周报/月报、专题报告和素材清单。
5. 回写链路具备白名单字段校验、冲突检测与审计结果。
6. 至少有一条端到端路径证明：
   - 热点事实生成
   - Obsidian 沉淀
   - 人工结构化修改
   - 回写 HotKey
   - 再次导出结果一致

# 待确认问题

1. `Event` 的事实归并策略是否继续基于当前 Topic 聚合链路演进，还是单独引入事件识别规则。
2. 首版是否需要为导出模板建立数据库侧模板实体，还是先以 Vault 中约定模板为主。
3. Web 与 Miniapp 后续是否需要消费知识中台的新对象读取接口。
