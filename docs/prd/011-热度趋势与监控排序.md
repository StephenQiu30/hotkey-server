# 热度、趋势与监控排序

## 目标

对不同来源指标进行版本化归一化，计算可解释、可重算、防单点操纵的 Event 热度、趋势和 Monitor 排序。

## 范围

- 实现来源 metric_capability_profile、可信度权重和管理员审核。
- 处理内容指标快照、窗口对齐、缺失值和来源特定指标。
- 计算 event_metric_snapshots、heat_score、trend_score 和趋势状态。
- 实现 Monitor 上下文排序，结合相关性、热度、趋势、新鲜度和用户反馈。
- 支持删除证据、算法升级和指定窗口的幂等重算。
- 在 Event API 返回分数版本、窗口和简化原因码。

## 非范围

- 不用 LLM 计算热度或判断唯一事实。
- 不直接横向比较不同来源的原始互动数。
- 不根据政治立场调整来源可信度。

## 功能要求

1. 只使用 active 的独立证据，duplicate、deleted、expired、invalid 不计分。
2. 缺失指标不视为零，其余分量按权重重新归一。
3. 独立来源、内容速度、来源广度、互动速度、新鲜度和可信度均有封顶。
4. 单一来源或单条 Content 不能无界推高 Event。
5. 相同 evidence_set_hash、window_end 和 heat_version 只产生一份有效快照。
6. 历史快照不可被算法升级原地改写。
7. 移除证据后热度、趋势、Monitor 排序和报告候选可重算。

## 交付物

- 指标归一化、热度、趋势、排序领域服务和 Repository。
- 管理配置、重算用例、Event/Monitor 查询字段与 OpenAPI。
- Schema、记录模型、算法版本和解释原因码。
- 缺失指标、防刷、确定性、证据删除和历史回放测试。

## 验收标准

- 同一输入、窗口和版本产生完全相同结果。
- 缺失指标 Fixture 不被错误惩罚或当作零。
- 单来源异常高互动不会突破封顶规则。
- 固定样本中的上升事件 Top 20 召回率与误报率形成版本化基线。
- API 与报告候选可说明排序主要原因和算法版本。

## 完成定义

PRD-015 可直接消费冻结的 Event 排序与快照，不自行重复计算热度。
