# 热度、趋势与监控排序

## 目标

对不同来源指标进行版本化归一化，计算可解释、可重算、防单点操纵的 Event 热度、趋势和 Monitor 排序。

## 范围

- 实现来源类型的版本化 `metric_capability_profile`、可信度权重和管理员审核；已发布档案不可原地改写，新版本必须带审核原因。
- 处理内容指标快照、窗口对齐、缺失值和来源特定指标。
- 计算 event_metric_snapshots、heat_score、trend_score 和趋势状态。
- 实现 Monitor 上下文排序，结合相关性、热度、趋势、新鲜度和用户反馈。
- 支持删除证据、算法升级和指定窗口的幂等重算。
- 在 Event API 返回分数版本、窗口和简化原因码。

## 非范围

- 不用 LLM 计算热度或判断唯一事实。
- 不直接横向比较不同来源的原始互动数。
- 不根据政治立场调整来源可信度。

## 功能要求

1. 只使用 active 的独立证据，duplicate、deleted、expired、invalid 不计分。
2. 缺失指标不视为零，其余分量按权重重新归一。
3. 独立来源、内容速度、来源广度、互动速度、新鲜度和可信度均有封顶。
4. 单一来源或单条 Content 不能无界推高 Event。
5. 相同 evidence_set_hash、window_end 和 heat_version 只产生一份有效快照。
6. 历史快照不可被算法升级原地改写。
7. 移除证据后热度、趋势、Monitor 排序和报告候选可重算。
8. `metric_capability_profile` 必须保存 source_type、profile_version、支持的指标、独立性策略、窗口、可信度权重和单项封顶；管理员只能创建新 draft、以理由发布或归档，所有已发布版本保持可审计且不可修改。
9. PLAN-011 负责同步、幂等的 Event 指标重算用例：从 active、非 duplicate、未 deleted/expired/invalid 的 EventContent 和其 `content_metric_snapshots` 取固定 1/6/24 小时窗口输入，按独立来源折叠，生成 `evidence_set_hash` 和所选发布档案集合的 `capability_profile_set_hash`，写入历史 `event_metric_snapshots`，并回写 Event 当前热度投影与 `monitor_events.final_score`。快照唯一/幂等身份固定为 `event_id + window_end + heat_algorithm_version + evidence_set_hash + capability_profile_set_hash`；档案切换即使证据和算法不变也必须生成新快照，旧快照不可改写。内容删除、duplicate 修正、事件 merge/split、指标快照变化和算法/档案版本切换都必须调用该用例；Cron/River 只由 PRD-013 负责触发或入队。
10. 指标归一化、热度、趋势与 Monitor 排序必须遵守 Design-010：同来源/同内容类型窗口化归一化与版本化 fallback、六分量热度、EMA/acceleration 趋势状态，以及 relevance 0.55、heat 0.25、trend 0.15、freshness 0.05 的排序和最低相关性阈值。快照必须同时保留 heat algorithm 和 capability profile set hash；算法或档案升级只追加新快照，并让当前 Event/Monitor 投影指向新配置的重算结果。

## 交付物

- 指标归一化、热度、趋势、排序领域服务和 Repository。
- 管理配置、重算用例、Event/Monitor 查询字段与 OpenAPI。
- Schema、记录模型、算法版本和解释原因码。
- 缺失指标、防刷、确定性、证据删除和历史回放测试。

## 验收标准

- 同一输入、窗口和版本产生完全相同结果。
- 缺失指标 Fixture 不被错误惩罚或当作零。
- 单来源异常高互动不会突破封顶规则。
- 固定人工标注的版本化样本中，上升事件的 Top 20 召回率与误报率形成可回归基线；每个版本保存期望排名、标签、原因和阈值。
- API 与报告候选可说明排序主要原因和算法版本。

## 完成定义

PRD-015 可直接消费冻结的 Event 排序与快照，不自行重复计算热度。实时调度、重试与长期线上质量不属于本 PRD，分别由 PRD-013 与 PRD-017 验收，不能阻塞本 PRD 的同步重算契约。
