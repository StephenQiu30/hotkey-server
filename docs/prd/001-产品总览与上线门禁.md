# 001 产品总览与上线门禁

## 背景

HotKey V1 面向个人创作者和研究者，解决用户在多个国内外平台反复刷信息、手动筛选热点、手动沉淀 Obsidian 笔记和手写日报周报的问题。产品不是实验 MVP，而是可直接上线使用的云端 SaaS。

V1 采用契约优先的平台内核：先稳定账号、授权、主题、采集、标准化、事件聚类、AI 摘要、Obsidian Git 同步、邮件订阅、审计删除和数据留存等核心域，再通过平台适配器接入不同信息源。

## 产品目标

1. 用户能在 Web 工作台完成账号登录、平台授权、主题配置、Git Vault 连接和邮件订阅。
2. 系统能按用户授权采集 X、Reddit、YouTube、Hacker News、微博、小红书、知乎、B站、微信公众号/RSS、新闻/RSS 的相关内容。
3. 系统能将采集内容整理为事件级热点：语义筛选、去重、聚类、热度评分、可信度评分、摘要、时间线和关键信号。
4. 系统能把事件笔记同步到用户指定 Git 仓库中的 Obsidian Vault 目录。
5. 系统能按用户时区和订阅时间发送固定模板日报和周报。
6. 小程序 V1 提供只读查看能力：今日热点、事件详情、同步状态和日报状态。

## 非目标

1. 不做团队协作、成员权限、企业舆情 SLA 和客户工单。
2. 不做可视化大屏、复杂报表和 BI 导出。
3. 不做可编辑邮件模板。
4. 不做创作稿件生成、标题生成、脚本生成和发布排程。
5. 不做 Obsidian 插件，V1 只支持 Git 同步 Vault。
6. 不做小程序配置端，所有配置在 Web 工作台完成。

## 上线级门禁

每个功能点必须满足以下条件才允许进入 Done：

1. 有对应 PRD，且功能范围、非范围、状态模型、失败状态和验收标准清晰。
2. 有 design 文档或 OpenAPI 契约变更说明，先冻结契约再实现。
3. TDD 红绿记录完整：先提交失败测试或可执行验收，再提交实现。
4. 服务端必须覆盖单元测试、集成测试和关键链路 E2E。
5. Web 关键流程必须覆盖 Playwright E2E。
6. 小程序只读流程必须覆盖构建验证和可执行端到端或等价自动化验收。
7. 外部平台能力必须同时覆盖确定性的 provider simulator E2E 和允许范围内的授权沙箱/测试账号 smoke。
8. 所有失败状态必须用户可见，不能静默吞错。
9. OpenAPI 是 server 到 web/miniapp 的事实源，客户端 API 类型必须由 OpenAPI 生成。
10. 数据最小化、撤权、删除和 MinIO 生命周期策略必须通过可执行验收。

## SDD 要求

1. 每个 ticket 开工前必须引用一个主 PRD。
2. 若涉及 API、状态机、数据模型或异步任务，必须先写 design 文档。
3. 若涉及 Web 或小程序调用后端，必须先更新 server OpenAPI。
4. design 文档必须包含状态模型、契约字段、幂等策略、失败策略、权限边界和迁移影响。
5. 未通过 design 审核的 ticket 不允许进入实现。

## TDD 要求

1. 功能行为变更必须先提交 `test:`，再提交 `impl:` 或 `feat:`。
2. 测试必须能在本地和 CI 中稳定复现。
3. 采集器、聚类、摘要、同步和邮件任务必须有确定性 fixture。
4. 外部平台不可作为唯一测试依据，必须有模拟服务或录制样本。
5. 不能写出只验证 happy path 的测试，必须覆盖授权失效、限流、重试、重复内容、删除和幂等。

## E2E 要求

V1 最小 E2E 闭环：

1. 用户注册并登录 Web。
2. 用户连接至少一个测试平台授权。
3. 用户创建主题和关键词。
4. 系统执行采集任务并写入标准化内容。
5. 系统完成相似度筛选、去重、事件聚类和 AI 摘要。
6. 系统把事件笔记提交到测试 Git 仓库。
7. 系统按订阅时间发送测试日报。
8. 用户在 Web 查看事件、同步状态和邮件状态。
9. 用户在小程序只读查看今日热点。
10. 用户撤权并删除数据，系统清理 token、缓存、对象、索引和任务。

## 关键指标

1. 主题命中内容的前 20 条人工相关性达到可上线验收阈值。
2. 相同事件重复内容能合并，不重复进入日报。
3. Git 同步成功率、邮件发送成功率和采集任务成功率可观测。
4. 用户可在 10 分钟内完成首次配置并看到第一份测试结果。
5. 删除账户后的用户可识别数据不可继续出现在搜索、日报、对象存储或任务队列中。
