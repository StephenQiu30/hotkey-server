# 004 平台适配器通用契约

## 功能目标

定义所有内容源适配器的统一产品契约，使 X、Reddit、YouTube、Hacker News、微博、小红书、知乎、B站、微信公众号/RSS 和新闻/RSS 都能接入统一采集、标准化、测试和观测流程。

## 用户故事

1. 作为用户，我看到每个平台的连接状态、采集范围和最近失败原因。
2. 作为用户，我不需要理解平台差异，也能得到统一的热点事件。
3. 作为研发，我可以新增平台适配器而不破坏核心事件处理链路。

## 范围

1. 统一适配器接口。
2. 统一采集任务输入输出。
3. 统一限流、重试、幂等、失败分类和健康状态。
4. 统一标准化内容字段。
5. 统一 provider simulator 测试能力。

## 非范围

1. 把所有平台强行做成相同授权方式。
2. 绕过平台规则或隐藏不可用状态。
3. 平台专属 UI 细节。

## 标准化内容字段

每条采集内容至少包含：`source_platform`、`source_account_id`、`source_item_id`、`canonical_url`、`author_name`、`author_handle`、`published_at`、`fetched_at`、`language`、`title`、`body_text`、`media_refs`、`engagement`、`raw_object_ref`、`visibility`、`license_hint`。

## 状态模型

适配器状态：`available`、`degraded`、`limited`、`disabled`。

采集任务状态：`queued`、`running`、`rate_limited`、`auth_failed`、`provider_failed`、`normalized`、`empty`、`completed`、`failed`。

失败分类：`auth`、`rate_limit`、`network`、`provider_schema`、`content_unavailable`、`policy_restricted`、`unknown`。

## 契约影响

OpenAPI 必须暴露平台清单、平台能力、连接状态、最近采集结果、限流状态、失败原因和用户可执行操作。内部服务必须提供统一 normalized content 契约给后续去重、相似度和事件聚类模块。

## SDD 要求

1. 先定义 adapter interface 和 normalized content schema。
2. 先定义失败分类与用户可见文案映射。
3. 先定义幂等键：用户、主题、平台、源内容 ID、时间窗口。
4. 先定义 provider simulator 的输入输出契约。

## TDD 要求

1. 先写适配器状态转换测试。
2. 先写重复采集幂等测试。
3. 先写限流、授权失败、schema 变化和空结果测试。
4. 每个平台适配器必须共享通用 contract test。

## E2E 验收门禁

1. provider simulator 返回多平台 fixture，系统完成统一标准化。
2. 任一平台限流时，其他平台采集不受影响。
3. 任一平台授权失效时，用户能在 Web 看到对应连接失效。
4. 同一 source item 重复返回时不会生成重复内容。

## 数据与权限

1. 原始响应写入 MinIO 时必须带用户、平台、采集任务和过期时间元数据。
2. 标准化内容不得泄漏其他用户授权数据。
3. 受限平台必须展示受限状态，不能伪装成成功。
