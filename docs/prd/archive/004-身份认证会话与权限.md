# 身份、认证、会话与权限

## 目标

交付可供全部业务 API 复用的用户、密码认证、可撤销会话和 admin/editor/viewer 授权能力。

## 范围

- 实现 identity 模块四层结构。
- 实现 users、user_preferences、auth_sessions、auth_refresh_tokens 的记录模型、Repository 和应用用例。
- 支持验证码注册、登录、刷新、退出、当前用户查询、密码修改、密码重置、用户停用、软删除与恢复。
- 使用 bcrypt、JWT v5 和 PostgreSQL 会话事实；Redis 仅保存验证码、验证票据、频率和短锁，不能成为认证正确性的事实源。
- 建立明确的 admin/editor/viewer 角色检查；首版不建立通用策略引擎。
- 发布 /api/v1/auth 和受限的 /api/v1/users 契约，以及仅限本地受控操作员的 bootstrap-admin 命令。

## 非范围

- 不实现第三方 OAuth、企业 SSO、多租户或前端页面。
- 不开放任意用户表通用 CRUD。
- 不把 JWT 作为不可撤销的唯一会话事实。
- 不为 Redis 不可用时建立内存验证码或会话替代事实。

## 功能要求

1. 邮箱写入前 trim/小写且仅在未删除用户内唯一；密码只保存 bcrypt 哈希，输入不得超过 72 字节。
2. 注册和重置密码均须先消费 Redis 中的单用途邮箱验证票据；六码验证码在 Redis 中只以绑定用途与规范化邮箱的服务端 HMAC 保存，HMAC 密钥不少于 32 字节；不存在邮箱、错误密码、禁用与删除账户在登录时不得可区分。
3. Access Token 有效期 15 分钟；Refresh Token 为 HttpOnly Cookie 内的不透明随机值，7 天滑动续期、30 天绝对上限。
4. 每次受保护请求均校验签名、活动用户和未撤销 PostgreSQL 会话；退出、改密、停用、软删除和 Refresh 重放立即使既有 Access Token 失效。
5. 两个并发刷新只有一个成功；仅未过期且未撤销的已消费 Token 被重放时撤销整个会话和其全部 Refresh Token，过期或已撤销 Token 只返回 session invalid。
6. admin 只能管理用户角色、状态、删除和恢复；系统必须保留一个未删除的 active admin，恢复用户默认 disabled。
7. 认证与授权失败使用统一 Result 和稳定数字业务码；Cookie、密码、Token、验证码与哈希不得进入响应、日志、trace 或审计快照。
8. 登录、刷新、重放、退出、密码操作和管理操作写入安全审计，且保留 request ID 与 trace ID；Redis、SMTP 或显式禁用 SMTP 只使新邮箱流程以 503 失败，不影响既有认证正确性。bootstrap-admin 与任意用户创建必须串行，确保管理员仅在用户表为空时创建。

## 交付物

- identity 领域、应用、PostgreSQL 适配器和 HTTP 路由。
- Schema、记录模型、OpenAPI、运行配置、错误码和架构门禁更新。
- 密码、JWT、会话轮换、重放撤销、RBAC、软删除恢复、真实 PostgreSQL/Redis 以及 Redis 不可用测试。
- 可用于其他模块的认证主体与授权接口。

## 验收标准

- 注册、登录、刷新、退出、改密、重置、停用、软删除与恢复完整链路通过。
- 无效密码、过期/错误算法 JWT、撤销会话、Refresh 重放、角色不足与最后管理员保护返回正确 HTTP 和业务码。
- 业务 JSON 不包含 request_id、密码哈希、会话密钥、Cookie、验证码、验证票据或内部错误。
- 真实 PostgreSQL/Redis 组合验证验证码 HMAC、票据原子性、过期 Refresh 不撤销会话、bootstrap/注册并发、审计关联字段与 SMTP 禁用；Redis 不可用时既有认证正确性不变，新邮箱流程确定返回 503。
- 所有 identity 表具备 Repository 契约和 Refresh 并发更新测试；Cookie 属性、OpenAPI、审计和日志脱敏可复核。

## 完成定义

PRD-005 及之后的公共业务 API 必须使用本任务提供的主体和授权能力。
