# 模块化单体、Schema基线与工程门禁

## 目标

把当前最小骨架补成可编译、可启动、可验证的模块化单体基线，并原子交付完整 Schema、记录模型和 Repository CRUD 契约。后续数据库任务只实现该既定模型的运行时与事务能力，不得把数据库事实源再次拆开。

## 当前差距

- Makefile 指向 cmd/hotkey，但该入口尚不存在。
- all、api、worker 角色只有局部骨架，没有完整生命周期和优雅停机契约。
- 结构校验仍要求旧的分目录 Schema，和目标设计不一致。
- `db/schema.sql`、完整记录模型和 Repository CRUD 契约尚未形成同一份可验证的数据库基线。

## 范围

- 建立 cmd/hotkey/main.go 和统一命令入口。
- 完成 all、api、worker 三种运行角色的 Fx 装配、启动失败处理和优雅停机。
- 保留 healthz，增加能反映必要依赖状态的 readyz。
- 固化 config、logging、clock、ID 和错误基础设施的装配边界。
- 修订 Makefile 和架构校验，使当前最小基线可以执行 lint、test、build、validate。
- 建立 `internal/modules/` 根目录和模块命名约束；实际业务模块及其四层目录只在对应 PRD 实现时创建，避免无职责空壳。
- 按设计 003 建立唯一、完整、可重入的 `db/schema.sql`：包含扩展、全部业务和运行表、关系、约束、索引与 River 基础设施；删除 `db/schema/` 分片与 manifest。
- 建立全部表的一一对应数据库记录模型，以及不绑定 GORM、pgx 或 HTTP 的统一 Repository CRUD **接口契约**；以可执行测试验证 Schema、模型和表目录同步。
- 以空 PostgreSQL 测试库连续执行 Schema 两次，并验证关键状态、分数、外键、唯一与幂等约束；本任务不创建应用连接池、GORM 运行时、事务实现或数据库命令。

## 非范围

- 不实现认证、采集或业务 API，也不实现业务用例或公共 CRUD API。
- 不接入 pgx/GORM 连接池、事务管理、游标分页、数据库兼容性命令或 Repository 具体实现；它们由 PRD-002 交付。
- 不引入 Kafka、微服务、动态插件和部署拓扑。
- 不为 api 与 worker 建立两套代码。

## 功能要求

1. 同一个二进制支持 all、api、worker，非法角色必须启动失败。
2. 配置加载失败、端口占用或必要生命周期钩子失败必须返回非零退出。
3. api 只启动 HTTP 所需组件；worker 不监听业务 HTTP 端口。
4. all 在一个进程启动 API、调度和 Worker 的生命周期占位装配。
5. 收到取消信号后停止接收新请求和新任务，再在超时内释放资源。
6. 业务模块不得依赖 Gin、GORM、River、MinIO 等具体基础设施类型。
7. `readyz` 聚合已注册的必要生命周期检查；F0 必须能够以注入失败检查证明未就绪不会误报为就绪，但不要求在本任务连接 PostgreSQL、MinIO 或 River。
8. 完整 Schema 与设计 003 的表目录、字段、状态、约束、索引和保留分类一致；空库连续执行两次均成功，不能保留分片 Schema、Migration、AutoMigrate 或第二套事实源。
9. 每张设计定义的业务表和运行表都有一个数据库记录模型；模型声明的表名和核心字段与完整 Schema 可自动核对。
10. 统一 CRUD 契约覆盖 Create、Get、List、Update、Delete/Archive 的适用语义与 `version` 乐观锁输入；运行历史的受限 Repository 不暴露任意 Update/Delete。具体数据库实现留给 PRD-002。
11. `/healthz` 和 `/readyz` 使用安全 Result、Request ID 与非敏感错误；它们是非 `/api/v1` 运维探针，不发布为 OpenAPI 操作。

## 交付物

- 可执行启动入口、角色解析、启动失败、端口冲突、生命周期失败和停止恢复测试。
- 完整 `db/schema.sql`、全部记录模型、Repository CRUD 接口契约，以及 Schema-模型一致性和空库约束测试。
- 更新后的 Makefile、环境变量样例和架构验证脚本。
- 目标目录边界测试与禁止依赖测试。
- README 中准确的启动命令和当前实现状态。

## 验收标准

- 正常：三种合法角色均可在测试装配中启动、提供预期组件并停止；healthz 返回统一 Result。
- 失败与边界：非法角色、无效配置、端口占用和必要生命周期钩子失败均返回非零退出；worker 不监听业务 HTTP 端口，api 不启动 Worker。
- 恢复：启动半途失败或取消时已启动资源被停止；取消后不再接收新 HTTP 请求或 Worker 工作，并在停机超时内完成释放。
- 并发：两个 API 实例竞争同一端口时只有一个可以启动，失败实例不会泄漏监听器或 goroutine。
- 就绪：readyz 在已注册的必要检查失败时返回统一的非敏感 503 Result，在全部检查通过时返回 200。
- Schema：在临时空 PostgreSQL 数据库连续执行完整 `db/schema.sql` 两次成功；业务表、运行表、扩展、索引、外键与约束数量和名称符合设计 003，`db/schema/`、Migration 和 AutoMigrate 均不存在。
- 数据完整性：非法状态和分数、错误外键、重复唯一键与幂等键均被数据库拒绝；重复执行 Schema 不产生重复对象；删除语义仅验证 Schema 所定义的软删除、关系硬删除和不可变运行历史保护。
- 一致性：每张表均有记录模型和受限/通用 Repository 契约，自动校验不允许模型表名、关键列或表目录漂移；本任务不连接应用运行时数据库，Repository 的具体 CRUD、并发写入和分页由 PRD-002 验收。
- API 边界：`/healthz` 与 `/readyz` 使用安全 Result 且不泄露依赖详情；它们不位于 `/api/v1`，并由后续 PRD-003 的 OpenAPI 校验验证排除。
- go test ./...、go build ./cmd/hotkey 和跨平台架构校验通过；仓库根目录没有构建产物。
- 新增代码不包含业务全局单例、旧 controller/service/repository 顶层目录或旧分表 Schema 目录。

## 完成定义

本任务完成后，PRD-002 可以在不改写启动骨架、Schema、记录模型或 Repository 契约的前提下接入 PostgreSQL 运行时、GORM、pgx 和事务基础设施。
