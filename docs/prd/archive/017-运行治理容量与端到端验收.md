# 运行治理、容量与端到端验收

## 目标

在不新增业务范围的前提下，完成运行查询、审计、保留清理、跨存储对账、容量标定、故障恢复和全链路验收。

## 范围

- 完成 operations 模块、audit_logs、retention_policies 和受限运行查询。
- 提供队列、采集、AI、Vault、报告、投递、检查点和依赖健康视图。
- 实现按设计保留期的批量清理、来源删除同步和数据库/MinIO/Vault 对账。
- 建立 P0、P1 的可重复本地验收 Fixture 和故障注入场景。
- 在目标容量基线上验证核心索引、分页、向量召回、Job 背压和存储增长。
- 完成安全、可观测、备份恢复和运维 Runbook。
- 收敛 make ci、OpenAPI、Schema、架构与 smoke 门禁。

## 非范围

- 不新增数据来源、模型任务、前端页面或部署平台。
- 不通过降低验收阈值掩盖实现缺陷。
- 不把验收脚本变成只检查文件存在的形式测试。

## 功能要求

1. 运行查询至少覆盖队列深度、最旧等待、成功率、重试率、隔离数、永久失败数和检查点滞后。
2. 清理任务使用批次、检查点和幂等键，不能长事务锁死在线链路。
3. 来源删除后 Content、Event 证据、热度、摘要、知识、报告候选同步失效。
4. 对账能检测并安全修复数据库、MinIO 与 Vault 的可修复不一致。
5. api/worker/all 在 PostgreSQL 重连、MinIO 故障、Provider 超时和 SMTP 故障下按设计降级。
6. 备份恢复演练保留业务事实、River 状态、对象引用和 Vault 修订关系。
7. 审计导出和运行 API 遵守管理员权限与敏感字段脱敏。

## 验收标准

- 固定标注集 Precision@20 不低于 80%。
- 跨语言事件聚类 F1 不低于 75%。
- 95% 正常内容在 60 分钟内形成或更新 Event。
- 同一采集窗口重跑不重复内容、匹配、事件证据、AI 运行、报告或投递。
- 单一来源失败不阻塞其他来源。
- 无 LLM、Vault、SMTP、Git、Kafka 和 Redis 时，P0 结构化热点查询仍可运行。
- 事件笔记、日报和周报可写入 Vault，人工修改不会被旧提案覆盖。
- 邮件可追踪重试，RSS/Atom 支持条件请求和 Token 轮换。
- 目标数据规模下核心查询、分页和任务吞吐达到已记录基线。

## 交付物

- Operations 用例、受限管理 API、保留、删除和对账任务。
- 容量数据生成器、性能基线、故障测试和端到端 smoke。
- OpenAPI、运维 Runbook、备份恢复步骤和长期验收证据。
- 全量 make ci 与本地依赖集成验证结果。

## 完成定义

所有 001–016 PRD 均为 done，系统门禁全部有可重复证据，工作区不存在未解释的目标设计与运行事实漂移。
