# 016 热点事件聚类与评分

## 功能目标

将通过主题相似度筛选的多平台内容聚合为事件，并计算热度、相关性、可信度、新鲜度和趋势变化，形成用户可阅读和可订阅的热点列表。

## 用户故事

1. 作为用户，我希望同一事件的多篇文章和多条社媒讨论合并展示。
2. 作为用户，我希望热点排序能反映热度和与我主题的相关性。
3. 作为用户，我希望看到事件为什么被认为重要。

## 范围

1. 内容到事件的聚类。
2. 事件标题生成前的候选标题选择。
3. 热度、相关性、可信度、新鲜度和趋势分计算。
4. 事件生命周期状态。
5. 事件来源列表和证据摘要。

## 非范围

1. 长篇研究报告。
2. 用户自定义评分公式。
3. 企业级风险预警 SLA。

## 状态模型

事件状态：`forming`、`active`、`updated`、`cooling`、`archived`、`suppressed`。

评分状态：`pending`、`scored`、`score_stale`、`score_failed`。

## 契约影响

OpenAPI 必须提供事件列表、详情、来源、评分、状态、更新时间和排序字段。事件详情必须能追溯到原始来源和去重关系。

## SDD 要求

1. 先定义事件聚类输入输出和聚类窗口。
2. 先定义评分字段、范围、默认权重和解释字段。
3. 先定义事件合并、拆分和归档规则。
4. 先定义低可信来源和单一来源事件的展示规则。

## TDD 要求

1. 先写同一事件多来源聚类测试。
2. 先写不同事件不可误合并测试。
3. 先写热度时间衰减测试。
4. 先写单一低可信来源评分较低测试。
5. 先写事件更新后日报只展示增量变化测试。

## E2E 验收门禁

1. 多平台 fixture 形成一个事件并展示来源证据。
2. 相似但不同事件保持分离。
3. 热度上升事件进入日报顶部。
4. 冷却事件不再重复占用日报主位。

## 数据与权限

1. 事件属于用户主题范围，不跨用户共享。
2. 事件保留长期摘要和评分，原文按留存策略清理。
3. 用户删除主题后事件索引和聚类关系清理。
