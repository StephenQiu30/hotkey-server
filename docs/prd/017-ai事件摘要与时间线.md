# 017 AI 事件摘要与时间线

## 功能目标

为热点事件生成结构化 AI 摘要，包括事件标题、简短摘要、时间线、关键信号、来源证据、风险提示和后续观察点，用于 Web 展示、Obsidian 笔记和邮件日报。

## 用户故事

1. 作为用户，我希望不用逐条阅读就能理解事件发生了什么。
2. 作为研究者，我希望摘要能保留来源证据和时间线。
3. 作为用户，我希望 AI 不确定时明确标注，而不是编造结论。

## 范围

1. 事件级摘要。
2. 时间线生成。
3. 关键信号提炼。
4. 来源证据引用。
5. 摘要失败重试和人工可见失败状态。
6. 模型调用成本和 token 上限控制。

## 非范围

1. 创作稿件生成。
2. 投资、医疗、法律结论建议。
3. 自动发布到社媒。

## 状态模型

摘要状态：`pending`、`generating`、`generated`、`needs_refresh`、`failed`、`suppressed`。

证据状态：`sufficient`、`single_source`、`low_confidence`、`conflicting`。

## 契约影响

OpenAPI 事件详情必须返回摘要状态、摘要内容、时间线、关键信号、证据列表、模型版本、生成时间、失败原因和刷新状态。

## SDD 要求

1. 先定义摘要 JSON schema。
2. 先定义 prompt 输入字段和最大上下文窗口。
3. 先定义模型失败、超时、内容安全和低证据的降级策略。
4. 先定义摘要版本和刷新规则。

## TDD 要求

1. 先写摘要 schema 校验测试。
2. 先写模型 mock 返回合法/非法 JSON 的测试。
3. 先写低证据事件标记测试。
4. 先写摘要刷新幂等测试。
5. 先写来源引用不能指向不存在内容的测试。

## E2E 验收门禁

1. 事件聚类完成后生成摘要、时间线和关键信号。
2. 模型失败时事件详情显示失败状态，并可重试。
3. 摘要内容同步进入 Obsidian 笔记和邮件日报。
4. 低证据事件在 Web、Obsidian 和邮件中标注置信度。

## 数据与权限

1. 模型输入不得包含授权 token。
2. 输出保存摘要和证据引用，原文仍按 30 天留存。
3. 用户删除事件或主题后摘要同步清理。
