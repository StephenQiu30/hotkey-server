# 任务执行与计划归档治理

## 目标

在不改变产品业务能力的前提下，把 Design-015 定义的单计划状态机落为仓库可验证的质量门禁。完成后，任何任务是否可开工、可归档以及索引是否真实，均可由一个本地命令和 CI 复核；PLAN-006 只有在该治理任务归档、相关设计接受和独立 Plan Review 完成后才可进入 `ready`。

## 范围

- 解析并校验 `docs/design`、`docs/prd`、`docs/plans`、`docs/acceptance` 与 `docs/operations` 的正式 frontmatter、文件链接、依赖图和状态组合。
- 校验 PRD/Plan 一一映射、Plan DAG、唯一 `in_progress`、归档任务对应的 accepted Acceptance、40 位提交 SHA 与 README 索引状态。
- 在 `make ci` 中运行文档治理校验；提供只针对已运行服务的、显式 opt-in 的 `make smoke`。
- 补齐 001–005 的规范化验收提交证据，修正 PLAN-013/017 的命令与 Operations 编号/文件动作，逐份将 PLAN-006–017 改为符合 Design-015 的可勾选 Task 契约。
- 保持归档文件的 canonical path，不新增第二套 archive 目录，不移动 001–005 的已归档 PRD/Plan。

## 非范围

- 不实现 RSS/HN 采集、River、AI、事件、报告、邮件或前端能力。
- 不改变现有业务 Schema、HTTP API、认证契约或已接受验收的实际结论。
- 不把 `make smoke` 纳入无服务进程的 `make ci`；CI 的固定门禁仍是可自举、可重复的测试与校验。
- 不将聊天记录、未提交 diff 或短 SHA 当作长期验收证据。

## 功能要求

1. 治理校验必须在仓库根目录运行，输出 `路径: 规则: 原因`，一次报告所有发现，不依赖网络或真实业务服务。
2. 每份正式文档必须具备 `docs/README.md` 规定的必填字段和值域；同层 `doc_no`、全仓 `canonical_path` 不得重复，且 canonical path 必须等于实际路径。
3. Markdown 相对链接、frontmatter 的 `inputs`/`downstream` 与 `depends_on` 必须可解析；URL 编码的本地链接按解码后的仓库路径校验。仅允许尚未 done/archived 的同编号 PRD/Plan 把未创建的 `docs/acceptance/<编号>-*.md` 作为终端 `downstream`，该例外不能用于 `inputs` 或其他缺失路径。
4. 每个 PRD 仅映射一个同编号 Plan，每个 Plan 仅映射一个同编号 PRD；Plan 依赖必须存在且无环。
5. 状态校验必须执行 Design-015 的开工与归档规则：最多一个 `in_progress`；`ready` 的前置 Plan 均为 `done`；`done`/`archived` 的 PRD/Plan 必有同编号 `result: accepted` Acceptance；Acceptance 必有可解析的 40 位 Git SHA。
6. 校验 README 索引表中的编号、文档成熟度、执行状态和审核状态与正文 frontmatter 一致，并校验 Plan 中的 Make 目标、`sh` 脚本路径真实存在。未归档 Plan 的 `make ci` 必须显式提供可丢弃的 `HOTKEY_TEST_DSN` 与 `HOTKEY_TEST_REDIS_URL`，不得依赖开发或生产环境。
7. `make docs-validate` 必须运行治理测试，`make ci` 必须包含该门禁；GitHub Actions 必须检出足以解析验收 SHA 的历史。
8. `make smoke` 必须要求显式 `HOTKEY_SMOKE_BASE_URL`，且只验证 `/healthz`、`/readyz`、`/api/v1/capabilities` 的 HTTP 成功和统一 Result 的 `code: 0`，不得启动、停止或修改服务和数据。
9. 001–005 Acceptance 的历史提交证据必须以可解析完整 SHA 保存；PLAN-013/017 不得引用不存在的命令，PLAN-017 不得复用 Operations `doc_no: 001` 或把现有文件标为创建。
10. PLAN-006–017 每份都必须列出准确 Create/Modify/Delete/Test 路径、输入/输出接口、逐动作 checkbox、红灯/绿灯/回归命令、提交边界与残余风险；只将满足即时依赖与审核条件的 PLAN-006 作为后续候选，其余保持 backlog。

## 验收标准

| 领域 | 必须证明的结果 |
|---|---|
| 文档解析 | 合法文档通过；缺字段、非法枚举、重复编号、错误 canonical path、断链与 URL 编码路径均以精确错误拒绝。 |
| 任务图与状态 | 缺失映射、循环依赖、错误 `ready`、多 `in_progress`、无 Acceptance 的 done/archived 文档和错误归档组合均被拒绝。 |
| 证据 | 001–005 的 Acceptance 使用完整可解析 SHA；短 SHA、非 Git SHA 与 `accepted_with_risk` 不得放行归档。 |
| 命令与 CI | `make docs-validate`、`make ci`、`make smoke` 的边界明确；CI 使用完整历史并通过质量门禁。 |
| 下游计划 | PLAN-013/017 已消除已知命令与 Operations 冲突；PLAN-006–017 均能由校验器检查任务路径、命令和最小执行结构。 |
| 归档 | Acceptance-018 记录受控红绿、全量门禁、独立复核和准确提交后，PRD/Plan-018 才可改为 `archived + done`。 |

## 完成定义

本 PRD 完成时，`make docs-validate` 与 `make ci` 均通过，Acceptance-018 为 `accepted` 且记录完整实施 SHA；PRD/Plan-018 在原路径归档。随后单独完成 Design-005/012 的接受、PRD-006/Plan-006 的独立审核，才允许 PLAN-006 从 backlog 进入 ready。

## 审核结论

2026-07-16，独立 Reviewer 审核 Design-015、本文档和 PLAN-018 v1.3：确认 future Acceptance 例外受限、任务依赖无环、006–017 已拆为可独立审核任务、全量门禁使用可丢弃 PostgreSQL/Redis 环境，结论为 approved。本文档因此从 `review/backlog` 进入 `accepted/ready`；实施开始后仍须将执行状态单独改为 `in_progress`。
